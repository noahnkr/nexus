# v1.0.1 — Fuzzy Entity Matching & Cross-Entity Dedupe

**Complexity:** 🔴 Complex — touches the connector resolution seam (the approval/entity-resolution invariant) and adds an agent-callable tool. Broken into 6 sequential tasks; each is independently testable.

---

## Context

**Why:** Today every lookup by a real-world string is exact-match only. Inbound connector events resolve to canonical entities *solely* via an exact three-column `external_ids` match (`resolution.py::_find_mapping`); on a miss, `write_lead` inserts a **new** row blindly with no check for an existing person (`entity_writers.py:42`). The agent's only name search is substring `ILIKE '%term%'` (`tools/entities.py::_like`). There is **no shared string normalization** (phone E.164 logic is trapped in `adapters/goto.py::_e164`; email/name have none).

The consequences: (1) the same person entered manually and synced from WelcomeHome/WellSky becomes two records; (2) the agent can't reliably find "Peggy Ellison" when the row says "Margaret Ellison"; (3) automations can't resolve a messy source string to an id. This matters **now**, before the Module 18a WelcomeHome backfill floods the DB with connector-created rows.

**Outcome:** A deterministic, explainable fuzzy-matching engine (Python/rapidfuzz, mirroring the `views/matching.py` house pattern) that powers three surfaces:
1. A **`find_entity`** lookup — exposed as both a safe agent/MCP tool and an automation function.
2. A **`scan_duplicates`** mechanism that flags near-duplicate records as plain-language **review tasks** (review-only; no auto-merge).
3. A **connector-resolution hook** so inbound events fuzzy-match before auto-creating: high confidence → link the external id to the existing entity; medium → review task; low → create (today's behavior).

**Decisions locked with the user:** review-task-only dedupe (no merge tool); Python/rapidfuzz engine (no Postgres extension, no migration); full connector-loop integration.

---

## Design principles (inherited from `views/matching.py`)

- **Deterministic & explainable.** No LLM in scoring. Re-run yields identical order. Every candidate carries `score` + plain-language `reasons`/`warnings`.
- **Weights & thresholds are in-seam constants**, not env config (one-client, explainability over tunability).
- **Single-pass over a tenant-scoped fetch.** Corpus is hundreds–low-thousands of rows per type; fetch candidates and score in-process, exactly as `rank_candidates` does. No index/extension needed. (Upgrade path if a tenant ever grows huge: pg_trgm GIN prefilter — documented, not built.)
- **Core mechanism / vertical content seam.** The scoring & normalization are business-agnostic core; the *set of matchable entity types and their columns* is the re-templating seam, a sibling to `entity_writers.py::WRITERS`, `tools/entities.py`, and `automations/entities.py::ENTITY_TABLES`.

## Task 1 — Shared normalization (`matching/normalize.py`)

Business-agnostic, pure, no DB. The foundation every other task builds on.

- `normalize_phone(raw) -> str | None` — port the E.164 logic from `adapters/goto.py::_e164` (strip non-digits, `+1` for bare 10-digit US, `+` for 11-digit leading-1). Return `None` on empty.
- `normalize_email(raw) -> str | None` — strip a display-name wrapper (`John <j@x.com>` → `j@x.com`), lowercase, trim. Closes the gap flagged in `adapters/gmail.py:64` where the raw `From` becomes the external id.
- `normalize_name(raw) -> str` — trim, collapse whitespace, casefold, strip accents via `unicodedata.normalize('NFKD', …)` + drop combining marks. Used only for comparison, never stored.
- `name_tokens(raw) -> frozenset[str]` — token set for order-independent comparison (handles "Ellison, Margaret" vs "Margaret Ellison").

**Refactor (optional, low-risk):** have `adapters/goto.py::_e164` delegate to `normalize_phone` so there's one phone normalizer. Keep the adapter's public name.

**Validation:** `backend/tests/test_matching_normalize.py` — pure offline unit tests (no DB, no env gate). Assert display-name stripping, accent folding ("José" ↔ "jose"), phone E.164 forms, token-set equality for reordered names.

## Task 2 — Fuzzy scoring engine (`matching/fuzzy.py`) + entity seam (`matching/entities.py`)

**Add dependency:** `rapidfuzz` to `backend/requirements.txt` (deterministic, C-backed, no transitive bloat).

**`matching/entities.py` — VERTICAL SEAM.** A `MatchSpec` dataclass and a registry:
```python
@dataclass(frozen=True)
class MatchSpec:
    entity_type: str      # "lead" | "client" | "resource" | "applicant" | "referral_partner"
    table: str
    name_col: str = "name"
    phone_col: str | None = "phone"
    email_col: str | None = "email"
    label: str            # "lead", "caregiver", …  (reuses automations/entities.py ENTITY_LABELS vocab)

MATCHABLE: dict[str, MatchSpec] = { … }   # the 5 person-like entities
```
Mirrors the `WRITERS` / `ENTITY_TABLES` seam so a re-template swaps this file. `resource` maps to caregivers; `referral_partner` and `client_contacts` included where useful.

**`matching/fuzzy.py` — CORE engine.** In-seam constants + scoring:
```python
WEIGHTS = {"email_exact": 60, "phone_exact": 55, "name_ratio_full": 40, "name_token_bonus": 10}
BANDS   = {"link": 85, "review": 60}   # >=link auto-link/confident; [review,link) → flag; <review → no match
```
- `score_pair(query: MatchQuery, row: dict, spec) -> MatchResult` where `MatchQuery` is normalized `{name, phone, email}` and `MatchResult = {score, reasons: list[str], warnings: list[str], band}`. Components:
  - Normalized email equal → `email_exact` + reason "Same email address".
  - Normalized phone equal → `phone_exact` + reason "Same phone number".
  - `rapidfuzz.fuzz.WRatio` / `token_sort_ratio` on normalized names, scaled by `name_ratio_full`; token-set overlap adds `name_token_bonus`. Reason: "Name 'Peggy Ellison' ≈ 'Margaret Ellison' (92%)".
  - Warning when name matches strongly but phone/email **conflict** (present and different) — "Names match but phone numbers differ".
- `find_candidates(conn, entity_type, query: dict, *, min_band="review", limit=10, exclude_id=None) -> list[MatchResult]` — fetch the tenant-scoped rows for `spec.table` (single pass, `dict_row` cursor inside the caller's tenant tx, exactly like `matching.py`), score each, drop below `min_band`, sort `(-score, name)` (stable), top-N. `exclude_id` skips self during dedupe.

**Validation:** `backend/tests/test_matching_fuzzy.py` — pure offline (feed candidate rows as dicts, no DB). Assert: exact-email dominates; reordered/accented names land in the `link` band; a name-only near-match lands in `review`; conflicting phone raises a warning; identical inputs → identical ordering (determinism).

## Task 3 — `find_entity` agent tool + automation function

One engine, two thin wrappers. **No changes to `execute_tool`, the chat loop, or the engine** — the tool flows through chat, MCP, and automation `tool` steps automatically once registered.

- **Tool** (register in `tools/entities.py`, the vertical tools seam, via `register(ToolDef(...))`, `safe=True`):
  - `name="find_entity"`, input `{query: str (required), entity_types?: string[] (default all MATCHABLE), limit?: int}`.
  - Handler builds a `MatchQuery` from the free-text `query` (a name, phone, or email — detect shape with the normalizers) and calls `find_candidates` across the requested specs; returns a `ToolResult` whose `data` lists `{entity_type, id, name, score, reasons}` and whose `summary` is plain-language ("Found 2 possible matches for 'peggy ellison': Margaret Ellison (lead, 92%)…"). Uses existing `_fetch_all`/`_jsonable` helpers.
  - Add to `tools/labels.py::TOOL_LABELS` ("Find a person"). Add one clause to `chat_service.py::PERSONA`'s tool paragraph so the agent prefers `find_entity` over `list_*` substring search when told a messy name.
- **Function** (register in `automations/functions.py` via `register_function(FunctionDef(...))`): `find_entity` returning the single best match id (or null) — pure/read-only, satisfies the "function has no external effect" contract. Lets a recipe resolve `{{trigger.payload.source}}` → an entity id for a downstream step. Renders in the M8 builder automatically from its `input_schema`.

**Validation:** `backend/tests/test_find_entity_tool.py` (gated on `NEXUS_APP_DB_URL`, `tenant_tx`/`asyncio.run` style per `test_matching.py`): seed a lead "Margaret Ellison"; assert `execute_tool(..., "find_entity", {"query": "Peggy Ellison"})` returns her in the `link`/`review` band and writes exactly one `tool.called` audit event. A function-path test resolves a messy string to her id.

## Task 4 — Cross-entity dedupe scan (`scan_duplicates`)

Review-task-only. A safe tool (callable by the agent, manually, or from an automation cron step) that finds near-duplicate pairs and raises plain-language review tasks. No merge, no migration, no new table.

- **Engine helper** in `matching/fuzzy.py`: `find_duplicate_pairs(conn, entity_type, *, min_band="review") -> list[pair]` — for each row, `find_candidates(..., exclude_id=self)` within the same type; emit unordered pairs above the band, de-duplicated. Skip pairs already linked by design (e.g. a `client.lead_id` pointing at the lead — that's a promotion, not a duplicate).
- **Tool** `scan_duplicates` (register in `tools/entities.py`, `safe=True`), input `{entity_types?: string[]}`. For each flagged pair with no existing open flag:
  1. `log_event(... event_type="duplicate.flagged", entity_type, entity_id=<survivor-candidate>, payload={summary, pair:[idA,idB], names:[…], score, reasons})` via `services/events.py`.
  2. Insert a `tasks` row: title "Possible duplicate: Margaret Ellison ↔ Peggy Ellison", description naming both records, their ids, and the match reasons in plain language, `originating_event_id` = the flagged event. (Tasks have no entity columns — the `duplicate.flagged` event payload is the structured link, matching how connector review tasks already lean on `originating_event_id`.)
  - **Idempotency:** before flagging, query open (non-`cancelled`/`done`) `duplicate.flagged` events whose `payload.pair` is the same unordered id set; skip if present. Re-running the scan never double-flags.
  - Returns a `ToolResult` summarizing how many pairs were flagged.
- Add `duplicate.flagged` to the read-time summary vocabulary if needed (`services/event_summaries.py`), and to `tools/labels.py`.

**Validation:** `backend/tests/test_dedupe_scan.py` (gated DB): insert two near-duplicate leads; run `scan_duplicates`; assert one review task + one `duplicate.flagged` event naming both. Run again → no second task (idempotent). Insert a genuinely distinct lead → not flagged.

## Task 5 — Connector-resolution fuzzy hook (`resolution.py`)

The full loop. Insert a fuzzy stage in `route_normalized_event` **between** the `_find_mapping` miss (line 59) and the writer branch (line 60), gated by a config flag so exact-match-only can be restored.

- **Config** (`config.py`, alongside the M18 connector flags): `nexus_connector_fuzzy_enabled: bool = True`.
- **New stage** (only when the flag is on, `ev.creates_entity` is true, and `ev.entity_type` is in `MATCHABLE`): build a `MatchQuery` from `ev.attributes` (name/phone/email) and call `find_candidates(conn, ev.entity_type, query, min_band="review", limit=3)`.
  - **`link` band (top score ≥ BANDS["link"], unambiguous):** don't create. Insert the `external_ids` mapping onto the **existing** entity id, bump `last_synced_at`, and `_log(..., resolution="linked", entity_id=existing)`. Add `"linked"` to the resolution vocabulary (new value alongside matched/created/task). Return `RouteOutcome("linked", …)`. Optionally backfill a missing phone/email on the existing row (skip if it would overwrite).
  - **`review` band (or a `link`-band tie between candidates):** don't create. Raise a review task ("Inbound {event_type} for '{name}' looks like existing {label} '{cand}' — link or create?") via a small extension of `_create_review_task` that names the candidate(s), and `_log(..., resolution="task")`. Return the task outcome.
  - **below `review` / no candidate:** fall through to the **existing** writer/create path unchanged.
- When the flag is off or the type isn't matchable, behavior is byte-for-byte today's — the exact-match and writer paths are untouched.

**Validation:** extend `backend/tests/test_connectors_resolution.py` (existing `app_db` + `set_tenant` sync style):
- Seed a lead; feed an inbound `lead.created` whose attributes fuzzy-match it strongly → outcome `linked`, an `external_ids` row now points at the existing lead, **no** new lead created.
- Medium-similarity inbound → outcome `task`, no create, candidate named in the task.
- Low-similarity inbound → outcome `created` (unchanged), new lead + mapping.
- Flag off → strong match still `created` (proves the escape hatch).

## What this plan deliberately does NOT do

- **No merge tool / no data mutation of duplicates** (user decision: review-only). The heavy FK-repointing merge is a future v1.0.x if the review queue proves it's needed.
- **No Postgres extension or migration.** Python-side scoring is correct and testable at this corpus size; pg_trgm is the documented scale-up path only.
- **No new env-tunable weights/thresholds.** In-seam constants, per the `views/matching.py` precedent.
