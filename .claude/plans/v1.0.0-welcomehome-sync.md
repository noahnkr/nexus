**Complexity: ⚠️ Medium** — the largest sub-plan of Module 18: it lands the shared connector-sync infrastructure (loop + ingest seam) and the full WelcomeHome integration. **Fallback split**: Tasks 1–4 (infrastructure, WH client, migration) stand alone and unblock 18b–18d; Tasks 5–7 (mapping, runner, backfill) are the WH content pass.

## Context

WelcomeHome is the CRM system of record (account 18754, one community, id 65648). The API has **no webhooks**; sync is polling `GET /api/exports/community/{cid}/table/{table}` (live paginated CSV, `filters[updated_at_after]`, `Link`-header cursor, ≤3 reuses of a cursor per minute) with JSON reference endpoints for config (`/stages`, `/activity_types`, `/lead_sources`, `/prospect_fields`). Auth: `Authorization: Token token={WELCOMEHOME_API_KEY}` (verified live, HTTP 200). User-locked scope: prospects + stages + backfill; activities/messages/calls → lead timelines; call transcriptions → RAG; residents + influencers/relationships. One-way inbound.

**Live stage vocabulary → Nexus lead statuses** (locked mapping; the Nexus funnel does NOT change this module — re-templating the Leads view to mirror WH stage names is a possible fast-follow, noted in Future Plans):

| WelcomeHome stage (`system_type`, position) | Nexus `leads.status` |
|---|---|
| Inquiry (`new_lead`, 0) | `new` |
| Contact Attempted (1), Contact Made (2) | `contacted` |
| Home Visit Scheduled (3), Home Visit Completed (`visit`, 4) | `qualified` |
| Start of Care (`move_in`, 5) | `converted` **[R] + client promotion (decision 11)** |
| discarded / close_reason set | `lost` |

The map is **config, not code**: keyed by `system_type` where present, else position band, so a renamed WH stage doesn't break it. Unknown stage ⇒ leave status unchanged + `connector.sync_failed`-style warning event (never crash).

## Key design decisions (locked in — do not re-litigate during build)

1. **Sync loop core** (`services/connectors/sync.py`): `SyncRunner` protocol (`source`, `enabled(settings)`, `run(conn, state) -> new_state`), a module registry, and `connectors_cycle()` invoked from a lifespan task every `NEXUS_CONNECTORS_POLL_SECONDS` (default 120). One `tenant_tx` per runner run; cursor state read/written on the `(tenant, source_system)` row of `connector_state` (state key layout per source, e.g. `{"cursors": {"Prospects": "...", "Activities": "..."}}`). Failure ⇒ `connector.sync_failed` event with plain summary + technical detail, runner skipped this cycle, loop continues. `connector_sync` traceable chain span per cycle with per-runner child spans. Justification: mirrors the proven M7 loop pattern exactly.
2. **Ingest seam** (`services/connectors/ingest.py`): move the post-verification body of the webhook route (receipt event → adapter.normalize → route_normalized_event, incl. the existing trace span) into `ingest_payload(source, payload, headers)`. The route keeps verify + HTTP concerns; runners call it directly. Behavior-identical: **all existing connector tests (`test_connector_adapters` / `test_connectors_resolution` / `test_webhook_ingress`) must pass unmodified.**
3. **[R] Resolution gains an update path — and leads get their single stage-writer first**: `NormalizedEvent` gets `updates_entity: bool = False` + reuses `attributes`; `entity_writers.py` gains `UPDATERS = {"lead": update_lead, "client": update_client}` (patch only non-null attributes). On a `matched` outcome with `updates_entity`, resolution applies the updater before logging. No updater registered ⇒ today's log-only behavior.
   **Correction**: the original plan routed stage changes "through `services/views/leads.update_lead_status`" — **no such function exists**. The stage-change logic (event emission + `supersede_sequence_runs`) currently lives *twice*: in `routers/leads.py` PATCH and in the `update_lead_status` tool handler (`services/tools/entities.py`). A connector updater would be a third copy. So this task first extracts the M15 pattern that arrived after 18a was planned — `views/clients.change_status` is the ONE writer of client status — into the leads seam: `services/views/leads.py::change_stage(conn, tenant_id, lead_id, to_status, *, source_system)` (event + supersede, caller's transaction). REST PATCH, the tool handler, and the connector updater all delegate; behavior-identical for existing callers (their tests pass unmodified). `lead.stage_changed` then lands with `source_system='welcomehome'` and sequences/timelines react for free.
4. **[R] Migration `20260730000000_entities_crm_sync.sql`** (vertical seam — **renamed**: the original `20260724000000` now sorts *before* four already-pushed migrations, `20260726…tenant_settings` through `20260729…workforce`; the new stamp must follow `20260729000000`): `alter table public.leads add column if not exists zip text, add column if not exists address text, add column if not exists background text;` plus new table `lead_contacts` **mirroring the M15 `client_contacts` shape** (`id uuid pk`, `tenant_id`, `lead_id fk → leads on delete cascade`, `name text not null`, `relationship text`, `phone text`, `email text`, `is_primary boolean default false`, `notes text`, `source text`, timestamps) with the standard guarded 4-policy RLS + `set_updated_at` (copy the `client_contacts` DDL pattern verbatim); no Realtime (timeline events cover UI freshness). Note: `clients` and `resources` already carry `address`/`zip` (M12) — only `leads` gains them; promotion (decision 11) copies them across. Residents: the *primary* resident's name/details feed the lead row (a WH "prospect" is the deal; the resident is the care recipient); additional residents and all influencers become `lead_contacts` rows (`relationship` = WH relationship label or `resident`). Justification: keeps `leads` the single pipeline entity (M9 unchanged) while giving related people a queryable home that promotes cleanly into `client_contacts`.
5. **WH client** (`wh_client.py`): httpx-based; `ping()`, `reference(name)` for the JSON config endpoints, and `export_pages(table, updated_at_after)` — an async generator yielding parsed CSV row-dicts page by page, following the `Link` header and sleeping to respect the 3/min cursor-reuse limit. Timeouts + bounded retries; raises a typed error the runner converts to `connector.sync_failed`.
6. **Mapping** (`wh_map.py`): pure functions, fixture-tested — `map_prospect(row, refs) -> (attributes, status)` (name from primary resident else prospect fields; phone/email; `source` = lead-source name; zip/address/background from resident/custom fields discovered at build time via `/prospect_fields`; `notes`), `map_activity(row, refs) -> NormalizedEvent` (`lead.activity_logged`, summary "«Type name»: first line of notes", `detail` carries type/direction/notes/completed_at; activity `record_type/record_id` links it to the prospect's external id), `map_contact(row)` for Influencers/extra Residents. `refs` (stages/activity_types/lead_sources/prospect_fields) are fetched once per cycle and cached in the runner. Which activity types represent messages/calls/transcriptions is **discovered at build time from the live `activity_types` list** (Task 5 records the chosen type→treatment table in code comments + README); transcription-bearing activities route their text to ingestion (decision 8).
7. **[R] Events used**: `lead.created` (auto-create, existing), `lead.updated` (update path), `lead.stage_changed` (via `change_stage`), `lead.activity_logged` (new), `client.created` (promotion, decision 11 — already a known type). `lead.activity_logged` is declared in `EVENT_ENTITY_TYPES` / `EVENT_PAYLOAD_FIELDS` (`detail.activity_type`, `detail.direction`, `detail.notes`) and `_KNOWN_EVENT_TYPES` so automations/tokens see it. **Frontend threading** (M13/M14a maps that postdate the original plan — fallbacks are graceful, but thread them for polish): `lib/events.ts` `SOURCE_ACCENT` gains a `welcomehome` accent (connector accent, alongside `webhook`); `lib/recipe.ts` `eventTypeLabel` gains `lead.activity_logged`. This amends the parent plan's "no frontend changes" note — two map entries, no components. Activity idempotency: the Activities cursor only moves after a committed batch, and each activity event's `detail.wh_activity_id` lets the runner skip rows already ingested in a partially-replayed page (checked against the last-batch id set kept in cursor state — cheap, no new table).
8. **[R] Transcriptions → RAG via the sanctioned M15 tag**: `services/ingestion` gains an `ingest_text(title, text, entity_type, entity_id, source)` entry (same parse→chunk→embed pipeline minus file parsing). The original plan's "storage_path null-able or a `text://` marker — check the DDL" hedge is resolved: `documents.storage_path` is nullable (M0 DDL) **and** M15 landed the sanctioned entity tag (`documents.entity_type`/`entity_id`, chunks stamped — `20260727000000_core_document_entity.sql`, already threaded through `services/ingestion.py`). `ingest_text` inserts the document row with `storage_path` null and the lead tag; chunks inherit it exactly like the M15 tagged-upload path. Only activities whose type is in the transcription set and whose notes exceed a threshold (~500 chars) are ingested. Known limitation (accepted): after promotion, transcripts stay tagged to the *lead* — the client profile's documents card lists client-tagged docs only; the transcript remains reachable via chat RAG and the lead timeline. Re-tagging on promotion → Future Plans if the office asks.
9. **Backfill** (`scripts/backfill_welcomehome.py`): operator-run, idempotent, resumable (its own cursor key in `connector_state`), ordered Prospects → Residents → Influencers → Activities, prints per-table counts, never prints record contents. Events for backfilled history are written with their original `occurred_at`. Rate-limit aware (the 3/min cursor rule). **[R]** Promotion during backfill (decision 11) is real: historical Start-of-Care prospects each create an `active` client. WH has no discharge signal, so long-ended engagements would land as active — the runbook therefore tells the operator to (a) set `welcomehome_backfill_since` to bound how far back promotion reaches, and (b) sweep the freshly created client list and discharge stale ones before trusting the census numbers. Justification: one-time import is an ops action, not a loop concern.
10. **New settings**: `welcomehome_api_key`, `welcomehome_community_id` (default `"all"`), `nexus_connectors_enabled` (default true), `nexus_connectors_poll_seconds` (default 120), `welcomehome_backfill_since` (optional ISO date bound for the Activities backfill **and** the promotion reach, decision 9). All optional; the WH runner is enabled only when the key is set.
11. **[R] NEW — Start-of-Care client promotion** (the M15 gap): when an applied WH stage move lands a lead on `converted` and **no `clients` row with that `lead_id` exists**, the lead updater promotes — inside the same transaction, wholly in the vertical seam (`entity_writers.py::write_client`, the M10 `move_stage` hire-promotion precedent):
    - `clients` row: `lead_id` set, `name`/`phone`/`email` from the primary resident (the care recipient — same source as the lead row), `address`/`zip` copied from the lead's WH-mapped values, status `active`; `payer`/`authorized_hours_per_week`/`care_summary`/`region_id` stay null (WH doesn't know them; the census buckets null payer as "unknown" by design — the office completes intake on the client profile).
    - `lead_contacts` rows for the lead are copied to `client_contacts` (same shape by construction, decision 4).
    - The WH resident id is registered in `external_ids` against the **client** entity, so later Resident-row changes resolve to the client and flow through `UPDATERS["client"]` (contact/address fields only — never status, payer, or authorized hours; client lifecycle belongs to the M15 seam).
    - Emits `client.created` with `source_system='welcomehome'` and a plain summary ("Client 'Margaret Ellison' created from WelcomeHome start of care"). The dormant declared `lead.converted` type stays dormant — `client.created` is the promotion event, matching M10's `resource.created`.
    - Idempotent: a re-synced/replayed `move_in` row or a backfill re-run finds the existing `lead_id` match and skips.
    - **Why this is in scope now**: M16's hours-won metric joins `clients.lead_id → leads.source`, and M15's census needs client rows — but nothing in the app writes `clients.lead_id` (the manual create route omits it). Without promotion, every CRM-driven conversion leaves Clients empty and Referrals blind. Human-path conversions stay manual (M15 behavior unchanged — a UI stage-move to `converted` does *not* promote; extending promotion to all `change_stage` callers is a deliberate product decision left to a fast-follow).
12. **[R] NEW — Referral-source discipline** (the M16 contract): WH `lead_sources` names are written to `leads.source` **verbatim and stably** — they are the exact-match enrichment key for `referral_partners`. The connector **never creates or links partner rows** (CLAUDE.md: "lead write paths and connector adapters never link to partner rows, they just keep writing source strings"); the WH `Referrers` / `MarketingTouchpoints` export tables are out of scope. An update must not blank or rewrite `source` unless WH itself changed the prospect's lead source. README documents that partners tracked on `/referrals` must be named exactly as the WH lead-source string (the Track button prefills it, so this is the happy path already).
