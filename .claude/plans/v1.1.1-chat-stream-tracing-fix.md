# Plan v1.1.1: Fix `NoneType … 'outputs'` on tool-calling chat turns

**Complexity: ✅ Simple** — a localized fix to one call site plus its error surface and tests. No schema, no seam, no new subsystem. Single-pass executable.

One caveat that shapes the whole plan: **the crash site is certain, the trigger is not reproducible offline.** Task 1 is therefore not test-driven in the usual red-then-green sense — the tripwire in Task 2 is what gives the fix a failing-before/passing-after signal, and Task 5 (live) is the real proof. Do not spend build time hunting for an offline repro of the original exception; that hunt was already run and is documented below.

## Context

**Why now.** Chat is the core surface and some questions fail outright in the live app. Reported 2026-07-21 with two examples: *"What is my most recent touch point with Barbara Noftz"* and *"Which caregiver has the phone number +16195550303"*. Both fail with `'NoneType' object has no attribute 'outputs'` rendered directly into the chat window.

**What the investigation established (2026-07-21).** `.outputs` appears nowhere in this codebase. A sweep of every `.outputs` attribute access in the installed dependency tree found exactly **two** that can execute during a chat turn, and they are the same code in the sync/async halves of the pinned `langsmith==0.10.4`:

`venv/Lib/site-packages/langsmith/wrappers/_anthropic.py`, `AsyncMessageStreamWrapper.text_stream` (line 337; sync twin at 426):

```python
@configured_traceable_text
async def _text_stream(**_):
    async for chunk in self._wrapped.text_stream:
        yield chunk
    run_tree = run_helpers.get_current_run_tree()   # may return None
    final_message = await self._wrapped.get_final_message()
    outputs = _message_to_outputs(final_message)
    run_tree.outputs = outputs                     # <-- AttributeError
```

`get_current_run_tree()` legitimately returns `None` in two documented cases: tracing disabled (`_setup_run` stores `None` in the run container's context, `run_helpers.py:1609`) and a dead weakref (`_PARENT_RUN_TREE_REF` holds `weakref.ref(run_tree)`, `_internal/_context.py:27`, deliberately weak to avoid leaking contexts captured by asyncio). Every other `get_current_run_tree()` caller in the SDK guards for `None` (e.g. `set_run_metadata`); this one does not. **The same line is unguarded in langsmith's current `main` branch, so upgrading the pin is not a fix.**

Everything else in the dependency tree that touches `.outputs` is evaluation/dataset code (`evaluation/*.py`, `client.py`, `beta/_evals.py`) — unreachable from a chat turn.

**What did not reproduce.** An offline harness (real `wrap_anthropic` over `httpx.MockTransport` serving canned SSE, Python 3.12.2, `LANGSMITH_TRACING=true`) was run across: a plain text response; a `tool_use`-only response with zero text blocks; the exact `chat_service` kwargs (two `system` blocks with `cache_control`, a `tools` array, `tool_choice` `auto`/`none`, `extra_body={"thinking": {"type": "disabled"}}`, replayed `tool_use`/`tool_result` history); a full two-iteration tool loop; consumption from inside a `@traceable` coroutine *and* from inside a `@traceable` async generator (matching `stream_chat_turn`); `gc.collect()` between every text delta; 25 concurrent turns; and both branches of `_process_async_iterator` (forcing `_runtime_override_active()` to `True`). **All passed.** The trigger therefore depends on live conditions the harness does not recreate — most plausibly context propagation through the Starlette/anyio SSE task, under real concurrency. That is precisely why the fix chosen removes the crash site rather than trying to prevent the `None`.

**What exists today.**
- `backend/app/services/chat_service.py:336` — `async for delta in stream.text_stream:` inside the `MAX_ITERS` agentic loop. **This is the only consumer of `.text_stream` in the repository**, and the only streaming Anthropic call anywhere (`summary.py`, `draft.py`, `engine.py` all use non-streaming `messages.create`, which has no `.outputs` deref and is unaffected).
- `backend/app/llm.py` — one lazily-built `wrap_anthropic(AsyncAnthropic(...))` shared by every caller.
- `backend/app/routers/chat.py:111` — `except Exception as exc: yield _sse("error", {"message": str(exc)})`. This is how the raw SDK exception string reached the user's screen. No router in the repo currently has a logger; services use `logging.getLogger("nexus.<area>")`.
- `frontend/src/lib/sse.ts:69` → `ChatPage.tsx:191` — the `error` frame's `message` is toasted and written into the assistant bubble verbatim. **No frontend change is needed**; a plain-language backend message flows straight through.
- `backend/tests/test_chat_tools.py` — the offline tool-loop test. Its `FakeStream` hand-rolls a `text_stream` property, so **the existing suite could never have caught this bug**: it fakes away the very wrapper that crashes. `test_chat_stop.py` imports `_install`/`FakeBlock`/`FakeMessage` from it, so both files move together.
- `.env` has `LANGSMITH_TRACING=true` and a real `LANGSMITH_API_KEY`, so tracing is live in this deployment.

**What the user locked in (2026-07-21).**
1. **Iterate the raw event stream** in `chat_service` instead of `.text_stream` — not an `llm.py` seam (one caller does not justify a layer) and not a monkeypatch of `langsmith` internals.
2. **Fix the error surface too**: log the traceback server-side, send the user a plain-language message.
3. **Add one real-wrapper regression test** (`wrap_anthropic` + `httpx.MockTransport`) rather than converting the whole chat suite or doing nothing.

**Non-goals.**
- **Upgrading or pinning around `langsmith`** — upstream `main` is unfixed, so a bump buys nothing. The pin stays at `0.10.4`.
- **An `llm.py` streaming seam** — explicitly declined. If a second streaming caller ever appears, that is the version to reconsider it.
- **Reporting the bug upstream / carrying a patched fork** — out of scope; the code change makes us independent of it.
- **Any frontend change** — `sse.ts` and `ChatPage.tsx` already render whatever `message` the backend sends.
- **Structured error codes in the SSE `error` frame** (e.g. `{code, message}`) — the contract stays `{message}`. If richer client-side error handling is wanted later it belongs in a chat-UX version, not here.
- **Extended thinking / `thinking` block streaming** — still deliberately disabled (`chat_service.py:334`); this version does not revisit it.

## Key design decisions (locked in — do not re-litigate during build)

1. **Consume `stream` (events), never `stream.text_stream`.** The wrapper traces the event iterator through a *different* path — `AsyncMessageStreamWrapper.__aiter__` → `configured_traceable` (`reduce_fn=_reduce_chat_chunks`) → `_TracedAsyncStream` — which contains **no `run_tree.outputs` assignment at all**. Verified in the harness: `_reduce_chat_chunks` accumulates the raw events through the SDK's own `accumulate_event` and produces the same `_message_to_outputs` payload, so **the LLM span is equivalent, not degraded**.

   ```python
   async with client.messages.stream(...) as stream:
       async for event in stream:
           if event.type == "content_block_delta" and getattr(event.delta, "type", None) == "text_delta":
               delta = event.delta.text
               if not citations_sent:
                   yield "citations", {"sources": agg_sources}
                   citations_sent = True
               streamed_text += delta
               yield "text", {"delta": delta}
       final = await stream.get_final_message()
   ```

   Use `getattr(event.delta, "type", None)` — `content_block_delta` also carries `input_json_delta` (tool arguments) and, on other block types, `signature_delta`/`thinking_delta`. Only `text_delta` is user-visible text.

2. **The SSE contract is unchanged**, byte for byte: `start → (tool, tool_result)* → citations → text* → done`. `citations` still fires exactly once, immediately before the first `text` frame, and `streamed_text` still accumulates only the current iteration's text (the stop-contract invariant from M15a). Every existing assertion in `test_chat_tools.py` / `test_chat_stop.py` must pass with its expectations unmodified.

3. **`get_final_message()` still closes each iteration.** The SDK accumulates state as events are iterated; awaiting the final message after the loop returns the same `Message` (with `stop_reason`, `usage`, and `content` blocks) as today. Confirmed in the harness for both `end_turn` and `tool_use` responses. `_dump_content(final.content)` and the `stop_reason != "tool_use"` break are untouched.

4. **A tripwire replaces the fake's `text_stream`.** In `test_chat_tools.py`, `FakeStream` gains an `__aiter__` yielding fake `content_block_delta` events, and its `text_stream` property becomes:

   ```python
   @property
   def text_stream(self):
       raise AssertionError(
           "chat must not consume .text_stream — it hits the unguarded "
           "run_tree.outputs deref in langsmith's Anthropic wrapper (v1.1.1)"
       )
   ```

   This is what makes the fix falsifiable: with Task 1 reverted, the whole chat suite fails loudly with a message naming the reason. It also permanently prevents a future edit from drifting back onto the buggy path.

5. **Errors: log the traceback, show plain language.** `routers/chat.py` gets `log = logging.getLogger("nexus.chat")` (matching the `nexus.<area>` convention used by services). The generic handler becomes:

   ```python
   except Exception:
       log.exception("chat turn failed (thread=%s)", thread_id)
       yield _sse("error", {"message": "Something went wrong answering that. Please try again."})
   ```

   `ThreadNotFound` keeps its own specific message. This satisfies the CLAUDE.md rule that user-facing surfaces carry plain language, not internals — and means the next internal failure is diagnosable from server logs rather than from a screenshot.

6. **The real-wrapper test is a CI guard, not a red-first test.** `tests/test_chat_stream_tracing.py` drives `stream_chat_turn` with a genuinely `wrap_anthropic`'d client over `httpx.MockTransport`. It is expected to **pass both before and after** Task 1 (the trigger does not reproduce offline). Its value is that it exercises the real wrapper on the real code path in CI, so a future langsmith bump that breaks streaming fails here instead of in the user's chat window. Task 2's tripwire is the fail-first signal.

7. **`LANGSMITH_ENDPOINT` is pointed at an unroutable address in the new test**, with a fake key and `LANGSMITH_TRACING=true`. Tracing must be genuinely *enabled* for the test to exercise the wrapper's traced path, but no run may actually be shipped anywhere. The SDK batches uploads on a background thread and swallows connection failures, so the test stays offline and green. Set and restore the env vars with `monkeypatch.setenv`.

### Task 1 — Consume the event stream in `chat_service`

Replace the `async for delta in stream.text_stream:` block in `stream_chat_turn` (`backend/app/services/chat_service.py`, ~line 336) with the event-iteration form from decision 1. Keep `citations_sent`, `streamed_text`, and the `yield "text"` frame identical.

Add a comment in the same register as the neighbouring `extra_body={"thinking": ...}` workaround note, stating: we iterate raw stream events because `langsmith`'s `AsyncMessageStreamWrapper.text_stream` assigns `run_tree.outputs` without a `None` guard (unfixed upstream as of 0.10.4), and the event path is traced separately via `_reduce_chat_chunks` with equivalent output. Update the module docstring's SSE section only if wording drifts — the contract itself does not change.

**Validation:** `cd backend && venv/Scripts/python.exe -m pytest tests/test_chat_tools.py tests/test_chat_stop.py -q` — green after Task 2 updates the fakes. (Expect failures *before* Task 2: the current `FakeStream` has no `__aiter__`. Do Tasks 1 and 2 together, then run.)

### Task 2 — Update the offline fakes and install the `.text_stream` tripwire

In `backend/tests/test_chat_tools.py`, rework `FakeStream`:
- add `async def __aiter__(self)` yielding one fake event per scripted delta, shaped like `content_block_delta` — a small `FakeEvent(type="content_block_delta", delta=FakeBlock("text_delta", text=d))` is enough, since `chat_service` only reads `.type`, `.delta.type`, and `.delta.text`;
- turn `text_stream` into the raising property from decision 4;
- leave `get_final_message`, `__aenter__`, `__aexit__` as they are.

`test_chat_stop.py` needs no edit — it imports `_install` and drives the same fakes.

**Validation:**

2. Fail-first proof: temporarily revert Task 1's edit, re-run, and confirm both files fail with the tripwire's `AssertionError` naming `.text_stream`. Restore Task 1.

### Task 3 — Real-wrapper regression test

Add `backend/tests/test_chat_stream_tracing.py`. It must use the **real** `wrap_anthropic` and the **real** `anthropic.AsyncAnthropic`, with only the HTTP transport faked:

- `monkeypatch.setenv` for `LANGSMITH_TRACING=true`, `LANGSMITH_API_KEY=lsv2_pt_fake`, `LANGSMITH_ENDPOINT=http://127.0.0.1:9/unroutable` (decision 7).
- Canned SSE bodies as module constants: one `tool_use`-only response (`stop_reason: tool_use`, a `content_block_start` of type `tool_use` plus an `input_json_delta`) and one text response (`stop_reason: end_turn`). Serve them in order from an `httpx.MockTransport` handler.
- Build the client as `wrap_anthropic(AsyncAnthropic(api_key="sk-fake", http_client=httpx.AsyncClient(transport=MockTransport(handler))))` and monkeypatch `cs.get_anthropic` to return it.
- Reuse `FakeConn` / `fake_tx` / `fake_execute_tool` by importing `_install` from `test_chat_tools` and then overriding only `get_anthropic`, so the DB and tool seams stay faked exactly as the other chat tests fake them.
- Drive a full two-iteration tool loop through `cs.stream_chat_turn` and collect the frames.

Assert: the turn completes without raising; the frame sequence is `start → tool → tool_result → citations → text* → done`; the concatenated `text` deltas equal the canned answer; and `execute_tool` was called once with the scripted tool name.

Add a docstring stating plainly that this test passes both before and after the v1.1.1 fix (decision 6) so a later reader does not mistake it for the reproduction.

**Validation:** `venv/Scripts/python.exe -m pytest tests/test_chat_stream_tracing.py -q` passes, and completes in under a few seconds with no network access (the LangSmith uploader's connection errors are logged by its background thread and must not fail the test — assert on the frames, never on log output).
