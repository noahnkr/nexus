## Context

GoTo Connect is the business phone system. Planning-time probe confirmed the OAuth client **rejects `client_credentials`** (401 "Unauthorized grant type") — the authorization-code flow with a one-time browser consent is mandatory; the refresh token then drives unattended access. Inbound call/SMS events come from a **Notification Channel** (user-locked: the WebSocket variant, which needs no public URL) subscribed to the Call Events API (`POST /call-events/v1/subscriptions`); channels expire and must be renewed. Outbound SMS goes through GoTo's Messaging API from the business line. The Module 3 placeholder adapter already normalizes `call.completed` / `sms.received` payloads with E.164 external ids; the bridge feeds it.

An open question the docs don't settle (the PRD flagged it): whether **inbound SMS** events are deliverable on the same notification channel or only via the Messaging API. Task 2 answers it empirically and takes the documented fallback (a Messaging-API poll runner) if channels don't carry SMS — the ingest path is identical either way.

## Key design decisions (locked in — do not re-litigate during build)

1. **OAuth bootstrap** (`scripts/goto_oauth.py`): starts a localhost redirect listener (port from `GOTO_CONNECT_REDIRECT_PORT`, default 8765 — **must match a redirect URI registered on the OAuth client**), prints the consent URL, exchanges the code at `https://identity.goto.com/oauth/token`, and prints `GOTO_CONNECT_REFRESH_TOKEN=...` for the operator to paste into `.env`. Never persists tokens itself. Scopes requested: the call-events read+notification-channel manage and messaging send/read scopes as named by the developer portal at build time (recorded in the script's docstring and README).
2. **Shared refresh helper** (`services/connectors/oauth.py`): `TokenSource(token_url, client_id, client_secret, refresh_token)` with an in-process cached access token (refresh at <60s remaining). Used by 18b and 18c — one implementation, two configs.
3. **Channel + subscription manager** (`goto_client.py`): create a WebSocket notification channel, `POST /call-events/v1/subscriptions` for call events (and SMS if supported — see context), persist `{channel_id, ws_url, expires_at, subscription_ids}` in `connector_state.state.goto`; re-create/renew when absent or near expiry. All idempotent — safe to run every cycle.
4. **WebSocket bridge** (`goto_bridge.py`, dep `websockets`): a long-lived task started by the sync loop (not per-cycle): connect to the channel's WS URL, keepalive pings, translate each received notification into the placeholder adapter's payload shape, and hand it to `ingest_payload("goto", ...)`. Reconnect with expo backoff; a dead channel triggers the manager's re-create on the next cycle. The HTTP webhook adapter stays untouched for a future deployed setup — same adapter, two deliveries.
5. **Real `send_sms`** (`services/messaging/goto_sms.py`): called from the existing gated tool handler — tool name, schema, gating, audit events all unchanged. Sender line resolved from `GOTO_CONNECT_LINE_ID` (or discovered from the account and cached; recorded in README). Without credentials the tool fails loudly with a plain message (today's placeholder behavior notes it's simulated — that note goes away).
6. **New settings**: `goto_connect_client_id`/`_client_secret` (present), `goto_connect_refresh_token`, `goto_connect_line_id`, `goto_connect_redirect_port`. Bridge/manager enabled only when id+secret+refresh token are all set.

## Tasks (strict order)

### Task 1 — OAuth bootstrap + refresh helper
Decisions 1–2 + settings.
**Validation**: offline — `TokenSource` refresh flow against a mocked token endpoint (caching, expiry slack, error surface). **Blocking ops step**: operator runs `python -m app.scripts.goto_oauth`, completes the browser consent, pastes `GOTO_CONNECT_REFRESH_TOKEN` into `.env`. Then the env-gated live test obtains an access token (asserted non-empty, never printed).

### Task 2 — Channel + subscription manager
Decision 3; empirically determine SMS deliverability on the channel and record the answer (+ fallback poll runner if needed) in code comments and README.
**Validation**: offline manager test against mocked endpoints (create-when-absent, renew-when-expiring, idempotent when healthy); gated live test creates a real channel + call-events subscription and stores state (then cleans up).

### Task 3 — WebSocket bridge runner
Decision 4.
**Validation**: offline — bridge test against a local fake WS server: a pushed call-event frame flows through `ingest_payload` and produces the receipt + `call.completed` events; disconnect triggers backoff + reconnect. **Live**: with the stack running, place a real call to the business line → within seconds the Event Log shows `call.completed` (matched lead or review task per the number), and an event-triggered automation on `call.completed` fires.

### Task 4 — Real outbound SMS
Decisions 5–6.
**Validation**: offline — handler test with mocked Messaging API (payload shape, error → plain tool error). **Live**: from chat, ask the agent to text a known number → gated task queues → approve in Tasks → a real SMS arrives; Event Log shows `action.queued → action.approved → tool.called`; LangSmith span chain intact.

### Task 5 — Wrap-up
README: GoTo section (bootstrap runbook incl. redirect-URI registration, scopes, line id, channel renewal behavior, SMS-delivery finding); `.env.example`; full pytest green.
**Validation**: full `pytest backend/tests` green; the Task 3 + Task 4 live walks recorded in PROGRESS.
