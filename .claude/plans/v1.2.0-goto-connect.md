# Plan v1.2.0: GoTo Connect — calls + SMS, the authoritative phone channel

> **Rewritten 2026-07-21** (supersedes the Module-18c-era draft): the communications tier (v1.1.0), the one-summary design (v1.1.4), the transcript requirement, the known-numbers guard, and the all-people phone matching all post-date the original. The OAuth findings from the 2026-07-20 probe carry over unchanged.

**Complexity: 🔴 Complex** — a new live connector with four moving parts (OAuth, a long-lived WebSocket bridge, transcript enrichment, real outbound SMS through the gated tool layer); CLAUDE.md defaults tool-layer versions to 🔴. Broken into **five ordered parts (A–E)**; each part is a safe stopping point, and **Part A ends in a hard gate** that can block the version without wasting downstream work.

## Context

**Why now.** GoTo Connect is the business phone system and becomes **the authoritative source for calls + SMS** (the one-source-per-channel framing that retired the old cross-source reconciler). Call transcripts and SMS land in the communications tier — and therefore in per-entity summaries and RAG — and `send_sms` stops being a placeholder. Gmail (v1.3.0) and the WelcomeHome comms retirement (v1.3.1) follow it.

**What the 2026-07-20 probe established (carried over).**
- The OAuth client **rejects `client_credentials`** (401 "Unauthorized grant type") — authorization-code with a one-time browser consent is mandatory; the refresh token then drives unattended access. Token endpoint: `https://identity.goto.com/oauth/token`.
- Inbound events come from a **Notification Channel** — the WebSocket variant needs no public URL — subscribed via `POST /call-events/v1/subscriptions`. **Channels expire and must be renewed.**
- Whether **inbound SMS** rides the same notification channel or only the Messaging API is undocumented; it gets answered empirically with a documented fallback (a Messaging-API poll in the goto runner). The ingest path is identical either way.
- Outbound SMS is the Messaging API (`messaging.v1.send`-scoped) from the business line.

**What exists.**
- `services/connectors/adapters/goto.py` — placeholder adapter (`call.completed` / `sms.received`, `_e164` normalization, caller number as external id, `entity_type="lead"` hardcoded — that assumption is removed this version).
- The connector chassis: `ingest_payload` seam, `resolution.py` (`matched`/`created`/`task` via `external_ids`; review task fallback), the sync loop (`sync.py::run_one`/`connectors_cycle`, runner registry, `connector_state` for cursors/channel state), and the lifespan background-task pattern (`main.py` starts `engine_loop` + `connectors_loop`; the WS bridge becomes a third).
- The communications tier: `ingest_communication` (store-all, `content_hash` dedup, `source_event_id` spine, `should_embed` — **sms never embeds**, long-form ≥ threshold does), `communication_chunks` as a rebuildable cache. Comms feed the merged smart summary (v1.1.4) and `search_communications` RAG.
- The gated tool layer: `tools/outbound.py::send_sms` — `safe=False`, `gate_describe`, `editable_fields=["body"]`, placeholder execution marked for replacement ("GoTo Connect messaging API… map `to`→contact, `body`→body").
- Frontend: `SOURCE_ACCENT` has no `goto` entry; v1.1.3's `eventIcon` already maps `call.*`/`sms.*` prefixes.
- `config.py` has **no GoTo settings yet** (the old draft's claim they exist is stale).

**What the user locked in (2026-07-21).**
1. ~~**Transcripts are a hard requirement.**~~ **SUPERSEDED 2026-07-21 — the gate was probed, failed, and the user relaxed it to metadata-only.** See "The A2 gate verdict" below. Calls now normalize to metadata (who / when / how long / which direction) with no body text. Everything else in the plan is unaffected: the comms tier, the timeline, and the summary all accept a call with no body; only the transcript sweep in Task 6 is dropped. If recording is ever enabled on the account, transcript enrichment is purely additive.
2. **Inbound calls/SMS resolve against all people entities** — leads, clients, caregivers (resources), and their contact rows (resolving to the parent entity). First match registers the `external_ids` mapping; ambiguity (two distinct entities share the number) goes to a review task, never a guess.

**Non-goals.**
- **An inbound-SMS conversational agent loop** (auto-replies) — no version assigned; `send_sms` remains human-gated.
- **Retiring WelcomeHome's call/SMS/email activities** — v1.3.1 (gated on this version + v1.3.0).
- **Gmail/email anything** — v1.3.0.
- **Webhook-channel (public URL) delivery** — the HTTP adapter contract stays compatible but no deployment work; this deployment uses the WebSocket channel.
- **Voicemail ingestion** — only if it arrives for free on the same channel events (recorded at build time either way); no dedicated machinery.
- **Historical call backfill** — the bridge ingests from go-live forward; GoTo's own history stays in GoTo.

## The A2 gate verdict (settled 2026-07-21 — do not re-probe)

The gate was answered from **history rather than a live call**, at the user's direction. The finding is negative and well-evidenced:

- **100 real calls over 90 days** from `GET /call-history/v1/calls`. Every record carries exactly `legId, originatorId, caller{name,number}, callee{name,number}, direction, startTime, answerTime, duration, hangupCause, ownerPhoneNumber` — **zero recording or transcript fields on any of them**.
- `/recording/v1/recordings/search` exists and is GET-only (405 on POST) but rejects **all 15 query grammars tried** (accountKey, windows in ISO and epoch, organizationId, userKey, date ranges, per-call ids) with a contentless 400. Every other recording path 404s, including `/recording/v1/settings` and any voice-admin recording surface.
- The account's live subscription list carries **no recording/transcript event types** — only `INCOMING_MESSAGE_SNIPPET`, `CONVERSATION_ASSIGNMENT_SUBSCRIPTION.*`, `SESSION_MANAGEMENT`, `INCOMING_CALLS`.
- **403 and 400 are distinguishable on this API**, which is what makes the conclusion safe: `/voicemail/v1/voicemails/search` and `/voicemail/v1/voicemails/transcriptions` both answer **403 `AUTHZ_INSUFFICIENT_SCOPE`** (real endpoints, scope not granted), while the recording surface answers 400. So `recording.v1.read` *is* granted and working — there is simply nothing recorded to read.

**Conclusion:** call recording is not producing artifacts on this account. That is an Admin setting or plan-tier question, not an API one, and no further probing will change it. **User decision: ship calls as metadata-only** rather than park the version.

**The one open lead, if transcripts are wanted later:** voicemail transcription is reachable — the endpoints exist and only the scope is missing. Adding a voicemail scope to `goto_client.SCOPES` and re-consenting would test it. Voicemail transcription is not call transcription, so it does not satisfy the original requirement on its own; it is recorded here so the finding is not lost.

## Key design decisions (locked in — do not re-litigate during build)

1. **Settings** (`config.py`, `.env` only — never the DB, per CLAUDE.md): `goto_connect_client_id`, `goto_connect_client_secret`, `goto_connect_refresh_token`, `goto_connect_line_id` (sender line for SMS; discoverable, see decision 8), `goto_connect_redirect_port` (default `8765`), `goto_ignored_numbers` (comma-separated E.164 list — the known-numbers guard). The goto runner and bridge activate only when id+secret+refresh are all set (the WelcomeHome enabled-when-credentialed pattern).

2. **OAuth bootstrap** (`app/scripts/goto_oauth.py`): starts a localhost listener on the redirect port (**the OAuth client must have `http://localhost:{port}` registered as a redirect URI**), prints the consent URL, exchanges the code at the token endpoint, prints `GOTO_CONNECT_REFRESH_TOKEN=…` for the operator to paste into `.env`. Never persists tokens itself. Scopes: call-events read + notification-channel manage + messaging send/read + recording/transcript read, exactly as named by the developer portal at build time (recorded in the script docstring and README).

3. **Shared `TokenSource`** (`services/connectors/oauth.py`): `TokenSource(token_url, client_id, client_secret, refresh_token)` with an in-process cached access token refreshed at <60s remaining. Written here, reused verbatim by v1.3.0 (one implementation, two configs).

4. **Transcript availability is a Part A gate, not a Part C surprise** (justification: user decision 1 — everything downstream assumes transcripts, so discover blockage first). The gated probe lists a recent call's recording/transcript through the account's recording API (exact endpoints from the developer portal at build time — the portal, not memory, is the authority). Outcome recorded in the plan-adjacent build notes: **available → proceed; unavailable → STOP**, report what the account is missing (recording enablement / plan tier), and the version waits.

5. **Real payload mapping is pure and separate** (`gt_map.py`, the `wh_map` pattern): translates real channel notifications into the adapter's normalized shapes; the adapter keeps its contract (verify + normalize). The placeholder shapes stay accepted (the ingress tests are the general contract). `_e164` moves up from the adapter into `gt_map` and gains the `direction` handling real payloads carry (inbound: counterpart = caller; outbound: counterpart = callee — the *counterpart* number, never the office line, is the resolution key).

6. **Known-numbers guard in `normalize`** (justification: plumbing must die before entity resolution — a WH-initiated call dials the office first, and that leg is not correspondence): any event whose counterpart number ∈ `goto_ignored_numbers` returns `ack_only` — receipt logged (audit trail intact), no resolution, no timeline, no comms. **Blocking input: the operator supplies WelcomeHome's provisional bridge number** for `.env` (visible in WH settings or the first guarded receipt).

7. **Phone-domain resolution across people entities** (user decision 2). `NormalizedEvent` gains `resolve_by: "id" | "phone"` (default `"id"` — every existing adapter untouched). For `"phone"` events, `resolution.py`:
   - looks up `external_ids` by `(source_system=category, external_id=E164)` **without an entity-type filter** (the mapping's entity_type is the answer);
   - on miss, calls the seam's `resolve_by_phone(conn, tenant_id, e164)` — a new `entity_writers.py` member (it knows the vertical tables): exact match against `leads.phone`, `clients.phone`, `resources.phone`, then `lead_contacts.phone`/`client_contacts.phone` resolving to the parent entity. Returns one match (→ core registers the mapping and proceeds `matched`), an ambiguity (**→ review task naming the candidates**), or nothing (→ the existing review task).
   Core stays business-agnostic: it calls the seam hook, it never names a table.

8. **Calls and SMS land in the communications tier via the runner's `after_commit`** (the WelcomeHome pattern — comms ingestion happens after the receipt/resolution transaction commits): SMS → `channel="sms"` (store-only by policy); calls → `channel="call"` with the **transcript as body**. Because transcripts are processed asynchronously by GoTo, `call.completed` first writes the timeline event + a `pending_transcripts` entry in `connector_state.state.goto`; each connector cycle drains it — fetch transcript → `ingest_communication` (embeds when long) → clear. A transcript still pending after the retry horizon (~24h) raises a review task (transcripts are required — silence is not an option). `goto_connect_line_id` discovery: if unset, the runner resolves the account's line id once and caches it in `connector_state` (the value is config, the cache is convenience).

9. **The WS bridge is a third lifespan task** (`goto_bridge.py`, dep `websockets`): connect to the channel's WS URL, keepalive, translate frames via `gt_map` → `ingest_payload("goto", …)`; exponential-backoff reconnect; a dead/expired channel is re-created by the **channel manager**, which runs inside the goto runner's per-cycle `run()` (create-when-absent, renew-when-near-expiry, idempotent; state `{channel_id, ws_url, expires_at, subscription_ids}` in `connector_state.state.goto`).

10. **Real `send_sms`** (`services/connectors/goto_sms.py`): the `tools/outbound.py` handler internals swap to a Messaging-API send via `TokenSource` — **name, schema, `safe=False`, `gate_describe`, `editable_fields` all unchanged**; without credentials it fails loudly with a plain message (the "[placeholder]" note dies). A successful send also writes the outbound message to the comms tier (`ingest_communication`, `channel="sms"`, `direction="outbound"`, entity from the approved action's context when known) — one interaction, one row, both directions.

11. **Frontend**: `SOURCE_ACCENT["goto"] = "bg-destructive"` (the shared came-from-outside accent, per the existing comment). Nothing else — v1.1.3 already gave `call.*`/`sms.*` their icons, and comms surface through the existing summary/timeline paths.

12. **Version fits minor** (a new connector subsystem). Plan name unchanged: `goto-connect`.
