# Plan v1.1.2: WelcomeHome stage reflection — one-to-one funnel

**Complexity: ⚠️ Medium** — a cross-stack vocabulary change (DB CHECK + data remap, five backend files, three frontend files, six test files) plus a one-time corrective re-sweep against the live WelcomeHome account. Every edit is mechanical and the seams already exist, but the stage keys thread through many files and the sweep touches real data — expect iteration on test expectations, not on design. Single-pass feasible; if it needs splitting, the natural cut is after Task 4 (backend green) with frontend + sweep as the second pass.

## Context

**Why now.** The funnel today collapses WelcomeHome's six stages into four worked Nexus stages (`new`/`contacted`/`qualified`/`converted`), so a lead's position in Nexus is coarser than what the office sees in their CRM: "Contact Attempted" and "Contact Made" both read *Contacted*; "Home Visit Scheduled" and "Home Visit Completed" both read *Qualified*. v1.1.2 makes the reflection one-to-one so every WelcomeHome stage has its own Nexus stage. It lands **before v1.2.0 deliberately** so WellSky begins writing into a settled stage set instead of one reshaped underneath it.

**What exists (verified against code and the dev DB, 2026-07-21).**
- `leads.status` CHECK constraint is named `leads_status_check` (confirmed via `pg_constraint`), values `new/contacted/qualified/converted/lost` (`supabase/migrations/20260715014926_entities_senior_care.sql:47`).
- Dev DB holds 100 leads (90 WelcomeHome-synced): `new` 37, `contacted` 24, `lost` 20, `converted` 11, `qualified` 8. **No `automations` rows carry a binding** — there are no stage-bound sequences to migrate.
- `services/views/leads.py` — `LEAD_STAGES` seam config + `change_stage()`, the single writer of `leads.status`. `funnel_metrics()` zero-fills from `LEAD_STAGES`, so it follows the config automatically.
- `services/connectors/wh_map.py` — `_SYSTEM_TYPE_STATUS` + `_POSITION_BANDS` map WH stages → Nexus statuses (config-not-code: `system_type` first, position bands for `system_type: "none"`). WH stage vocabulary (from the live account, v1.0.0): Inquiry (0, `new_lead`) · Contact Attempted (1) · Contact Made (2) · Home Visit Scheduled (3) · Home Visit Completed (4, `visit`) · Start of Care (5, `move_in`); closed/discarded → `lost`.
- `services/connectors/entity_writers.py` — `LEAD_STATUSES` tuple; `update_lead` patches fields then routes status through `change_stage`. **The patch path writes and logs `lead.updated` even when the incoming values equal the stored ones** — invisible in normal operation (the runner's `updated_at` watermark only re-sends changed records) but it would spam ~90 spurious timeline events during this version's corrective full re-sweep.
- `services/views/referrals.py:54` — `_IN_PIPELINE_STATUSES = ("new", "contacted", "qualified")`.
- `services/tools/entities.py` — `LEAD_STATUSES` list (line 38) + status vocabulary in the `update_lead_status` tool description (~line 1074) and the text-to-SQL schema doc (~line 1283).
- Frontend: `lib/api.ts` `LeadStatus` union · `lib/leads.ts` `LEAD_STAGES` + `sequenceStages` · `components/leads/LeadMetrics.tsx:48` hardcodes the in-pipeline keys. `StageSelect`/`LeadInfoCard`/the board columns all derive from `LEAD_STAGES` and need no edit.
- `wh_runner.py` cursors: one watermark per table under `connector_state.state["cursors"]` (`Prospects`, `Activities`). `services/connectors/sync.py` exposes `run_one(runner, tenant_id)` for driving a single sweep from a script.
- Automation binding validation is shape-only (`routers/automations.py::_validate_binding`) — new stage keys need no core change. `services/automations/entities.py` reads entities with `select *` — no change.
- Core `_validate_binding`, `supersede_sequence_runs`, and the sequence trigger convention (`lib/leads.ts::buildLeadTrigger`) are stage-key-agnostic; only the seam configs name stages.

**What the user locked in (2026-07-21).**
1. **Stage keys keep the existing names where they already correspond; labels stay Nexus-plain.** The seven-stage funnel is: `new` (New) → `contact_attempted` (Contact Attempted) → `contacted` (Contacted) → `visit_scheduled` (Visit Scheduled) → `visit_completed` (Visit Completed) → `converted` (Converted), plus terminal `lost` (Lost). Only `qualified` disappears (remapped to `visit_scheduled`).
2. **`lost` is an archive stage.** No sequence attaches to it; it stays out of `sequenceStages`. Leads land there when they've said don't-contact or the inquiry is simply not applicable.
3. **No new lead columns.** Close-reason context lives in SMS history and notes; nothing is added to the schema beyond the status values.

**Non-goals.**
- **Any change to `lost` behavior** — it remains a plain terminal stage (decision 2).
- **WellSky anything** — v1.2.0.
- **Timeline rendering improvements** — v1.1.3.
- **Renaming `new`/`contacted`/`converted` keys or labels to WelcomeHome's names** — explicitly declined (decision 1).
- **Re-sweeping Activities** — the corrective sweep resets the `Prospects` cursor only; activities are already imported and deduped.

## Key design decisions (locked in — do not re-litigate during build)

1. **The seven-stage funnel, in order** (justification: one-to-one with WelcomeHome while keeping every already-correct key stable):

   | Nexus key (`leads.status`) | Label | WelcomeHome stage | Terminal |
   |---|---|---|---|
   | `new` | New | Inquiry (`new_lead`, pos 0) | no |
   | `contact_attempted` | Contact Attempted | Contact Attempted (pos 1) | no |
   | `contacted` | Contacted | Contact Made (pos 2) | no |
   | `visit_scheduled` | Visit Scheduled | Home Visit Scheduled (pos 3) | no |
   | `visit_completed` | Visit Completed | Home Visit Completed (`visit`, pos 4) | no |
   | `converted` | Converted | Start of Care (`move_in`, pos 5) | yes |
   | `lost` | Lost | closed / discarded | yes |

2. **Migration remaps `qualified` → `visit_scheduled`** (the conservative half of the old bucket — we can't know locally whether the visit happened; the corrective sweep then sets WH-sourced leads precisely). Existing `contacted` rows are left alone for the same reason (they may be Attempted or Made); the sweep corrects them. New entity-seam migration file `supabase/migrations/20260801000000_entities_lead_stages.sql`, in the additive/idempotent house pattern:

   ```sql
   alter table public.leads drop constraint if exists leads_status_check;
   update public.leads set status = 'visit_scheduled' where status = 'qualified';
   alter table public.leads add constraint leads_status_check
     check (status in ('new','contact_attempted','contacted','visit_scheduled',
                       'visit_completed','converted','lost'));
   ```

   Drop → remap → re-add is safely re-runnable. Core tables untouched — this is the re-templating seam.

3. **`wh_map` becomes exactly one-to-one** (justification: that is the version): `_SYSTEM_TYPE_STATUS = {"new_lead": "new", "visit": "visit_completed", "move_in": "converted"}`; `_POSITION_BANDS = ((0, "new"), (1, "contact_attempted"), (2, "contacted"), (3, "visit_scheduled"), (4, "visit_completed"))`, position > 4 → `converted`. The renamed-stage resilience design (system_type first, bands as fallback, unknown → `None`/leave-alone) is unchanged. Update the module docstring's mapping table.

4. **`update_lead` gains no-op patch suppression** (justification: the corrective sweep re-sends ~90 unchanged prospects; without this, each logs a spurious `lead.updated` timeline event — and every future full re-sweep, including WellSky's, benefits). Fetch the current values of `LEAD_PATCH_FIELDS` for the lead, restrict the patch to fields whose incoming value **differs**, and skip the UPDATE + `lead.updated` event entirely when nothing differs. Status handling (`change_stage`, itself already no-op-safe) and contact syncing are unchanged.

5. **One corrective re-sweep, Prospects cursor only** (justification: the runner's watermark means untouched prospects would otherwise keep their coarse pre-v1.1.2 status indefinitely; the whole point is a settled stage set before WellSky). A build-step script clears `state["cursors"]["Prospects"]` (leaves `Activities`) in the `welcomehome` `connector_state` row and drives `sync.run_one()` once. Resulting `lead.stage_changed` events are honest corrections and stay in the timeline; decision 4 keeps `lead.updated` noise out.

6. **In-pipeline = non-terminal, derived — never a fresh hardcoded list.** `referrals._IN_PIPELINE_STATUSES` becomes the five non-terminal keys (derive from `LEAD_STAGES` or state the five explicitly with a comment pointing at the seam config); `LeadMetrics.tsx` derives its in-pipeline filter from the `LEAD_STAGES` `terminal` flags instead of naming keys. Prevents the next stage-set change from silently missing a surface.

7. **`sequenceStages` = the six worked stages** (`new`, `contact_attempted`, `contacted`, `visit_scheduled`, `visit_completed`, `converted`); `lost` stays excluded (user decision 2). The trigger convention (`new` → `lead.created`, others → `lead.stage_changed` pinned to `payload.to`) is generic and unchanged.

8. **Version/scope check:** patch is right — this tweaks the existing leads capability and the existing WelcomeHome mapping; no new subsystem, no framework change. Plan name: `wh-stage-reflection`.
