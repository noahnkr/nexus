# Plan v1.1.0: Communications tier & RAG hygiene

**Complexity: 🔴 Complex** — a new knowledge subsystem (a core table + its own retrieval path + a derived-knowledge profile) that also reworks the v1.0.0 connector transcript path. It is broken into **three ordered parts**; each stands alone as a fallback split point:

- **Part 1 (Tasks 1–3)** — the communications substrate: store, ingest policy, retrieval + tool. Delivers a working tier even if Parts 2–3 slip.
- **Part 2 (Tasks 4–5)** — migrate the WelcomeHome transcript path onto the substrate. Cleanly severable; the documents corpus becomes files-only here.
- **Part 3 (Tasks 6–8)** — the per-entity communication profile, the split history-seed, and the docs wrap-up.

The design is **strictly additive**: `documents`, `document_chunks`, and `search_documents` are **not touched** — every existing document/retrieval test must pass unmodified. This was the user-locked choice (2026-07-20): separate `communication_chunks` (not a generalized chunk table), a separate `search_communications` tool (not a unified search), and **all** WelcomeHome activity prose → communications (documents = files only).

## Context

**Why now.** This is the foundational knowledge-tier rework, and it lands *before* the messaging connectors (v1.3.0 GoTo SMS, v1.4.0 Gmail) so they build into the right substrate rather than each inventing its own message store. The PRD already defines the three-tier model (§ Knowledge & Retrieval): **Documents** (curated, durable files), **Communications** (messages/calls/emails — own store, timeline-linked always, embedded *selectively*), **Derived knowledge** (per-entity summaries + communication profiles via `entity_summaries`). v1.1.0 builds tier 2 and the comm-profile half of tier 3.

**What exists today.**
- `documents` / `document_chunks` (`core_tables.sql`): the curated RAG corpus. `document_chunks` has `embedding vector(1024)`, an HNSW index, and `entity_type`/`entity_id`/`source_system`/`metadata`. Retrieval is `retrieval.retrieve_chunks` (top-8 cosine over document_chunks), surfaced by the `search_documents` tool.
- `services/ingestion.py::ingest_text(tenant_id, title, text, *, entity_type, entity_id, source, external_id)` — the v1.0.0 (18a) entry that turns already-extracted text into a `documents` row + chunks. **Its only callers are the WelcomeHome path** (`wh_runner.after_commit`, `scripts/backfill_welcomehome.py`) and `test_ingest_text.py`. It becomes dead code after this version.
- WelcomeHome narrative activities (`Call`/`Email`/`Note`/`Assessment`/`Other`, and separately `Text`) currently route via `wh_map.is_narrative` + `map_activity(...)["narrative"]`: `≥ 500` chars → `ingest_text` → a `documents` row tagged to the lead; everything else is a `lead.activity_logged` timeline event only. `Text` (SMS) is timeline-only regardless of length. **This is the "v1.0.0 transcript path" that moves.**
- `entity_summaries` (`entity_summaries.sql`): PK `(tenant_id, entity_type, entity_id)` — **one summary row per entity**. `services/views/summary.py` reads/writes it (the on-demand smart-summary cache) via `get_or_generate_entity_summary` / `regenerate_entity_summary`.
- `ingest_payload` (`services/connectors/ingest.py`) writes the timeline event inside `route_normalized_event`; it returns per-outcome **counts**, not the event id. `log_event` returns the row id.

**What the user locked in (2026-07-20).**
1. **Separate `communication_chunks` table + its own HNSW index.** `documents`/`document_chunks` are untouched; retrieval merges two top-k searches by distance. The discriminator is the table (result tagged `kind='communication'`) plus a `source` column for provenance.
2. **Separate `search_communications` tool.** `search_documents` stays document-only (the curated corpus stays pristine, per the PRD "never pollute" principle); the agent picks the tool by whether the question is about files or conversations.
3. **All WelcomeHome activity prose → communications.** `Call`/`Email`/`Text`/`Note`/`Assessment`/`Other` all become `communications` rows (store-all); the documents corpus is reserved for uploaded files and connector file attachments (e.g. v1.2.0 WellSky DocumentReference). Assessments included.

**Non-goals** (name the destination version):
- **New message/email event types** (`message.received`, `email.received`) and their connectors — **v1.3.0** (GoTo) and **v1.4.0** (Gmail). v1.1.0 reuses the existing `lead.activity_logged` spine for WelcomeHome and does **not** add event types.
- **A manual "log a communication" UI** — out of scope; `ingest_communication` serves connector/seed callers only this version. No new outbound tool, no gate involvement (the tier is inbound/read-only).
- **Re-tagging or migrating existing production transcript rows.** The WelcomeHome live backfill has **not run in production** (it is a pending operator step — see `PROGRESS.md` "Carried-over pending validations"), so there are no production `documents` transcript rows to migrate. This is a *go-forward* change plus the split-pass backfill rewrite; any dev/test `documents` rows from earlier runs are disposable.
- **A background retention/pruning job.** CLAUDE.md forbids scheduled re-ingestion. "Retention" here is a *documented policy* (below), not a cron.
- **Chat-thread (`chat_messages`) mining.** The comm profile summarizes an entity's *communications* history (calls/emails/messages), not the office↔agent chat threads.

**Retention policy (documented, not a job).** `communications` rows are **durable** — the timeline and audit trail need them, so nothing auto-prunes them. The embedding index (`communication_chunks`) is a **derived, rebuildable cache**: the `communications.embedded` flag makes (re-)embedding idempotent, and a store-only communication can be embedded later without a schema change. Store ≠ embed. At this corpus's scale (low hundreds), no volume machinery is warranted (CLAUDE.md scale discipline).

## Surface-coverage note — what changes per surface

| Surface | Effect |
|---|---|
| Chat / agent | Gains `search_communications` (safe, read-only); `search_documents` unchanged. Documents corpus no longer contains WelcomeHome transcripts. |
| Lead / client profile | Gains an on-demand **Communication profile** card (tone/responsiveness/topics), cached under `entity_summaries` `kind='comm_profile'` alongside the existing smart summary. |
| Lead timeline / Event Log | Unchanged — WelcomeHome activities still emit `lead.activity_logged`; the communication row links to that event (`source_event_id`). No new event types. |
| Documents (Knowledge) view | Now files-only. WelcomeHome transcripts no longer appear as documents. |
| Connectors (v1.3/v1.4) | Plug into `ingest_communication` instead of inventing a store. |
| Automations | No change — no new event types, no `automations/entities.py` edits. |

## Key design decisions (locked in — do not re-litigate during build)

1. **Core migration `20260731000000_core_communications.sql`** (core, business-agnostic — like `documents`, it carries no vertical concept). Two new core tables + one additive `entity_summaries` change. Standard guarded 4-policy RLS (`enable` + `force`) + explicit grants to `authenticated, anon`, copied verbatim from the `documents` pattern in `core_rls.sql`. Idempotent (`if not exists`, guarded PK swap). **No Realtime publication** — the timeline is events-driven, so communications need no live channel (the `lead_contacts` precedent, 18a).

   ```sql
   -- communications: the message/call/email store. Timeline-linked always,
   -- embedded selectively. entity_type/entity_id reference the canonical entity
   -- map (like events.entity_type) — core never interprets the values.
   create table if not exists public.communications (
     id              uuid primary key default gen_random_uuid(),
     tenant_id       uuid not null references public.tenants(id),
     channel         text not null check (channel in ('call','email','sms','note','other')),
     direction       text check (direction in ('inbound','outbound')),      -- null = unknown
     occurred_at     timestamptz not null,                                   -- source time, not ingest time
     subject         text,
     body            text not null,
     entity_type     text,
     entity_id       uuid,
     source          text not null,                                          -- 'welcomehome' | 'goto' | 'gmail' | 'manual' | ...
     external_id     text,                                                   -- connector id, for idempotency
     content_hash    text,                                                   -- cross-source dedup key
     source_event_id uuid references public.events(id),                     -- event-as-spine link
     embedded        boolean not null default false,                        -- store ≠ embed
     metadata        jsonb not null default '{}',
     created_at      timestamptz not null default now(),
     updated_at      timestamptz not null default now()
   );
   create unique index if not exists communications_source_external_idx
     on public.communications (tenant_id, source, external_id)
     where external_id is not null;                                          -- idempotent re-sync
   create index if not exists communications_entity_idx
     on public.communications (tenant_id, entity_type, entity_id);
   create index if not exists communications_occurred_idx
     on public.communications (tenant_id, occurred_at desc);
   create index if not exists communications_hash_idx
     on public.communications (tenant_id, content_hash);
   create trigger communications_set_updated_at
     before update on public.communications
     for each row execute function app.set_updated_at();

   -- communication_chunks: the SELECTIVELY-embedded retrieval unit. Mirrors
   -- document_chunks; its own HNSW index. Only long-form comms get rows here.
   create table if not exists public.communication_chunks (
     id               uuid primary key default gen_random_uuid(),
     tenant_id        uuid not null references public.tenants(id),
     communication_id uuid not null references public.communications(id) on delete cascade,
     chunk_index      int not null,
     chunk_text       text not null,
     embedding        vector(1024),
     entity_type      text,
     entity_id        uuid,
     source           text,
     metadata         jsonb not null default '{}',
     created_at       timestamptz not null default now(),
     unique (communication_id, chunk_index)
   );
   create index if not exists communication_chunks_tenant_idx
     on public.communication_chunks (tenant_id);
   create index if not exists communication_chunks_entity_idx
     on public.communication_chunks (tenant_id, entity_type, entity_id);
   create index if not exists communication_chunks_embedding_idx
     on public.communication_chunks using hnsw (embedding vector_cosine_ops);

   -- entity_summaries: add a `kind` discriminator so a comm profile coexists with
   -- the smart summary. Existing rows default to 'smart_summary'. Guarded PK swap.
   alter table public.entity_summaries
     add column if not exists kind text not null default 'smart_summary';
   do $$
   begin
     if exists (select 1 from pg_constraint where conname = 'entity_summaries_pkey') then
       alter table public.entity_summaries drop constraint entity_summaries_pkey;
     end if;
     if not exists (select 1 from pg_constraint where conname = 'entity_summaries_pkey') then
       alter table public.entity_summaries
         add constraint entity_summaries_pkey
         primary key (tenant_id, entity_type, entity_id, kind);
     end if;
   end $$;
   ```
   *Justification:* additive and isolated — the documents corpus is untouched, and `entity_summaries` gains a discriminator rather than a parallel table, exactly as the seam is meant to grow (PRD tier 3).

2. **`services/communications.py`** — the new core service. Business-agnostic; no vertical concepts.
   - `should_embed(channel: str, body: str) -> bool` — the **embed-selectively policy**. Default: `len(body.strip()) >= EMBED_MIN_CHARS` (module constant `EMBED_MIN_CHARS = 500`, carrying over the empirically-set WelcomeHome narrative threshold) **and** `channel != 'sms'` (short-message channels stay store-only regardless — an SMS is a message, not a retrievable document). Pure; unit-testable.
   - `content_hash(channel, direction, occurred_at, body) -> str` — `sha256` over the normalized tuple, for cross-source dedup.
   - `ingest_communication(tenant_id, *, channel, direction, occurred_at, body, entity_type=None, entity_id=None, subject=None, source, external_id=None, source_event_id=None, embed=None) -> str | None` — **store-all, embed-selectively**:
     1. `body` blank → return `None` (nothing to store).
     2. **Idempotency:** if `external_id` given and a row exists for `(source, external_id)` → return its id, no writes.
     3. **Cross-source dedup:** else if a row for the same `entity` with the same `content_hash` exists from a **different** `source` → return that id (deduped), no new row.
     4. Insert the `communications` row (store-all), capturing `source_event_id`.
     5. `embed` (defaulting to `should_embed(channel, body)`) → chunk `body` (`chunking.chunk_text`), insert `communication_chunks` (stamped with the parent's `entity_type`/`entity_id`/`source`), `embed_documents` the chunk texts, write vectors back, set `communications.embedded = true`. Embedding round-trips happen **outside** any held connector transaction (the `ingest_text` discipline — the caller invokes this in `after_commit`).
     - Does **not** emit its own timeline event — the caller owns the spine event and passes `source_event_id`. Raises on failure (the caller logs + skips; a bad comm must not cost a sync its cursor — the `ingest_text` contract).
   *Justification:* one seam for every message source; the store/embed split lives here so connectors stay dumb.

3. **Retrieval — `services/retrieval.py` gains `retrieve_communications`** (mirror of `retrieve_chunks`, over `communication_chunks` joined to `communications` for the citation label). Returns `{chunk_id, communication_id, chunk_index, chunk_text, channel, occurred_at, source, entity_type, entity_id}`. Same `TOP_K=8` clamp, same `embedding is not null` filter, same tenant-by-RLS path. **`retrieve_chunks` is not modified.** No unified merge in v1.1.0 (separate tool per decision 2); a future unified search can merge the two if wanted.

4. **Tool — `services/tools/communications.py::search_communications`** (registered via `register`, `safe=True`, read-only). Mirrors `search_documents`: `query` + `top_k` (1–8) + injected `start_index`. Sources carry `kind='communication'`, a plain label (`f"{Channel} · {occurred_at:%b %-d, %Y}"`, e.g. "Call · Jul 3, 2026"), `communication_id`, snippet, and `source`. Description steers the agent: *"Search past communications (calls, emails, messages, notes) by semantic similarity … Use for questions about what was said or discussed with someone, as opposed to `search_documents` for uploaded files."* Auto-exposed to chat and MCP by the registry (no MCP-specific branch — CLAUDE.md). `search_documents` unchanged.

5. **WelcomeHome mapping — `wh_map.py`.** Replace the embed-or-not `narrative` flag with a **communication payload for every message/interaction activity** (store-all). `map_activity` returns, alongside the existing timeline fields, a `communication` dict when the type carries prose/interaction:
   - **Channel map** (in-seam constant): `Call → 'call'`, `Email → 'email'`, `Text → 'sms'`, `Note → 'note'`, `Assessment → 'other'`, `Other → 'other'`. Types not in the map (Home Visit, Appointment, …) get **no** communication (timeline event only, as today). `SYSTEM_ACTIVITY_TYPES` still skipped wholesale.
   - The `communication` dict: `{channel, direction, occurred_at, body: notes, external_id: "wh:activity:{id}", subject: None}`. The **embed decision leaves the map** — `should_embed` (length) now decides embedding, so the old `NARRATIVE_MIN_CHARS`/`is_narrative` gate is deleted (the 500-char threshold lives once, in `communications.EMBED_MIN_CHARS`). A `Text`/`sms` of any length is stored, never embedded (decision 2).
   *Justification:* store-all means the map stops gatekeeping storage; it only classifies channel. The threshold is a policy, in the policy module.

6. **WelcomeHome runner + backfill — event-as-spine linkage.** `wh_runner._sync_activities` collects `pending_communications` (replacing `pending` transcripts): `(channel, direction, occurred_at, body, lead_id, external_id)`, gated on `mapped.get("communication") and result.get("matched")` and a resolved `lead_id` (same discipline as today — an activity that resolved to a review task has no lead to tag). `after_commit` then, per pending comm: **resolve `source_event_id`** by looking up the activity's timeline event —
   ```sql
   select id from public.events
    where entity_type = 'lead' and entity_id = %s
      and event_type = 'lead.activity_logged'
      and payload->'detail'->>'wh_activity_id' = %s
    order by created_at desc limit 1
   ```
   — then call `ingest_communication(..., source_event_id=<id or None>)`. Remove the `ingest_text` import/call. `scripts/backfill_welcomehome.py` swaps the same way (see decision 8 for its pass structure). **Delete `services/ingestion.py::ingest_text` and `test_ingest_text.py`** — dead after this migration (verified: only these callers). `process_document` (file uploads) is untouched.
   *Justification:* the event stays the spine; the comm carries a real FK to it. `wh_activity_id` is already on the event's `detail`, so the lookup is cheap and needs no plumbing changes to `ingest_payload`.

7. **Per-entity communication profile — `entity_summaries` `kind='comm_profile'`, via `summary.py`.** Generalize the cache helpers in `services/views/summary.py` to take `kind: str = 'smart_summary'` (threaded into the insert columns and the `on conflict (tenant_id, entity_type, entity_id, kind)` target) — the existing smart-summary callers pass nothing and keep working. Add a **comm-profile builder** that assembles the entity's recent `communications` (channel/direction/occurred_at/body-snippet, oldest first, capped at ~20) into a block and generates prose with a comm-specific `prompt_intro` (*"Describe how this person communicates — tone, responsiveness, preferred channel, recurring topics — for a coordinator about to reach out. Base it only on the messages provided."*), `span_name='comm_profile'`, reusing `generate_entity_summary`'s model call and `SummaryUnavailable` (503-without-key) shape. New endpoints mirror the smart-summary routes: `GET /api/leads/{id}/comm-profile` (+ `POST …/regenerate`) and the same for clients, delegating to `get_or_generate` / `regenerate` with `kind='comm_profile'`. Frontend: a **Communication profile** card on `LeadProfilePage`/`ClientProfilePage` reusing the existing smart-summary card component (generate-on-open, Regenerate button, quiet 503 notice) — additive, no new component.
   *Justification:* tone/style is a summary problem, not a retrieval one (PRD tier 3); the seam already exists, so this is a discriminator + a second prompt, not new machinery.

8. **Split history-seed — `scripts/backfill_welcomehome.py` activity phase in three passes.** A large historical import must not embed one comm at a time inline. Restructure the activity backfill into: **(1) structured pass** — write every communication row via `ingest_communication(..., embed=False)` (store-all, fast, no embedding round-trips); **(2) batched embed pass** — for the rows `should_embed` selects, chunk + `embed_documents` in batches and write `communication_chunks` + set `embedded=true`; **(3) summary pass** — generate the comm profile per affected entity (`kind='comm_profile'`). Each pass is idempotent and resumable via the backfill's existing `connector_state` cursor key; a re-run finds existing rows/embeddings and skips. Prints per-pass counts, never record contents (the 18a backfill discipline).
   *Justification:* the three passes are the efficient shape for bulk history and the reusable template every future connector seed follows.
