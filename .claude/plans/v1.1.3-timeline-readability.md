# Plan v1.1.3: Entity timeline readability

**Complexity: ⚠️ Medium** — frontend-heavy readability work (a shared detail renderer, row upgrades on two surfaces, two pure utilities) plus one small backend write-side fix. No schema, no seam, no framework change, and every logic piece is a pure unit-testable function — but visual work invites iteration, so budget a browser-driven polish loop. Single-pass feasible; the natural fallback split is after Task 3 (utilities + shared renderer landed) with the row upgrades and polish as a second pass.

## Context

**Why now.** Lead timelines are now the office's read on what WelcomeHome has been doing (864 of the ~1,100 lead-timeline events are synced CRM activities), and they are hard to read: every row is a one-line summary with a raw-JSON expander. Two concrete defects ship with this pass: **email bodies render raw HTML tags** in the summary line and detail, and **long timeline entries get cut off** — the write path truncates summaries at ~120 chars even though the full text is stored in the event detail.

**Payload analysis (dev DB, 2026-07-21 — the analysis the roadmap called for, done at plan time).**
- Lead-timeline event types by volume: `lead.activity_logged` 864 · `lead.updated` 183 · `automation.run_started/completed` 121 · `lead.created` 36 · `lead.stage_changed` 22. Client/resource timelines add `credential.added/updated/removed`, `resource.status_changed`, `resource.updated`, `client.updated/created`, `call.completed`.
- `lead.activity_logged` payload shape is stable: `{summary, resolution, external_id, detail: {notes, direction, activity_type, completed_at, wh_activity_id}}`.
- Activity types: **Email 431** (notes avg 203 / max 1,578 chars; **332 contain HTML tags**) · **Call 264** (avg 379 / max 3,045) · **Text 120** · **Note 71** (avg 1,292 / max 4,547) · **Assessment 19** · **Other 6** · **"Thank You Note" 2** (an unmapped type with `null` notes — unknown types and missing fields must never crash the renderer).
- **398 of 913** stored activity summaries sit at the 120-char truncation cap — the "cut off" defect. The full text is always in `detail.notes`.
- Stored summaries with HTML look like: `"Email (outbound): <b>Come See Us at the DuPage County Fair This Week!</b><br><br>Hi! …"` — the HTML defect, baked in at write time.
- Other recurring payload shapes: `lead.stage_changed {summary, from, to}` · `lead.updated {summary, fields[]}` or `{summary, detail, external_id, resolution}` · `automation.run_* {summary, automation_name, automation_id, run_id}`.

**What exists.**
- `frontend/src/components/events/EntityTimeline.tsx` — the shared profile timeline (leads, clients, caregivers use it verbatim). Rows: chevron, one-line `event.summary`, source badge, mono `event_type`, relative time; expanded = raw `JSON.stringify(payload)` in a `<pre>`.
- `frontend/src/components/events/EventRow.tsx` — the Event Log row (v0.15): already has `eventIcon()` per type-prefix, `sourceAccent()` bar, `eventTypeLabel`/`sourceLabel`; expanded = the same raw-JSON `<pre>`.
- `frontend/src/lib/events.ts` — `fallbackSummary`, `SOURCE_ACCENT`, `ICON_BY_PREFIX` (note: **no `call` or `credential` prefix entries** — those events currently fall through to the `CircleAlert` fallback icon), `matchesFilters`.
- `backend/app/services/connectors/wh_map.py::activity_summary` — builds the stored one-liner: first line of notes, truncated to 120 chars, **no HTML handling**.
- `backend/app/services/event_summaries.py` — read-time summary derivation; `payload.summary` always wins (writer convention). Stored events are immutable — **never backfilled or mutated** (CLAUDE.md), so the 864 existing summaries keep their HTML/truncation forever; any fix for them must be at display time.
- Frontend tests are vitest unit tests over pure `src/lib/*.ts` functions (`clients.test.ts`, `recipe.test.ts`, …) — no component-testing harness. Backend: `tests/test_wh_map.py` covers `activity_summary`.

**What the user locked in (2026-07-21).**
1. **Email/HTML bodies convert to plain text** — a small dependency-free HTML-to-text utility (tags stripped, `<br>`/`<p>`/`<div>` become line breaks, entities decoded). No sanitized-rich-HTML rendering.
2. **Flat list, icons + accents** — per-row icons (per-activity-type for WelcomeHome activities), the Event Log's source-accent bar, multi-line clamped previews with expand. **No day-group headers.**

**Non-goals.**
- **Day-group date headers** — explicitly declined (decision 2).
- **Rendering sanitized rich HTML** — explicitly declined (decision 1); no DOMPurify dependency.
- **Backfilling or rewriting stored event summaries** — events are immutable (CLAUDE.md); the read path fixes display for existing rows, the write path fixes future rows.
- **Normalizing HTML in the communications tier** (`communications.body` stays verbatim as ingested) — channel/shape normalization is v1.5.0's canonical-action work.
- **Smart-summary / comm-profile card changes** — v1.1.4.
- **New backend endpoints or event schema changes** — none needed; everything renders from the payload the API already returns.

## Key design decisions (locked in — do not re-litigate during build)

1. **`htmlToText()` in a new `frontend/src/lib/text.ts`** (justification: both defects and the renderer need one shared, unit-testable conversion; no dependency). Behavior: replace `<br>`, `</p>`, `</div>`, `</li>`, `</tr>`, `</h1-6>` with `\n`; strip all remaining tags; decode the common entities (`&amp;` `&lt;` `&gt;` `&quot;` `&#39;`/`&apos;` `&nbsp;` plus numeric `&#NNN;`); collapse 3+ consecutive newlines to 2; trim. Idempotent on plain text (a string with no tags/entities passes through unchanged), safe on `null`/`undefined` (returns `""`).

2. **`eventDisplay(ev)` in `lib/events.ts`** — the read-time derivation that fixes both defects for the 864 existing events (justification: stored summaries are immutable; display must derive from `detail`, which always holds the full text). Returns `{title, body}`:
   - For `*.activity_logged` with a `detail.activity_type`: `title` is built client-side — `"Email (outbound)"` style from `activity_type` + `direction` (never the stored summary, killing its HTML and truncation in one move); `body` = `htmlToText(detail.notes)`, full length, or `null` when notes are absent.
   - For every other type: `title` = `fallbackSummary(ev)` (the server summary), `body` = `null`.
   - Never throws on missing/odd payload shapes ("Thank You Note" has `null` notes).

3. **Per-activity-type icons + prefix-map gaps filled** (justification: the icon is the "what happened" key; CRM activities are the bulk of the timeline). `eventIcon(eventType, payload?)` gains an optional payload: for `*.activity_logged`, pick by `detail.activity_type` — Email → `Mail`, Call → `Phone`, Text → `MessageSquare`, Note → `StickyNote`, Assessment → `ClipboardCheck`, anything else → `FileText`. Existing single-arg callers stay valid. Also add missing prefixes to `ICON_BY_PREFIX`: `call: Phone`, `credential: ShieldCheck`, `communication: MessagesSquare`, `email: Mail`, `sms: MessageSquare` — client/caregiver timeline events stop falling through to the alert icon.

4. **A shared `EventDetail` component** (`components/events/EventDetail.tsx`) replaces both raw-JSON expanders (justification: "best-effort renderer for the variable JSON detail", written once, both surfaces). Best-effort, crash-proof rendering of `payload`:
   - **Long-text block**: the derived `body` (decision 2) — or, generically, any of `detail.notes` / `error` string over ~80 chars — rendered `whitespace-pre-wrap break-words` (no horizontal cut-off, ever).
   - **Known-key grid**: label/value rows for the recurring scalar keys the analysis found — `direction` (Direction), `completed_at`/`occurred_at` (Occurred, formatted date), `from`→`to` (Stage, rendered "From → To"), `fields` (Fields changed, joined), `automation_name` (Automation), `resolution` (Resolution). Unknown scalars under `detail` render as humanized-key/value rows.
   - **Technical detail**: everything else (ids, nested objects, the raw payload) behind a small collapsed "Technical detail" toggle showing the raw-JSON `<pre>` exactly as today — the Event Log keeps its raw access (CLAUDE.md puts technical detail there), profile timelines keep it one click further away.

5. **`TimelineRow` (EntityTimeline) upgrade — flat list** (justification: user decision 2): per-row icon (decision 3) + `sourceAccent` bar (reused from `lib/events.ts`), title from `eventDisplay`, and — when a `body` exists — a **3-line clamped preview** (`line-clamp-3`) of the body; the chevron expands to the full `EventDetail`. Meta line (source badge, mono type, relative time) stays. Layout stays compact (profile pages are dense).

6. **`EventRow` (Event Log) keeps its row, swaps its expander** (justification: the row already has icons/accents/labels from v0.15; the defect there is the expanded raw JSON): expanded state renders `EventDetail` (structured first, raw JSON under the toggle). No other Event Log changes.

7. **Write-side fix in `wh_map.activity_summary`** (justification: future stored summaries should be clean everywhere they surface — chat, tasks, traces — not just where the frontend re-derives): a small private `_html_to_text()` (regex tag-strip + entity decode + `<br>`→space, mirroring decision 1's semantics in miniature) applied to notes before the first-line/120-char logic. The truncation cap itself stays — the stored summary is a one-liner by design; the full text lives in `detail.notes` and the UI now shows it.

8. **No backend read-path or API changes** — `event_summaries.py`, routers, and schemas untouched. Version fits **patch**: a readability/defect pass on existing surfaces.
