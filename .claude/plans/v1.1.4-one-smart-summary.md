# Plan v1.1.4: One smart summary per entity

**Complexity: ✅ Simple** — a seam-local merge (`services/views/summary.py`), four endpoint removals, a two-card-to-one frontend cleanup, and test rework. No schema change, no new subsystem, no framework surface. Single-pass executable; the only iteration risk is prompt wording, which is deliberately locked below.

## Context

**Why now.** Every entity profile today shows two AI cards: the smart summary (record + recent activity) and, on leads and clients, a separate communication profile (tone/responsiveness/channel from the comms tier). Two cards mean two caches, two Regenerate buttons, and a split "at a glance" read. v1.1.4 merges them: **one summary per entity** covering the record, activity, communication history, and significant facts. It is sequenced before the connector versions deliberately — it depends on the comms *tier* (v1.1.0), not on which connector fills it; coverage thickens automatically as v1.2.0/v1.3.0 land.

**What exists (verified against code and the dev DB, 2026-07-21).**
- `services/views/summary.py` — the seam, two halves:
  - *Generic smart summary*: `generate_entity_summary` (record block + last 20 events via `summarize_event`, one `fast_model` call, `@traceable` chain span), `get_or_generate_…`/`regenerate_…` over the `entity_summaries` cache (`_read_cache`/`_write_cache`, `kind` param defaulting `"smart_summary"`).
  - *Comm profile (v1.1.0)*: `COMM_PROFILE_KIND`, `_recent_communications` (last 30 comms via plain SQL, bodies trimmed to 300 chars), `generate_comm_profile` (placeholder without a model call when no comms), `get_or_generate_comm_profile`, `regenerate_comm_profile`.
- Three verticals call the generic half with their own intro: `LEAD_SUMMARY_INTRO` (`views/leads.py`), `CLIENT_SUMMARY_INTRO` (`views/clients.py`), `APPLICANT_SUMMARY_INTRO` (`views/caregivers.py`). **Caregivers/applicants have no comm-profile endpoints** — they gain comms coverage automatically once the generic generator includes it.
- Comm-profile endpoints: `GET/POST /api/leads/{id}/comm-profile[/regenerate]` (`routers/leads.py:357-395`) and the same pair on `routers/clients.py`. Both reuse the summary response models — no dedicated schema to remove.
- Frontend: `SmartSummary.tsx` is already generic (label/icon/closure props). `LeadProfilePage.tsx:122-129` and `ClientProfilePage.tsx` (same pattern) each render a second `<SmartSummary label="Communication profile" …>` card over four `api.ts` functions (`get/regenerateLeadCommProfile`, `get/regenerateClientCommProfile`).
- `app/scripts/backfill_welcomehome.py` pass 3 calls `regenerate_comm_profile` for each touched lead.
- Tests: `test_comm_profile.py` (the comm-profile half's unit coverage), `test_communications.py:80` and `test_backfill_seed.py:59` (cleanup/assertions over both `kind`s).
- Dev DB: `entity_summaries` holds 29 `smart_summary` + 20 `comm_profile` rows; `communications` holds 277 rows across 67 entities — enough real data for a meaningful merged summary on WH-active leads.
- CLAUDE.md and PRD both describe the two-kind arrangement ("one entity holds both a `smart_summary` and a `comm_profile`") — doc sync required.

**What the roadmap locked in.** One "at a glance" section per entity; leads and clients lose the second card and its `comm-profile` endpoints; caregivers gain comms coverage in the summary they already have; the summary draws on correspondence **through the communications tier directly, not a second cached artifact**; seam-local — the `entity_summaries.kind` **column stays**. Retires the two-cache half of the *Summary freshness* backlog item (one cache, one Regenerate).

**Non-goals.**
- **Summary freshness/invalidation machinery** — the single cache still refreshes only on manual Regenerate; the remaining backlog item stays in the backlog.
- **Embedding/reranked retrieval for the summary's comms context** — recency SQL over `communications` is the mechanism (scale discipline: no retrieval complexity for a per-entity recap of ≤30 messages). `search_communications` (chat RAG) is untouched.
- **Dropping the `entity_summaries.kind` column or any migration** — the schema stays; only cache *rows* of the retired kind are cleaned up (rebuildable data, not a migration).
- **Which sources fill the comms tier** — v1.2.0/v1.3.0/v1.3.1's business; thin coverage now is expected and fine (no-comms entities get a plain "(no correspondence…)" block, not a degraded summary).

## Key design decisions (locked in — do not re-litigate during build)

1. **The comms block joins the generic generator** (justification: this is what makes every vertical — including caregivers — gain coverage with zero per-vertical code). `_recent_communications` moves up into the generic half of `summary.py`, unchanged in behavior (last 30, recency SQL, 300-char snippets, oldest-first). `generate_entity_summary`'s `user_content` gains a third section:

   ```
   Record:
   …
   Recent activity (oldest first):
   …
   Recent correspondence (oldest first):
   <comms lines>            — or —          (no correspondence on record yet)
   ```

2. **The comm-profile's angle folds into the system guidance, lightly** (justification: tone/responsiveness was the second card's value; it belongs in the one summary when the data supports it). The generic system prompt (after the vertical intro) becomes: *"Write 3-5 short sentences of plain prose for a busy office coordinator. Cover who they are, what has happened, and — when the correspondence shows it — how they communicate (tone, preferred channel, responsiveness). No preamble, no bullet points, no headings. Base it only on the facts provided; do not invent details."* `_MAX_TOKENS` stays 400. Vertical intros in `views/leads.py` / `views/clients.py` / `views/caregivers.py` each get one added sentence naming communication history as in-scope for their entity kind; their existing voice and span names stay.

3. **The comm-profile half of `summary.py` is deleted** — `COMM_PROFILE_KIND`, `COMM_PROFILE_INTRO`, `generate_comm_profile`, `get_or_generate_comm_profile`, `regenerate_comm_profile` (justification: "not a second cached artifact" is the version). The cache helpers keep their `kind` parameter and `"smart_summary"` default — the column stays, and a future derived kind plugs back in without seam surgery.

4. **Four endpoints removed** — both `comm-profile` pairs on `routers/leads.py` and `routers/clients.py` (justification: the roadmap names them). No schema removals (they reused the summary response models). The summary endpoints themselves are untouched.

5. **Backfill pass 3 becomes cache invalidation** (justification: the pass existed to pre-build derived knowledge; with one lazy cache, the right backfill behavior is "touched leads regenerate on next open" at zero API cost during the backfill): `backfill_welcomehome.py` deletes touched leads' `smart_summary` rows instead of calling the removed `regenerate_comm_profile`. Its progress line and docstring update to match.

6. **One-time cache hygiene in the build, by script not migration** (justification: `entity_summaries` is a rebuildable cache; deleting rows is ops, not schema): delete all `kind='comm_profile'` rows (20) **and** all `kind='smart_summary'` rows (29), so every profile's first open after this version generates the merged summary rather than serving a pre-merge cached one indefinitely. ~30 lazy fast-model regenerations is negligible cost.

7. **Frontend: remove the second card, keep `SmartSummary` generic** (justification: the component's closure-prop design was built for reuse; nothing about it changes): drop the second `<SmartSummary>` from `LeadProfilePage.tsx` and `ClientProfilePage.tsx` (and the now-unused `MessageSquare` import), remove the four comm-profile functions from `api.ts`.

8. **Version fits patch** — a refinement of the existing summary capability; nothing new stands up. Plan name: `one-smart-summary`.
