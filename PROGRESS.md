Plan: `.claude/plans/v1.3.0-google-workspace.md`. **The authoritative source for email.** Lead intake stays WelcomeHome's job — Gmail never creates leads. **Ops step (NOT DONE — blocks going live): GCP OAuth client + consent → `GOOGLE_*` in `.env`** (self-service).
- `[x]` Task 1 — Google OAuth bootstrap + `google_client.py`. `tests/test_google_client.py` **16 passed, 1 skipped** (the live profile check, correctly gated on credentials). Reuses v1.2.0's `TokenSource` verbatim — one implementation, two configs. No `google-api-python-client`: six REST endpoints do not justify that dependency tree. Scopes are least-privilege (`gmail.readonly`, not `.modify` — this mirrors the mailbox, it never changes it), asserted by a test so widening one is a deliberate act. **Two Google-specific traps encoded:** the consent URL needs `access_type=offline` (no refresh token at all without it) *and* `prompt=consent` (Google issues a refresh token only on the first consent per client+account, so a re-run would otherwise silently produce none); and Google wants client credentials in the form body where GoTo wants HTTP Basic.
- `[x]` Task 2 — Gmail poll runner + mapper. `tests/test_gm_map.py` **31 passed**, `tests/test_gmail_runner.py` **11 passed**. `gm_map` parses what real mail actually looks like — the body is a recursive MIME tree, so `multipart/alternative` prefers plain text and falls back to stripped HTML (v1.1.3's lesson applied at the source), and **base64URL not base64**, which would otherwise silently fail on most messages. No backfill: a first run adopts the mailbox's position and imports nothing. **The cursor advances past a message that cannot be fetched** — the opposite of the WelcomeHome runner's overlapping watermarks, and deliberate: Gmail's history ids are exact, so there is no window to re-cover, and a poison message would otherwise wedge every email behind it forever.
- `[x]` Task 3 — Real `send_email`. `tests/test_gmail_send.py` **11 passed**. Gate contract asserted untouched. Built through `EmailMessage` rather than string concatenation so a subject with an accent in it is not what breaks outbound mail.
- `[x]` Task 4 — Calendar poll runner. **Reopened and fixed 2026-07-22** after the live run proved it could not sync a real calendar; now `tests/test_gcal_runner.py` **15 passed** and verified against the real calendar (token captured, incremental sync begins). Fix detail under _Carried-over pending validations_ ①. `syncToken` and `timeMin` are mutually exclusive (sending both is a 400); `nextSyncToken` appears only on the LAST page, so the runner pages to exhaustion — stopping early would drop the token and force a full re-window every cycle. A 410 re-windows rather than stalling; re-delivered events are idempotent by external id. Cancelled events are ingested, not dropped — a cancelled visit is exactly what someone needs to see. **The bug is that "pages to exhaustion" is capped at 10 pages and this calendar needs more** — see _Carried-over pending validations_.
- `[x]` Task 5 — Calendar tools. `tests/test_calendar_tools.py` **13 passed**. Safe `list_calendar_events`, gated `create_calendar_event` (an event is visible outside the system the moment it exists and emails an invitation). `editable_fields=["title","description"]` — an approver may reword, but moving the time or changing who is invited would change *what* was approved. New event types declared in the automations vocabulary.

## Carried-over pending validations

### First live Google/GoTo run — 2026-07-22

Four defects, all invisible to the test suite, all found within minutes of pointing the app at the real account. **All four are now fixed** — the Calendar `syncToken`, GoTo call subscriptions, GoTo SMS subscriptions, and the Windows event-loop policy (now set in `backend/run.py`, the entrypoint the README documents). Their detail lives in the commit messages and code docstrings rather than here. What the run left open:

**⚠️ The call frame shape is the biggest open risk on v1.2.0, and the subscription fix widened it.** The bridge and `gt_map` were built around a `REPORT_SUMMARY` payload; every fixture in `test_gt_map.py` / `test_goto_bridge.py` / `test_goto_resolution.py` is that shape, and the call-history records they came from are real but are *not* notification frames. Since the 2026-07-22 fix, calls arrive as **`ENDING`** frames instead. `frame_kind` already classifies `ENDING` as a call and `gt_map` scans structurally for numbers rather than trusting field names, so the expected worst case is a degraded "no counterpart found → ack-only receipt" rather than wrong data on someone's record — but **nothing has confirmed an `ENDING` payload carries the participants, direction and duration `map_call` reads.** The first live call settles it, and the fixture should be replaced with a captured frame then. This is the single highest-value thing left on v1.2.0.

**Left behind by the run, and it breaks two tests:** the live run wrote `gmail` and `gcal` rows into `connector_state` for the demo tenant, and `test_connectors_resolution::test_connector_state_unique_per_source` / `::test_connector_state_tenant_isolation` both insert a `'gmail'` row expecting the *first* insert to succeed. They now fail on a `UniqueViolation` that has nothing to do with the code. Deleting the two rows fixes it (a fresh run re-adopts the cursor with no backfill, so nothing is lost). **The deeper point: those tests will fail on any deployment where the Gmail connector has actually run** — they should use a fixture-owned source name or clean up after themselves rather than assuming a bare table.
