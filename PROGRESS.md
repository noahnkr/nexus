### First live Google/GoTo run — 2026-07-22 (four defects, none fixed yet)

All four were invisible to the test suite and appeared within minutes of pointing the app at the real account. Recorded here with enough detail to fix without re-deriving. **① is fixed and live-verified; ②–④ are still open and need routing to versions.**

- **① FIXED 2026-07-22 — v1.3.0 Calendar runner could never store a `syncToken` on a real calendar, so there was no incremental sync, ever.** *Fix: the window is now bounded forward as well as back (`timeMax = now + 90 days`), the page cap is 40 rather than 10, and hitting the cap with pages still pending raises `CalendarWindowTooLarge` instead of silently storing a token-less sweep. **Verified against the real calendar**: 143 events instead of 2,501, `syncToken` captured, and the incremental follow-up returned 0 changes with a fresh token — incremental sync actually begins now. Two regression tests added (`test_a_first_run_bounds_the_window_forward_as_well_as_back`, `test_truncating_at_the_page_cap_raises_instead_of_storing_no_token`) plus cap-boundary coverage; `test_gcal_runner.py` 15 passed.* Original diagnosis, kept because the failure mode is the interesting part — observed: `connector_state.gcal` = `{}` after a sweep that ingested **2,501 events** in one cycle, including recurring instances dated 2032, 2035 and 2039. Mechanism, all confirmed in code: a first run sets `timeMin = now − 7 days` with **no `timeMax`**, and `google_client.calendar_events` sends `singleEvents: True`, so every future recurrence of every series is expanded. `gcal_runner._MAX_PAGES = 10` × `maxResults = 250` = **exactly the 2,500 ceiling that was hit**. Google returns `nextSyncToken` only on the *last* page — which the runner's own comment (line 97) states correctly — so hitting the page cap means the last page is never reached, `next_token` stays `None`, and `{}` is stored. The next cycle re-windows and repeats the identical 2,500-event sweep. **Why the tests miss it:** the fixtures fit in a single page, so the cap is never reached and the token always arrives. A fix needs a `timeMax` bound (the runner never passes the parameter, though the client already supports it), a higher page cap, or both — plus a test whose fixture spans more pages than the cap.
- **② v1.2.0 GoTo call subscriptions fail on every single cycle — the phone channel is not working in production.** `POST /call-events/v1/subscriptions` → `400 BAD_REQUEST`, `"The Notification Channel must be a WebSocket channel"`, constraint on `Events[0]`. Surfaces as a `connector.sync_failed` event each cycle. **Two plausible causes were probed live and both disproven:** `channelId` parses correctly (it is top-level in the real response — only `channelURL` is nested under `channelData`, which the code already handles), and subscribing *after* opening the WebSocket fails identically, so it is not a "channel not yet connected" ordering problem. Cause still unknown — the `Events[0]` constraint points at the `REPORT_SUMMARY` event name rather than the channel.
- **③ v1.2.0 GoTo inbound SMS never subscribes, and fails silently.** `POST /messaging/v1/subscriptions` → `400`, `{"field":"ownerPhoneNumber","constraint":"Invalid"}` — GoTo rejects `+16303602784` in the E.164 form the config documents as required. **The failure handling is the worse half:** it is swallowed as a log warning and returns **zero subscription ids**, and the "no ids" path treats that as success. Had ② not been failing loudly beside it, the channel would have looked healthy while no text ever arrived. Whatever fixes the number format should also make a zero-id subscription a hard failure.
- **④ v1.1.5's Windows startup fix is inert — the bug it claims to fix is still there.** `main.py:25` sets `WindowsSelectorEventLoopPolicy` at import, but uvicorn 0.34 calls `config.setup_event_loop()` and then `asyncio.run(self.serve())` in `Server.run()` — and `serve()` is what calls `config.load()` to import the app. **The loop already exists before `main.py` is imported**, so the policy set there applies to nothing. Plain `python -m uvicorn app.main:app` still dies with `PoolTimeout: pool initialization incomplete after 30.0 sec`. The documented command works only because it carries `--reload`: uvicorn's own `asyncio_setup()` sets the selector policy when `use_subprocess` is true. A real fix has to set the policy before uvicorn builds the loop — an entrypoint script (`python run.py`) rather than a module-level statement in the app.
