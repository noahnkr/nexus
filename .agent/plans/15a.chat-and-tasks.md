# Plan 15a: Chat & Task Completion

**Complexity: ⚠️ Medium** — one careful backend change (cancellation-safe chat persistence), one seam extension (approve-with-edits through `services/approvals.py`), and otherwise frontend composition over existing patterns (drawer per `VisitDrawer`, labels per M13). No migrations, no new deps, no engine changes. **Fallback split point**: Tasks 1–2 (chat) and Tasks 3–5 (tasks + event log) are independent halves.

Parent: `.agent/plans/15.finishing-touches.md` (context + locked decisions live there; this plan is self-contained for execution).

## Context

- **Chat can't be stopped**: `lib/sse.ts::streamChat` already accepts an optional `AbortSignal` (4th param) but `ChatPage.send` never passes one, and the backend generator (`routers/chat.py::post_message` → `services/chat_service.py::stream_chat_turn`) has no cancellation handling — a client disconnect cancels the generator mid-turn and can leave the thread history ending on a `user` message, which breaks the next turn's verbatim replay (Messages API requires alternating roles).
- **Input alignment**: `MessageInput` uses `items-end` with a default-height icon button next to a `rows={1}` textarea — the button overshoots the input and the single-line placeholder isn't vertically centered.
- **Approvals are approve-verbatim-or-reject**: `approvals.py::approve_action` executes the stored `tool_input` untouched; the office user can't fix a typo in a drafted SMS without rejecting and re-asking.
- **Task cards leak JSON**: `ApprovalCard` renders `tool_input` behind a "Technical detail" `<pre>`; cards have no type iconography; the "· Originating event" separator dot is inside the link text (the reported centering bug at `TaskCard.tsx:113`).
- **Event log** is functional but visually flat; M13's `eventTypeLabel` exists in `lib/recipe.ts` and is not yet used by `EventRow`.

## Key design decisions (locked in — do not re-litigate during build)

1. **Stop-stream contract** (backend): wrap the body of `stream_chat_turn`'s agentic loop so that on `asyncio.CancelledError` (client disconnected / Starlette cancelled the generator) the accumulated streamed text of the in-flight assistant turn is persisted as the final assistant message — `metadata` gains `{"stopped": true}` plus the usual usage/model keys — inside `asyncio.shield(...)` on a **fresh** `tenant_tx` (the old one may be mid-rollback), then the error re-raises. If no text streamed yet, persist a single empty text block with `stopped: true`. Mid-tool-iteration cancels are already safe (the assistant tool_use + tool_result messages persist together at iteration end; nothing partial is written), so the handler only persists the *text* accumulated from the current stream. Log a `chat.message.stopped` event (plain `payload.summary`: "Response stopped by the user") instead of `chat.message.completed`. Justification: replay validity is the hard requirement; everything else is bookkeeping.
2. **Stop-stream contract** (frontend): `ChatPage.send` creates an `AbortController` held in a ref, passes `controller.signal` to `streamChat`, and `MessageInput` morphs its send button into a stop button (Square icon) while `streaming` — clicking it aborts. On abort: flush the rAF buffer, mark the bubble `streaming: false` with a muted "— stopped" annotation (a `stopped` flag on `UIMessage`, also derived on history reload from `metadata.stopped`), no error toast (an abort is not an error — catch and swallow `AbortError`).
3. **Input alignment fix**: give the button and the single-row textarea the same height (`h-10` both; textarea `min-h-10` with vertical padding that centers one line) and keep `items-end` so multi-line growth still bottom-aligns the button. No layout re-architecture.
4. **Formatting = PERSONA guidance + prose CSS**: extend PERSONA with an output-structure paragraph (use headings/bullet lists/GFM tables when the user asks for a structured artifact — e.g. a care plan or comparison; prose for conversational answers). `remark-gfm` already parses tables — verify `.prose-chat` styles them acceptably in both themes and wrap tables in an `overflow-x-auto` container in `Markdown.tsx` so wide tables never break the chat column. No PDF/export (deferred, parent non-goal).
5. **Approve-with-edits**: `ToolDef` gains `editable_fields: list[str] | None = None`; set `["body"]` on `send_sms`, `["subject", "body"]` on `send_email` (only outbound messaging for now — the seam supports any tool later). `PendingActionOut` gains `editable_fields` resolved at read time from the registry (never stored). The approve endpoint body gains optional `tool_input: dict`; the router 422s if it changes any key outside `editable_fields` (compare against the stored input) or if a value is not a non-empty string. `approve_action(..., edited_input=...)` updates the stored `tool_input` before executing (so the audit `tool.called` row and the executed handler both see the final text), and the `action.approved` event payload + action `result` gain `{"edited": true, "edited_fields": [...]}` with the original input preserved in the event payload (`"original_input"`). Justification: one execution path, full audit of what changed.
6. **`TaskDrawer`** (`components/tasks/TaskDrawer.tsx`, `VisitDrawer` pattern from `components/schedule/VisitDrawer.tsx`): clicking a task card opens a right-side drawer with — type icon + title, status/priority controls (reusing the card's transition logic), description, and for each pending action a **clean labeled rendering** of `tool_input` (per-field labels: To / Message / Subject; unknown tools fall back to humanized key–value rows, never raw JSON), editable fields rendered as prefilled inputs/textareas when `editable_fields` includes them and the action is pending, Approve (sends edits when dirty) / Reject with note, resolved-state line, "View history" and "Originating event" links, and a single collapsed "Technical detail" expander at the drawer bottom (the only raw-JSON surface). `ApprovalCard` on the card itself slims to a one-line status chip + the description; its JSON expander is deleted.
7. **Task-type icons/labels** live in a new `lib/tasks.ts`: derive a `{icon, label, tone}` from the task's first pending action's `tool_name` (`send_sms` → MessageSquareText "Text message", `send_email` → Mail "Email", `create_schedule`/`cancel_schedule` → CalendarDays "Scheduling", `update_*_status`/`update_applicant_stage`/`assign_caregiver`/`record_call_out` → UserRound "Record update") and default ListTodo "Task" for plain tasks. Used by `TaskCard` (leading icon + label chip) and `TaskDrawer` header.
8. **Dot fix**: in `TaskCard`, the `"· Originating event"` link becomes a standalone muted `<span>·</span>` separator followed by the link — same fix pattern as the existing `"· Due"` span.
9. **Event-log readability**: `EventRow` gains a per-source accent (small colored dot or left border tick — map in `lib/events.ts`: user/chat/automation/mcp/webhook sources → semantic token colors) and a type icon + plain-language type label via `eventTypeLabel` (already in `lib/recipe.ts`), raw type demoted to mono secondary text. `EventFilters` already shows labels (M13). No backend change — summaries stay server-derived.
