- **`12b.schedule-board.md`** (⚠️ Medium) — the `/schedule` week-board page (caregivers as rows, open-shifts row), visit drawer with call-out → ranked candidates → assign → gated SMS notify, create dialog with weekly repeat, the minimal roster drawer, Home open-shifts card, and the documented call-out automation recipe.

## Context

Module 12 (rescoped by the user 2026-07-18, replacing the deferred "matching/decision harness + scheduling" backlog item) delivers the staffing story: *suggest the optimal caregiver for a visit* and *when a caregiver calls out, find qualified replacements fast*. Today `schedules` exists (one-off visits, `resource_id NOT NULL`, statuses `scheduled/completed/cancelled/no_show`) with gated `create_schedule`/`cancel_schedule` tools and seed data — but there is **no schedules UI, no notion of an unfilled shift, and no matching logic anywhere**. Clients and caregivers have no address or language/trait fields, so "windshield time" and cultural fit have no data to score on yet.

**User-locked decisions (2026-07-18):**
1. **Week board UI**: a `/schedule` page — week calendar grid, caregivers as rows, visits as status-colored blocks, an Open-shifts row pinned on top, click-to-act drawer. The natural home for call-out/assign actions.
2. **Add address & trait fields**: `clients`/`resources` gain `address`, `zip`, `languages[]`, and preference/trait tags so the match score covers geography (zip-level) and compatibility. True drive-time (maps data) stays in Future Plans.
3. **Call-out flow = owner picks, then SMS**: call-out frees the visit; the system produces a ranked candidate list; the owner assigns one; a **gated** `send_sms` notifies the chosen caregiver (approved in Tasks). No inbound-SMS handling exists, so no broadcast/reply loop — the broadcast variant is a documented recipe a user can build, not a shipped path.
4. **Repeat-weekly expansion**: the create-visit dialog offers "repeat weekly until <date>" (capped at 12 additional weeks); the backend expands into individual `schedules` rows at create time. No recurrence tables.

**Design decisions locked at planning time** (conventional, not re-litigated): the generic phase-pipeline *decision harness* from the old backlog item stays deferred — matching here is a deterministic scoring service + safe tool using the existing approval gate, no new harness abstraction. Matching weights are constants in the vertical seam file (no config UI). The board is the third sanctioned vertical surface alongside Leads/Caregivers — pattern-core, content-seam, recorded in CLAUDE.md.

**Non-goals** (name the destination): true distance/drive-time scoring via a maps/geocoding connector (Future Plans); inbound SMS / offer-acceptance loop (Future Plans, needs telephony connector work); EVV clock-in/out (Future Plans); caregiver utilization/retention views (Future Plans); recurrence tables or per-visit series editing (Future Plans); authorized-vs-delivered hours census (Future Plans); any client-profile UI (still no sanctioned Clients view); seeded/starter automations (M8 user-lock stands — the call-out recipe is documented, not shipped).

## Shared conventions (both sub-plans)

- **Shift model**: `resource_id` becomes nullable — `NULL` + status `open` *is* the open shift. A call-out keeps the original row (`called_out`, resource retained — the record of who called out) and creates a linked replacement row (`open`, `replaces_schedule_id` → original). Assignment fills an `open` row → `scheduled`. One table, no parallel shift structures.
- **One write path per transition**: `services/views/schedule.py` owns `create_visits` (with weekly expansion), `assign`, `call_out`, `cancel`, and status bookkeeping — REST routes and tool handlers both delegate (the `move_stage()` precedent). Every transition emits a plain-summary event: `schedule.created` / `schedule.assigned` / `schedule.called_out` / `schedule.cancelled` / `schedule.updated`, with the caller's `source_system` (tool paths via the invocation contextvar, the `update_lead_status` precedent).
- **Matching is deterministic and explainable**: `services/views/matching.py` ranks candidates with hard disqualifiers (missing required qualification, hard time conflict, the caregiver who called out) and scored components (availability fit, zip/region geography, continuity, language/trait overlap, weekly-load balance), returning plain-language `reasons`/`warnings` per candidate. No LLM in the ranking.
- **All agent access through the registry**: safe `find_available_caregivers` (read-only rank), gated `record_call_out` and `assign_caregiver`; existing `create_schedule`/`cancel_schedule` re-pointed at the seam functions. The UI's "notify by text" runs `send_sms` through `execute_tool` (`source_system='user'`) and surfaces the queued gated task — one seam, full audit trail.
- Plain language everywhere; matching breakdowns as reason chips, never raw scores tables; raw JSON only behind technical expanders. No new frontend deps, no new env vars, no engine changes.
