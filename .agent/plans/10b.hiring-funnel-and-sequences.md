# Plan 10b: Caregivers View — hiring funnel, metrics & per-stage sequences

**Complexity: ⚠️ Medium** — the payoff sub-plan for the pattern investment: no new schema, no engine changes, no new shared components. It instantiates M9's generic funnel/builder pieces with a caregivers view config and adds one metrics endpoint. Fallback split point: Task 1 (backend, curl-testable) vs Tasks 2–4 (frontend + live walk).

Parent: `.agent/plans/10.caregivers-view.md`. Requires 10a built (applicants API, `applicant.stage_changed`, `views/caregivers.py`) and Module 9 built (`components/pipeline/FunnelStrip`, view-config-driven `StageSequencePage` + `lib/pipeline.ts`, `automations.binding` API).

## Context

After 10a, recruiters can track applicants but can't see the pipeline shape or automate the PRD's marquee flows — the accepted/denied emails. This sub-plan completes PRD interface #8: the hiring funnel strip with per-stage sequence chips, hiring metrics widgets, and per-stage sequences as bound automations. Everything runs on mechanisms that already exist generically: binding (`{"view":"caregivers","stage":…}`), the shared stage builder, the M7 engine, the approval gate for outbound email.

**User-locked** (parent plan): scoring deferred to M11 — sequences here branch on plain applicant fields (e.g. `entity.stage`, `trigger.payload.to`) only.

**Non-goals**: scoring functions/conditions (M11); roster/credential views (Future Plans); sequence analytics beyond the Center's run info; multi-sequence-per-stage; any change to `FunnelStrip`/`StageSequencePage` beyond what a view config drives — if the caregivers instantiation needs a shared-component change, make it a generic prop/config, never a `view === "caregivers"` branch.

## Key design decisions (locked in — do not re-litigate during build)

1. **`GET /api/applicants/metrics`** (`routers/applicants.py` → `views/caregivers.py`, mirroring the leads metrics shape): `{stages: [{stage, count}…] (all six), hire_rate (hired ÷ total, %), new_last_7_days, avg_days_to_hire (mean of stage_changed-to-hired event time − applicant created_at, null if none), top_sources (top 5)}`. Plain SQL over `applicants` + `events`; empty tenant → zeroes/nulls, never a 500.
2. **Funnel strip instantiation** (`CaregiversPage`): the generic `components/pipeline/FunnelStrip` fed by the caregivers `PipelineViewConfig` from `lib/caregivers.ts` — five worked segments + `rejected`, counts from the metrics endpoint, segment click ↔ the existing `stage` URL param (replacing 10a's chip row). **`sequenceStages` includes all six stages — `rejected` gets a chip** (denied email), the deliberate divergence from leads' chip-less `lost`; it's config data, not component logic. Sequence chips map from `GET /api/automations?view=caregivers` by `binding.stage`; chip click → `/caregivers/stages/{stage}/sequence`.
3. **`HiringMetrics`** (`components/caregivers/HiringMetrics.tsx`, mirrors `LeadMetrics`): four `StatCard`s — In pipeline (non-terminal count), Hire rate, New this week, Avg days to hire — plus the small top-sources card. Semantic tones; no charts.
4. **Stage sequence builder = shared page + config, no new page**: route `/caregivers/stages/{stage}/sequence` renders M9's `StageSequencePage` with the caregivers config — trigger convention (stage `applied` → event `applicant.created`; others → `applicant.stage_changed` + managed `trigger.payload.to eq <stage>` condition), tool palette `send_email`/`send_sms`/`create_task`, default names "Caregivers · Offer sequence", binding `{"view":"caregivers","stage"}`, new sequences start paused with the activate toggle. Duplicate-stage save → the existing 409 surfaced plainly.
5. **Center awareness is config, not code**: `describeBinding()` already renders "Caregivers · Offer" generically; the Edit-rerouting map (binding.view → stage-builder route) gains the caregivers entry in `lib/caregivers.ts`'s config registration — `lib/recipe.ts` and `routers/automations.py` stay untouched.
6. **No backend automation changes**: binding validation, uniqueness 409, `?view=` filter, and the vocabulary endpoint are already generic (9b). If a gap appears, fix it generically in the M9-established files with a test.
