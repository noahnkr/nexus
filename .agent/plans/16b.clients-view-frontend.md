# Plan 16b: Clients View & Care Overview Frontend

**Complexity: ⚠️ Medium** — two new pages and a card set that reuse the M9/M10 directory/profile patterns and the M12b drawer patterns wholesale, plus small additions to two existing schedule components. No backend work (16a's API is complete), no migrations, no new deps. **Fallback split point**: Tasks 1–3 (directory + profile) stand alone; Tasks 4–5 (documents card, board EVV) are a clean second pass.

Parent: `.agent/plans/16.client-care-oversight.md`. Build after 16a.

## Context

16a shipped: `GET/POST /api/clients` (+ `metrics`/`facets`), `GET/PATCH /api/clients/{id}`, contacts CRUD, cached client summary endpoints, `POST /api/schedule/{id}/check-in|check-out` (feed rows now carry `check_in_at`/`check_out_at`/`evv`), and tagged document upload/list. The frontend has every pattern this module needs: `CaregiversPage`/`CaregiverProfilePage` (directory + profile with tag editors), `LeadMetrics`/`HiringMetrics` (StatCard strips), shared `SmartSummary` + `EntityTimeline`, `VisitDrawer`/`VisitBlock`, `ui/Select` (M13 — never native `<select>`), and Realtime debounced-refetch wiring on every view page.

Client statuses are `active`/`hospital_hold`/`discharged`; payers `private_pay`/`medicaid`/`ltc_insurance`/`va`/`other`.

## Key design decisions (locked in — do not re-litigate during build)

1. **`lib/clients.ts`**: `CLIENT_STATUS_META` (label + semantic tone: active=success, hospital_hold=warning, discharged=muted), `PAYER_LABELS` (Private pay, Medicaid, LTC insurance, VA, Other, Unknown), `fmtHours(n)` ("38.5 h"), `fmtDuration(checkIn, checkOut)`. Pure module, vitest-covered.
2. **`/clients` directory page** (`ClientsPage.tsx`): `PageHeader` + `CensusStrip` + filter row (status chips via meta, payer + region `Select`s, search) ↔ URL params (leads-page pattern), `ClientsTable` (name, status pill, payer, region, authorized hrs/wk, this-week scheduled hrs), `ClientCreateDialog` (name required; phone/email/payer/region/authorized hours optional), Realtime on `clients` (debounced refetch), row-click → profile. Nav entry **Clients** (`HeartPulse` icon) between Caregivers and Schedule.
3. **`CensusStrip`**: four `StatCard`s — Active clients; Authorized hrs/wk; Scheduled this week (open/unfilled hours as the subline); Delivered this week (delivery-rate subline, **warning tone when `leakage_hours > 0`** — the revenue-leakage number is the point of the census). Below, two compact chip rows: by-payer and by-region counts (clicking a chip applies that filter). Data from `GET /api/clients/metrics` — no client-side math beyond formatting.
4. **`/clients/{id}` profile page** (`ClientProfilePage.tsx`), caregiver-profile layout (two-column, cards):
   - `SmartSummary` (shared component, `entityType="client"`).
   - `ClientInfoCard`: contact/address/zip inline edit + `languages`/`preferences` tag editors (caregiver-drawer pattern) → `PATCH`.
   - `CareCard`: status `Select` (meta dots; **confirm dialog on discharge** — hire-confirm precedent), payer `Select`, authorized hrs/wk numeric input, region `Select`, `care_summary` textarea with save.
   - `HoursCard`: this week's authorized / scheduled / delivered as three labeled bars (delivered bar warning-toned when short of authorized), leakage line ("6 h under authorized this week").
   - `ContactsCard`: family contacts list (primary star first), add/edit/delete dialogs, relationship + phone/email.
   - `CaregiversCard`: distinct assigned caregivers with next-visit line, linking to the schedule board.
   - `ClientVisitsCard`: next 5 upcoming + last 5 past visits, status pills + EVV badges (late/missed amber; actual duration shown when clocked), "Open in schedule" linking `/schedule?week=` of that visit.
   - `EntityTimeline` (`entityType="client"`).
5. **`ClientDocumentsCard`** (care plans & records): lists the client's tagged documents (status pill, relative time), upload button reusing the ingestion dropzone flow with `entity_type='client'`/`entity_id` **preset invisibly** — the user just picks a file; delete with confirm. A muted explainer: "Documents here are searchable in chat and tied to this client." The Ingestion page itself is untouched (15b reworks it — no collision).
6. **Schedule board EVV surfaces**: `VisitBlock` gets a compact amber `late`/`missed` badge from the feed's server-computed `evv` field (no client-side rule math). `VisitDrawer` gets state-driven **Check in** / **Check out** actions on scheduled visits (check-out completes the visit — drawer reflects the transition), actual times displayed once recorded ("Checked in 8:04 · out 12:14 · 4h 10m"), and the existing outcome buttons remain for unclocked bookkeeping.
7. **No Home changes, no new metrics math client-side** — every number renders from 16a responses.
