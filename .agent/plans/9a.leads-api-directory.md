## Context

Leads exist (Module 0 table, webhook auto-creation from 3b, gated chat tools from 5a) but have no human surface: no list, no profile, no manual entry, and stage changes leave no first-class entity event. This sub-plan ships the directory and profile halves of PRD interface #7 and establishes `lead.stage_changed` ‚Äî the event 9b's per-stage sequences trigger on. The funnel strip, metrics widgets, and sequence builder are 9b; this page ships with a plain stage chip-row filter that 9b replaces.

**User-locked** (parent plan): create + stage moves + basic-field edits as human REST writes; no delete; smart summary on-demand with no persistence.

**Non-goals**: funnel strip/metrics/sequences (9b); lead scoring (M10); webhook stage moves (connectors keep recording `lead.updated` without moving stages); bulk operations/CSV import; editing `requirements` jsonb beyond display (structured requirements editing waits for a real need).

## Key design decisions (locked in ‚Äî do not re-litigate during build)

1. **Stage config in the vertical seam** ‚Äî `backend/app/services/views/leads.py` exports `LEAD_STAGES`: ordered `[new, contacted, qualified, converted]` + terminal drop-off `lost`, each with a plain label ("New", "Contacted", ‚Ä¶). `frontend/src/lib/leads.ts` mirrors it (labels, order, badge tones: converted = success, lost = muted, others = info/default). *Why*: stages already live in `leads.status` (Module 0 CHECK); config-not-schema keeps the funnel re-templatable.
2. **`routers/leads.py`** (vertical seam, JWT tenant-scoped like every `/api` route):
   - `GET /api/leads` ‚Äî filters `status`, `source`, `q` (ilike over name/phone/email), `limit` (default 50, cap 100) + `offset`, `order by created_at desc`; each row carries `region_name` (left join). Offset pagination, not keyset ‚Äî a directory sorts and pages small tenant-local sets. Response `{leads, total}` so the UI can page.
   - `GET /api/leads/facets` ‚Äî distinct non-null sources + `{id, name}` regions (create/edit selectors and the source filter).
   - `POST /api/leads` ‚Äî `name` required; `phone`/`email`/`source`/`region_id` optional (422 plain message on missing name / bad region id). Status always starts `'new'`. Emits `lead.created` (`source_system='user'`, entity-linked, `payload.summary` "Lead 'X' created manually").
   - `GET /api/leads/{id}` ‚Äî full row + `region_name`.
   - `PATCH /api/leads/{id}` ‚Äî accepts `name`, `phone`, `email`, `source`, `region_id`, `status`. A `status` change validates against `LEAD_STAGES` values and emits **`lead.stage_changed`** (`payload: {from, to, summary: "Lead 'X' moved from New to Contacted"}`); other field changes emit one `lead.updated` (summary names the changed fields). No-op PATCHes emit nothing (tasks-router precedent). **No DELETE route.**
   *Why REST writes, not tools*: the approval gate exists for agent-initiated effects; a human clicking their own UI is the approver ‚Äî exactly the Tasks-page precedent (`source_system='user'`).
3. **`update_lead_status` tool handler also emits `lead.stage_changed`** (after its UPDATE, same payload shape, `source_system` = the caller's, via `log_event` on the handler's conn). *Why*: chat/MCP/automation stage moves must appear in the lead's timeline and (chat/MCP only ‚Äî M7 loop guard skips automation-sourced events, deliberately) fire 9b's sequences. Read the current status before updating so `from` is truthful.
4. **Smart summary: `GET /api/leads/{id}/summary`** ‚Äî split view-agnostic from vertical *(amended at M10 planning: M10 reuses the machinery for applicants)*: `services/views/summary.py` holds the generic helper `generate_entity_summary(conn, entity_row, entity_type, entity_id, prompt_intro, span_name)` ‚Äî loads the last 20 entity events (summaries via `event_summaries.derive_summary` ‚Äî plain language in, plain language out), one `settings.fast_model` call, `@traceable` chain span, returns `{summary, generated_at}`, raises a typed no-key error the router maps to 503 (8b draft-endpoint precedent). `services/views/leads.py` supplies only the lead prompt intro (2‚Äì4 sentences: who the lead is, where they came from, what has happened, likely next step) and the `lead_summary` span name. **No persistence.** Plain text output ‚Äî not a Pydantic structured call; the CLAUDE.md rule covers *structured* outputs and this is prose.
5. **Migration `20260718000000_leads_view_infra.sql`** (vertical): add `leads` to the `supabase_realtime` publication, guarded/idempotent (eventlog-migration precedent). No other schema change.
6. **`/leads` directory page** (`LeadsPage.tsx`): `PageHeader` + "New lead" button; stage chip-row filter (All + five stages, counts omitted ‚Äî 9b's funnel strip brings counts) + source select + search input, all ‚Üî URL params (events-page precedent); `LeadsTable` (name, stage badge, source, region, age relative-time, contact) with row click ‚Üí `/leads/{id}`; `LeadCreateDialog` (name/phone/email/source/region via facets); Realtime `postgres_changes` on `leads` with the debounced-refetch pattern; `EmptyState` for zero leads. Nav entry: **Leads** (lucide `Filter` ‚Äî the funnel glyph) between Tasks and Automations.
7. **`/leads/{id}` profile page** (`LeadProfilePage.tsx`): header (name, stage badge, back link); `SmartSummary` card at top ‚Äî auto-fetches on mount, skeleton while loading, regenerate button, quiet inline notice on 503 (never blocks the page); `LeadInfoCard` ‚Äî inline editable basic fields (Input on click or an Edit toggle, PATCH on save, toast on error) + `StageSelect` (select of stage labels; change PATCHes status and refetches ‚Äî the timeline gains the stage event); requirements jsonb shown read-only behind a technical expander; `EntityTimeline` (new shared component under `components/events/`) ‚Äî compact feed of `GET /api/events?entity_type=lead&entity_id=‚Ä¶` (summary, source, relative time, payload behind expander, Load more via keyset cursor). *Why a page, not a drawer*: profiles are deep-linkable and M10 reuses the layout.
