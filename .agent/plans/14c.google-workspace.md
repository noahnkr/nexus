# Plan 14c: Gmail & Google Calendar Integration

**Complexity: ⚠️ Medium** — builds on 14a's sync loop/ingest seam and 14b's shared OAuth helper. **Fallback split**: Tasks 1–3 (Gmail: poll, attachments, real send) stand alone; Tasks 4–5 (Calendar poll + tools) are a clean second pass.

See the parent plan `.agent/plans/14.external-connectors.md` for shared conventions. **Requires 14a Tasks 1–4 and 14b Task 1 built** (sync loop, ingest seam, `oauth.py`).

## Context

User-locked: **polling with sync cursors**, not Pub/Sub push — no public URL or GCP Pub/Sub setup needed; the fetch-back code is identical to what push would call, so push later is additive. The M3 placeholder adapters already normalize `email.received` (Gmail, Pub/Sub-envelope shape) and `calendar.event.updated` (GCal); the poll runners produce the *decoded* shapes directly. Scope: read + send email (attachments → ingestion), calendar read + event creation via tools usable from chat/MCP/automations. Gmail's `historyId` comes from `users.getProfile`/`history.list`; Calendar increments come from `events.list` with a stored `syncToken` (HTTP 410 ⇒ drop token, full resync).

## Key design decisions (locked in — do not re-litigate during build)

1. **OAuth bootstrap** (`scripts/google_oauth.py`): same pattern as 14b's — localhost redirect listener, consent URL, code exchange at `https://oauth2.googleapis.com/token`, prints `GOOGLE_REFRESH_TOKEN=...`. Scopes: `gmail.readonly`, `gmail.send`, `calendar` (constants in the script). Requires a GCP OAuth client (**blocking ops step**: the operator creates a "Web application" client with `http://localhost:{port}` redirect, sets `GOOGLE_CLIENT_ID`/`GOOGLE_CLIENT_SECRET` in `.env`, and completes consent with the business Google account; testing-mode consent screens need the account added as a test user — documented in README).
2. **Google client** (`google_client.py`): thin httpx wrappers over the REST endpoints actually used (`users/me/profile`, `users/me/history`, `users/me/messages/{id}`, `users/me/messages/{id}/attachments/{id}`, `users/me/messages/send`, `calendars/primary/events` list/insert), authenticated by a `TokenSource` from 14b's `oauth.py`. No google-api-python-client.
3. **Adapters accept the decoded shape**: `adapters/gmail.py` treats a payload that already carries `messages` at top level the same as one inside the Pub/Sub envelope (`_decode_data` falls through); `adapters/gcal.py` already reads `payload["events"]` — the runner passes headers with `x-goog-resource-state: exists`. Placeholder fixtures keep passing; push delivery later reuses the same adapters unchanged.
4. **Gmail runner** (`gmail_runner.py`): cursor `state.gmail.history_id` (bootstrap: current profile `historyId` — **no historical backfill**; email history is not imported, only new mail from activation forward, matching the "mirror, don't re-platform" scope). Each cycle: `history.list` → collect added message ids → `messages.get` (metadata + structure) → build the decoded payload for `ingest_payload("gmail", ...)`. Inbound-only filter: skip messages with the `SENT` label. Attachments: for resolvable senders (the `email.received` event matched a lead), each attachment ≤10 MB with a parseable MIME type is downloaded and fed to 14a's `ingest_text`/file pipeline (`services/ingestion`), entity-tagged to the lead; unmatched senders' attachments are skipped (the review task from resolution is the surface for those). Justification: ingest-what-you-can-attribute keeps the corpus tenant-clean.
5. **Real `send_email`** (`services/messaging/gmail_send.py`): RFC 2822 build (To/Subject/plain body; From = the authenticated account), base64url `messages/send`; called from the existing gated tool handler — name, schema, gate, audit unchanged. A successful send also writes an `email.sent` event (declared type) linked to the recipient's lead when resolvable, so timelines show outbound mail.
6. **Calendar runner** (`gcal_runner.py`): cursor `state.gcal.sync_token`; initial run stores a token via a bounded window list (`timeMin` = now−7d) rather than importing deep history; 410 ⇒ clear token, re-window. Changed events → `ingest_payload("gcal", ...)` → `calendar.event.updated` (existing declared type, entity resolution against `external_ids` category `calendar`, unresolvable ⇒ review task per the standard path).
7. **Calendar tools** (registry-registered in `services/tools/entities.py`, so chat/MCP/automations and the M8 vocabulary get them for free): safe `list_calendar_events` (start/end ISO range → id/title/start/end/attendees) and gated `create_calendar_event` (title, start, end, attendees?, description?; `gate_describe` names the title + time in plain language). Successful creation writes `calendar.event.created` (new declared type) and an `external_ids` calendar mapping when the event was created for a known entity (optional `lead_id` input). No update/delete tools this module (Future Plans until a real need).
8. **New settings**: `google_client_id`, `google_client_secret`, `google_refresh_token`, `gmail_attachment_max_mb` (default 10). Runners/tools enabled only when all three credentials are set; tools fail loudly with a plain message otherwise.

## Tasks (strict order)

### Task 1 — Google OAuth bootstrap + client
Decisions 1–2, 8. **Blocking ops step**: GCP OAuth client + consent (see decision 1); operator pastes the three `GOOGLE_*` values into `.env`.
**Validation**: offline client tests against mocked endpoints; env-gated live test fetches the Gmail profile and asserts a non-empty `historyId` (never printed beyond presence).

### Task 2 — Gmail poll runner + attachments → ingestion
Decisions 3–4.
**Validation**: offline — fixture history pages: new inbound message → `email.received` via `ingest_payload` with the receipt event; `SENT`-labeled skipped; second cycle from the advanced cursor fetches nothing; attachment on a matched sender lands a document row entity-tagged to the lead (mocked download). **Live**: email the business inbox (with a small PDF) from a known lead's address → within one poll the timeline shows `email.received` and the attachment appears in Ingestion, retrievable in chat.

### Task 3 — Real `send_email`
Decision 5.
**Validation**: offline — RFC 2822 build + payload shape test with mocked send; gated live: from chat, draft + approve an email to a test address you control → real delivery; `email.sent` on the lead timeline; `action.queued → approved → tool.called` trail intact.

### Task 4 — Calendar poll runner
Decision 6.
**Validation**: offline — fixture sync pages incl. a 410-then-resync sequence; changed event → `calendar.event.updated` through the standard resolution path (unmatched ⇒ review task). **Live**: create/modify an event in the business Google Calendar → event appears in the Event Log within one poll.

### Task 5 — Calendar tools
Decision 7.
**Validation**: gated tool tests — `list_calendar_events` (mocked client) returns the normalized shape; `create_calendar_event` queues when unapproved and, via the approvals path, executes with the mocked client + writes `calendar.event.created` and the calendar `external_ids` row when `lead_id` given. **Live**: in chat, "schedule a tour with «lead» Friday 2pm" → gated task names it in plain language → approve → the event exists in Google Calendar.

### Task 6 — Wrap-up
README: Google section (GCP client runbook, scopes, consent test-user note, poll behavior, attachment policy); `.env.example`; full pytest green; `npm run build` verified.
**Validation**: full `pytest backend/tests` green; the Task 2/3/5 live walks recorded in PROGRESS; LangSmith shows `connector_sync` cycles with gmail/gcal child spans.

## Execution notes

- **Blocking inputs**: GCP OAuth client creation + one-time consent (Task 1); a test email address the operator controls (Task 3); the business Google account signed in for the live walks.
- **Reuse**: `TokenSource` from 14b `oauth.py`; sync-runner registration from 14a `sync.py`; gated-tool + approvals test patterns from Module 5 tests; ingestion pipeline entry from 14a's `ingest_text`.
- **Scope discipline**: no email backfill; no Pub/Sub/push; no calendar UI; no update/delete calendar tools; attachment ingestion only for attributed senders; `send_email` gate semantics untouched.
