# Plan 15: Finishing Touches (parent)

**Complexity: 🔴 Complex** — nine loosely-coupled PRD bullets spanning every surface of the app, plus a new **core table** (`tenant_settings`), an approval-seam extension (approve-with-edits), and two automations-framework additions (a `formula` function and a `run_automation` agent tool — framework involvement keeps the 🔴 default per CLAUDE.md). Per the planning rule it is **broken into three sub-plans**, each independently ⚠️ Medium:

- **`15a.chat-and-tasks.md`** (⚠️ Medium) — stop/cancel chat streaming (with a cancellation-safe persistence contract), chat input alignment fix, document-style output formatting, approve-with-edits (edit a drafted SMS/email before approving), a task detail drawer that renders action data cleanly (no raw JSON on cards), task-type icons/labels, the metadata-line dot fix, and event-log readability (icons, plain type labels, source accents).
- **`15b.shell-settings-knowledge.md`** (⚠️ Medium) — `tenant_settings` core migration + settings API, agent-instructions injection into the chat system prompt, sidebar collapse to an icon rail, mobile-responsive shell (drawer nav) + core-page sweep, the `/settings` page (profile edit, workspace, appearance), and the Ingestion → **Knowledge** rename (Documents tab with a detail drawer + Instructions tab).
- **`15c.automation-touches.md`** (⚠️ Medium) — the `formula` calculation function (hand-rolled safe expression parser) + a token-aware `FormulaEditor` replacing the raw-JSON weighted-score form, manual-trigger automations getting a **Run** button instead of the Active/Paused toggle, and the safe `run_automation` tool so chat/MCP agents can start manual automations.

Build order: **15a → 15b → 15c**. 15a and 15b are independent, but 15b's mobile sweep must cover 15a's new task drawer, so 15a lands first. 15c is fully independent and framework-adjacent — building it last keeps the riskiest changes isolated. If any sub-plan stalls, the others still ship whole.

**Requires Modules 7–13 built** (all are): the approval gate (edit-before-approve extends `approvals.py`), the engine + waker (`run_automation` parks a run for the waker to advance), the M11 field catalog + `TokenText` (the formula editor is token-based), and the M13 `ui/Select` (every new dropdown uses it).

## Context

Modules 0–13 are code complete; Module 14 (real connectors) is planned but not built — nothing here depends on it. Module 15 is the polish-and-usability pass the user collected while using the app: chat can't be stopped mid-stream (`streamChat` accepts an `AbortSignal` that no caller passes), the send button doesn't line up with the input, approval cards dump raw `tool_input` JSON behind a "Technical detail" toggle, a queued SMS/email must be approved verbatim or rejected (no editing the draft), tasks give no visual cue of their type, the sidebar is fixed-width desktop-only, there is no settings surface at all (and no per-tenant settings storage — config is env-only), agent behavior can't be customized, manual automations awkwardly show Active/Paused, and the "Calculate a score" step renders as raw JSON (the M11b `CalculationEditor` named in PROGRESS was never actually committed — `weighted_score` args fall through to `SchemaForm`'s raw-JSON object fallback).

**User-locked decisions (2026-07-19):**
1. **Three sub-plans** as listed above (the recommended split).
2. **Rich output = formatting only**: improve generation guidance and markdown rendering (tables, headings, document-style answers like a care plan); any PDF/export story is **deferred to Future Plans** — no backend PDF service, no print-export in this module.
3. **Ingestion → Knowledge, two tabs**: Documents (existing upload/list plus richer viewing — detail drawer with chunk/status info and delete) and Instructions (free-text agent instructions plus a small set of structured fields like tone). Stored per-tenant in a new core settings table, injected into the chat system prompt.
4. **Mobile = core pages**: responsive shell (sidebar becomes a drawer behind a hamburger) and proper small-screen layouts for Home, Chat, Tasks, Leads/Caregivers, Event Log, Settings, Knowledge. The schedule board and automation builder stay desktop-first with reliable horizontal scroll — bespoke mobile layouts for them go to Future Plans.

**Design decisions locked at planning time** (do not re-litigate during build):
1. **`tenant_settings` is a core table** (one row per tenant, `settings jsonb`, 4-policy RLS): user-facing workspace/agent preferences. Env vars remain the home of infra config and credentials — this table never holds secrets. Read/write through a `services/settings.py` seam; whitelisted keys only.
2. **Agent instructions never weaken safety**: the tenant's instructions are appended to the chat system prompt *after* the core PERSONA (which keeps the gating rules); they customize tone/content, not the approval gate or tool semantics.
3. **Approve-with-edits goes through the existing seam**: `ToolDef` gains `editable_fields` (initially `send_sms.body`, `send_email.subject/body`); the approve endpoint accepts an edited `tool_input` restricted to those keys; `services/approvals.py` stays the only caller of the bypass, and the `action.approved` event records that (and which fields) were edited. Never a second execution path.
4. **Stop-stream must leave valid history**: chat messages replay verbatim to the Messages API (roles must alternate), so a client abort persists the partial assistant text (or an empty text block) with `metadata.stopped=true` before the generator dies — the next turn always sees a well-formed thread.
5. **`run_automation` is a safe tool restricted to manual-trigger automations**: starting a run has no direct external effect (gated steps still park for approval), and the handler refuses when invoked with `source_system='automation'` (extends the automations-don't-trigger-automations rule to the tool layer). Because a tool handler's transaction commits only after `execute_tool` returns, the tool creates the run **deferred** (`status='waiting'`, `wake_at=now()`) and the M7b waker advances it within one poll — no new execution machinery.
6. **The formula function is a parser, not eval**: `formula` evaluates `+ - * / ( )`, unary minus, and `round()` over numbers via a hand-rolled recursive-descent parser; field values arrive through the existing `{{token}}` template rendering. `weighted_score` stays registered so existing recipes keep running; the builder's "Run a calculation" now leads with `formula`.
7. **The task drawer follows the `VisitDrawer` pattern** (M12b): card click opens a right-side drawer; clean labeled rendering of the action (To / Message / Subject…), editable drafts, approve/reject, history links; raw JSON survives only in a technical expander at the drawer's bottom. Cards lose the JSON toggle entirely.

**Non-goals** (name the destination): PDF/print export of chat output (Future Plans); mobile layouts for the schedule board and automation builder (Future Plans); inbound SMS/email conversation loops (Module 14b/c scope and beyond); per-user (as opposed to per-tenant) preferences (Future Plans); tenant administration / user management UI (Future Plans); editing non-whitelisted tool args at approval time (extend `editable_fields` later if a real need appears); real SMS/email delivery (Module 14 — the placeholder handlers stay); any LLM in conditions, formulas, or ranking (never — CLAUDE.md).
