## Context

The current frontend is functional shadcn-default: a 224px sidebar with four flat nav links, per-page `h-14` headers, stock zinc-ish palette, plain-text chat bubbles with a `whitespace-pre-wrap` div, and `scrollIntoView`-on-every-message autoscroll. Views exist for Chat (`/`), Ingestion, Tasks, Event Log. After 6a, everything sits behind login and `useAuth().session.user` is available to render.

User-locked (see parent): Home lands at `/` and Chat moves to `/chat`; Home is a **light** landing page — greeting, at-a-glance widgets, recent activity, quick actions — explicitly *not* a needs-attention queue (Tasks already is one) and *not* feature-heavy; it should establish the widget-grid pattern future modules extend. PRD locks: markdown rendering + better text streaming for chat; use the frontend-design plugin; non-technical-user rule (plain language, no raw JSON outside sanctioned expanders).

**Non-goals**: needs-attention queue / cross-surface aggregation on Home (future Home-widget pass, once M7 workflows and M8 harness outcomes exist to aggregate); Settings view (env-config phase); dashboard customization/drag-drop widgets (future); charts or analytics beyond simple counts (nothing to chart yet); virtualized lists or chat pagination (corpus is small — scale discipline); citation superscript/hover-preview rework (M10 retrieval work revisits citations); no backend behaviour changes beyond D2's read-only endpoint.

## Key design decisions (locked in — do not re-litigate during build)

1. **Routes**: `/login` (6a) · `/` → `HomePage` · `/chat` → `ChatPage` · `/ingestion`, `/tasks`, `/events` unchanged · unknown paths `<Navigate to="/">`. Nav order: Home, Chat, Tasks, Ingestion, Event Log. *Why*: user-locked landing; Tasks promoted above Ingestion because it's the daily-triage surface.
2. **`GET /api/home/summary`** (new `routers/home.py`, standard `tenant_conn`, Pydantic model in `schemas.py`): one round-trip returning `{open_tasks, pending_approvals, documents: {ready, processing, failed}, events_today}` — four scalar subqueries over core tables (`tasks` status in pending/in_progress; `pending_actions` status=pending; `documents` grouped by status; `events` where `created_at >= date_trunc('day', now())`). Read-only, business-agnostic, RLS does the filtering. *Why*: the list endpoints have no counts, and Home shouldn't fire five paginated requests to render four numbers. Recent activity intentionally reuses `GET /api/events?limit=6` — no duplicate feed endpoint.
3. **HomePage composition** (a CSS grid of self-contained widget cards — the extension seam future modules add to):
   - Greeting header: time-of-day salutation + user's email local-part (`useAuth`), long-form date. No functionality.
   - Four `StatCard`s from D2 (label, count, icon, sublabel like "2 processing" under documents), each a link into its view (`/tasks?status=pending,in_progress`, `/tasks` for approvals, `/ingestion`, `/events`).
   - `RecentActivity`: last 6 event summaries (existing `api.listEvents`) with source badge + relative time, footer link "Open Event Log". Summaries only — no payloads, no expanders (that's the Event Log's job).
   - `QuickActions`: New chat → `/chat` (fresh thread state), Upload document → `/ingestion`, New task → `/tasks?create=1` (TasksPage opens its existing `CreateTaskDialog` when the param is present, then strips it).
   - Realtime is **not** wired on Home this phase — counts load on mount; it's a landing page, not a monitor. *Why*: keeps Home light per the user's direction.
4. **Design foundation** (this is where the frontend-design skill drives; specifics below are floors, not ceilings):
   - Typography: `@fontsource-variable/inter` self-hosted, `font-sans` set to Inter with system fallbacks; type scale tightened (13px base UI chrome, larger/relaxed chat prose).
   - Palette: keep the existing HSL token architecture in `index.css` but redesign the values — a confident accent hue (not default zinc), audited light *and* dark themes, consistent semantic tokens for the existing status colors (amber=queued/pending, green=done/ready, red=failed/error already appear across views — name them as tokens instead of ad-hoc Tailwind classes).
   - Primitives: `PageHeader` (title, description, action slot — replaces the four hand-rolled `h-14` headers), `EmptyState` (icon + message + optional action — replaces bare muted-text divs), consistent `Skeleton` loading states on every list surface.
   - `@tailwindcss/typography` for rendered markdown, tuned via `prose` overrides to match the palette.
5. **Shell overhaul** (`AppShell.tsx` + new `UserMenu.tsx`): restyled sidebar (brand mark, grouped nav with refined active/hover states, optional collapse-to-icons if it earns its keep), sidebar footer becomes a `UserMenu` — signed-in email, theme toggle, sign out (relocating 6a's temporary button). Page content column gets a consistent max-width/padding rhythm via `PageHeader`. Mobile stays out of scope; keep the existing desktop-first layout responsive enough not to break at laptop widths.
6. **Chat markdown**: new `components/chat/Markdown.tsx` wrapping `react-markdown` + `remark-gfm` (tables, lists, task lists, strikethrough), `prose prose-sm` styling, styled code blocks (`overflow-x-auto`, monospace, subtle surface) — **assistant messages only**; user bubbles stay plain text. Citation markers like `[1]` continue to render as literal text (SourceList below the bubble is unchanged). History reload renders through the same component. *Why raw-text user bubbles*: user input isn't markdown-authored; rendering it would surprise.
7. **Streaming smoothness**: buffer incoming SSE text deltas in a ref and flush to state once per `requestAnimationFrame` (collapses re-render-per-token and keeps react-markdown reparses bounded); autoscroll becomes *pinned-aware* — track whether the user is within ~80px of the bottom via `onScroll`, only auto-follow when pinned, and show a "jump to latest" affordance when unpinned during a stream; keep the block cursor while streaming.
8. **Polish sweep** (Ingestion, Tasks, Event Log — presentational only, zero logic/API changes): adopt `PageHeader`/`EmptyState`/`Skeleton`/semantic status tokens, normalize spacing, badges, table and card styles to the new system; ThreadList in chat gets the same treatment. Any behaviour change found tempting here gets written down for a future module instead of done.
9. **New frontend deps** (only these): `react-markdown`, `remark-gfm`, `@tailwindcss/typography`, `@fontsource-variable/inter`. No state library, no animation library — Tailwind transitions suffice.
