# Plan 8b: Automations Center — recipe builder & agent drafting

**Complexity: ⚠️ Medium** — two focused backend endpoints (one static-ish vocabulary read, one structured-LLM draft) and one rich frontend page. The builder is the most intricate UI in the app (schema-driven forms, sentence editing, step reordering), but it edits a single JSON document against an API that already validates everything — the server is the safety net, the UI only has to be honest. Fallback split point: Tasks 1–2 (both endpoints, curl-testable) vs Tasks 3–6 (builder UI + drafting flow).

Parent: `.agent/plans/8.automations-center.md`. Requires 8a built (routes, read-mode recipe components, `lib/recipe.ts`).

## Context

8a ships viewing/managing; this sub-plan ships creating/editing: the monday.com-style **sentence + step list** builder (user-locked) on dedicated pages (user-locked), fed by a vocabulary endpoint so the UI is data-driven (new tools/functions appear in the builder with zero frontend changes — the M9/M10 seam), plus the **describe → agent drafts → review in builder** flow (user-locked: drafts are never persisted by the agent).

Everything the vocabulary endpoint serves already exists in registries: `ToolDef.name/description/input_schema/safe`, `FunctionDef.name/description/input_schema` (added to plan 7a at M8 planning time), recipe operators/step shapes (`services/automations/recipe.py`), observed event types (`GET /api/events/facets` pattern), model options (`settings.chat_model`/`fast_model`).

**Non-goals**: templates gallery (user-locked: none); chat/MCP drafting tools (future); constrained per-stage builders (M9/M10 — they reuse these components); drag-and-drop (up/down buttons); client-side schema validation beyond required/type hints (the server 422s with plain messages — display them, don't duplicate the validator).

## Key design decisions (locked in — do not re-litigate during build)

1. **`GET /api/automations/vocabulary`** (router addition, `tenant_conn`): one payload the whole builder renders from —
   ```json
   {
     "triggers": {"event_types": [...], "source_systems": [...]},   // union of core-known types + observed events facets, automation-sourced excluded
     "tools": [{"name","label","description","input_schema","safe"}],
     "functions": [{"name","description","input_schema"}],
     "operators": ["eq","neq","gt","gte","lt","lte","contains","not_contains","exists","not_exists"],
     "generate_models": ["default","fast"],
     "field_roots": ["trigger","entity","context"]
   }
   ```
   `label` = the plain-language tool label (reuse chat's `TOOL_LABELS` map — lift it from `chat_service.py` into the tool layer if needed rather than duplicating). *Why*: the builder must not hardcode the vocabulary — M9/M10 add functions/tools and they must simply appear.
2. **`POST /api/automations/draft {"description": str}`** (`services/automations/draft.py`): Anthropic call (`settings.chat_model`) with a system prompt embedding the vocabulary (D1's payload), the recipe JSON shapes, and drafting rules (prefer gated tools for outreach, always name/describe the automation plainly, template with `{{…}}` paths); **Pydantic-validated structured output** (`AutomationDraft{name, description, trigger, conditions, steps, explanation}`) via a forced tool-use/structured-output call per the CLAUDE.md Pydantic rule; the result additionally passes `validate_recipe()`. On validation failure: **one retry** with the error message appended; still failing ⇒ 422 with a plain "couldn't draft that — try describing it differently" (+ technical detail in the payload). Traced as an `automation_draft` chain span. **Never writes to the DB** (user-locked); returns the draft for the builder. 503 without `ANTHROPIC_API_KEY`.
3. **Builder pages** (`AutomationBuilderPage.tsx`, routes `/automations/new` + `/automations/{id}/edit`): full-page layout per the user-locked shape —
   - **WHEN**: sentence line with chips — trigger type select (event/cron/manual); event ⇒ event-type + optional source-system selects from vocabulary; cron ⇒ expression input with next-3-fires preview (client-side croniter-alike is overkill — server already validates; preview via a tiny `GET /api/automations/cron-preview?expr=` helper added to the router, 422 on bad expressions, reusing croniter).
   - **IF**: `ConditionChips` (8a's component, edit mode on): rows of field-path input (with root suggestions from `field_roots`) / operator select / value input; add/remove; renders as sentence chips.
   - **THEN**: `StepList` of `StepCard`s (8a's components, edit mode): add-step menu (5 types), per-type forms — tool: tool select (grouped safe/gated, gated shows the amber chip) then `SchemaForm` fields; delay: number + unit; condition: nested `ConditionChips` (stop-if-false explained inline); function: select + `SchemaForm`; generate: prompt textarea + `save_as` + model toggle. Reorder via up/down; remove with confirm when the form is dirty.
   - **`SchemaForm.tsx`**: renders JSON Schema `properties` → string/number/boolean/enum inputs (required markers from `required`); every text field gets `TemplateInsert` — a popover inserting `{{trigger.payload.…}}`/`{{entity.…}}`/`{{context.<save_as>}}` paths (context suggestions from earlier steps' `save_as` values). Unknown schema constructs degrade to a raw-JSON field, never a crash.
   - Footer: live `describeRecipe()` sentence preview, Save (create ⇒ `paused` + toast linking to detail; edit ⇒ PATCH), Save & activate. Server 422 messages render inline at the top — the server is the validator of record.
4. **Edit mode + guard UX**: `/automations/{id}/edit` loads the definition into the same state; a 409 from 8a's edit guard renders a banner ("N runs in flight") with buttons: view runs (detail page) / cancel runs (8a's endpoint, confirm first, then retry save).
5. **Draft flow** (`DraftBox.tsx` on `/automations/new`): a prominent "Describe what you want to automate" textarea + Draft button → `POST /draft` (loading state ~seconds) → on success the builder state is **replaced** by the draft (confirm if the form was dirty), with the agent's `explanation` shown in an info banner ("review before saving — nothing is created yet"); the user edits/saves through the normal path. On 422, the plain failure message + keep typed description for retry.
6. **No new frontend deps** — selects/popovers from existing shadcn-style primitives; no form library (single-document state in the page, prop-drilled to cards).

## Execution notes

- **Blocking user input**: none new — drafting and the live walk need the standing `ANTHROPIC_API_KEY` (+ `NEXUS_WEBHOOK_SECRET`, office user) already required by M7's live walk.
- Reuse: structured-output pattern — follow the CLAUDE.md Pydantic rule with a forced tool-choice call (see how `chat_service.py` builds Anthropic requests; keep `draft.py` self-contained otherwise); `TOOL_LABELS` (lift, don't copy); facets query shape (`routers/events.py`); 8a's read-mode components and `lib/recipe.ts`; monkeypatched-Anthropic test pattern (search existing tests for anthropic mocking before inventing one).
- The vocabulary endpoint is the M9/M10 seam: constrained builders will filter this same payload — resist baking any vertical assumptions (no "lead" specifics anywhere in 8b code).
- Scope discipline: no templates, no chat integration, no dnd; do not re-implement `validate_recipe` client-side — required/type hints only, server messages are the validator; `SchemaForm` must degrade gracefully on schema constructs it doesn't know (raw JSON field), never block a save the server would accept.
- Non-technical-user rule: sentence preview, tool labels, 422 messages, and the draft explanation are plain language; recipe JSON stays behind the technical toggle.
