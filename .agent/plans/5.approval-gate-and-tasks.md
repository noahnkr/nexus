# Plan 5: Approval Gate & Task System (parent)

**Complexity: 🔴 Complex** — CLAUDE.md defaults Module 5 to complex. It spans two concerns with a hard dependency between them: the backend gate mechanics (queue instead of refuse, approval-triggered execution, the first write tools) and the Tasks interface (PRD interface #4) that humans use to clear the queue. Per the planning rule it is **broken into two sub-plans**:

- **`5a.approval-gate-backend.md`** (⚠️ Medium) — gate path in `execute_tool`, approval/rejection execution engine, first write tools (entity writes, placeholder sends, safe `create_task`), Tasks & approvals API.
- **`5b.tasks-interface.md`** (⚠️ Medium) — Tasks page with approval cards, manual task creation, status transitions, Realtime live updates; chat surfaces "queued for approval" distinctly.

Build order: **5a then 5b** — unlike Module 3's independent halves, 5b consumes 5a's API and cannot start meaningfully before 5a's Task 5 (the routers) exists. If 5a stalls mid-way, its Tasks-API task is a clean fallback split point (the gate works headless via curl before any UI exists).

## Context

Modules 0–4 delivered everything this module hangs off:

- `tasks` and `pending_actions` tables already exist (Module 0, `20260715014924_core_tables.sql`) with the right status vocabularies (`pending/approved/rejected/executed/failed` on actions) and full 4-policy RLS — Module 5 adds three columns and Realtime publication, no new tables.
- `execute_tool()` (`backend/app/services/tools/core.py`) is the single audited seam; it currently **hard-refuses** unsafe tools with a "Module 5 isn't here yet" message. This module replaces that branch with the queue path. No tool is unsafe today — this module ships the first write tools.
- `resolution.py` already creates review tasks (the pattern to mirror); the Event Log (Module 4) will display the new gate-lifecycle events with zero changes because writers set `payload.summary`.

**User-locked decisions (2026-07-16):**
1. **Write tools = entity writes + placeholder sends.** Vertical-seam entity write tools (`update_lead_status`, `update_client_status`, `create_schedule`, `cancel_schedule`) plus core gated `send_sms`/`send_email` whose *execution* is a log-only placeholder until real GoTo/Gmail credentials arrive — mirrors Module 3's placeholder-adapter approach and proves the gate on genuinely external actions.
2. **Full simple task management** on the Tasks page: list + filters, status transitions, manual creation, originating-event drill-down into the Event Log, and inline approve/reject cards.
3. **Safe `create_task` tool** — the agent can create internal coordination tasks immediately (no external effect ⇒ `safe=True`), from chat and MCP alike.

**Non-goals** (name the module): unified needs-attention queue, auth, and `resolved_by` as a real user identity (M6 — free-text/null until then); n8n workflow-created tasks and real outbound send credentials (M7); fuzzy matching / decision harness that *feeds* this gate with ambiguous cases (M8); business-specific task views (M9, out of repo); no pending-action expiry/TTL this phase (age is surfaced in the UI instead); no pausing a chat turn to await approval — the turn ends with "queued", approval executes later, out-of-band (PRD: "approval triggers real execution").

## Shared conventions (both sub-plans)

- Gate lifecycle event types (core vocabulary, written via `log_event` with `payload.summary`): `action.queued` (written before the task, which points back at it via `originating_event_id`), `action.approved` and `action.rejected` (linked `entity_type='task'`). Approved execution *additionally* writes the standard `tool.called` row by routing through `execute_tool` with the approved-action bypass — handlers are never called directly (CLAUDE.md seam rule holds).
- A queued gated call is **not an error**: the model receives a successful `tool_result` saying the action is queued (with task id), so it tells the user plainly; `is_error=True` stays reserved for real failures.
- Plain-language everywhere: gated task titles come from each tool's `gate_describe(conn, args)` (async, may look up entity names); raw `tool_input` JSON appears only in expandable technical detail, never in summary lines.
- No new env vars. One migration (Realtime publication + three `pending_actions` columns).
