# Plan 8a: Automations Center — grid, run history & management

**Complexity: ⚠️ Medium** — thin backend additions to Module 7's API plus two new pages composed from established patterns (cards + Realtime refetch + drawer). The only stateful subtlety is run cancellation interacting with a pending approval. Fallback split point: Task 1 (backend, curl-testable) vs Tasks 2–4 (frontend).

Parent: `.agent/plans/8.automations-center.md`. Requires Module 7 (7a + 7b) built.

## Context

After Module 7, automations and runs are managed only via curl. This sub-plan ships the management surface: see every automation at a glance, pause/resume/delete, inspect run history with a per-step timeline (`automation_runs.step_log`, added to the 7a plan for exactly this), and cancel stuck runs. Creation/editing UI is 8b — this page's "New automation" affordance lands there.

**Non-goals**: builder pages, vocabulary/draft endpoints (8b); metrics beyond counts (M9/M10); bulk operations; run retry (a failed run is terminal — re-trigger instead).

## Key design decisions (locked in — do not re-litigate during build)

1. **`POST /api/automation-runs/{id}/cancel`** (router addition): allowed while `running`/`waiting`/`waiting_approval`; 409 on terminal states, 404 unknown. If the run is `waiting_approval` with a still-`pending` action, cancellation goes **through `approvals.reject_action`** (note "automation run cancelled by user") — its 7b hook cancels the run, resolving action + task + run through the one sanctioned seam. Otherwise the router flips the run to `cancelled` (+`finished_at`) and writes `automation.run_cancelled` with a plain summary. `resolved_by`/canceller from `get_current_user`. *Why*: never leave an orphaned pending approval behind a dead run, and never bypass the approvals seam.
2. **Definition-edit guard** (change inside the existing PATCH): editing `trigger`/`conditions`/`steps` while active runs exist (`running`/`waiting`/`waiting_approval`) ⇒ **409** with a plain message ("N runs are in flight — cancel them or let them finish"). `name`/`description`/`status` edits are always allowed. *Why*: 7a's engine reads steps from the row at each advance; guarding edits is cheaper and clearer than run-against-snapshot versioning.
3. **List enrichment** (existing `GET /api/automations`): each row gains `active_runs` (count in the three active states) and `last_run` (`{status, finished_at|created_at}` via a lateral join) — what the grid card renders. Keyset/limit semantics unchanged.
4. **Home summary extension**: `GET /api/home/summary` gains `automations: {active, runs_today}` (active-status automations; runs created today); `HomePage` gains one `StatCard` (Zap icon, semantic tone by health: info normally, warning if any run `failed` today — derived client-side from a `failed_today` count included in the payload). Fulfils the M6 "widget grid extends in M8" note.
5. **`/automations` grid page** (`AutomationsPage.tsx` + `AutomationCard.tsx`): responsive card grid (the Home widget-grid feel, not a table). Card: name, status pill (active = success tone, paused = muted), plain-language trigger line from `describeRecipe()` (`lib/recipe.ts` — shared with 8b: "When a lead is created from WelcomeHome · 4 steps · requires approval"), amber approval chip when any step's tool is gated (tool safety comes from the 8b vocabulary endpoint later — until then derive from a hardcoded gated-tool name list? **No**: include `requires_approval: bool` in D3's list enrichment, computed server-side against the registry), `active_runs` badge, last-run line with relative time, pause/resume switch (PATCH status, optimistic), overflow menu: View (detail), Edit (routes to 8b's page — render disabled with a "coming in 8b" tooltip until it exists), Delete (confirm dialog → DELETE). "New automation" button routes to `/automations/new` (8b; disabled placeholder until then). Nav entry: **Automations** (Zap icon) between Tasks and Ingestion.
6. **`/automations/{id}` detail page** (`AutomationDetailPage.tsx`): header (name, status pill, pause/resume, Edit/Delete), a read-only recipe summary (sentence line + step list *rendered by the same components 8b will make editable* — build them display-first here: `TriggerSentence`, `ConditionChips`, `StepCard` in read mode), raw recipe JSON behind an expandable technical toggle, and **Runs**: newest-first list (`RunList.tsx` — status pill, started/finished relative times, error line when failed, trigger source), row click opens `RunTimeline.tsx` drawer: `step_log` entries as a vertical timeline (index, plain summary, status icon, timestamp), the run `context` behind a technical expander, and a **Cancel run** button for active states (D1).
7. **Realtime**: subscribe `postgres_changes` on `automations` + `automation_runs` (both already in the publication, 7a) with the `TasksPage.tsx` debounced-refetch pattern — grid counts and run lists move live while the engine works.
8. **No new frontend deps; no schema changes.** All backend work is router/schema-level on M7's tables.
