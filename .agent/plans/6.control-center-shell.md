# Plan 6: Control Center Shell & Visual Overhaul (parent)

**Complexity: 🔴 Complex** — Module 6 is not on CLAUDE.md's default-complex list, but it spans two heavy concerns with a hard dependency between them: real auth (a change that touches every API route, every API test, and every frontend fetch) and a full visual overhaul of the shell plus all four existing views. Per the planning rule it is **broken into two sub-plans**:

- **`6a.supabase-auth.md`** (⚠️ Medium) — Supabase Auth end to end: backend JWT verification replacing the env-tenant seam, machine-path tenant seam for webhooks/MCP, login page + session plumbing + authenticated fetch, Realtime via the real session, `resolved_by` becomes a real identity.
- **`6b.shell-home-overhaul.md`** (⚠️ Medium) — the Control Center shell redesign, the new Home landing page (light widgets, not a queue), route restructure (Home at `/`, Chat at `/chat`), chat markdown + streaming QoL, and a polish sweep across Ingestion/Tasks/Event Log. Uses the frontend-design plugin.

Build order: **6a then 6b** — 6b's shell renders the signed-in user and sign-out, and its Home page sits behind 6a's route guard. If 6a stalls, its backend half (Tasks 1–3) is a clean fallback split point: the API is JWT-protected and testable via curl/pytest before any frontend auth exists.

## Context

Modules 0–5 delivered a working app with four views (Chat, Ingestion, Tasks, Event Log) behind **no auth at all**: the backend trusts `NEXUS_TENANT_ID` from env (`deps.get_tenant_id()`), the frontend mints Realtime tokens through a dev seam (`GET /api/auth/realtime-token`), and every file involved carries a "Module 6 replaces this" comment. The groundwork is already in place:

- `app.current_tenant_id()` (Module 0) resolves tenant from the JWT claim `app_metadata.tenant_id` *first*, GUC second — RLS already understands real Supabase tokens.
- The test harness (`conftest.mint_tenant_jwt`) mints HS256 tokens with `SUPABASE_JWT_SECRET`; PostgREST and Realtime accept them, so the project's **legacy HS256 secret is still trusted**.
- Verified empirically (2026-07-16): the hosted project's JWKS (`/auth/v1/.well-known/jwks.json`) serves an **ES256 signing key** — real Supabase-Auth-issued access tokens are ES256-signed, so backend verification must handle both ES256 (JWKS) and HS256 (secret; test harness + Realtime seam heritage).
- `mcp_server.py:57` calls `get_tenant_id()` as a plain function and `routers/webhooks.py:35` uses it as a dependency — both are machine paths with their own credentials (static bearer / HMAC signature) and must **not** switch to user JWTs.

**User-locked decisions (2026-07-16):**
1. **Home lands at `/`; Chat moves to `/chat`.** The Home page must feel like a *home*, not a duplicated needs-attention queue — Tasks already serves that purpose. Light at-a-glance widgets (counts, recent activity, quick actions) laid out as a widget grid that future modules can extend; deliberately low functionality this phase.
2. **Email + password** sign-in (Supabase Auth). One office user created via dashboard with `app_metadata.tenant_id` set — a one-time ops step.
3. **`/mcp` keeps the static bearer token** (`NEXUS_MCP_TOKEN`). MCP consumers are machines (Claude clients now, n8n nodes in M7); per-client OAuth is deferred until a real external-client need exists.

**Non-goals** (name the module): needs-attention queue on Home (deferred to a future Home-widget pass once M7/M8 create more signal; Tasks stays the queue); Settings interface (config stays env-var per PRD Out of Scope); multi-user management, roles, invites, password-reset flows (dashboard-managed; multi-tenant admin tooling is explicitly post-validation); n8n embedding (M7); MCP OAuth (future need-driven); retrieval changes (M10); business-specific views (M9, out of repo).

## Shared conventions (both sub-plans)

- **Fail closed everywhere**: no valid bearer token ⇒ 401 on every `/api` route except the three sanctioned openings — `/healthz`, `/api/webhooks/{source}` (HMAC-verified), `/mcp` (static bearer). A valid token without `app_metadata.tenant_id` ⇒ 403.
- The tenant seam splits in two, both in `deps.py`: `get_tenant_id` (request-scoped, verified JWT — the user surface) and `get_machine_tenant_id` (env `NEXUS_TENANT_ID` — webhooks + MCP only). No other code reads `settings.nexus_tenant_id` at request time.
- Frontend: **every** `/api` call carries `Authorization: Bearer <session access token>` via one shared helper; Supabase Realtime rides the real session (supabase-js auto-forwards it) — the minted-token dev seam is deleted, not kept as fallback.
- Visual overhaul keeps the shadcn/Tailwind token system (`index.css` HSL variables, light + dark) — restyle by changing token values and components, not by introducing a second styling system. Load the `frontend-design` skill before the design tasks.
- No new core tables, no entity-schema changes. One tiny read-only endpoint (`GET /api/home/summary`) and no new env vars.
