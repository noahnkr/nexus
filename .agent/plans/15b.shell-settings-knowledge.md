**Complexity: ⚠️ Medium** — one small core migration (`tenant_settings`), one new API pair, one prompt-injection seam, and a broad-but-mechanical responsive/navigation sweep. No engine, tool, or approval changes. **Fallback split point**: Tasks 1–2 (settings backend + injection) stand alone; Tasks 3–6 (shell/pages) are a clean second pass.

Parent: `.agent/plans/15.finishing-touches.md`. Build after 15a (the mobile sweep in Task 6 must cover 15a's task drawer).

## Context

There is no per-tenant settings storage anywhere — config is env-only (CLAUDE.md: infra config stays env; this module adds *user-facing preferences*, a different thing). The chat system prompt is the hardcoded `PERSONA` in `services/chat_service.py` (one cached system block; tools carry the second cache breakpoint). The sidebar is a fixed `w-60` rail in `AppShell.tsx` with no collapse and no mobile affordance; `UserMenu` has theme + sign-out only. The Ingestion page is an upload dropzone + document table; the documents API already supports detail (`GET /api/documents/{id}` returns chunk previews) and delete — the UI just never exposes them.

**User-locked (2026-07-19)**: Ingestion renames to **Knowledge** with Documents + Instructions tabs; instructions are per-tenant (new core table) and injected into the chat prompt; mobile covers the core pages with a drawer nav, while the schedule board and builder stay desktop-first with horizontal scroll.

## Key design decisions (locked in — do not re-litigate during build)

1. **Migration `20260726000000_core_tenant_settings.sql`** (CORE, not vertical):
   ```sql
   create table public.tenant_settings (
     id uuid primary key default gen_random_uuid(),
     tenant_id uuid not null unique references public.tenants(id) on delete cascade,
     settings jsonb not null default '{}'::jsonb,
     created_at timestamptz not null default now(),
     updated_at timestamptz not null default now()
   );
   ```
   plus the standard 4-policy tenant RLS and the shared `set_updated_at` trigger. One row per tenant (`unique`), lazily created on first write. Justification: a single jsonb row keeps the schema stable as preference keys evolve; the whitelist lives in code.
2. **Settings seam `services/settings.py`**: `SETTINGS_KEYS` whitelist with per-key validation — v1 keys: `workspace_name` (str ≤ 80), `agent_instructions` (str ≤ 4000), `agent_tone` (one of `"balanced" | "professional" | "friendly" | "concise"`, default `"balanced"`). `get_settings(conn) -> dict` returns defaults merged with the stored row; `update_settings(conn, tenant_id, patch) -> dict` upserts, rejects unknown keys/invalid values with plain messages, and logs a `settings.updated` event whose `payload.summary` names the changed keys ("Settings updated: agent instructions, tone") — **never the values** (instructions may contain private detail; the technical payload also stores only the key list). Justification: whitelist-in-code keeps the jsonb from becoming a junk drawer; key-only audit keeps the event log clean.
3. **API `routers/settings.py`**: `GET /api/settings` and `PATCH /api/settings` (partial update), standard `tenant_conn`/`get_tenant_id` deps, 422 on validation failures. No machine access — user JWT only.
4. **Chat injection**: `stream_chat_turn` loads settings once per turn; when `agent_instructions` is non-empty (or tone ≠ balanced) it appends a **second** system block after PERSONA: a framed preamble ("The business owner has customized how you should respond. Follow these preferences where they don't conflict with the rules above:") + tone sentence + instructions text. `cache_control` moves to the last system block (the API caches the prefix; two text blocks + tools stays within the 4-breakpoint budget). PERSONA remains first and unmodified — instructions can shape tone/content, never the gating rules (parent decision 2). Justification: cheap, cacheable, and safety-ordered.
5. **Sidebar collapse**: `AppShell` gains a `collapsed` state persisted to `localStorage("nexus.sidebar")` — collapsed = `w-14` icon rail (nav items icon-only with `title` tooltips, brand mark only, `UserMenu` collapses to the avatar button). Toggle button (PanelLeftClose/PanelLeftOpen) in the sidebar header. Desktop only (≥ md).
6. **Mobile shell (< md)**: the sidebar becomes an overlay drawer — a slim top bar (h-12: hamburger + brand + `UserMenu` avatar) replaces it; the drawer slides over content with a scrim, closes on nav. Pages keep working unchanged; the shell is the only structural change. Use Tailwind's `md:` variants — no JS media-query library.
7. **`/settings` page**, linked from a new "Settings" (gear) item in the `UserMenu` popover: sections — **Profile** (display name → `supabase.auth.updateUser({ data: { display_name } })`, email read-only, change password with confirm field via `updateUser({ password })`, plain error surfacing), **Workspace** (`workspace_name` → PATCH `/api/settings`; used in the Home greeting when set), **Agent** (a pointer card linking to `/knowledge?tab=instructions` — the editor lives there per the user's choice), **Appearance** (theme via the existing `useTheme`). No new backend for profile — supabase-js handles it.
8. **Knowledge page**: route `/knowledge` (redirect `/ingestion` → `/knowledge` for old links), nav item renamed **Knowledge** (BookOpen icon), `?tab=documents|instructions` (default documents).
   - **Documents tab**: existing dropzone + table, plus a row-click `DocumentDrawer` — filename, status/error, created/updated, chunk count + first chunk previews (from `GET /api/documents/{id}`), and Delete with confirm (existing `DELETE`, toast + Realtime refresh).
   - **Instructions tab**: `agent_instructions` textarea (char counter, Save button + saved toast), `agent_tone` via `ui/Select`, and a muted explainer of exactly where these apply ("Applied to every chat response. Documents remain the assistant's knowledge; instructions shape its behavior."). Saves via PATCH `/api/settings`.
9. **Mobile sweep = layout, not redesign**: each core page (Home, Chat, Tasks, Leads, Caregivers, Event Log, Settings, Knowledge) gets small-screen passes — grids collapse to single column, filter rows wrap, tables get `overflow-x-auto` wrappers, the chat thread list becomes a collapsible panel, drawers go full-width, and full-height pages use `dvh`-safe heights. Schedule board + builder: guarantee horizontal scroll containers and nothing more.
