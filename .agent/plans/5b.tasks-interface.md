# Plan 5b: Tasks Interface — approval UI, task management, chat surfacing

**Complexity: ⚠️ Medium** — one new page with four components, an additive SSE field, and a Realtime subscription reusing the established pattern. No migrations, no new env vars. Depends on 5a's API being built first.

Parent: `.agent/plans/5.approval-gate-and-tasks.md`. Backend contracts this plan consumes are locked in `.agent/plans/5a.approval-gate-backend.md` D9 (endpoints) and D5 (approve/reject semantics).

## Context

After 5a: gated tool calls queue as `tasks` (+ `pending_actions`), the approvals engine executes on approve, the Tasks API exists, and `tasks`/`pending_actions` are in the Realtime publication. The frontend has an established shell (`AppShell` nav — Chat / Ingestion / Event Log), the Realtime token/channel pattern (`IngestionPage`, `EventLogPage`), the opaque-cursor "Load more" pattern, and chat tool chips (`ToolActivity.tsx`) fed by SSE `tool`/`tool_result` events.

User-locked (see parent): **full simple task management** — list + filters, status transitions, manual creation, originating-event drill-down, inline approve/reject cards.

**Non-goals**: needs-attention aggregation across surfaces and any nav rework (M6 Control Center shell); assignee identity/auth (`resolved_by` stays anonymous until M6); task editing beyond status (title/description immutable this phase); no pause-and-resume of a chat turn on approval — approval outcomes surface in Tasks/Event Log, not back into the thread.

## Key design decisions (locked in — do not re-litigate during build)

1. **Route `/tasks`, nav item "Tasks"** (ListTodo icon) between Ingestion and Event Log in `AppShell` — M6 reworks the shell; this is just a fourth entry, exactly like Module 4 added Event Log.
2. **`lib/api.ts` additions**: `listTasks(params)`, `createTask(body)`, `patchTask(id, status)`, `approveAction(id)`, `rejectAction(id, note?)` + `Task`/`PendingAction` types mirroring 5a's `TaskOut`/`PendingActionOut`.
3. **`pages/TasksPage.tsx`** — status tabs ("Open" = `pending,in_progress`, default; "Done"; "Cancelled"; "All") + priority select, both ↔ URL query params (same pattern as `EventLogPage`); "Load more" via `next_cursor`; "New task" button opening `CreateTaskDialog`. Realtime: one channel subscribing to `postgres_changes` INSERT+UPDATE on `tasks` **and** `pending_actions` (reuse the `EventLogPage` token pattern); on any change, refetch the first page if the change matches the active tab (tasks are mutable, so refetch-on-signal beats client-side row patching — simpler and always consistent; fine at this scale).
4. **`components/tasks/TaskCard.tsx`** — title, plain-language description, priority + status badges, relative age, due date when set; status-transition buttons per 5a's allowed transitions (Start / Done / Cancel; hidden on terminal states, and Done/Cancel disabled with a hint while an action is `pending` — the API 409s anyway, the UI just explains); "View history" link → `/events?entity_type=task&entity_id={id}` and, when `originating_event_id` is set, a link to the originating event's drill-down (the Module 4 URL-addressable filters do the work).
5. **`components/tasks/ApprovalCard.tsx`** — rendered inside a TaskCard when it has pending_actions: tool label (reuse the plain-language naming approach of chat's `TOOL_LABELS` — a small local map, no raw tool names for known tools), the task's plain description as the "what will happen" line, Approve / Reject buttons (Reject opens an optional-note input); while resolving, buttons disable; on response, show the outcome inline — `result.summary` on success, `result.error` plainly phrased on failure; resolved actions render as a muted outcome line (Approved → executed / failed, Rejected + note). Raw `tool_input` JSON lives **only** behind an expandable "Technical detail" toggle (pretty-printed, same as `EventRow`).
6. **`components/tasks/TaskFilters.tsx`** (tabs + priority select) and **`CreateTaskDialog.tsx`** (title required, description, priority select defaulting `normal`, optional due date) — reuse existing `ui/` primitives (`button`, `input`, `textarea`, `badge`, `card`); add a minimal `ui/dialog.tsx` (or inline overlay) in the existing shadcn style if none exists.
7. **Chat surfacing of queued actions**: `chat_service.py` adds an additive `queued: true` field to the `tool_result` SSE event (and to `metadata.tool_calls` entries for history reload) when the result data is `{"status": "queued", ...}` — the SSE contract stays additive, exactly like Module 2's extension. `ToolActivity.tsx` styles queued chips distinctly (clock icon, amber) and links to `/tasks`; the model's own text already explains the queue per 5a's PERSONA. No other SSE change.
