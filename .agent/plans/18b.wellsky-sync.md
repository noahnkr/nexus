# Plan 18b: WellSky Personal Care Sync

**Complexity: ⚠️ Medium** — one connector, but it feeds three vertical surfaces (M12 board, M15 clients/census/EVV, M17 roster) and adds document ingestion; everything builds and tests offline against recorded fixtures because **no API credentials exist yet**. **Fallback split**: Tasks 1–3 (client + people sync) stand alone; Tasks 4–5 (schedules + EVV) are the second pass; Task 6 (documents) is cleanly severable.

See the parent plan `.agent/plans/18.external-connectors.md` for shared conventions (single ingest seam, sync loop, secrets-in-env, fixtures discipline). **Requires 18a Tasks 1–4 built** (sync loop `services/connectors/sync.py`, ingest seam `ingest_payload`, `NormalizedEvent.updates_entity` + `UPDATERS` resolution update path). This plan supersedes the 2026-07-18 WellSky-direct deferral (user decision 2026-07-20): the M12/M15/M17 surfaces built since the deferral need line-of-business data the WelcomeHome sync cannot carry.

## Context

WellSky Personal Care (formerly ClearCare) is the line-of-business system: active clients, the caregiver roster, scheduled shifts, and EVV clock data live there. The **Connect API** was researched empirically 2026-07-20 from the public spec (`https://apidocs.clearcareonline.com/` → ReDoc over `swagger.yaml`; spec + key sub-files archived in the planning session):

- **Base**: `https://connect.clearcareonline.com/v1/`. Quirk: list/create endpoints **require a trailing slash** (`GET /v1/patients/` works, `/v1/patients` 404s); GET-by-id with query params must **omit** it.
- **Auth**: `POST /oauth/accesstoken`, JSON body `{client_id, client_secret, grant_type: "client_credentials"}` → short-lived `access_token` (+ `expires_in`); **no refresh tokens** — re-request on expiry. Bearer header on every call. Credentials are partner-gated (granted by a WellSky account rep) — **not yet in hand; every live validation in this plan is credential-gated**.
- **Patients** (= prospects *and* active clients): `POST /v1/patients/_search/` supports `created`/`updated` filters with `lt/gt/le/ge/eq` prefixes (e.g. `{"updated": "ge2026-07-01T00:00"}`) plus `_count`/`_page` pagination and `totalRecords` → **incremental watermark polling works**. GET-by-id returns the full resource (FHIR-ish: HumanName, telecom, address, `communication` languages, deactivation/deceased fields).
- **Practitioners** (= caregivers *and* applicants): same `_search` watermark + pagination; `is_hired` search param separates the roster from applicants.
- **Appointment** (scheduled shifts): `POST /v1/appointment/_search/` **has no modified-since** — requires (`clientId` or `caregiverId`) + (`startDate` [+`additionalDays` ≤6] or `weekNo` YYYYWW or `monthNo` YYYYMM). Shift sync must sweep per-entity time windows and diff.
- **Encounter** (care logs — actual shifts with clock data; the EVV source M15 anticipated): `POST /v1/encounter/_search/` by `startDate`/`endDate`/`clientId`/`caregiverId` + `_count`/`_page`. Clock-in/out **write** endpoints exist but are out of scope (one-way inbound).
- **RelatedPerson**: patient family contacts (`/v1/relatedperson/{patient_id}` etc.) → `client_contacts`.
- **DocumentReference**: client files with download URLs; search-by-profile `POST /v1/documentReferences/_profile/` body `{"reference": "Patient/##"}`.
- **Subscriptions**: webhook notifications with a **minimal payload** `{"resourceType": "Patient", "id": 3675284}` needing a fetch-back, delivered to a client-provided public HTTPS URL — unusable on localhost; **polling via the 18a sync loop is the mechanism now**, subscriptions are the documented deployed-mode upgrade (the existing `adapters/wellsky.py` webhook path stays for it).
- **Certificates API is commented out of the spec (disabled)** → M17 credential data **cannot** come from WellSky; credentials stay human-entered. Not a scope decision — an API fact.
- **Rate limits**: soft — ≤100 req/s, "don't use the API for batch operations". Our scale (tens of clients/caregivers, 120s cycles) is far below any limit; no pacing machinery needed (unlike 18a's 3/min cursor rule).

**User-locked scope (2026-07-20)**: active clients only (prospect-status patients skipped — leads stay WelcomeHome's domain per 18a); hired caregivers only (`is_hired` — the M10 pipeline stays the only writer of hiring stages); **full schedule sync** — appointments into `schedules` + encounter clock data into the M15 EVV columns, WellSky authoritative for the board, with the divergence risk documented (board actions in Nexus are local-only, no write-back); **DocumentReference ingestion included** — new/changed client files download into RAG entity-tagged to the client.

**Non-goals** (name the destination): write-back of any kind incl. clock-in/out POSTs (future module on a real driver); WellSky prospects → leads (WelcomeHome's domain, 18a); applicants sync (M10 owns hiring); Subscriptions webhook mode (deployed-mode future, documented in the adapter); CarePlan/Condition/Goal clinical resources → structured care plans (Future Plans, with M15's structured-care-plan editor); credential sync (impossible — API disabled); Invoice/ChargeItem billing data (Future Plans, with M15's billing export).

## Key design decisions (locked in — do not re-litigate during build)

1. **WS client** (`ws_client.py`): httpx-based. In-process token cache — `POST /oauth/accesstoken` on first use and whenever <60s of `expires_in` remain (client-credentials re-request, *not* a refresh-token flow — 18c's `oauth.py` helper doesn't apply). Surface: `search(resource, body, *, count=100, page=1)` (posts to `/v1/{resource}/_search/`, returns `(rows, total_records)`), `search_all(resource, body)` (async generator walking `_page` until exhausted), `get(resource, id)`, `token()`. Trailing-slash rules encoded once in URL construction. Timeouts + bounded retries on 429/5xx; typed `WellSkyError` the runner converts to `connector.sync_failed`. Justification: mirrors `wh_client.py` exactly, minus pagination pacing the API doesn't need.
2. **Settings**: `wellsky_client_id`, `wellsky_client_secret` (runner enabled only when both set), `wellsky_base_url` (default `https://connect.clearcareonline.com`), `wellsky_agency_id` (optional — passed as `agencyId` where the API validates it), `wellsky_schedule_horizon_weeks` (default 2 — appointment sweep reach), `wellsky_encounter_lookback_days` (default 7 — clock-data window), `wellsky_document_max_mb` (default 10). All optional; no runner registers without credentials (the 18a convention).
3. **Runner** (`ws_runner.py`), registered in 18a's `SyncRunner` registry, source `wellsky`. Per cycle, dependency order: **patients → related persons → practitioners → appointments → encounters → documents**. Cursor state on the `(tenant, 'wellsky')` `connector_state` row: `{"watermarks": {"patients": iso, "practitioners": iso, "relatedperson": iso, "documents": iso}, "windows": {"appointments_swept_through": iso, "encounters_swept_through": iso}, "documents_ingested": {file_id: updated_iso}}`. Watermark tables use `{"updated": "ge{watermark}"}` searches; each batch commits before its watermark advances (the 18a idempotency discipline). Every fetched resource flows through `ingest_payload("wellsky", resource, headers={})` — fetch → translate → same receipt → normalize → resolve path; **never a second inbound path**.
4. **Mapping** (`ws_map.py`): pure, fixture-tested functions producing `NormalizedEvent`s for the real `adapters/wellsky.py` normalize (decision 9). `map_patient` → client attributes (name from HumanName text, phone/email from telecom, address/zip, languages from `communication`, notes) and **skips non-active patients** (prospect-status/deceased/deactivated discriminator confirmed at build time against fixtures; deactivated/deceased patients that are already-linked clients map to a `discharged` status change instead of a skip). `map_practitioner` (hired only) → resource attributes. `map_appointment` → schedule attributes (WS client/caregiver ids as external refs, start/end, status). `map_encounter` → clock stamps + completion for the matched schedule. `map_related_person` → contact rows. Unknown/malformed rows degrade to ack-only or review tasks, never a crash.
5. **Cross-connector identity — link-or-create writers**: a WS patient may already exist as a Nexus client (18a's Start-of-Care promotion, or manual creation); same for board-seeded caregivers. The vertical writers (`entity_writers.py`) gain link-or-create discipline for `wellsky`-fed creates: exact normalized-phone match first, then exact case-insensitive name match, **among rows lacking an `ehr` external id** → link (return the existing entity id; resolution inserts the `external_ids` mapping as usual). No match → create. Multiple matches → the writer raises a typed ambiguity error the runner converts to a per-row review task + skip (no business write — the resolution `task` outcome's discipline). Justification: without this, every WelcomeHome-promoted client duplicates on first WellSky sync.
6. **Schedule single-writer rule holds**: `services/views/schedule.py` gains `sync_upsert_visit(conn, tenant_id, attrs, *, source_system)` — the **seam stays the only writer of schedule state**; the connector's schedule writer/updater delegates to it (CLAUDE.md's M12 rule extended, not bypassed). Semantics: participants resolved via `external_ids` (`ehr` category — unresolvable client/caregiver ⇒ review task, no write); create as `scheduled` (or `open` when WS shows no caregiver) with `schedule.created`; changed start/end/assignment ⇒ `schedule.updated`; WS cancellation ⇒ seam `cancel`. EVV: encounter clock stamps apply through the existing seam `check_in`/`check_out` (no-op when stamps are unchanged, so re-sweeps don't re-emit events); check-out completes the visit — census delivered-hours and read-time EVV flags then work on real data (M15's explicit anticipation: "connector-fed clock-ins land in these same columns via Module 18's ingest path").
7. **Window sweeps** (appointments/encounters have no modified-since): appointments — per synced client (`clientId` from the `ehr` external id), `weekNo` for the current week through `wellsky_schedule_horizon_weeks` ahead, every cycle; DB diff decides create/update/no-op, so an unchanged week emits nothing. Encounters — one windowed search per cycle covering the trailing `wellsky_encounter_lookback_days`, diffed the same way. Scale check: ~30 clients × 3 weeks ≈ 90 requests per 120s cycle — trivial against the API's guidance.
8. **Documents** (user-locked in): per synced client, `documentReferences` search-by-profile; files not in the `documents_ingested` ledger (or with a newer `updated`) download via the returned URL and enter the **existing ingestion pipeline** entity-tagged to the client (the M15 tag; chunks inherit — same path as 18a's `ingest_text`, but with real file bytes through the parser registry). Only supported formats (PDF/DOCX/HTML/MD/TXT) ≤ `wellsky_document_max_mb` are ingested; others are skipped with a logged note in the receipt event. The ledger lives in `connector_state` (documents aren't canonical entities, so `external_ids` is not the home; corpus is low-hundreds — jsonb is fine).
9. **Real adapter** (`adapters/wellsky.py`): placeholder internals replaced — `normalize` maps full resources by `resourceType` through `ws_map` (Patient → client create/update, Practitioner → resource create/update, Appointment/Encounter → schedule create/update, RelatedPerson → client-contact update on the matched client). The docstring documents Subscriptions fetch-back as the deployed-mode upgrade (minimal `{resourceType, id}` payload → `ws_client.get` → same normalize). Category stays `ehr` (`external_ids` CHECK already allows it — verified in the M0 migration). Justification: adapter-file internals replace, the seam never changes (the M3b promise).
10. **Events**: reuses existing types only — `client.created`/`client.updated`/`client.status_changed` (via the M15 seam's `change_status` for deactivation→discharged), `resource.created`/`resource.updated`, `schedule.created`/`schedule.updated`/`schedule.cancelled`/`schedule.checked_in`/`schedule.checked_out`, document lifecycle events from ingestion — all `source_system='wellsky'`, dispatcher-eligible. **No new event types, no `automations/entities.py` changes, no migration** (clients/resources/schedules already carry every column this sync writes — the M12/M15 schema was built for this). Frontend: `lib/events.ts` `SOURCE_ACCENT` gains `wellsky` alongside 18a's `welcomehome` entry.
11. **No new tools, no gate involvement**: this is pure inbound sync. Outbound effects stay with the existing gated tools; nothing here touches the approval gate or the tool registry.

## Tasks (strict order)

### Task 1 — Config + WS client + fixtures
Settings (decision 2); `ws_client.py` (decision 1); sanitized JSON fixtures modeled on the spec's response examples (patients/practitioners/appointment/encounter/relatedperson/documentReference — invented names only).
**Validation**: offline `test_ws_client.py` — token cache requests once and re-requests after simulated expiry; `search_all` walks a 3-page fixture set via `_page`/`totalRecords`; trailing-slash rules pinned (list URL ends with `/`, id-with-params doesn't); 429 retries then raises `WellSkyError`. Credential-gated `test_ws_live_token` (skipif no `WELLSKY_CLIENT_ID`) asserts a live token grant.

### Task 2 — Mapping
`ws_map.py` (decision 4) with the active-only / hired-only filters and the deactivation→discharge distinction.
**Validation**: offline `test_ws_map.py` — patient fixture → client attributes (name/phone/zip/languages), prospect-status fixture → skipped, deceased fixture → status change; practitioner `is_hired` filter; appointment → schedule attrs with external participant refs; encounter → clock stamps; related person → contact row.

### Task 3 — People sync: writers, link-or-create, runner watermarks
`entity_writers.py` link-or-create for client/resource + client-contact upsert (decision 5); `ws_runner.py` patients/relatedperson/practitioners stages with `updated ge` watermarks through `ingest_payload` (decision 3); real adapter normalize for these resource types (decision 9).
**Validation**: gated tests — a WS patient matching an existing promoted client by phone **links** (no duplicate row, `ehr` external id inserted); an unmatched patient creates with `client.created` from `wellsky`; ambiguous name match → review task, no write; two-cycle runner test over fixture pages — second cycle fetches only post-watermark rows, zero duplicate events; practitioner sync updates a board-seeded caregiver in place.

### Task 4 — Schedule seam extension + EVV
`views/schedule.py::sync_upsert_visit` (decision 6); schedule writer/updater delegating to it; encounter clock application via seam `check_in`/`check_out`.
**Validation**: gated `test_ws_schedule_sync.py` — appointment fixture creates a `scheduled` visit with resolved participants and `schedule.created`; re-applying the same fixture is a no-op (no second event); a moved start time emits one `schedule.updated`; unresolvable caregiver ⇒ review task and no row; encounter stamps clock-in/out exactly once across two sweeps, visit completes, and the M15 census `delivered` figure moves (delta-asserted, the M15 test convention).

### Task 5 — Window sweeps
Appointment per-client `weekNo` horizon sweep + encounter lookback sweep with DB diffing (decision 7); sweep markers in cursor state.
**Validation**: offline runner test — a two-cycle sweep over fixture weeks where cycle 2's fixtures move one shift and cancel another yields exactly one `schedule.updated` + one `schedule.cancelled` and nothing else; horizon setting respected (a shift beyond the horizon is untouched).

### Task 6 — Documents into RAG
DocumentReference sweep + download + ingestion entity-tagged to the client, with the `documents_ingested` ledger, format/size caps (decision 8).
**Validation**: offline test with a faked download — new file ingests (document row entity-tagged `client`, chunks stamped), unchanged file skips on the next cycle, oversized/unsupported files skip with the logged note; gated retrieval test — the ingested fixture text is retrievable via `search_documents` scoped to the tenant.

### Task 7 — Wrap-up
README WellSky section (env vars, scope table incl. the explicit not-synced list — prospects, applicants, credentials [API disabled], write-back; the board-divergence caveat; the subscriptions upgrade path); `.env.example`; `lib/events.ts` accent; full `pytest backend/tests` green; `npm run build` clean.
**Validation**: full pytest green offline. **Credential-gated live walk checklist** (recorded in the README, run when WellSky grants API access): live token; patient search returns the agency's clients; one full sync cycle populates clients/roster/board; a clock-out in WellSky lands `schedule.checked_out` within one poll and the census delivered-hours move; a client file appears in RAG; LangSmith shows the `connector_sync` cycle span with a `wellsky` child.
