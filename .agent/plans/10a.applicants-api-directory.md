# Plan 10a: Caregivers View — applicants model, API, directory & profiles

**Complexity: ⚠️ Medium** — a 1:1 re-instantiation of 9a's shape (migration → router → summary → directory → profile) with two additions that carry the real design weight: the new entity type threaded through every vertical seam, and the atomic hired→caregiver promotion. Fallback split point: Tasks 1–3 (backend, curl-testable) vs Tasks 4–5 (frontend).

Parent: `.agent/plans/10.caregivers-view.md`. Requires Module 9 built (9a's router/page patterns, `EntityTimeline`, `services/views/summary.py`, `StageSelect`/`SmartSummary` components). 10b builds on this sub-plan's API and page.

## Context

Caregiver recruiting has no system surface at all: `resources` is the active roster (no stage column), and applicants don't exist in the schema. This sub-plan stands up the applicants entity end-to-end — table, seams, tools, events, API, directory, profile — plus the promotion path onto the roster. The funnel strip, metrics, and sequences are 10b; this page ships with the plain stage chip-row filter that 10b replaces (the 9a precedent exactly).

**User-locked** (parent plan): applicants table + promotion; 5 stages + terminal `rejected`; automatic atomic promotion on `hired`; scoring deferred to M11.

**Non-goals**: metrics/funnel/sequences (10b); scoring (M11); webhook applicant ingestion (future connector work — no `applicant` entry in `connectors/entity_writers.py` this module); availability editing (read-only expander); roster UI beyond the hired banner; bulk import.

## Key design decisions (locked in — do not re-litigate during build)

1. **Migration `20260719000000_entities_applicants.sql`** (vertical seam file, additive to the M0 entity migration):
   - `applicants`: `id uuid pk`, `tenant_id` FK, `name text not null`, `phone`, `email`, `source`, `stage text not null default 'applied'` CHECK in (`applied`,`screening`,`interview`,`offer`,`hired`,`rejected`), `qualification_ids uuid[] default '{}'`, `region_ids uuid[] default '{}'`, `availability jsonb default '{}'`, `notes text`, `created_at`/`updated_at`; `(tenant_id, stage)` index; `set_updated_at` trigger; the standard four-policy RLS block (M0 precedent).
   - `alter table public.resources add column if not exists applicant_id uuid references public.applicants(id);` — promotion provenance, the `clients.lead_id` precedent applied to the promoted row.
   - `applicants` added to the `supabase_realtime` publication (guarded/idempotent).
   - `seed.sql`: a handful of applicants across stages (idempotent `on conflict (id) do update`, the schedule-seed precedent) so every later task has data.
   *Why quals/regions/availability on applicants*: promotion copies them verbatim onto the caregiver row, and recruiters capture them during screening anyway.
2. **Seam updates** (all existing vertical files):
   - `services/automations/entities.py`: `ENTITY_TABLES["applicant"] = "applicants"` (recipes can condition on `entity.stage`).
   - `services/tools/entities.py`: read tools `list_applicants` (stage/source/`q` filters, `_limit` cap) and `get_applicant` (qualification/region id arrays resolved to names via the existing `_name_maps`); **gated** `update_applicant_stage` (`safe=False`, `gate_describe` → "Move applicant 'X' to offer") whose handler delegates to `move_stage()` (D3) — never its own UPDATE. `APPLICANT_STAGES` constant beside the other status lists; `SQL_SCHEMA_DOC` gains the applicants table.
   - `services/tools/labels.py`: labels for the three new tools; `routers/automations.py` `_KNOWN_EVENT_TYPES` gains `applicant.created` + `applicant.stage_changed` (list already carries vertical names — precedent).
3. **`move_stage()` in `services/views/caregivers.py`** — the single stage-moving path (REST PATCH and tool handler both call it, on the caller's tenant-scoped conn/tx):
   - Validates the target stage; no-op returns without events.
   - Updates the row, emits `applicant.stage_changed` `{from, to, summary}` (plain: "Applicant 'X' moved from Interview to Offer") with the caller's `source_system`.
   - On `hired`, **iff no `resources` row already carries this `applicant_id`**: inserts the caregiver row (name/phone/email/qualification_ids/region_ids/availability copied), emits `resource.created` ("Applicant 'X' hired — caregiver record created"), all in the caller's transaction — atomic with the stage move. Moving out of `hired` never deletes the caregiver; re-entering `hired` with provenance set skips re-promotion (documented, tested).
   *Why one function*: one event emitter, one promotion path — chat/MCP-approved moves and UI moves are indistinguishable in the timeline and can't diverge.
4. **`routers/applicants.py`** (vertical seam, mirrors `routers/leads.py` shape): `GET /api/applicants` (filters `stage`/`source`/`q` over name/phone/email, `limit` default 50 cap 100 + `offset`, `{applicants, total}`, `created_at desc`); `GET /api/applicants/facets` (distinct sources + `{id,name}` regions + qualifications — the create/edit selectors); `POST /api/applicants` (name required, stage always `applied`, emits `applicant.created` `source_system='user'`, entity-linked plain summary); `GET /api/applicants/{id}` (quals/regions resolved to names); `PATCH /api/applicants/{id}` (basic fields + `qualification_ids`/`region_ids` + `notes` → one `applicant.updated` naming changed fields; `stage` → `move_stage()`); **no DELETE**. `GET /api/applicants/{id}/summary` → the view-agnostic helper `services/views/summary.py` (9a amendment) with a hiring-flavored prompt from `views/caregivers.py` (who they are, source, stage, what's happened, sensible next step); `{summary, generated_at}`, no persistence, 503 without key, `applicant_summary` trace span.
5. **`/caregivers` directory page** (`CaregiversPage.tsx`): `PageHeader` + "New applicant"; stage chip-row (All + six stages — 10b replaces with the funnel strip) + source select + search ↔ URL params; `ApplicantsTable` (name, stage badge, source, qualifications count, age, contact) with row click → `/caregivers/{id}`; `ApplicantCreateDialog` (name/phone/email/source + qualification/region multi-select checkboxes from facets); Realtime `postgres_changes` on `applicants` (debounced refetch); `EmptyState`. Nav: **Caregivers** (lucide `Users`) directly after Leads.
6. **`/caregivers/{id}` profile page** (`CaregiverProfilePage.tsx`, mirrors `LeadProfilePage`): `SmartSummary` card (reused component, pointed at the applicants summary endpoint); `ApplicantInfoCard` — inline-editable contact/source + qualification/region checkboxes (PATCH on save), `notes` textarea, availability jsonb behind a read-only technical expander; `StageSelect` (reused) with a **confirm dialog when the target is `hired`** ("This creates a caregiver record for X — continue?"), and afterwards a success-tone hired banner naming the created caregiver (from the PATCH response, which includes the promoted `resource_id`/name); `EntityTimeline` with `entityType="applicant"`. *Why a page*: deep-linkable, the M9/M10 pattern contract.

## Tasks

**Validation**: gated tests (extend `test_schema.py`/`test_tools_entities.py`/`test_tools_write.py` or start `test_applicants_api.py`): applicants rows visible only to their tenant (RLS probe zero); stage CHECK rejects a bad value; `list_applicants` filters by stage; `get_applicant` resolves qualification names; `update_applicant_stage` queues a gated action whose task title names the applicant; vocabulary endpoint lists the new tools + `applicant.*` event types.

### Task 2 — Router + `move_stage()` + promotion
D3 + D4 (all but summary).

**Validation**: gated `test_applicants_api.py`: POST → stage `applied` + `applicant.created` entity-linked; PATCH stage → `applicant.stage_changed` truthful `{from,to}`; PATCH to `hired` → `resources` row exists with copied quals/regions + `applicant_id` provenance + `resource.created` event, response carries the new resource id; second pass to `hired` (move out, move back) does **not** create a duplicate resource; field PATCH → one `applicant.updated`; no-op PATCH emits nothing; filters/`total`/offset/cap; 401 without token; RLS probe zero. Approved `update_applicant_stage` execution (approvals-engine fixture from `test_approval_gate.py`) leaves the same `applicant.stage_changed` — the shared path proven.

### Task 3 — Hiring smart summary
D4 (summary endpoint), reusing `services/views/summary.py`.

**Validation**: `test_applicant_summary.py` (mirror `test_lead_summary.py`): mocked client → `{summary, generated_at}`, prompt includes applicant name + stage + a seeded event summary; 503 without key; 404 unknown id.

### Task 4 — Directory page + create dialog + nav
D5.

**Validation**: browser (running stack): seeded applicants render with stage badges; chip/search/source filters work and round-trip the URL; creating an applicant (with two qualifications ticked) lands it in the table and the Event Log; Realtime: a second-tab stage change appears without refresh. `npm run build` clean.

### Task 5 — Profile page: info edit, stage moves, hire confirm, timeline

**Validation**: browser: profile deep-link renders; smart summary populates after skeleton; editing qualifications persists (`applicant.updated` on the timeline); moving to `offer` updates badge + timeline; moving to `hired` shows the confirm dialog, then the hired banner naming the new caregiver, and the timeline shows `applicant.stage_changed` + `resource.created`.
