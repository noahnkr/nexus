## Context

Clients are an M0 table with M12a matching fields (`address`/`zip`/`languages`/`preferences`) but no view surface, no contacts, no payer/authorized-hours, and statuses (`active`/`paused`/`ended`) that don't match how home care actually runs (`active`/`hospital_hold`/`discharged`). Schedules have the full M12 status machine and single-writer seam but no clock data. `document_chunks` carries `entity_type`/`entity_id`; the parent `documents` table doesn't, so "this client's documents" is unqueryable. The seed (`supabase/seed.sql`) has three clients (Walter Grimes active, Estelle Ferraro active, Frank Delgado paused) with `requirements.hours_per_week` jsonb values ready to promote into a real column.

User-locked (2026-07-19): in-app EVV-lite (read-time flags, no detector loop); care plans = tagged documents + RAG + one `care_summary` field; delivered hours = actuals fallback scheduled; backend/frontend split.

## Key design decisions (locked in â€” do not re-litigate during build)

1. **Migration `20260727000000_core_document_entity.sql`** (CORE):
   ```sql
   alter table public.documents
     add column if not exists entity_type text,
     add column if not exists entity_id   uuid;
   create index if not exists documents_entity_idx
     on public.documents (tenant_id, entity_type, entity_id);
   ```
   Nullable, business-agnostic (canonical-entity reference like `events`/`document_chunks`). No RLS change (existing policies cover the table).
2. **Migration `20260727000001_entities_client_oversight.sql`** (VERTICAL seam, additive + idempotent like the M12a migration):
   - `clients` gains `region_id uuid references public.regions(id)`, `payer text` with CHECK `in ('private_pay','medicaid','ltc_insurance','va','other')` (nullable = unknown), `authorized_hours_per_week numeric(5,1)` CHECK `>= 0` (nullable), `care_summary text`.
   - Status rename with data migration **before** the new CHECK: `update â€¦ set status='hospital_hold' where status='paused'`, `'discharged' where 'ended'`; drop `clients_status_check`, re-add with `('active','hospital_hold','discharged')`.
   - New `client_contacts` table: `id`, `tenant_id`, `client_id references clients(id) on delete cascade`, `name text not null`, `relationship text`, `phone`, `email`, `is_primary boolean default false`, `notes text`, timestamps; standard 4-policy RLS + `set_updated_at` trigger + `(tenant_id, client_id)` index.
   - `schedules` gains `check_in_at timestamptz`, `check_out_at timestamptz` with coherence CHECKs: `check_out_at is null or check_in_at is not null`, and `check_out_at is null or check_out_at > check_in_at`.
   - `clients` and `client_contacts` added to the `supabase_realtime` publication (guarded, M12a precedent).
   - `seed.sql`: promote `requirements.hours_per_week` â†’ `authorized_hours_per_week` (40/25/12), payers (`private_pay`/`medicaid`/`ltc_insurance`), region ids, one-line `care_summary` each, two `client_contacts` (one primary), and check-in/out stamps on one seeded completed visit (actual â‰  scheduled duration so the census math is visibly "actuals-first"). Keep `on conflict (id) do update` idempotency.
3. **Clients view seam `services/views/clients.py`** (re-templating-seam member):
   - `CLIENT_STATUSES` config (key/label: Active, Hospital hold, Discharged) + `status_label()`; `CLIENT_PAYERS` (key/label map for the five payers).
   - `change_status(conn, tenant_id, source_system, client_id, status)` â€” the ONE writer of `clients.status` (schedule-seam precedent): validates, no-op returns without an event, emits `client.status_changed` with plain summary ("Client 'Walter Grimes' placed on hospital hold") and `{"from": â€¦, "to": â€¦}` payload. Raises `ClientError(not_found=â€¦)` mapped to 404/422 by callers.
   - `census_metrics(conn, week_start=None) -> dict` â€” Monday-week window (default: current week): `active_clients`, `by_region` `[{region_id, region, count}]`, `by_payer` `[{payer, count}]` (active clients only, null payer bucketed as `"unknown"`), `authorized_hours`, `scheduled_hours` (statuses `scheduled`/`completed`/`no_show`/`called_out` in-window), `delivered_hours` (completed: `coalesce(check_out_at - check_in_at, end_time - start_time)` â€” only when both stamps exist, else scheduled duration), `open_hours` (unfilled), `leakage_hours = max(authorized - delivered, 0)`, `delivery_rate` (% of authorized, null when authorized = 0). Hours as floats rounded to 0.1. Visits counted by `start_time` in-window.
   - `client_week_hours(conn, client_id, week_start=None) -> dict` â€” same math scoped to one client (profile card + `get_client` tool).
   - `EVV_GRACE_MINUTES = 15`; `evv_flag(visit_row, now) -> str | None` â€” `'late'` when `scheduled` + no check-in + now > start + grace; `'missed'` when `scheduled` + no check-in + now > end; else `None`. Pure function, exported for the board feed.
4. **Schedule seam gains `check_in` / `check_out`** (`services/views/schedule.py`, same shape as the five existing transitions): `check_in(conn, schedule_id, source_system, at=None)` â€” valid only on a `scheduled` visit with a caregiver, not already checked in; `at` defaults `now()`; emits `schedule.checked_in`. `check_out(conn, schedule_id, source_system, at=None)` â€” requires a prior check-in, no prior check-out, `at > check_in_at`; sets `check_out_at` **and** `status='completed'`; emits `schedule.checked_out` with the actual duration in the summary ("Maria Santos checked out of the visit with Walter Grimes â€” 4h 10m"). `ScheduleError` for every rejection.
5. **Tools** (all through the registry; handlers take the tenant-scoped conn, never `tenant_id`):
   - `update_client_status` rewired: status enum becomes the new three; handler delegates to `change_status` (still gated; `gate_describe` unchanged in spirit).
   - NEW gated `record_visit_check_in` / `record_visit_check_out` (`schedule_id`, optional ISO `time`): delegate to the seam transitions; `gate_describe` names client + window ("Record check-in for the visit with Walter Grimes on Tue Jul 21"). Gated because they change billing-relevant care records (M5 entity-write precedent).
   - NEW safe `get_census` (no args) â†’ `census_metrics` with a plain-language content line ("14 active clients; 380 authorized hrs/wk, 342 delivered this week (90%)").
   - `_get_client` enriched: care fields, contacts list, `client_week_hours` block (upcoming schedules already included).
   - Labels for the three new tools in `services/tools/labels.py`.
6. **REST `routers/clients.py`** (applicants-router shape: JWT tenant-scoped, `source_system='user'`, no delete):
   - `GET /api/clients` â€” filters `status`/`payer`/`region_id`/`q` (name/phone/email ilike), `limit`/`offset` (50/100 caps) â†’ `ClientPage {clients, total}` with `region_name` resolved.
   - `GET /api/clients/metrics` â€” `census_metrics` (optional `?week=YYYY-MM-DD` Monday, board precedent). `GET /api/clients/facets` â€” observed payers, regions, statuses. (Literal paths registered before `/{client_id}` â€” applicants-router gotcha.)
   - `POST /api/clients` â†’ 201, `client.created` event. `GET /api/clients/{id}` â†’ `ClientDetail`: fields + contacts + distinct assigned caregivers (from non-terminal schedules Â± 30 days, `{resource_id, name, next_visit}`) + `client_week_hours` + tagged documents `[{id, filename, status, created_at}]`.
   - `PATCH /api/clients/{id}` â€” basic fields (`name`,`phone`,`email`,`address`,`zip`,`languages`,`preferences`,`care_summary`,`payer`,`authorized_hours_per_week`,`region_id`) â†’ one `client.updated` naming changed fields; `status` via `change_status`. No-op PATCH emits nothing.
   - Contacts: `POST /api/clients/{id}/contacts`, `PATCH`/`DELETE /api/clients/{id}/contacts/{contact_id}` â€” each emits `client.updated` on the **client** with a plain summary ("Family contact 'Susan Grimes (daughter)' added for Walter Grimes"); setting `is_primary` clears the previous primary in the same tx.
   - `GET /api/clients/{id}/summary` + `POST â€¦/summary/regenerate` â€” shared `services/views/summary.py` cache (`entity_type='client'`, `CLIENT_SUMMARY_INTRO`, span `client_summary`, 503 without key; applicants precedent verbatim).
7. **Schedule router additions**: `POST /api/schedule/{id}/check-in` and `/check-out` (optional `{time}` body) â†’ seam; board feed rows gain `check_in_at`/`check_out_at` and a server-computed `evv` field (`late`/`missed`/`null`).
8. **Document tagging**: upload (`POST /api/documents`) accepts optional `entity_type` + `entity_id` form fields â€” `entity_type` must be a key of `services/automations/entities.ENTITY_TABLES`, the row must exist (RLS-scoped) â†’ 422 otherwise; stored on the document row, passed through `process_document`, stamped on every chunk (replacing the current `entity_type='document'` stamp **only when a tag is present** â€” untagged uploads keep today's behavior). `GET /api/documents` gains optional `entity_type`/`entity_id` filters; `DocumentOut` gains both fields.
9. **Event-type threading**: `client.status_changed`, `schedule.checked_in`, `schedule.checked_out` join `routers/automations.py`'s known event types and `services/automations/entities.py`'s eventâ†’entity map (client/schedule); `services/event_summaries.py` needs no new templates (every new writer sets `payload.summary`).
