# Plan 18a: Workforce & Compliance Backend

**Complexity: ⚠️ Medium** — one vertical migration, a seam + router following the M16/M17 precedents, one safe read tool, two surgical behavior changes (inactive exclusion), and an engine-run recipe test on existing machinery. **Fallback split point**: Tasks 1–2 (schema + seam) stand alone; Tasks 3–5 (tool + API + recipe) are a clean second pass.

Parent: `.agent/plans/18.workforce-compliance.md`. Build first; 18b consumes this API.

## Context

`resources` has no status flag and no dated credentials; `qualifications` is an undated per-tenant vocabulary and `resources.qualification_ids uuid[]` is the matching input. The schedule router serves the roster with `hours_this_week` (`_roster` + `matching.week_hours_map`) and a minimal PATCH (`resource.updated`); `matching.py` has `_hm` (time parsing) and `rank_candidates`; the M7 engine + cron trigger and the M8 create API are live (181+ tests). Seeded resources: three caregivers with availability jsonb and qualification arrays.

User-locked (2026-07-19): roster as a `/caregivers` tab; credentials = dated rows per (caregiver, qualification); retention/at-risk deferred; safe tool + daily digest recipe.

## Key design decisions (locked in — do not re-litigate during build)

1. **Migration `20260729000000_entities_workforce.sql`** (VERTICAL seam, additive + idempotent):
   ```sql
   alter table public.resources
     add column if not exists status text not null default 'active'
       check (status in ('active','inactive'));

   create table if not exists public.resource_credentials (
     id               uuid primary key default gen_random_uuid(),
     tenant_id        uuid not null references public.tenants(id),
     resource_id      uuid not null references public.resources(id) on delete cascade,
     qualification_id uuid not null references public.qualifications(id),
     issued_at        date,
     expires_at       date,          -- null = does not expire
     notes            text,
     created_at       timestamptz not null default now(),
     updated_at       timestamptz not null default now(),
     unique (tenant_id, resource_id, qualification_id)
   );
   ```
   Standard 4-policy RLS + `set_updated_at` trigger + `(tenant_id, resource_id)` index; `resource_credentials` added to the `supabase_realtime` publication (guarded). The status CHECK is a fresh named constraint (`resources_status_check`).
2. **Seed** (additive; `on conflict … do update` idempotency; **relative dates** via `current_date + interval` so they never drift — the schedule-seed precedent): three credential rows on existing caregivers — one valid (expires +180 days), one expiring (+30 days), one expired (−10 days) — plus one no-expiry row (background check). No new resources/clients/visits — M16/M17 seed expectations untouched.
3. **Workforce seam `services/views/workforce.py`** (re-templating-seam member):
   - `EXPIRING_DAYS = 60`; `credential_status(expires_at, today=None) -> 'valid'|'expiring'|'expired'|'no_expiry'` — pure function.
   - `available_week_hours(availability) -> float | None` — sums the jsonb day windows via `matching._hm`; `None` when no windows declared.
   - `roster_rows(conn, ref) -> list[dict]` — every resource (active AND inactive; the Roster tab is the one place inactive appear) with `hours_this_week` (reuse `week_hours_map`), `available_hours`, `utilization` (% or null), and its credentials `[{id, qualification_id, qualification_name, issued_at, expires_at, status, notes}]`.
   - `roster_metrics(conn, ref) -> dict` — `active_count`, `inactive_count`, `avg_utilization` (active caregivers with declared availability only), `expiring_count`, `expired_count` (distinct credentials, active caregivers only — an inactive caregiver's lapsed CPR isn't a compliance emergency).
   - `expiring_credentials(conn, days_ahead) -> list[dict]` — active caregivers' credentials with `expires_at <= current_date + days_ahead` (includes already-expired), ordered soonest-first, each row plain-language ready (`caregiver`, `credential`, `expires_at`, `days_left`).
4. **Inactive exclusion** (two surgical edits): `matching.rank_candidates` resource query gains `where status = 'active'`; `routers/schedule.py::_roster` gains the same filter (board + candidates never see inactive caregivers). The workforce seam's `roster_rows` is the only surface listing everyone.
5. **Status changes**: the existing roster PATCH (`routers/schedule.py`) — still the single `resources` writer — accepts `status` (`active`/`inactive`); a status change emits **`resource.status_changed`** (from/to + plain summary "Caregiver 'Maria Santos' deactivated"), other field edits keep emitting `resource.updated`; one no-op rule as today.
6. **Safe tool `list_expiring_credentials`** (`services/tools/entities.py`, `safe=True`): input `{days_ahead?: int}` (default 60, clamped 1–365) → seam `expiring_credentials`; result `{credentials: [...], count}` with a plain content line ("2 credentials need attention: Maria Santos's CPR expires in 30 days; …"). Label in `labels.py`. No gated tools this module — credential CRUD is human-UI (REST) territory.
7. **REST `routers/workforce.py`** (JWT tenant-scoped, `source_system='user'`, wired in `main.py`):
   - `GET /api/workforce/roster?week=` → `{metrics, caregivers}` (decision 3 rows + metrics; Monday-week param, board precedent).
   - `POST /api/workforce/credentials` (`resource_id`, `qualification_id`, dates, notes; duplicate pair → 409; both ids validated RLS-scoped) → `credential.added`.
   - `PATCH /api/workforce/credentials/{id}` (dates/notes) → `credential.updated` naming changed fields; no-op emits nothing. `DELETE /api/workforce/credentials/{id}` → `credential.removed`.
   - All three events carry `entity_type='resource'`, `entity_id=resource_id`, and a plain `payload.summary` ("CPR credential for Maria Santos updated (expires Sep 12)").
8. **Threading**: `SQL_SCHEMA_DOC` gains `resources.status` + `resource_credentials`; known event types (`routers/automations.py`) gain `resource.status_changed`, `credential.added`, `credential.updated`, `credential.removed`; the automation entity map (`services/automations/entities.py`) maps the `credential.` prefix → `resource`. Update any vocabulary/catalog assertions that pin lists.
9. **Digest recipe = docs + test** (never seeded; standard create path only): README publishes the recipe —
   WHEN cron `0 7 * * *` → step `tool list_expiring_credentials {days_ahead: 60}` save-as `expiry` → IF `{{steps.expiry.count}} > 0` → step `tool create_task` (digest title/description from `{{steps.expiry.…}}`). The pytest creates it via `POST /api/automations`, forces `next_fire_at` due, drives `scheduler` cycles, and asserts the created digest task (and no task on a tenant with no expiring credentials).
