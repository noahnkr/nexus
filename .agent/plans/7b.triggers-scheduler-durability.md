# Plan 7b: Automations — triggers, scheduler & durable runs

**Complexity: ⚠️ Medium** — no new tables and no new endpoints; this is background machinery over 7a's engine: four small loops (dispatcher, cron, waker, recovery) plus one surgical hook in `services/approvals.py`. The risk is concurrency/liveness subtleties (cursor progress, missed wakes, double-fires), which is why every loop exposes a synchronous `*_once()` tick function that tests drive deterministically — the `while True` wrappers stay trivial.

Parent: `.agent/plans/7.core-automations-framework.md`. Requires 7a complete (tables, engine, API all exist).

## Context

After 7a, runs exist but nothing moves them without a request: event/cron automations never fire on their own, `waiting` runs never wake, `waiting_approval` runs never resume. This sub-plan adds the in-app engine loops and the approval hook. Patterns already in place: the lifespan hosts a long-running service (`session_manager.run()` in `main.py`); `connector_state` is the durable-cursor store (tenant-scoped, RLS — verify its exact columns at build, it was made for Gmail `historyId`-style cursors in 3b); `deps.get_machine_tenant_id()` is the machine-path tenant seam; `tenant_tx` serves code outside the request cycle.

User-locked (see parent): fail + review task; one active run per entity (7a enforces); loop guard — automation-emitted events are never re-dispatched.

**Non-goals**: UI/Realtime consumers of run status (M8); multi-tenant engine iteration (post-validation; the single documented seam is the tenant loop in D1); per-automation schedules beyond cron expressions; wait-for-event steps (only time-based waits this phase — event-reactivity belongs to triggers); notification/push on failure beyond the review task.

## Key design decisions (locked in — do not re-litigate during build)

1. **Engine loops live in the FastAPI lifespan** (`services/automations/scheduler.py`): a single asyncio task started/cancelled in `main.py`'s lifespan alongside the MCP session manager — one process, one pool (scale discipline; no worker deployment). Each cycle runs the four phases in order: dispatch → cron → wake → recover, all under `tenant_id = get_machine_tenant_id()` (single-tenant phase; a multi-tenant deployment later wraps the cycle in a tenant loop — this is the seam, documented in the module docstring). Guards: the task starts only when `settings.nexus_automations_enabled` (default `true`) **and** the pool is open; it never starts under tests (httpx `ASGITransport` doesn't run the lifespan — the existing test posture).
2. **Every loop phase is a testable tick**: `dispatch_once()`, `tick_cron_once()`, `wake_due_once()`, `recover_stale_once()` — each opens its own `tenant_tx`(s), does one bounded pass, returns counts. The background task is just `while True: run_cycle(); sleep(settings.nexus_automations_poll_seconds)` (default 5) with a broad try/except that logs and continues (a bad cycle must not kill the loop).
3. **Event dispatcher with a durable cursor** (`dispatch_once`): reads `events` with `id`-keyset pagination past the persisted cursor (stored in `connector_state` under a reserved source key, e.g. `_automations`, value `{"last_event_id": …}` — matching whatever shape 3b established; verify at build). For each new event: skip if `source_system='automation'` (**loop guard**, user-locked); match against `active` automations with an `event` trigger where `trigger.event_type == event.event_type` and (if set) `trigger.source_system == event.source_system`; each match ⇒ `start_run` (entity from the event's `entity_type`/`entity_id`) then `advance_run`. The cursor advances **after** the batch commits, initialized to the current `max(events.id)` on first ever run (don't replay history). Batch cap per tick (e.g. 200) so a burst can't starve the other phases.
   *Why polling over LISTEN/NOTIFY*: survives restarts by construction (cursor), no trigger DDL on the immutable events table, and a 5s lag is invisible for funnel automations.
2b. **Events `id` ordering note**: keyset on `id` requires monotonically increasing ids — if `events.id` is `gen_random_uuid()` (verify at build), keyset on `(created_at, id)` instead and store both in the cursor (the Module 4 pagination pattern).
4. **Cron triggers** (`tick_cron_once`): `automations.next_fire_at` is the schedule state — set on create/activate (7a leaves it null; this task backfills on activation) via `croniter(expr, now).get_next()`. Tick claims due rows (`status='active' and trigger->>'type'='cron' and next_fire_at <= now()` … `for update skip locked`), starts+advances a run for each (no entity ⇒ no concurrency guard), and advances `next_fire_at` to the next occurrence **before** running the recipe (a slow run must not double-fire). PATCHes that change the expression or re-activate recompute `next_fire_at` (7a's PATCH gains that two-liner here).
5. **Waker** (`wake_due_once`): claims `waiting` runs with `wake_at <= now()` (`for update skip locked`), flips them `running`, calls `advance_run`. Bounded batch like the dispatcher.
6. **Approval resume/cancel** — the only change outside `services/automations/`: at the end of `approvals.approve_action` / `reject_action`, if the action row has `automation_run_id`:
   - approved + `executed` ⇒ merge the tool result into the run's `context` (honoring the paused step's `save_as`), bump `step_index`, flip `running`, `advance_run` — the run continues in the same request, exactly like a chat-approved action executes in-request.
   - approved + `failed` ⇒ run `status='failed'` + `error` + `automation.run_failed` event, **no second review task** (Module 5 already leaves the failed action's task pending — one human surface per failure).
   - rejected ⇒ run `status='cancelled'` + `automation.run_cancelled` event (summary names the automation and the rejecting `resolved_by`).
   `engine.py` exposes `resume_after_approval(...)`/`cancel_after_rejection(...)` so approvals stays thin and the run-state machine stays in one file.
7. **Recovery sweep** (`recover_stale_once`): a run stuck in `running` with `updated_at` older than a threshold (setting, default 10 min) means a process died mid-advance — 7a's one-tx-per-step design makes re-entry safe, so the sweep just calls `advance_run` on it. Also re-arms cron automations whose `next_fire_at` passed while the app was down (they fire once on catch-up, not once per missed slot — croniter from `now`).
8. **Config** (`config.py`): `nexus_automations_enabled: bool = True`, `nexus_automations_poll_seconds: float = 5`, `nexus_automations_stale_minutes: int = 10`. Env-overridable, documented in `.env.example` as optional.

## Execution notes

- **Blocking user input**: none new — the live walk needs the standing env (`NEXUS_APP_DB_URL`, `ANTHROPIC_API_KEY`, `NEXUS_WEBHOOK_SECRET`, the Module 6 office user for the browser approval).
- Reuse: `connector_state` upsert/read helpers from 3b (`services/connectors/…` — check where the cursor helpers landed; if adapter-local, lift a tiny shared helper rather than duplicating); `for update skip locked` claim pattern is new — keep it inside the tick functions only; signed-webhook snippet (README / `test_webhook_ingress.py`) for the live walk; `tenant_tx` for everything the loops touch.
- Concurrency invariants to preserve: cursor advances only after its batch commits; `next_fire_at` advances before recipe execution; every claim uses `skip locked`; `advance_run` stays idempotent per step (7a's per-step tx) — if a test reveals a step replay, fix the engine, not the test.
- Scope discipline: no UI, no new endpoints (beyond the PATCH recompute inside the existing route), no agent tools (M8); no wait-for-event step; do not attempt multi-tenant iteration — leave the documented seam; approvals changes limited to the end-of-function hook.
