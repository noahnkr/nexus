# Plan 8: Automations Center (parent)

**Complexity: 🔴 Complex** — Module 8 is not on CLAUDE.md's default-complex list, but it pairs the richest UI built so far (a dynamic recipe builder rendering per-tool forms from JSON Schemas) with the first structured-LLM product feature (agent-drafted recipes). Per the planning rule it is **broken into two sub-plans**:

- **`8a.center-management.md`** (⚠️ Medium) — the management surface: `/automations` grid (status, plain-language trigger lines, pause/resume, delete), automation detail with run history and a per-step run timeline, run cancellation, the definition-edit guard, Realtime live updates, Home widget, nav entry. Small backend additions; no creation UI.
- **`8b.recipe-builder.md`** (⚠️ Medium) — creation and editing: the vocabulary endpoint, the sentence + step-list builder pages (`/automations/new`, `/automations/{id}/edit`), and the describe-it → agent-drafts → review-in-builder flow.

Build order: **8a then 8b** — 8a is immediately useful against automations created via curl (Module 7's live-walk artifacts) and establishes the routes/components 8b's builder links into. If 8b stalls, its vocabulary + draft endpoints (Tasks 1–2) are a clean fallback split — both are backend-testable before any builder UI exists.

**Requires Module 7 built** (7a + 7b): the engine, the automations/runs API, the Realtime publication rows, `step_log`, and `FunctionDef.input_schema` (both added to plan 7a at M8 planning time) are all consumed here.

## Context

Module 7 delivers a headless engine: automations exist, trigger, run, park, and resume — but only curl can see or manage them. This module is PRD interface #6, the monday.com-inspired generic surface. The constrained per-stage builders in M9/M10 are *not* this — they compose the same engine with a narrower UI; this module's components (step cards, condition editor, vocabulary endpoint) should be built expecting that reuse.

What exists to build on: the M7 REST API (CRUD, manual run, runs list/detail) with plain-language 422 validation errors; `automations` + `automation_runs` in the Realtime publication; `ToolDef.input_schema`/`safe` and `FunctionDef.input_schema` for data-driven forms; `GET /api/events/facets` for observed event types; the M6 design system (`PageHeader`, `EmptyState`, semantic status tokens, `StatCard`, Realtime-refetch pattern in `TasksPage.tsx`); Pydantic-structured LLM output is a standing CLAUDE.md rule and `llm.get_anthropic()` is traced.

**User-locked decisions (2026-07-16):**
1. **Builder layout = sentence + step list**: WHEN and IF render as an editable sentence line with inline chips; THEN is a vertical, reorderable list of step cards.
2. **Dedicated builder pages** (`/automations/new`, `/automations/{id}/edit`) — no modal; deep-linkable so M9/M10 can route into it.
3. **Agent drafting = draft → review in builder**: the draft endpoint returns an unsaved, validated recipe; the builder prefills; the user saves through the normal create path (server revalidates). A recipe never exists without a human having seen it.
4. **No starter templates this phase** — blank builder + agent drafting cover the blank-page problem; M9/M10 bring the prescribed sequences.

**Non-goals** (name the module): per-stage constrained sequence builders and funnel views (M9/M10); agent-facing automation tools in chat/MCP (future — drafting here is a dedicated endpoint, not a chat tool); starter-template gallery (dropped, user-locked); recipe versioning/run-against-snapshot semantics (the edit guard makes it unnecessary this phase); drag-and-drop step reordering (up/down buttons; a dnd lib is not warranted); automation analytics beyond run counts (M9/M10 dashboards own metrics); editing `connector_state` or engine settings from the UI (env-config phase).

## Shared conventions (both sub-plans)

- The Center *manages* recipes; it never executes anything — every effect stays behind the M7 API. No new execution paths, no direct engine imports in routers beyond what M7 exposed.
- Plain language on every surface: trigger/condition/step summaries derive from a shared frontend `describeRecipe()` util (also used by 8b's sentence line); raw recipe JSON appears only behind an expandable technical toggle (the Event Log expander precedent).
- Gated tools render an amber "requires approval" chip wherever they appear (grid cards, step cards, run timelines) — office staff must never be surprised by a paused run.
- Drafts are never persisted by the agent (user-locked #3): `POST /api/automations/draft` is read-only w.r.t. the database; the standard create path is the only writer. This is durable posture, recorded in CLAUDE.md.
- New frontend deps: none. The builder is shadcn-style primitives + existing patterns; no form library, no dnd library.
