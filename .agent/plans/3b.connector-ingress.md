# Plan 3b: Connector Ingress & Entity Resolution (sub-plan of Module 3)

**Complexity: ⚠️ Medium.** Linear once the seam is right, but it carries the module's only migration and the resolution routing has three outcome branches to get exactly right. Fallback split point: Tasks 1–4 (ingress + resolution working with one inline test adapter) are a complete, shippable core; Task 5 (the five placeholder adapters) is additive. Parent context and integration research: `.agent/plans/3.mcp-and-connectors.md` — the per-system delivery research there is **normative** for the adapter docstrings.

## Context

Every inbound connector event must resolve to a canonical entity via `external_ids` before writing anywhere else (CLAUDE.md). Module 0 built `external_ids` (unique `(tenant_id, source_system, external_id)`, category CHECK `crm/phone/ehr/email/manual`), immutable `events`, and `tasks`. This sub-plan builds the single webhook ingress, the adapter seam, and the resolution router that enforces that rule — validated with placeholder payloads for five real systems (WelcomeHome, GoTo Connect, WellSky, Gmail, Google Calendar) whose real integration flows are documented in each adapter file.

**User-locked decisions**: placeholder payloads now, real adapters slot into the same seam later; auto-create for events that explicitly represent a new entity, review-task fallback for unresolvable references; poll-based sources enter through this same ingress (an external poller POSTs), keeping the core webhook-shaped.

**Non-goals**: outbound actions — send SMS/email, write back to CRM/EHR (M5 gated tools); applying field-level *updates* from `.updated` events onto business rows (real adapters own their schema mapping; this phase records the event linked to the resolved entity); fuzzy matching/dedup (M8); scheduled pollers (M7, via n8n); Event Log UI (M4); live OAuth/credential config for the five systems (post-module ops); frontend changes.

## Key design decisions (locked in — do not re-litigate during build)

1. **One migration, core, `supabase/migrations/<ts>_connector_infra.sql`:**
   - `connector_state` — durable connector cursors/renewals (Gmail `historyId`, GCal `syncToken`, GoTo channel ids + expiries): `id uuid pk default gen_random_uuid()`, `tenant_id uuid not null references tenants(id)`, `source_system text not null`, `state jsonb not null default '{}'`, `created_at`/`updated_at` (+ `set_updated_at` trigger), `unique (tenant_id, source_system)`. Four-policy tenant RLS identical to the other core tables (match `core_rls.sql` style). Placeholder adapters don't read it yet — it exists so real adapters have their seam.
   - `external_ids.source_system` CHECK constraint gains `'calendar'` (drop + re-add the constraint; existing rows untouched).
2. **Source naming**: connector names `welcomehome|goto|wellsky|gmail|gcal` are the URL segment and `events.source_system`; each adapter declares a `category` (`crm|phone|ehr|email|calendar`) used for `external_ids.source_system`. One live system per category this phase (parent plan records the re-templating caveat).
3. **Adapter seam (`services/connectors/base.py`)**:
   - `NormalizedEvent` dataclass: `event_type` (canonical, e.g. `lead.created`), `entity_type` (`lead|client|resource|schedule`), `external_id: str`, `summary: str` (plain language — reaches tasks/UI), `attributes: dict` (canonical field values; used only on auto-create), `creates_entity: bool = False`, `occurred_at: str | None`, `detail: dict` (technical payload for the event row).
   - `NormalizedResult`: `ack_only: bool = False`, `events: list[NormalizedEvent]`.
   - `ConnectorAdapter` protocol/base: `source: str`, `category: str`, `verify(headers, body: bytes) -> bool` (default implementation: hex HMAC-SHA256 of the raw body with `settings.nexus_webhook_secret`, compared constant-time against the `X-Nexus-Signature` header — real adapters override with their platform's scheme: GoTo signature keys, Google channel token/Pub/Sub OIDC, WelcomeHome/WellSky per their docs), `async normalize(payload: dict, headers) -> NormalizedResult` (async because real Gmail/GCal adapters do the fetch-back API call here).
   - `registry.py`: `register_adapter()`/`get_adapter(source)`; `__init__.py` imports all adapter modules (bootstrap, mirroring `services/tools/__init__`).
4. **Resolution router (`services/connectors/resolution.py`, core, generic)** — `route_normalized_event(conn, tenant_id, adapter, ev, originating_event_id) -> outcome`:
   - Look up `external_ids` by `(category, ev.entity_type, ev.external_id)` on the tenant-scoped conn (RLS filters tenant).
   - **Matched** → update `last_synced_at`; outcome `matched` with `entity_id`.
   - **Miss + `creates_entity`** → call the entity writer for `ev.entity_type` (D5) to insert the canonical row from `ev.attributes`, insert the `external_ids` mapping, outcome `created`.
   - **Miss + reference** → insert a `tasks` row (`title` = ev.summary prefixed "Review: ", `description` = plain-language "…couldn't be matched to a known {entity_type}", `originating_event_id` = the `webhook.received` event id, priority `normal`), outcome `task`.
   - Every outcome writes one `events` row: `source_system` = connector name, `event_type` = `ev.event_type`, `entity_type`/`entity_id` set when resolved/created, `payload` = `{summary, external_id, resolution: matched|created|task, detail}`. Written via `services/events.log_event`.
5. **Entity writers (`services/connectors/entity_writers.py`) — the vertical seam**, sibling to `tools/entities.py`: `WRITERS: dict[str, Callable]` mapping `entity_type` → `async write(conn, attributes) -> entity_id`. This instantiation ships **`lead` only** (name required; contact/source/status/region optional with safe defaults — parameterized INSERT on the tenant-scoped conn, RLS supplies nothing, `tenant_id` comes from the GUC-scoped insert exactly like other services: pass tenant_id explicitly to the INSERT as existing services do). An `entity_type` without a writer that claims `creates_entity` falls back to the task outcome (never a 500). A new vertical replaces this file alongside the entity migration and `tools/entities.py`.
6. **Ingress (`routers/webhooks.py`)** — `POST /api/webhooks/{source}`:
   1. `get_adapter(source)` or 404.
   2. `adapter.verify(headers, raw_body)` or 401 — **before any DB write** (unauthenticated garbage never touches the DB). 401 also when `nexus_webhook_secret` is unset (fail closed).
   3. Parse JSON (400 on malformed) → open the request's tenant-scoped conn → write the raw-receipt `events` row (`event_type='webhook.received'`, `source_system`=connector, payload = raw body JSON) and capture its id.
   4. `await adapter.normalize(payload, headers)`; if `ack_only` → 200 `{"status": "ack"}` (the receipt row stays — handshakes are audit-worthy).
   5. Route each `NormalizedEvent` through D4; respond 200 `{"received": n, "matched": n, "created": n, "tasks": n}`.
   Endpoint uses the standard `tenant_conn` dependency — single-tenant ingress this phase, same seam as everything (Module 6 revisits identity).
7. **Placeholder adapters (`services/connectors/adapters/`)** — one file per source; each **docstring documents the real flow from the parent plan's research table** (auth, subscription/watch setup, delivery shape, fetch-back, cursors in `connector_state`, renewal cadence) and states exactly what the real adapter will change (verify + normalize internals only). Placeholder payloads are realistic in *shape* but carry data inline (as if fetch-back already ran):
   - `welcomehome.py` (crm): `lead.created` (**creates_entity**, attributes → lead writer), `lead.updated` (reference), `tour.scheduled` (reference to lead). External id = WelcomeHome prospect id.
   - `goto.py` (phone): `call.completed` (reference; external_id = normalized E.164 caller number — matching a known lead/client requires a seeded `external_ids` phone mapping), `sms.received` (reference). Unknown number → review task, per the locked decision.
   - `wellsky.py` (ehr): FHIR-shaped placeholders — `client.updated` (Patient, reference), `schedule.updated` (Encounter, reference), `resource.updated` (Practitioner, reference). External id = FHIR resource id.
   - `gmail.py` (email): accepts the Pub/Sub push envelope shape (`{message: {data: base64, messageId}, subscription}`); decodes `data`; placeholder data carries `{emailAddress, historyId, messages: [{from, subject, snippet}]}` inline; emits `email.received` per message (reference; external_id = sender address). Docstring: real adapter stores/advances `historyId` in `connector_state` and calls `users.history.list`.
   - `gcal.py` (calendar): reads `X-Goog-Resource-State` header — `sync` → **ack_only**; `exists` → placeholder body carries changed events inline `[{id, summary, start, end, attendees}]`; emits `calendar.event.updated` (reference; entity_type `schedule`, external_id = calendar event id). Docstring: real adapter's body is empty — fetch-back via `events.list` + `syncToken` cursor.
8. **Config**: `nexus_webhook_secret: str = ""` in `config.py`. One shared placeholder secret; real adapters bring per-platform verification so per-source secrets arrive with them.
9. **Observability**: `@traceable(run_type="chain", name="webhook_ingress")` on the router's processing function so LangSmith shows ingress → resolution spans.

## Tasks

### Task 1 — Migration: `connector_state` + `calendar` category
Write and push the migration per D1 (match the DDL/RLS style of `20260715014924_core_tables.sql` / `20260715014925_core_rls.sql`).

**Validation**: gated schema test in `test_connectors_resolution.py` (existing PostgREST/`db` fixture pattern from `test_schema.py`): `connector_state` insert under demo tenant succeeds and is invisible to the probe tenant; duplicate `(tenant, source_system)` rejected; `external_ids` insert with `source_system='calendar'` succeeds (and `'bogus'` still fails).

### Task 2 — Adapter framework
`base.py`, `registry.py`, `__init__.py` per D3. Include the shared HMAC helper `sign(body: bytes) -> str` used by both `verify` and the tests.

**Validation**: offline `test_webhook_ingress.py::test_hmac_verify` — correct signature accepted, wrong signature/absent header/empty-secret all rejected; comparison is `hmac.compare_digest` (assert by reading the constant, or simply cover both branches).

### Task 3 — Resolution + entity writers
`resolution.py` + `entity_writers.py` per D4/D5.

**Validation**: gated `test_connectors_resolution.py`, driving `route_normalized_event` directly on a tenant-scoped conn:
- Seed an `external_ids` row for a seeded lead → matched outcome, `last_synced_at` updated, event row `resolution='matched'` linked to the entity.
- `lead.created` with `creates_entity` + attributes → new `leads` row exists with the attributes, `external_ids` mapping exists, event `resolution='created'`.
- Reference to an unknown external id → `tasks` row with `originating_event_id` set and a plain-language title/description (no JSON in either), event `resolution='task'`, and **no** business-table row.
- `creates_entity` for an entity_type with no writer → task outcome, not an exception.
- Probe-tenant `external_ids` rows never match under the demo tenant GUC.

### Task 4 — Webhook ingress router
`routers/webhooks.py` per D6, wired into `main.py`. For this task's tests, register a throwaway in-test adapter (fixture) so ingress is validated independently of Task 5.

**Validation**: `test_webhook_ingress.py` — offline: unknown source → 404; bad signature → 401; malformed JSON with valid signature → 400. Gated: valid signed POST through the throwaway adapter → 200 with correct counts, `webhook.received` event row + one row per normalized event; `ack_only` payload → `{"status":"ack"}` and only the receipt row.

### Task 5 — Five placeholder adapters + fixtures
Adapters per D7, fixture payloads under `tests/fixtures/webhooks/`, each adapter docstring carrying its real-flow documentation (research table in the parent plan is the source of truth — auth, subscription setup, delivery shape, cursors, renewals, what the real adapter changes).

**Validation**: gated `test_connector_adapters.py`, one signed end-to-end POST per source using the fixtures:
- `welcomehome` `lead.created` → new lead row + external_ids mapping (`source_system='crm'`) + events.
- `goto` `call.completed` from a phone number pre-seeded in `external_ids` → matched to that lead; `sms.received` from an unknown number → review task, no business write.
- `wellsky` `client.updated` for a pre-seeded EHR external id → matched to the client.
- `gmail` Pub/Sub envelope (base64 round-trip exercised) → `email.received` event; unknown sender → task.
- `gcal` `X-Goog-Resource-State: sync` → ack-only; `exists` fixture with a pre-seeded calendar external id → matched to a schedule (`source_system='calendar'`).
- Cross-cutting asserts: every POST produced exactly one `webhook.received` row; all event rows use the connector name as `source_system`; every task title/description is plain language.

Tick PROGRESS.md 3b boxes; mark Module 3 complete if 3a done.

## Execution notes

- **Blocking user input**: the migration push needs the linked Supabase project (same flow as Modules 0–1: `supabase db push`). Gated tests need `NEXUS_APP_DB_URL`; live validation additionally needs `NEXUS_WEBHOOK_SECRET` set and the Module 1 keys for the chat check.
- Reuse: `tenant_tx`/`tenant_conn` (`app/db.py`), `log_event` (`services/events.py`), `_jsonable` (`services/tools/core.py`) if needed for detail payloads, `_require`/`db`/`set_tenant` fixtures (`tests/conftest.py`), DDL/RLS house style (`supabase/migrations/*core*`), bootstrap-import pattern (`services/tools/__init__.py`).
- Tests that seed `external_ids`/`connector_state` rows must clean up after themselves (or use throwaway external ids namespaced by test run) so reruns stay green — follow whatever cleanup convention `test_tools_entities.py` established.
- Scope discipline: no outbound calls to any real system, no OAuth flows, no update-application onto business rows from `.updated` events, no fuzzy matching. If resolution feels like it needs "smart" matching, that is Module 8; the answer here is the task fallback.
- Non-technical-user rule: `NormalizedEvent.summary` and task titles/descriptions are plain language; raw payloads live only in `events.payload` and LangSmith.
