# Plan 5a: Approval Gate — backend engine, write tools, Tasks API

**Complexity: ⚠️ Medium** — one guarded migration, a surgical change to an existing seam (`execute_tool`), one new service, three small tool modules, one router. Linear and single-pass feasible, but the gate/approval state machine has enough edge cases (double-resolve, handler failure, task/action coupling) that iteration on tests is expected. Fallback split point: Tasks 1–4 (gate works headless) vs Task 5–6 (API + wrap-up).

Parent: `.agent/plans/5.approval-gate-and-tasks.md` (context + locked decisions live there; this plan is self-contained for execution).

## Context

`execute_tool()` (`backend/app/services/tools/core.py`) refuses `safe=False` tools today (`GATED_MESSAGE`, `core.py:120-132`); no unsafe tool exists yet. `tasks`/`pending_actions` exist since Module 0 with RLS and the status vocabularies below — verified live:

- `tasks.status`: `pending / in_progress / done / cancelled`; `priority`: `low / normal / high / urgent`; `originating_event_id` FK to `events`.
- `pending_actions`: `task_id` FK (not null), `tool_name`, `tool_input jsonb`, `status pending / approved / rejected / executed / failed`, `resolved_at`, timestamps + `set_updated_at` trigger. UPDATE policies exist on both tables.

User-locked (see parent): entity writes + placeholder sends as the gated tool set; safe `create_task`. Locked here as design decisions: synchronous execution on approve (D5), task/action status coupling (D6), no expiry.

**Non-goals**: frontend (5b); auth/`resolved_by` identity (M6); real SMS/email delivery (M7 credentials + real adapters); retry/re-queue of failed actions (a failed action is terminal — re-ask the agent); n8n callers (M7, they arrive through MCP → same seam).

## Key design decisions (locked in — do not re-litigate during build)

1. **Migration `supabase/migrations/<ts>_task_approval_infra.sql`** (one file, idempotent):
   - `alter table public.pending_actions add column if not exists source_system text not null default 'chat'` (who queued it — `chat`/`mcp`/`workflow`), `add column if not exists resolved_by text` (free text until M6 auth), `add column if not exists result jsonb` (execution outcome: `{summary, error?}` — the UI reads this without joining `events`).
   - Add `public.tasks` and `public.pending_actions` to the `supabase_realtime` publication, guarded DO-block exactly like `20260716000001_eventlog_realtime.sql` (CLAUDE.md: Realtime for task/event queue updates).
   *Why*: smallest schema delta that lets the UI render outcomes and live-update; everything else was built in Module 0.
2. **`ToolDef` gains `gate_describe: Callable[[conn, dict], Awaitable[str]] | None = None`** — async so it can look up entity names ("Update lead 'Margaret Ellison' to contacted", not a UUID). Used for the task title and `action.queued` summary. Fallback when absent or raising: `"Approve: {tool_name}"` + generic description. It runs *before* queueing, inside the same transaction, and must be read-only.
3. **Gate path replaces the refusal branch in `execute_tool`**: for `safe=False` (and no bypass, D4) —
   - Write order (all in the caller's transaction): **event → task → action**. First the `action.queued` event via `log_event` (it returns the id, same trick as `resolution.py`): `source_system` = caller's, no entity linkage (the task doesn't exist yet), payload = `{summary: "Queued for approval: {describe}", tool_name, input}`. Then the `tasks` row: `title = "Approve: " + gate_describe(...)`, `description` = plain-language explanation naming the requesting source (`chat`/`mcp`), `priority='high'` (a human-requested action is stalled on it), `originating_event_id` = that event — task→event is the drill-down path, matching `resolution.py` review tasks. Then the `pending_actions` row (`task_id`, `tool_name`, `tool_input=args`, `source_system`).
   - return `ToolResult(summary="Queued for approval: {describe} (task created)", data={"status": "queued", "task_id": ..., "pending_action_id": ...}, is_error=False)` — the model reports it plainly; **not** an error.
   - Exactly one `events` row per gated call (the `action.queued` row) — the `tool.called` row is written only when the tool actually executes (post-approval). `_audit` is therefore skipped on the gate path.
4. **Approved execution goes through `execute_tool` with a keyword-only bypass**: `execute_tool(conn, tenant_id, name, args, source_system=..., approved_action_id: str | None = None)`. When set, the gate branch is skipped and the `tool.called` audit payload includes `pending_action_id` — one seam for every execution (CLAUDE.md), uniform `tool.called` audit across direct and approved runs. Only `services/approvals.py` may pass it.
5. **`services/approvals.py`** — `approve_action(conn, tenant_id, action_id, resolved_by=None)` / `reject_action(conn, tenant_id, action_id, resolved_by=None, note=None)`:
   - Lock the row: `select … from pending_actions where id=%s for update`; missing → `ActionNotFound`; status ≠ `pending` → `ActionAlreadyResolved` (router → 404/409). Synchronous execution in-request (locked): at this scale a tool handler is fast; no background queue.
   - Approve: set `status='approved'`, then execute via D4's bypass (`source_system` = the action's stored `source_system` so the audit reflects the original requester); on success → `status='executed'`, task → `done` + `resolved_at`; on `is_error` result → `status='failed'`, task **stays `pending`** (visible, human decides: cancel or re-ask the agent). Either way `result = {summary, error?}`, `resolved_at=now()`, `resolved_by`; write `action.approved` event (`entity_type='task'`, `entity_id=task_id`, `payload.summary` = "Approved: {task title} — {outcome summary}", payload carries `pending_action_id`, `outcome`).
   - Reject: `status='rejected'`, task → `cancelled` + `resolved_at`, `result={"summary": note or "Rejected"}`; `action.rejected` event, same linkage. No execution.
6. **Task/action coupling is one-way and minimal**: resolving an action drives its task (D5); the reverse — `PATCH`ing a task that still has a `pending` action to `done`/`cancelled` — is a 409 ("resolve the approval first"). No cascade deletes, no orphan cleanup.
7. **Write tools — vertical seam additions to `services/tools/entities.py`** (same file per CLAUDE.md; keeps the seam one-file-per-layer), all `safe=False`, all with `gate_describe` doing name lookups, all validating inputs with the existing `_require_uuid`/`ToolInputError` helpers *inside the handler* (so bad args still fail cleanly post-approval):
   - `update_lead_status(lead_id, status)` — status ∈ `LEAD_STATUSES`.
   - `update_client_status(client_id, status)` — status ∈ `CLIENT_STATUSES`.
   - `create_schedule(resource_id, client_id, start_time, end_time)` — ISO-8601, end > start; returns the new schedule id.
   - `cancel_schedule(schedule_id)` — sets status `cancelled` (refuse if already `completed`/`cancelled`).
   Handlers return a plain `ToolResult.summary` naming the entities ("Updated lead 'Margaret Ellison' to contacted").
8. **Core tool modules** (business-agnostic):
   - `services/tools/tasks.py`: `create_task(title, description?, priority?, due_at?)`, **`safe=True`** (internal coordination, no external effect — user-locked); inserts the task and returns its id; summary "Created task: {title}".
   - `services/tools/outbound.py`: `send_sms(to, body)`, `send_email(to, subject, body)`, `safe=False`. **Placeholder execution** (locked): the handler validates inputs, performs *no* external call, and returns `summary="[placeholder] Would send SMS to {to}: “{body[:60]}”"` with `data={"delivered": false, "placeholder": true}` — the file's docstring documents the real GoTo (`messaging.v1.send`) / Gmail (`users.messages.send`) flows the way Module 3's adapters do, so wiring credentials later replaces only handler internals.
   - Bootstrap: `services/tools/__init__.py` imports gain `tasks`, `outbound` after `entities` — registry order stays fixed, cache breakpoint still lands on the last tool.
9. **Tasks & approvals API — `routers/tasks.py`**, standard `tenant_conn` dependency, RLS does the filtering, Pydantic models in `schemas.py`, router in `main.py`:
   - `GET /api/tasks?status=&priority=&cursor=&limit=50` — `status` accepts a comma-separated list (UI tabs need `pending,in_progress`); keyset pagination `(created_at desc, id desc)` with the Module 4 opaque-cursor pattern; limit capped at 100. Each `TaskOut` embeds its `pending_actions` array (0..n; includes `tool_name`, `status`, `result`, `created_at`, `resolved_at` — `tool_input` included too: it feeds the UI's *expandable* technical detail only).
   - `POST /api/tasks {title, description?, priority?, due_at?}` → 201 TaskOut; writes a `task.created` event (`source_system='user'`, `payload.summary` = "Task created: {title}") — manual creation is auditable like everything else.
   - `PATCH /api/tasks/{id} {status}` — allowed transitions: `pending↔in_progress`, `pending|in_progress → done|cancelled`; terminal states immutable (409); 409 while a `pending` action exists (D6); sets `resolved_at` on `done`/`cancelled`; writes `task.status_changed` event.
   - `POST /api/pending-actions/{id}/approve` / `POST /api/pending-actions/{id}/reject {note?}` → the D5 engine; response = refreshed `PendingActionOut` + its task (the UI shows the outcome immediately); 404 unknown, 409 already-resolved.
10. **Chat integration is prompt-level only** (mechanics are 5b): `PERSONA` in `chat_service.py` gains an "action tools" paragraph — actions queue for human approval, tell the user a task was created, never promise the action already happened; `TOOL_LABELS` gains the seven new tools. `event_summaries.py` needs **no change** (all new writers set `payload.summary`).

## Tasks

### Task 1 — Migration: pending_actions columns + Realtime publication
Write and `supabase db push` the migration per D1.

**Validation**: gated checks in `backend/tests/test_tasks_api.py` (`db` fixture, direct SQL): `pg_publication_tables` lists `tasks` and `pending_actions`; `information_schema.columns` shows `source_system`/`resolved_by`/`result` on `pending_actions`. Re-running the DO-blocks is a no-op.

### Task 2 — Gate path in `execute_tool`
`core.py` per D2/D3/D4: `gate_describe` on `ToolDef`, queue path (event → task → action, non-error result), `approved_action_id` bypass, `GATED_MESSAGE` removed. Register a throwaway unsafe tool inside the test (not in the app registry) to exercise the path before Task 4's real tools exist.

**Validation**: gated `backend/tests/test_approval_gate.py::test_gated_call_queues` — calling the unsafe test tool via `execute_tool` returns `is_error=False` with `status="queued"`; exactly one new `events` row (`action.queued`, with `payload.summary`); a `tasks` row (`priority='high'`, `originating_event_id` = that event) and a `pending_actions` row (`status='pending'`, `tool_input` = args, `source_system='chat'`) exist; the tool's handler did **not** run (side-effect flag on the test tool).

### Task 3 — Approvals engine
`services/approvals.py` per D5/D6.

**Validation**: gated tests in `test_approval_gate.py`:
- approve → handler ran once; action `executed` with `result.summary`; task `done` + `resolved_at`; events gained `action.approved` **and** a `tool.called` whose payload carries `pending_action_id`;
- reject → handler never ran; action `rejected` (note in `result`), task `cancelled`, `action.rejected` event;
- double-resolve → `ActionAlreadyResolved` (second call), row unchanged;
- failing handler (test tool raises) → action `failed` with `result.error`, task still `pending`, `action.approved` event records the failed outcome.

### Task 4 — Write tools
`entities.py` additions, `tools/tasks.py`, `tools/outbound.py`, bootstrap order per D7/D8.

**Validation**: gated `backend/tests/test_tools_write.py` —
- `create_task` executes immediately (safe): row exists, summary plain;
- `update_lead_status` called via `execute_tool` queues (unsafe) with a task title containing the seeded lead's *name* (gate_describe lookup), and after `approve_action` the lead row's status actually changed;
- `create_schedule` rejects `end <= start` cleanly post-approval (action `failed`, plain error); `cancel_schedule` refuses a completed schedule;
- `send_sms` queues, and approval yields `result.summary` starting `[placeholder]` with no external call;
- registry: `anthropic_tool_defs()` includes all new tools, breakpoint still on the last entry only. Seed data cleanup follows the delete-what-you-seeded convention (events are immutable — use throwaway source markers, as in `test_events_api.py`).

### Task 5 — Tasks & approvals API
`routers/tasks.py` + `schemas.py` + `main.py` per D9.

**Validation**: gated `backend/tests/test_tasks_api.py` (app-client pattern from `test_chat_api.py`): create → list round-trip; comma-separated status filter; keyset pages walk all rows exactly once; PATCH transitions incl. 409 on terminal and 409 while an action is `pending`; approve endpoint executes and returns the refreshed action + task; reject endpoint; 404/409 cases; probe-tenant rows invisible (RLS through the API); `limit=500` capped at 100.

### Task 6 — Chat prompt + wrap-up
PERSONA paragraph + TOOL_LABELS per D10. README: new endpoints section (no new env vars). Tick PROGRESS boxes for 5a.

**Validation**: full `pytest backend/tests` green in one run (gated tests skip cleanly without env). Quick live check (running backend): via MCP or curl, invoke `update_lead_status` through chat and confirm the SSE `tool_result` summary says queued — full browser validation belongs to 5b.

## Execution notes

- **Blocking user input**: none new — standing Module 1 env (`NEXUS_APP_DB_URL` for gated tests, `SUPABASE_DB_URL` + linked project for `supabase db push`, `ANTHROPIC_API_KEY` only for the live chat check).
- Reuse: `log_event` returns the event id (`services/events.py`, used by `resolution.py`); review-task insert pattern (`services/connectors/resolution.py:_create_review_task`); opaque cursor + facets patterns (`routers/events.py`); `_require_uuid`/`_limit`/`ToolInputError` helpers (`services/tools/entities.py`); fixtures `_require`/`db`/`app_db`/`set_tenant` (`tests/conftest.py`); guarded-publication SQL (`20260716000001_eventlog_realtime.sql`).
- Scope discipline: do not touch the SSE contract or frontend (5b); do not build retry/expiry for actions; do not add an MCP-specific branch anywhere — MCP calls hit the gate automatically via `execute_tool` with `source_system='mcp'` and that is the whole story; `run_report`/read tools stay `safe=True` untouched.
- Non-technical-user rule: task titles/descriptions and every `payload.summary` are plain language; `tool_input` JSON is data for the UI's expander, never for a summary line.
