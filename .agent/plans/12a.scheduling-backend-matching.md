# Plan 12a: Smart Staffing — scheduling backend & matching engine

**Complexity: ⚠️ Medium** — no new entity *type* (schedules/clients/resources all exist), but real design weight in three places: schema surgery on a live table (nullable `resource_id`, new statuses), the transition seam that must stay coherent across REST/tools/approvals, and the deterministic matching engine. Fallback split point: Tasks 1–3 (schema + transitions + matching, all curl/pytest-testable) vs Tasks 4–5 (tools + REST).

Parent: `.agent/plans/12.smart-staffing-scheduling.md`. Requires Modules 7–11 built. 12b renders this sub-plan's API.

## Context

`schedules` today: one-off visits, `resource_id NOT NULL`, statuses `scheduled/completed/cancelled/no_show`, time-relative seeds, gated `create_schedule`/`cancel_schedule` tools with inline handlers, no REST routes, no matching logic. Clients/resources have no address, zip, language, or trait fields. This sub-plan makes the schema able to represent unfilled and called-out shifts, centralizes every transition in one seam with events, builds the explainable deterministic ranker, and exposes it all as tools + REST.

**User-locked** (parent plan): open shifts + call-outs in the one `schedules` table; address/zip/languages/traits added to clients and resources; owner-picks-then-gated-SMS flow; repeat-weekly expansion at create (≤12 extra weeks).

**Non-goals**: all UI (12b); drive-time/geocoding, inbound SMS, EVV, recurrence tables (Future Plans); no decision-harness abstraction; no matching-weight config UI (constants in the seam file).

## Key design decisions (locked in — do not re-litigate during build)

1. **Migration `20260723000000_entities_scheduling.sql`** (vertical seam, additive/idempotent):
   - `schedules`: `alter column resource_id drop not null`; status CHECK recreated as `('open','scheduled','called_out','completed','cancelled','no_show')`; coherence CHECKs — `open` ⇒ `resource_id is null`, and `scheduled/called_out/completed/no_show` ⇒ `resource_id is not null`; new columns `required_qualification_ids uuid[] not null default '{}'`, `replaces_schedule_id uuid references public.schedules(id)`, `notes text`; partial index `(tenant_id, start_time) where status='open'`.
   - `clients`: `address text`, `zip text`, `languages text[] not null default '{}'`, `preferences text[] not null default '{}'` (free tags: "female caregiver", "no dogs", …).
   - `resources`: `address text`, `zip text`, `languages text[] not null default '{}'`, `traits text[] not null default '{}'` (the caregiver-side tags `preferences` match against).
   - `schedules` added to the `supabase_realtime` publication (guarded/idempotent — the M4/M5 precedent).
   - `seed.sql`: zips (some matching, some only region-covered), languages (`en` everywhere, `es`/`tl` mixed), a couple of traits/preferences; one future `open` shift with a required qualification; one future `called_out` visit + its linked `open` replacement; required quals on two scheduled visits. Times stay relative via the existing `on conflict (id) do update` block (extend it to the new rows/columns).
   *Why one table*: an open shift is just a visit nobody holds yet — a parallel shifts table would duplicate RLS, Realtime, events, and the board query for zero modeling gain.
2. **Transition seam `services/views/schedule.py`** — the only writer of schedule state (REST routes and tool handlers both delegate; runs on the caller's tenant-scoped conn/tx, never opens its own; tool paths pick up `source_system` via the invocation contextvar, the `update_lead_status` precedent):
   - `create_visits(conn, *, client_id, resource_id=None, start, end, required_qualification_ids=[], notes=None, repeat_weekly_until=None, source_system)` → validates client/resource exist, `end > start`, status derived (`open` iff no resource); expands weekly occurrences through `repeat_weekly_until` capped at **12 additional rows** (422/`ToolInputError` beyond); for an assigned caregiver, any hard overlap with their existing `scheduled/called_out` visits rejects the whole series (all-or-nothing, one tx); emits `schedule.created` per row ("Visit for Walter Grimes with Alicia Moreno, Tue Jul 21 8:00–12:00" / "Open shift for Walter Grimes …").
   - `assign(conn, schedule_id, resource_id, source_system)` → valid on `open` (fill) and `scheduled` (reassign); qualification gaps and availability mismatches are **warnings in the result, not blocks** (the owner outranks the score); hard time-overlap with the caregiver's other visits **rejects**; sets `scheduled`, emits `schedule.assigned` ("Open shift for Walter Grimes on Wed filled by Carmen Ruiz").
   - `call_out(conn, schedule_id, source_system)` → valid only on `scheduled` visits that haven't ended; sets `called_out` (resource retained — who called out is queryable), inserts the replacement row (same client/window/required quals/notes, `resource_id NULL`, status `open`, `replaces_schedule_id`); emits `schedule.called_out` on the original (payload carries `replacement_schedule_id`, summary "Alicia Moreno called out of Tue 8:00–12:00 with Walter Grimes — open shift created") **and** `schedule.created` on the replacement. The `schedule.called_out` event is the automation trigger for call-out sequences.
   - `cancel(conn, schedule_id, source_system)` → `scheduled`/`open` → `cancelled`, emits `schedule.cancelled`; cancelling a replacement does not resurrect the original.
   - `set_outcome(conn, schedule_id, status, source_system)` → past-visit bookkeeping to `completed`/`no_show`, emits `schedule.updated`.
3. **Matching engine `services/views/matching.py`** — `rank_candidates(conn, schedule_row) -> list[dict]`, deterministic, one pass over the roster (fine at this scale — low tens of caregivers):
   - **Disqualifiers** (excluded, with a stated reason available for tests): missing any `required_qualification_ids`; hard overlap with an existing `scheduled`/`called_out` visit; being the resource on the visit this shift replaces.
   - **Score components** (module-level `WEIGHTS` dict, plain-language reason string per earned component): availability fit +30 (visit window inside a declared weekday range; otherwise 0 + warning "outside declared availability" — extra shifts happen); geography +20 same zip as client, +12 client zip in the caregiver's regions' `zip_codes`, else 0 + warning; continuity +5 per past `completed` visit with this client, capped +20; language overlap +10; `traits`∩`preferences` +5 each capped +10; load balance from the caregiver's scheduled hours this ISO week — would-exceed-40h ⇒ −15 + warning "would push over 40h", under-20h ⇒ +5.
   - Returns top **10**: `{resource_id, name, phone, score, reasons[], warnings[]}` sorted score-desc, name-asc tiebreak.
   *Why constants not config*: one client, explainability over tunability; re-templating swaps this seam file wholesale anyway.
4. **Tools** (`services/tools/entities.py` + labels; all through the registry as ever):
   - **Safe** `find_available_caregivers` — inputs `schedule_id` *or* (`client_id` + `start_time` + `end_time` [+ `required_qualification_ids`]); wraps `rank_candidates` (ad-hoc inputs build a transient pseudo-row); summary text names the top 3 with their reasons; structured data carries the full list. Safe ⇒ callable from chat/MCP and as an automation tool step whose result lands in run context (`{{steps.N.candidates.0.name}}` — the M11 catalog picks it up from observed payloads).
   - **Gated** `record_call_out` (inputs `schedule_id`; `gate_describe` → "Record a call-out for Alicia Moreno's Tue 8:00–12:00 visit with Walter Grimes and open a replacement shift") → delegates to `call_out()`. **Gated** `assign_caregiver` (`schedule_id`, `resource_id`; describe names both parties) → delegates to `assign()`.
   - Existing `create_schedule`/`cancel_schedule` handlers rewired to delegate to `create_visits`/`cancel` (create gains optional `resource_id`, `required_qualification_ids`, `repeat_weekly_until`; its describe says "open shift" when unassigned). No behavior branch outside the seam.
   - Seam threading: `ENTITY_TABLES["schedule"] = "schedules"` (`services/automations/entities.py`) so recipes can condition on `entity.status`; `SQL_SCHEMA_DOC` updated (new columns); `_KNOWN_EVENT_TYPES` gains the five `schedule.*` types; labels for the three new tools.
5. **REST `routers/schedule.py`** (vertical seam, wired in `main.py`, JWT-gated via `deps.get_tenant_id`, `source_system='user'`):
   - `GET /api/schedule?week=YYYY-MM-DD` → one board payload: `{week_start, visits[], caregivers[]}` — visits joined to client/resource names + resolved qualification names for the Mon–Sun window; caregivers = full roster (id, name, phone, zip, languages, traits, availability, `hours_this_week`) so empty rows render.
   - `POST /api/schedules` (create/expand per D2, 201 with all created rows), `PATCH /api/schedules/{id}` (time/notes/required-qual edits on `open`/`scheduled` + outcome statuses via `set_outcome`; anything stage-like refuses — transitions have verbs), `POST /api/schedules/{id}/call-out`, `POST /api/schedules/{id}/assign` `{resource_id}` (response includes warnings), `GET /api/schedules/{id}/candidates` (rank for an `open` shift; 409 otherwise). **No DELETE** — cancel is the terminal verb.
   - `GET /api/roster` + `PATCH /api/roster/{id}` (contact/address/zip/languages/traits/availability; emits one `resource.updated` naming changed fields) — the minimal roster-editing surface 12b's drawer needs; a full roster view stays in Future Plans.
   - `POST /api/schedules/{id}/notify` `{resource_id, message}` → runs `send_sms` through **`execute_tool`** (`source_system='user'`) → returns the queued gated action/task id (the 5b chat-queued precedent, now from a page). *Why gated even for a human click*: `send_sms` is an external effect executed by the system, and one seam means one audit trail — the CLAUDE.md human-UI exemption covers entity record writes, not outbound messaging.

## Tasks

### Task 1 — Migration + seeds + seam threading
D1 + the D4 seam threading (`ENTITY_TABLES`, `SQL_SCHEMA_DOC`, `_KNOWN_EVENT_TYPES`). Push with `supabase db push`, re-seed.

**Validation**: gated tests (start `test_schedule_api.py` schema section): an `open` row with a resource_id violates the coherence CHECK (and vice-versa for `scheduled` with NULL); seeds contain ≥1 open shift and a called-out pair linked by `replaces_schedule_id`; RLS probe tenant sees zero schedules; vocabulary endpoint lists `schedule.*` event types and the M11 field catalog shows schedule entity fields.

### Task 2 — Transition seam + events
D2 (`services/views/schedule.py`).

**Validation**: gated `test_schedule_api.py` (seam called directly on a test tx, the `move_stage` test precedent): create unassigned → `open` + `schedule.created`; create with `repeat_weekly_until` +3 weeks → 4 rows, all events, one tx; 13-week repeat rejects; overlap in week 3 of an assigned series rejects the whole series; `assign` on open → `scheduled` + `schedule.assigned`, hard-conflict rejects, missing-qual returns a warning but succeeds; `call_out` → original `called_out` + linked `open` replacement + both events, payload carries `replacement_schedule_id`; `call_out` on a completed visit rejects; `cancel` + `set_outcome` emit their events.

### Task 3 — Matching engine
D3 (`services/views/matching.py`).

**Validation**: gated `test_matching.py` against purpose-built fixture rows: a caregiver missing a required qualification is absent; the caregiver with an overlapping visit is absent; the called-out caregiver is absent from their replacement's candidates; same-zip beats region-only-zip beats no-geography (holding all else equal); continuity cap honored at +20; a candidate pushed past 40h scores −15 and carries the overtime warning; reasons/warnings are plain sentences (assert no raw field names like `qualification_ids`); deterministic order across two runs.

### Task 4 — Tools
D4 (safe `find_available_caregivers`, gated `record_call_out`/`assign_caregiver`, delegate rewires, labels).

**Validation**: gated `test_schedule_tools.py`: `find_available_caregivers` by `schedule_id` and by ad-hoc window both return ranked candidates with a plain summary and write a `tool.called` audit row; `record_call_out` queues a gated action whose task title names caregiver + client + time; approving it (approvals fixture from `test_approval_gate.py`) executes the call-out and leaves the same `schedule.called_out` event as the REST path; `create_schedule` without `resource_id` queues with an "open shift" description and creates an `open` row on approval; unknown schedule id → clean tool error, never a 500.

### Task 5 — Schedule & roster REST API
D5 (`routers/schedule.py`).

**Validation**: gated `test_schedule_api.py` (API section): week feed returns the seeded board (open shift present, caregiver `hours_this_week` correct, quals resolved to names); POST expands a 3-week series; call-out endpoint round-trips (drawer payload shape); assign returns warnings on a qual gap; `candidates` on a non-open visit → 409; roster PATCH emits one `resource.updated` naming changed fields; `notify` returns a queued action id and the task exists with `send_sms` payload; every route 401s without a token; RLS probe isolation on the week feed.

- Seam functions never open `tenant_tx` — tool handlers already hold one via `execute_tool`, routes hold theirs.
- Scope discipline: no UI (12b); no automation recipes seeded; no changes under `services/automations/` beyond `ENTITY_TABLES`; weights stay constants; no geocoding, ever, this module.
