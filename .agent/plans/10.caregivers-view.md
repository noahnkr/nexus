# Plan 10: Caregivers View & Hiring Process (parent)

**Complexity: 🔴 Complex** — mostly a re-instantiation of Module 9's pattern, but it is the first module to add a **new entity type across every vertical seam at once** (entity migration, `ENTITY_TABLES`, entity read/write tools, views seam, event vocabulary) plus atomic hire-promotion logic. Per the planning rule it is **broken into two sub-plans**:

- **`10a.applicants-api-directory.md`** (⚠️ Medium) — the `applicants` table + seams, the applicants REST API with `applicant.stage_changed` events and hired→caregiver promotion, the hiring smart summary, and the `/caregivers` directory + profile pages.
- **`10b.hiring-funnel-and-sequences.md`** (⚠️ Medium) — hiring metrics, the funnel strip instantiation, and per-stage sequences (accepted/denied emails) via the shared stage builder.

Build order: **10a then 10b** — 10a stands alone as a working applicant tracker; 10b hangs the funnel and sequences off it. If 10b stalls, its Task 1 (metrics backend) is a clean fallback split.

**Requires Module 9 built**: the generic pipeline pieces (`components/pipeline/FunnelStrip`, the view-config-driven `StageSequencePage` + `lib/pipeline.ts`, `EntityTimeline`, `services/views/summary.py`) and the core `automations.binding` mechanism all come from M9. (The 9a/9b plans were amended at M10 planning time — M9 unbuilt, amendments free — to place those pieces generically from the start.) Also requires Modules 7–8 (engine, Center, builder components), as M9 did.

## Context

Module 9 instantiates the pipeline-view pattern for leads; this module — PRD interface #8, the second and final sanctioned vertical view — re-instantiates it for caregiver recruiting, proving the pattern is core while content is seam. The structural difference from M9: leads already existed as a table with a status column, but **applicants don't exist in the schema at all** — `resources` is the active caregiver roster (name, contact, qualification_ids, region_ids, availability) with no stage column. M10 therefore adds the applicants entity end-to-end and a promotion path onto the roster, mirroring how `clients.lead_id` records lead conversion.

What exists to build on: the full M9 file set (leads router/views-seam/pages as the 1:1 template), `resources` + `qualifications` + `regions` tables and their read tools, `send_email`/`send_sms` gated placeholder tools + safe `create_task`, the M7 engine + binding + M8 builder, `EntityTimeline`, and the shared summary helper.

**User-locked decisions (2026-07-17):**
1. **Data model = `applicants` table + promotion**: a new vertical table carries the hiring pipeline; hire promotes onto `resources` (leads→clients precedent). The roster stays clean for scheduling and M11 matching.
2. **Stages = `applied → screening → interview → offer → hired`, terminal `rejected`** (rejected mirrors leads' `lost` as the drop-off).
3. **Promotion = automatic and atomic**: moving an applicant to `hired` creates the `resources` row in the same transaction (copying name/contact/qualifications/regions/availability) and emits `resource.created`. The human moving the stage is the approver — no extra ceremony.
4. **Scoring deferred to Module 11** — no scoring function, no score column, no score UI this module. The matching/decision harness owns applicant/lead scoring.

**Non-goals** (name the module): scoring (M11, user-locked); caregiver roster/utilization and credential-expiry views (Future Plans backlog); webhook/ATS applicant ingestion — manual creation only this module; a future ATS connector registers an `applicant` entity writer in the connector seam (post-M10, connector work); scheduling/EVV surfaces (Future Plans); structured availability editing (read-only expander here; editing arrives with the scheduling backlog item); bulk import; any resources/caregiver-roster UI beyond the hired-banner link.

## Shared conventions (both sub-plans)

- **Stages live in `applicants.stage`**; the ordered config (labels Applied/Screening/Interview/Offer/Hired/Rejected, tones: hired = success, rejected = muted) lives in `services/views/caregivers.py` + `frontend/src/lib/caregivers.ts` — the caregivers instance of M9's seam convention.
- **One stage-moving path**: `move_stage()` in `services/views/caregivers.py` validates the target, updates the row, emits `applicant.stage_changed` `{from, to, summary}`, and performs hired-promotion. Both the REST PATCH and the new gated `update_applicant_stage` tool delegate to it — one event emitter, one promotion path.
- **Sequence trigger convention mirrors M9**: stage `applied` → trigger `applicant.created`; every other stage → `applicant.stage_changed` + the managed `trigger.payload.to` condition. Bindings are `{"view":"caregivers","stage":…}` on the M9 core column — zero schema work here.
- **`rejected` gets a sequence chip** (unlike leads' `lost`): the PRD's automated denied email is the marquee use case. Chip-bearing stages are per-view config (`sequenceStages` in the pipeline view config), never hardcoded in shared components.
- Plain language everywhere; raw JSON (availability, payloads) only behind technical expanders. No new frontend deps, no new env vars, no engine changes.
