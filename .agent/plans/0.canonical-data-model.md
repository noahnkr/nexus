# Plan 0: Canonical Data Model

**Complexity: ⚠️ Medium** — mostly declarative SQL, single-pass feasible, but RLS correctness is subtle and must be *proven* by tests, not assumed. No LLM/agent surface involved, so it does not trip the 🔴 rule for Modules 3/5/8.

**Goal**: stand up the Supabase schema every other module builds on — core tables (identical across deployments), the senior-care entity instantiation (isolated so re-templating touches only its files), tenant RLS at the Postgres level, and seed data.

**Non-goals**: no backend/frontend code beyond a thin test harness; no ingestion, chat, or tools. Auth *users* come in Module 6 — this module only needs tenancy plumbing that auth will plug into.

---

## Key design decisions (locked in — do not re-litigate during build)

1. **Hosted Supabase project + Supabase CLI migrations.** Local `supabase start` requires Docker, which we don't assume on this Windows machine. Migrations live in `supabase/migrations/`, applied with `supabase db push`. Seed in `supabase/seed.sql` applied manually via `supabase db push --include-seed` or psql.
2. **Migration file separation** (the re-templating seam, per CLAUDE.md):
   - `<ts>_core_foundation.sql` — extensions, `tenants`, helper functions, triggers
   - `<ts>_core_tables.sql` — `external_ids`, `documents`, `document_chunks`, `events`, `tasks`, `pending_actions`
   - `<ts>_core_rls.sql` — RLS for all core tables
   - `<ts>_entities_senior_care.sql` — `regions`, `qualifications`, `leads`, `clients`, `resources`, `schedules` **and their RLS policies** (a new vertical replaces only this file)
3. **Tenant resolution**: helper function `app.current_tenant_id()` returns, in order: JWT claim `app_metadata.tenant_id` (for Supabase client / Realtime access), else GUC `request.app.tenant_id` (for the FastAPI backend, which will `SET LOCAL` it per request in Module 1+), else NULL (policy then denies). All RLS policies are `tenant_id = app.current_tenant_id()`.
4. **IDs**: `uuid` PK, `default gen_random_uuid()`. Timestamps `timestamptz default now()`. `updated_at` maintained by a shared trigger function `app.set_updated_at()`.
5. **Status fields**: `text` + `CHECK` constraints (easier to evolve than enums). Values:
   - `tasks.status`: `pending | in_progress | done | cancelled`; `tasks.priority`: `low | normal | high | urgent`
   - `pending_actions.status`: `pending | approved | rejected | executed | failed`
   - `leads.status`: `new | contacted | qualified | converted | lost`; `clients.status`: `active | paused | ended`; `schedules.status`: `scheduled | completed | cancelled | no_show`
6. **Embeddings**: `vector(1024)` (Voyage `voyage-3.5` default dim), nullable (chunks exist before embedding completes). HNSW cosine index created now — cheap at this scale.
7. **`events` immutability**: enforced two ways — no UPDATE/DELETE RLS policies exist, *and* a `BEFORE UPDATE OR DELETE` trigger raises an exception (protects even privileged roles from accidental mutation).
8. **`documents` parent table is core** (chunks need a parent; ingestion in Module 1 fills it): `id, tenant_id, filename, mime_type, storage_path, status (uploaded|processing|ready|failed), error, created_at, updated_at`.
9. **Test harness**: Python `venv` under `backend/`, `pytest` + `supabase-py` + `psycopg[binary]`, connecting with (a) two locally-signed tenant JWTs against PostgREST for RLS tests, (b) the direct Postgres connection string for structural/trigger tests. This seeds the `backend/` skeleton Module 1 will grow into.

## Tasks

### Task 1 — Project scaffolding
1. Create `.gitignore` (`.env`, `backend/venv/`, `__pycache__/`, `node_modules/`).
2. `supabase init` in repo root (installs `supabase/config.toml`). Install Supabase CLI via `scoop`/`winget` if missing.
3. User-provided step (blocking input): a hosted Supabase project must exist. Need from the user: project ref, DB password, anon key, service-role key, JWT secret. Record required names in `.env.example`:
   ```
   SUPABASE_URL=
   SUPABASE_ANON_KEY=
   SUPABASE_SERVICE_ROLE_KEY=
   SUPABASE_DB_URL=            # postgres connection string (session pooler)
   SUPABASE_JWT_SECRET=
   NEXUS_TENANT_ID=            # the single active tenant's uuid, set after seeding
   ```
4. `supabase link --project-ref <ref>`.

**Validation**: `supabase migration list` runs cleanly against the linked project (empty list, no auth errors).

### Task 2 — Core foundation migration
`core_foundation.sql`:
1. `create extension if not exists vector;` (pgcrypto/gen_random_uuid is built-in on PG15+).
2. `create schema if not exists app;`
3. `tenants` table: `id uuid pk`, `name text not null`, `created_at`. (No RLS-by-tenant on itself; RLS: a session may `select` only its own row.)
4. `app.current_tenant_id() returns uuid` — `security definer`, `stable`; reads `auth.jwt() -> 'app_metadata' ->> 'tenant_id'`, falls back to `nullif(current_setting('request.app.tenant_id', true), '')`. Wrap `auth.jwt()` in an exception-safe form so it also works over direct psycopg connections where the `auth` schema function errors without a request context.
5. `app.set_updated_at()` trigger function.
6. `app.forbid_mutation()` trigger function (raises exception; used by `events`).

**Validation**: `supabase db push` succeeds; `select app.current_tenant_id()` over psycopg returns NULL, and returns the set value after `set request.app.tenant_id = '<uuid>'` (covered later by `test_schema.py`).

### Task 3 — Core tables migration
`core_tables.sql`, all with `tenant_id uuid not null references tenants(id)`, `created_at`, and (where mutable) `updated_at` + trigger:
1. `external_ids` — `entity_type text not null`, `entity_id uuid not null`, `source_system text not null check (source_system in ('crm','phone','ehr','email','manual'))`, `external_id text not null`, `last_synced_at timestamptz`. Unique `(tenant_id, source_system, external_id)`; index on `(tenant_id, entity_type, entity_id)`. (`entity_id` is *not* an FK — it points into per-vertical tables that vary; integrity is application-enforced by design.)
2. `documents` — per design decision 8.
3. `document_chunks` — `document_id uuid not null references documents(id) on delete cascade`, `chunk_index int not null`, `chunk_text text not null`, `embedding vector(1024)`, `entity_type text`, `entity_id uuid`, `source_system text`, metadata `jsonb default '{}'`. Unique `(document_id, chunk_index)`; HNSW index `using hnsw (embedding vector_cosine_ops)`.
4. `events` — `source_system text not null`, `event_type text not null`, `entity_type text`, `entity_id uuid`, `payload jsonb not null default '{}'`, `created_at`. **No updated_at.** Attach `forbid_mutation` trigger. Index `(tenant_id, created_at desc)` and `(tenant_id, entity_type, entity_id)`.
5. `tasks` — `title text not null`, `description text`, `status`/`priority` checks per decision 5, `originating_event_id uuid references events(id)`, `assigned_to text`, `due_at timestamptz`, `resolved_at timestamptz`.
6. `pending_actions` — `task_id uuid not null references tasks(id)`, `tool_name text not null`, `tool_input jsonb not null`, `status` check per decision 5, `resolved_at timestamptz`.

**Validation**: `test_schema.py` — over psycopg: all 7 core tables exist; inserting a task with `status='bogus'` raises; `updated_at` changes after an UPDATE on `tasks`.

### Task 4 — Entity tables migration (senior-care instantiation)
`entities_senior_care.sql` — all tables `tenant_id`-scoped with `created_at`/`updated_at`:
1. `regions` — `name text not null`, `zip_codes text[] not null default '{}'`.
2. `qualifications` — `name text not null`, `description text`. Unique `(tenant_id, name)`.
3. `leads` — `name`, `phone`, `email`, `source text`, `status` check, `region_id uuid references regions(id)`, `requirements jsonb default '{}'` (free-form care requirements: hours/week, needed qualification names, notes).
4. `clients` — `lead_id uuid references leads(id)`, `name not null`, `phone`, `email`, `status` check, `requirements jsonb default '{}'`.
5. `resources` (caregivers) — `name not null`, `phone`, `email`, `qualification_ids uuid[] default '{}'`, `region_ids uuid[] default '{}'`, `availability jsonb default '{}'` (weekday→time-range map; `availability_ref` from the PRD realized as inline jsonb this phase).
6. `schedules` — `resource_id uuid not null references resources(id)`, `client_id uuid not null references clients(id)`, `start_time`/`end_time timestamptz not null` with `check (end_time > start_time)`, `status` check.
7. **RLS for these six tables lives in this same file** (enable + force RLS, the four standard policies from Task 5) so a vertical swap is one-file.

**Validation**: `test_schema.py` extension — entity tables exist; inserting a schedule with `end_time <= start_time` raises; inserting a lead with a `tenant_id` that doesn't exist raises FK error.

### Task 5 — RLS migration (core tables)
`core_rls.sql`:
1. For every core table: `alter table … enable row level security; alter table … force row level security;`
2. Standard policies per table, all `to authenticated, anon` roles with `using/with check (tenant_id = app.current_tenant_id())`:
   - SELECT + INSERT + UPDATE + DELETE for `tasks`, `pending_actions`, `documents`, `document_chunks`, `external_ids`
   - SELECT + INSERT **only** for `events` (no update/delete policies exist)
   - `tenants`: SELECT only, `using (id = app.current_tenant_id())`
3. Note in a comment: the FastAPI backend (Module 1+) must connect as a **non-bypass role** (`authenticated` via PostgREST or a dedicated role) and `SET LOCAL request.app.tenant_id` per request; the service-role key bypasses RLS and is reserved for migrations/ops.

**Validation**: `test_rls.py` — mint two JWTs (HS256 with `SUPABASE_JWT_SECRET`, `role=authenticated`, `app_metadata.tenant_id` = tenant A / tenant B) and use each against PostgREST via supabase-py:
- Tenant A sees seeded tenant-A leads; tenant B gets zero rows from the same query.
- Tenant B cannot INSERT a task with tenant A's `tenant_id` (RLS `with check` rejects).
- An unauthenticated/claimless request sees zero rows everywhere.
- `test_events_immutable.py` — UPDATE and DELETE on an event row fail both via PostgREST (no policy) *and* via direct psycopg as table owner (trigger raises).

### Task 6 — Seed data
`supabase/seed.sql` (idempotent — fixed UUIDs + `on conflict do nothing`):
1. One tenant: `Nexus Demo Care Co.` with a fixed UUID (documented in `.env.example` as `NEXUS_TENANT_ID`).
2. 3 regions (with real-looking zip lists), 5 qualifications (e.g., CNA, dementia care, hoyer lift), 6 leads across all statuses, 3 clients (two converted from leads), 5 resources with overlapping regions/qualifications, 8 schedules (past + upcoming, mixed statuses), 3 `external_ids` rows mapping two leads and a client to fake CRM ids, 5 events (e.g., `lead.created`, `call.received`) with plain jsonb payloads, 2 tasks (one linked to an event) and 1 `pending_actions` row in `pending` — so Module 5's approval-gate UI has something real on day one.
3. A **second tenant** (`RLS Probe Tenant`, fixed UUID) with one lead and one task — exists solely so RLS tests prove cross-tenant invisibility against real rows.

**Validation**: seed applies twice without error (idempotency); `test_rls.py` row-count assertions above pass against seeded data.

### Task 7 — Vector smoke test + wrap-up
1. `test_vector.py` — over psycopg as tenant A: insert a document + 3 chunks with hand-made 1024-dim vectors, run `order by embedding <=> $1 limit 1`, assert the nearest chunk is the expected one; confirm the HNSW index exists in `pg_indexes`. Clean up rows after.
2. Update `PROGRESS.md`: Module 0 → `[x]` with a one-line summary and date.
3. Update `README.md` "Getting Started" with the actual steps (CLI install, link, push, seed, run tests).
4. Commit.

**Validation**: `pytest backend/tests` — full suite green in one run.

## Execution notes

- **Blocking user input**: Task 1 step 3 needs Supabase project credentials. Everything up to `supabase link` can be built without them; tests cannot run until they exist.
- **Order matters**: Tasks 2→3→5 are ordered migrations; Task 4's file must sort after Task 2 (needs `app.*` helpers) — CLI timestamps handle this if created in order.
- **Do not** add retrieval extras (tsvector columns, hybrid-search plumbing) here — that's Module 10, and CLAUDE.md's scale-discipline rule says keep Module 0 minimal.
