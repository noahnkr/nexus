# Plan 9b: Leads View — funnel, metrics & per-stage sequences

**Complexity: ⚠️ Medium** — one small core migration, additive API changes, and a constrained builder that *composes* 8b's already-built components rather than inventing new ones. The subtle part is the binding/trigger convention; it is fully specified below. Fallback split point: Tasks 1–2 (backend, curl-testable) vs Tasks 3–5 (frontend + live walk).

Parent: `.agent/plans/9.leads-view.md`. Requires 9a built (the `/leads` page, `lead.stage_changed`, `services/views/leads.py`) and Modules 7–8 (engine, automations API, builder components, vocabulary endpoint).

## Context

After 9a, staff can work leads one at a time but can't see the funnel shape or attach outreach to a stage. This sub-plan completes PRD interface #7: the clickable funnel strip with per-stage sequence chips, conversion metrics widgets, and the per-stage outreach sequence builder — deliberately less flexible than M8's free-form builder because the trigger is fixed by the stage. A sequence is an ordinary M7 automation tagged with a **binding**; the engine, approval gate, run history, and Automations Center all apply to it unchanged.

**User-locked** (parent plan): funnel strip filters the directory (no kanban); linkage via core `automations.binding` jsonb with one-sequence-per-stage.

**Non-goals**: caregivers instantiation (M10 — reuses binding + these components); lead scoring functions (M10); sequence analytics beyond the run info the Center already shows; multi-sequence-per-stage; drag-and-drop; editing a bound sequence's trigger/binding from the generic M8 builder (the Center links bound automations to the stage builder instead).

## Key design decisions (locked in — do not re-litigate during build)

1. **Migration `20260718000001_automation_binding.sql`** (core): `alter table public.automations add column if not exists binding jsonb;` (null = unbound, the default) + partial unique index `automations_binding_uniq on (tenant_id, (binding->>'view'), (binding->>'stage')) where binding is not null`. *Why core*: "this automation belongs to surface X, slot Y" is business-agnostic; M10 binds `{"view":"caregivers","stage":…}` with zero schema work.
2. **Binding API** (additive, `routers/automations.py` + `schemas.py`): create/PATCH accept optional `binding` — validated generically (object, non-empty string values, must contain `view`; ≤4 keys; 422 plain message otherwise); a unique-index violation surfaces as **409** "That stage already has a sequence — edit it instead."; `GET /api/automations` gains optional `view` query param (`binding->>'view' = :view`); `AutomationOut.binding` returned everywhere. The edit guard (8a) applies to bound automations unchanged. *Why generic validation*: core never interprets vertical stage names; the unique index and the builder enforce meaning.
3. **Sequence trigger convention** (generated by the stage builder, never hand-assembled by the user): stage **`new`** → trigger `{"type":"event","event_type":"lead.created"}` (entering the funnel *is* creation — webhook auto-creates and manual creates both fire it); every **other stage** → `{"type":"event","event_type":"lead.stage_changed"}` + a **managed first condition** `{"field":"trigger.payload.to","op":"eq","value":"<stage>"}` that the constrained builder renders as part of the fixed sentence, not as an editable chip. Documented edge: moving a lead *back* to `new` re-fires nothing (stage_changed to `new` has no listener); automation-sourced stage changes never dispatch (M7 loop guard) — sequences cannot chain.
4. **Funnel metrics: `GET /api/leads/metrics`** (`routers/leads.py` → `services/views/leads.py`): `{stages: [{stage, count}…] (all five), conversion_rate (converted ÷ total leads, %), new_last_7_days, avg_days_to_convert (mean of stage_changed-to-converted event time − lead created_at, null if none), top_sources ([{source, count}] top 5)}`. One endpoint, plain SQL over `leads` + `events` — no materialized views at this scale.
5. **`FunnelStrip`** (replaces 9a's chip row on `LeadsPage`) — **built generic in `components/pipeline/FunnelStrip.tsx`** *(amended at M10 planning: M10 renders the same component with a caregivers config)*: prop-driven — ordered stage descriptors (key, label, tone), counts, active filter key + `onSelect`, and a per-stage chip descriptor (none/active/paused + route). Segments render label, count, % of non-zero total as weight (min-width clamped), active-filter highlight; click toggles the existing `status` URL param. Which stages carry a **sequence chip** (⚡ active accent / paused muted / dashed "＋ sequence") is the *config's* `sequenceStages` — for leads: the four funnel stages, `lost` excluded (drop-off, not worked; M10's `rejected` differs, by config not component logic). Chips sourced from `GET /api/automations?view=leads` mapped by `binding.stage`; click routes to `/leads/stages/{stage}/sequence`. Refetch bound automations with the existing Realtime `automations` subscription pattern.
6. **`LeadMetrics`** row beside/above the strip: four `StatCard`s — In pipeline (non-terminal count), Conversion rate, New this week, Avg days to convert — plus a small top-sources list card. Semantic tones only; no charts this phase.
7. **`/leads/stages/{stage}/sequence`** (`StageSequencePage.tsx`) — **built view-config-driven from the start** *(amended at M10 planning: M10 adds a route + config, no new page)*: a `PipelineViewConfig` type in **`lib/pipeline.ts`** carries view name + labels, ordered stage descriptors + `sequenceStages`, entity event names, a trigger-convention builder (stage → trigger + managed condition), the tool allowlist, route helpers, and default sequence naming; **`lib/leads.ts` supplies the leads instance** (events `lead.created`/`lead.stage_changed`, allowlist `send_sms`/`send_email`/`create_task`, names "Leads · Contacted sequence") and registers it in a view-config map keyed by `binding.view`. The page itself: fixed sentence header "**When a lead enters *Contacted***" (no `TriggerSentence` editing — trigger + managed condition from D3 constructed on save, shown as prose); optional extra IF conditions via `ConditionChips`; THEN via 8b's `StepList`/`StepCard`/`SchemaForm`/`TemplateInsert` with the vocabulary filtered client-side by the config's allowlist; editable name/description. Save = standard `POST /api/automations` (or PATCH in edit mode) with `binding: {view:"leads", stage}` — server revalidates everything; new sequences start **paused** with an activate toggle on the page (M8 create precedent). Existing-sequence mode loads by binding, offers pause/resume, delete (confirm), and reuses 8a's edit-guard 409 UX (`cancelRunsAndRetry`). A 409 on save (double-create race) surfaces the plain message and links to the existing sequence.
8. **Center awareness**: `AutomationCard` and `AutomationDetailPage` show a binding chip ("Leads · Contacted") when `binding` is set, and their Edit affordance for a *registered* `binding.view` routes to that view's stage builder instead of the generic builder — `describeBinding()` + the rerouting both read the view-config map in `lib/pipeline.ts` (leads registers itself in `lib/leads.ts`; `lib/recipe.ts` stays vertical-free). *Why*: one editing surface per recipe — the generic builder would let the trigger drift from the binding.

- The M7 engine is untouched. If a gap appears (e.g. run entity linkage on `lead.stage_changed` events), fix it in `services/automations/` with a test — events already carry `entity_type`/`entity_id`, and the dispatcher copies them onto the run.
- Scope discipline: no new step types, no charts, no per-stage analytics beyond D4's endpoint, no M10 content.
