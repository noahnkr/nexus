# Plan 7a: Automations ‚Äî recipe model, tables & synchronous engine core

**Complexity: ‚öÝÔ∏è Medium** ‚Äî one migration, one new service package, one router. No background machinery (that's 7b): everything here executes synchronously inside a request or a test transaction, which keeps iteration tight. The engine's step semantics (gate pause, per-step commits, fail path) carry the risk; the validation matrix covers each step type explicitly. Fallback split point: Tasks 1‚Äì3 (engine works headless from tests) vs Tasks 4‚Äì5 (API + wrap-up).

Parent: `.agent/plans/7.core-automations-framework.md` (context + locked decisions live there; this plan is self-contained for execution).

## Context

The engine composes existing seams ‚Äî `execute_tool` (audited, gated), `log_event`, `get_anthropic()`, `tenant_tx` ‚Äî into a step runner over a declarative recipe. Nothing in this sub-plan runs in the background: a manual trigger executes the run in-request until it completes, parks at a `delay` (status `waiting`, `wake_at` set ‚Äî 7b wakes it), or pauses at a gated tool (status `waiting_approval` ‚Äî 7b resumes it).

User-locked (see parent): declarative-only conditions; fail + review task; one active run per (automation, entity); REST only. Locked here as design decisions: the recipe shapes (D2), per-step transactions (D5), the vertical entity-lookup seam (D4).

**Non-goals**: background loops, cron, event dispatch, approval resume (all 7b); UI and agent tools (M8); branching (linear stop-guards only); retries; per-automation concurrency overrides.

## Key design decisions (locked in ‚Äî do not re-litigate during build)

1. **Migration `supabase/migrations/<ts>_automations_infra.sql`** (one file, idempotent, house 4-policy RLS + `set_updated_at` on both tables):
   - `automations`: `id uuid pk default gen_random_uuid()`, `tenant_id` FK, `name text not null`, `description text`, `status text not null default 'paused' check (status in ('active','paused'))`, `trigger jsonb not null`, `conditions jsonb not null default '[]'`, `steps jsonb not null default '[]'`, `next_fire_at timestamptz` (cron bookkeeping, maintained from 7b), `created_by text`, timestamps.
   - `automation_runs`: `id`, `tenant_id`, `automation_id` FK `on delete cascade`, `status text not null default 'running' check (status in ('running','waiting','waiting_approval','completed','failed','cancelled'))`, `trigger_event_id` FK `events` (nullable), `entity_type text`, `entity_id uuid`, `context jsonb not null default '{}'`, `step_index int not null default 0`, `step_log jsonb not null default '[]'` (per-step trail ‚Äî M8's run-detail timeline reads this), `wake_at timestamptz`, `error text`, `finished_at timestamptz`, timestamps.
   - Concurrency guard (user-locked): `create unique index ... on automation_runs (tenant_id, automation_id, entity_type, entity_id) where status in ('running','waiting','waiting_approval') and entity_id is not null`.
   - Waker index: `(tenant_id, status, wake_at)`.
   - `alter table pending_actions add column if not exists automation_run_id uuid references automation_runs(id)` ‚Äî how 7b's approval hook finds the paused run.
   - Add `automations` + `automation_runs` to the `supabase_realtime` publication (guarded DO-block) ‚Äî M8's grid live-updates for free.
   *Why*: both are core tables (business-agnostic), so they live in a core migration and join the CLAUDE.md core-tables list.
2. **Recipe vocabulary** (`services/automations/recipe.py`, Pydantic discriminated unions; `validate_recipe()` raises plain-language `RecipeError`s):
   - Trigger: `{"type":"event","event_type":str,"source_system":str?}` ¬∑ `{"type":"cron","expression":str}` (5-field, validated with `croniter`) ¬∑ `{"type":"manual"}`.
   - Condition: `{"field":str,"op":str,"value":any?}` with `op ‚àà eq,neq,gt,gte,lt,lte,contains,not_contains,exists,not_exists`; `field` paths root at `trigger.` (the triggering event: `trigger.event_type`, `trigger.payload.‚Ä¶`), `entity.` (the linked canonical row), `context.` (accumulated step outputs). Top-level `conditions` is an AND-list evaluated at run start.
   - Steps (max 20):
     - `{"type":"tool","tool":str,"input":{‚Ä¶},"save_as":str?}` ‚Äî `tool` must exist in the tool registry at validation time; `input` values may contain `{{path}}` templates; on success `result.data` lands in `context[save_as]`.
     - `{"type":"delay","minutes":int}` (also `hours`/`days`, exactly one, ‚â•1) ‚Äî parks the run.
     - `{"type":"condition","conditions":[‚Ä¶],"on_false":"stop"}` ‚Äî mid-sequence guard; false ‚áí run completes early (recorded in the completion event). Linear only; no else-branch this phase.
     - `{"type":"function","function":str,"args":{‚Ä¶},"save_as":str?}` ‚Äî must exist in the function registry.
     - `{"type":"generate","prompt":str,"save_as":str,"model":"default"|"fast"?}` ‚Äî LLM text generation; `fast` maps to a cheap model setting (Haiku), `default` to `settings.chat_model`.
   *Why*: exactly the vocabulary the user specified (MCP tools, delays, waits, conditionals, custom functions, content generation), declaratively renderable by M8's sentence builder.
3. **Templates** (`services/automations/templates.py`): `render(value, scope)` deep-renders dicts/lists/strings, substituting `{{trigger.payload.name}}`-style paths from `{trigger, entity, context}`; non-string full-value references (`"{{context.score}}"` alone) preserve type. **Unresolvable path ‚áí `TemplateError` ‚áí run fails** (fail-loud beats sending an SMS with a blank name). No expressions, no eval.
4. **Entity lookup ‚Äî vertical seam** (`services/automations/entities.py`): `ENTITY_TABLES: dict[str, str]` mapping `entity_type` ‚Üí table (this instantiation: `lead`/`client`/`resource`/`schedule` ‚Üí their tables) + `get_entity(conn, entity_type, entity_id) -> dict | None` (single-row select, `_jsonable`-coerced). Feeds `entity.*` in conditions/templates. Unknown type ‚áí empty scope (conditions on `entity.*` then fail `exists`). *Why a new seam file*: read tools return tool-shaped summaries; conditions need the raw row; the map mirrors `entity_writers.py` and re-templates the same way.
5. **Engine** (`services/automations/engine.py`):
   - `start_run(conn, tenant_id, automation, *, trigger_event=None, entity_type=None, entity_id=None) -> run_id | None`: build scope (trigger event row ‚Üí `trigger.`, entity via D4 ‚Üí `entity.`), evaluate entry conditions ‚Äî all-true or no-run (condition-false start is **not** an event; it's normal filtering); insert the run row ‚Äî a unique-index violation (23505) ‚áí log `automation.run_skipped` event and return `None` (user-locked concurrency); log `automation.run_started`.
   - `advance_run(tenant_id, run_id)`: loops from `step_index`, **one `tenant_tx` per step** ‚Äî each step's effects + `context`/`step_index` bump commit together, so a crash resumes at the next step without replaying side effects. Step semantics:
     - `tool` ‚Üí `execute_tool(conn, tenant, name, rendered_input, source_system='automation')`. Result `data.status=='queued'` (the gate) ‚áí stamp `pending_actions.automation_run_id`, run `status='waiting_approval'`, stop (7b resumes). `is_error` ‚áí fail path. Else `save_as` ‚Üí context, next step.
     - `delay` ‚Üí `wake_at = now() + interval`, `status='waiting'`, stop (7b wakes; in 7a tests, wake_at is forced past and `advance_run` re-called).
     - `condition` ‚Üí all-true ‚áí next step; false ‚áí `status='completed'` with the completion event noting the stop (`"stopped early: condition not met at step N"`).
     - `function` ‚Üí registry handler; result ‚Üí context.
     - `generate` ‚Üí `get_anthropic().messages.create` (plain text, no tools, `max_tokens` capped); text ‚Üí `context[save_as]`.
   - Every step outcome appends one entry to `step_log` in the same per-step transaction: `{"index", "type", "summary", "status": "ok"|"queued"|"waiting"|"stopped"|"failed", "at"}` ‚Äî plain-language `summary` (the tool result summary, "waited 2 days", "generated message", ‚Ä¶) so M8 can render a run timeline without reconstructing it from events.
   - Exhausted steps ‚áí `status='completed'`, `finished_at`, `automation.run_completed`.
   - **Fail path** (user-locked): `status='failed'`, `error`, `finished_at`; `automation.run_failed` event; plain-language review task (`resolution.py` pattern) titled `"Automation failed: {name}"`, `originating_event_id` = the run_failed event. Exception: a gated tool that later fails **post-approval** already leaves its action `failed` + task pending (Module 5) ‚Äî 7b marks the run failed without a second task.
   - `@traceable(run_type="chain", name="automation_run")` around `advance_run` so LangSmith shows the step sequence.
6. **Function registry** (`services/automations/functions.py`): `FunctionDef(name, description, input_schema, handler(conn, args) -> jsonable)` + `register_function`/`get_function` (registry.py pattern) ‚Äî `input_schema` is JSON Schema like `ToolDef`'s, so M8's builder renders function-step forms the same way it renders tool forms. Core starters (deliberately minimal): `now()` ‚Üí ISO timestamp; `days_since(date)` ‚Üí int. Vertical scoring functions (M10) register here without core changes. Functions are safe-by-definition (no gate) ‚Äî anything with external effect must be a *tool*.
7. **API** (`routers/automations.py`, `tenant_conn`, Pydantic models in `schemas.py`, router in `main.py`):
   - `GET /api/automations?status=` ‚Äî list (name, status, trigger type, counts of active runs), newest first.
   - `POST /api/automations` ‚Äî create (validated via D2; 422 with the plain-language `RecipeError` message); status defaults `paused`.
   - `GET /api/automations/{id}` ‚Äî full definition.
   - `PATCH /api/automations/{id}` ‚Äî partial update of `name/description/status/trigger/conditions/steps`; definition changes revalidate; `status` flips active/paused (paused = no new runs; in-flight runs finish).
   - `DELETE /api/automations/{id}` ‚Äî 204 (runs cascade; events remain).
   - `POST /api/automations/{id}/run` `{entity_type?, entity_id?}` ‚Äî manual trigger (also allowed for event/cron automations ‚Äî it's the test/debug path): starts + advances synchronously in-request; returns the run row (may be `completed`, `waiting`, `waiting_approval`, or `failed`); 409 if skipped by the concurrency guard.
   - `GET /api/automations/{id}/runs` + `GET /api/automation-runs/{id}` ‚Äî run history/detail (status, step_index, context, error, timestamps).
   *Why runs in-request*: at this scale a recipe without delays finishes in well under a second (approvals set the precedent for synchronous execution).
8. **Dependencies/config**: `croniter` added to `requirements.txt` (validation needs it now; 7b schedules with it). New setting `fast_model: str = "claude-haiku-4-5-20251001"` for `generate`'s `fast` option. No new env *requirements* (both models overridable like `chat_model`).

## Tasks

### Task 1 ‚Äî Migration
D1: tables, RLS, indexes, `pending_actions.automation_run_id`, Realtime publication. `supabase db push`.

**Validation**: gated checks in `backend/tests/test_automations_api.py` (`db` fixture, direct SQL): both tables exist with the expected columns/checks; the partial unique index exists; `pg_publication_tables` lists both; `pending_actions.automation_run_id` present. Re-running the guarded blocks is a no-op.

### Task 2 ‚Äî Recipe models, templates, functions
D2 + D3 + D6 (and the D4 seam file). Offline ‚Äî no DB.

**Validation**: `backend/tests/test_recipe.py` (offline): each trigger/step shape round-trips; bad shapes produce plain-language errors (unknown tool name, unknown function, zero/negative delay, >20 steps, bad cron expression, bad op); template rendering resolves nested paths, preserves types on full-value refs, deep-renders dict/list, raises `TemplateError` on a missing path; `days_since` computes correctly.

### Task 3 ‚Äî Engine core
D5 (+ D4 lookup wired). Register a throwaway safe tool and a throwaway gated tool inside tests (the `test_approval_gate.py` trick) plus the real `send_sms` placeholder for one realistic path.

**Validation**: gated `backend/tests/test_automation_engine.py`:
- linear run: tool (safe, `save_as`) ‚Üí function ‚Üí condition (true) ‚Üí completes; `context` holds both outputs; `step_log` has one plain-language entry per step; events show `run_started` + `tool.called` (`source_system='automation'`) + `run_completed`;
- condition-false mid-sequence ‚áí `completed` early, later steps never ran;
- entry conditions false ‚áí no run row, no events;
- delay step ‚áí `waiting` + future `wake_at`; forcing `wake_at` past and re-calling `advance_run` completes the run;
- gated tool ‚áí `waiting_approval`, `pending_actions` row carries `automation_run_id`, run parked (resume is 7b ‚Äî not tested here);
- failing tool step ‚áí run `failed` with `error`, `automation.run_failed` event, review task created and linked;
- template referencing a missing path ‚áí fail path (not a crash);
- concurrency: second `start_run` for the same (automation, entity) while the first is `waiting` ‚áí `None` + `automation.run_skipped` event; different entity ‚áí runs.

### Task 4 ‚Äî Automations & runs API
D7 + schemas + router wiring.

**Validation**: gated `backend/tests/test_automations_api.py` (TestClient + `auth_headers`): create‚Üíget‚Üílist round-trip; invalid recipe ‚áí 422 with the plain message; PATCH flips status and revalidates definitions (bad edit ‚áí 422, row unchanged); manual run of a no-delay recipe returns a `completed` run with context; manual run hitting the guard ‚áí 409; runs list/detail; DELETE cascades runs; probe-tenant token sees nothing (RLS); no header ‚áí 401.

## Execution notes

- **Blocking user input**: none new ‚Äî standing env (`SUPABASE_DB_URL` + linked project for `db push`, `NEXUS_APP_DB_URL` for gated tests; `ANTHROPIC_API_KEY` only if a test exercises `generate` live ‚Äî otherwise mock/skip the LLM call, keeping `generate` coverage in 7b's live walk).
- Reuse: registry pattern (`services/tools/registry.py`) for functions; throwaway-tool registration and gate assertions (`tests/test_approval_gate.py`); review-task insert (`services/connectors/resolution.py`); `_jsonable` (`services/tools/core.py`); guarded-publication SQL (`20260716000002_task_approval_infra.sql`); TestClient + `auth_headers` (`tests/test_auth.py`, `test_home_api.py`); opaque-cursor pagination only if runs lists need it (start with plain `limit`, cap 100).
- Scope discipline: no background loops, no cron *scheduling* (validation only), no dispatcher, no approvals changes (7b); no agent tools, no UI (M8); engine touches tools **only** through `execute_tool`; do not add branching or retries however tempting mid-build.
- Non-technical-user rule: `RecipeError` messages, run summaries, and review-task titles are plain language ‚Äî recipe JSON belongs to the API and M8's technical expanders.
