## Context

The PRD's Module 18 summary: caregiver roster/utilization (headcount, active vs inactive, hours-this-week, utilization %), a credential expiry tracker (CPR, TB test, background check, license — "an expired credential can mean a caregiver legally can't work a shift"), and a retention/at-risk view. What exists: `resources` has contact/quals/regions/availability/address/zip/languages/traits but **no status flag and no dated credentials** — `qualifications` is an undated vocabulary and `resources.qualification_ids uuid[]` is a boolean "has it". The schedule router already serves a roster with `hours_this_week` (`_roster`, `week_hours_map`) and a minimal roster PATCH emitting `resource.updated`; `matching.availability_covers`/`_hm` parse the availability jsonb; the M11 `days_until` function and the M7 cron trigger make the expiry automation cheap; `CaregiverDrawer` (M12b) already edits contact/tags/availability.

**User-locked decisions (2026-07-19):**
1. **Tabs on `/caregivers`**: one people surface — Pipeline (existing hiring funnel, unchanged) | Roster (new), `?tab=` URL param. No new nav entry.
2. **Dated rows on qualifications**: new `resource_credentials` table — one row per (caregiver, qualification) with issued/expires dates + notes. `resources.qualification_ids` stays the matching input; credentials add dated evidence on top (only tracked ones need dates). Expiry status derived at read time: valid / expiring ≤60d / expired / no-expiry (null date).
3. **Retention/at-risk view DEFERRED** (Future Plans, with `hire_date`/tenure): this module ships roster + credentials + the expiry automation only.
4. **Safe tool + digest recipe**: safe `list_expiring_credentials(days_ahead)` tool; a documented daily-cron recipe calls it, IF count > 0 THEN a digest task (optionally a gated notify). Validated by a pytest that runs the recipe through the real engine. Per-credential granularity stays on the roster's read-time badges (per-credential `credential.expiring` events → Future Plans if a real need appears).

**Design decisions locked at planning time** (do not re-litigate during build):
1. **`resources.status`** (`active`/`inactive`, default active): inactive caregivers are excluded from matching candidates and the schedule-board roster — they exist only on the Roster tab (and history). Status changes emit a dedicated `resource.status_changed` event (stage-changed convention) so automations can trigger on deactivation; the existing roster PATCH stays the single resources writer.
2. **Read-time credential status, in-seam constants** (EVV-flag precedent): `EXPIRING_DAYS = 60`; no stored status column, no detector loop, no LLM.
3. **Utilization is deterministic seam math**: available hrs/wk parsed from the availability jsonb (reusing `matching._hm`); utilization % = scheduled hours this week ÷ available hours; caregivers with no declared availability get `null` (not 0%).
4. **Credential events ride the resource entity**: `credential.added` / `credential.updated` / `credential.removed` with `entity_type='resource'` so they land on the caregiver's timeline; the `credential.` prefix maps to `resource` in the automation entity map.
5. **The digest recipe is documentation + test, never seeded**: README publishes the recipe JSON (cron daily → tool `list_expiring_credentials` save-as → IF count > 0 → `create_task` digest); a pytest creates it through the standard API and drives one engine cycle to a created task. The standard create path stays the only writer of `automations` rows (CLAUDE.md).

**Non-goals** (name the destination): retention/at-risk flags + `hire_date`/tenure (Future Plans — user decision 2026-07-19); per-credential `credential.expiring` events with dedup state (Future Plans); credential document uploads (works today via M16's tagged uploads once built — no special wiring here); payroll/HR export (Future Plans); applicant-side credential capture during hiring (Module 10 scope if revisited); scheduling hard-blocks on expired credentials (matching already warns on missing qualifications — a hard block is a policy change deferred to Future Plans).
