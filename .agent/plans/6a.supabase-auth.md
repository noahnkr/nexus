# Plan 6a: Supabase Auth & Tenant Identity

**Complexity: ⚠️ Medium** — one focused rewrite of a deliberately narrow seam (`deps.py`), a mechanical sweep of API tests, and a small but cross-cutting frontend change (session provider + one fetch helper). No migrations, no new tables. The risk is coverage, not depth: every route and every test that touches the API is affected, so the validation matrix matters more than any single piece. Fallback split point: Tasks 1–3 (backend JWT-protected, testable via curl) vs Tasks 4–5 (frontend + ops).

Parent: `.agent/plans/6.control-center-shell.md` (context + locked decisions live there; this plan is self-contained for execution).

## Context

Today `deps.get_tenant_id()` returns `settings.nexus_tenant_id` unconditionally — anyone who can reach the API is the demo tenant. Everything downstream is already JWT-ready:

- `app.current_tenant_id()` resolves `auth.jwt() -> app_metadata ->> tenant_id` *before* the GUC, so RLS accepts real Supabase tokens as-is (Module 0).
- `conftest.mint_tenant_jwt()` mints HS256 tokens with `SUPABASE_JWT_SECRET` that PostgREST/Realtime accept — the legacy shared secret is still trusted by the hosted project.
- **Verified empirically**: the project's JWKS (`{SUPABASE_URL}/auth/v1/.well-known/jwks.json`) serves one **ES256** key — real Supabase-Auth access tokens are ES256-signed. Backend verification must accept both ES256 (JWKS) and HS256 (secret).
- Direct `get_tenant_id` consumers: `db.tenant_conn` (dependency chain → all data routers), `routers/documents.py`, `routers/chat.py`, `routers/tasks.py` (explicit `Depends`), `routers/webhooks.py:35` (machine path — must NOT become JWT), `services/mcp_server.py:57` (plain function call — must NOT become JWT), `routers/auth.py` (dev seam — deleted by this plan).
- Frontend Realtime auth is the dev seam: three pages call `api.getRealtimeToken()` + `supabase.realtime.setAuth(token)` (`IngestionPage.tsx:33`, `TasksPage.tsx:71`, `EventLogPage.tsx:92`).

User-locked (see parent): email + password; `/mcp` keeps the static bearer token; webhook ingress keeps HMAC verification.

**Non-goals**: sign-up/invite/password-reset UI (the one office user is dashboard-managed this phase); roles/permissions beyond tenant membership; multi-tenant user admin (post-validation); MCP OAuth; any UI restyling (6b — Task 4's login page is functional shadcn, 6b makes it beautiful).

## Key design decisions (locked in — do not re-litigate during build)

1. **`deps.get_tenant_id` becomes a request dependency verifying the Supabase JWT.** Signature `async def get_tenant_id(request: Request) -> str` — every existing `Depends(get_tenant_id)` call site (including the chain through `db.tenant_conn`) keeps working unchanged. Behaviour:
   - Extract `Authorization: Bearer <token>`; missing/malformed ⇒ **401**.
   - Dispatch on the unverified JWT header: `alg == "HS256"` ⇒ verify with `settings.supabase_jwt_secret`; `alg == "ES256"` (or any non-HS alg with a `kid`) ⇒ verify against the project JWKS via `jwt.PyJWKClient` (module-level singleton, `cache_keys=True`, URL `{SUPABASE_URL}/auth/v1/.well-known/jwks.json`). Both paths: `algorithms` pinned explicitly, `audience="authenticated"`, `exp` enforced. Any verification failure ⇒ **401** (fail closed — including unset secret/URL).
   - `app_metadata.tenant_id` missing/empty on a *valid* token ⇒ **403** ("no tenant claim") — distinguishes a real user we can't scope from an intruder.
   *Why*: HS256 keeps the offline test harness and any legacy tokens working; JWKS handles what Supabase Auth actually issues; header dispatch means no config knob to choose between them.
2. **`deps.get_current_user`** — same verification (decoded claims shared per-request via `request.state` so the token is verified once), returns `{"id": sub, "email": email}`. Used by Task 3; available to any future router.
3. **`deps.get_machine_tenant_id()`** — plain function returning `settings.nexus_tenant_id`; the *only* remaining reader of the env tenant at runtime. `routers/webhooks.py` switches its `Depends(get_tenant_id)` to it (its caller authenticates by HMAC signature, not JWT); `services/mcp_server.py:57` calls it directly (its ASGI wrapper already enforces `NEXUS_MCP_TOKEN`). Docstring notes the future: multi-tenant connector routing maps source→tenant via connector config, not env.
4. **The Realtime dev seam is deleted**: `routers/auth.py` removed (and its `main.py` include), `api.getRealtimeToken` removed, the three `setAuth` call sites removed. The frontend's signed-in supabase-js client forwards the session token to Realtime automatically (v2 behaviour on `SIGNED_IN`/`TOKEN_REFRESHED`), and the session token carries `app_metadata.tenant_id`, so `postgres_changes` RLS keeps working with zero replacement code.
5. **PyJWT gains its crypto extra**: `requirements.txt` + `cryptography` (ES256 verification; PyJWT is already pinned). No other backend deps.
6. **Frontend session layer** (`lib/auth.tsx`): `AuthProvider` owning `session`/`loading` via `supabase.auth.onAuthStateChange` + initial `getSession()`; `useAuth()` hook; `supabase.ts` flips to `persistSession: true, autoRefreshToken: true`. `LoginPage` (`/login`): email + password → `signInWithPassword`, error toast on failure, redirect to `/` on success (or the originally requested path). `RequireAuth` wrapper around the `AppShell` route: `loading` ⇒ splash, no session ⇒ `<Navigate to="/login">`. Sign-out button (temporary placement in the shell sidebar footer; 6b restyles it into the user menu) calls `supabase.auth.signOut()`.
7. **One authenticated-fetch helper**: `lib/api.ts` exports `authFetch(input, init?)` that awaits `supabase.auth.getSession()` and injects `Authorization: Bearer <access_token>`; every `api.*` method and `sse.streamChat` (and the FormData upload — headers merge, content-type untouched) route through it. A 401 response triggers `supabase.auth.signOut()` (expired/revoked session ⇒ back to login via the provider). No scattered header literals.
8. **`resolved_by` becomes real identity**: `routers/tasks.py` approve/reject endpoints add `user = Depends(get_current_user)` and pass `resolved_by=user["email"]` into the existing `approve_action`/`reject_action` parameters (Module 5 built the plumbing; this fills it).
9. **Test harness**: `conftest.py` gains `auth_headers` / `auth_headers_probe` fixtures (`{"Authorization": f"Bearer {mint_tenant_jwt(tenant)}"}` — HS256 path, no network). Every TestClient-based API test sends them; webhook/MCP tests are untouched (their auth is signature/bearer). New `test_auth.py` owns the verification matrix.
10. **Ops step (one-time, blocking live validation)**: create the office user in the Supabase dashboard (Authentication → Add user, email + password, auto-confirm), then attach the tenant claim:
    ```sql
    update auth.users
       set raw_app_meta_data = coalesce(raw_app_meta_data, '{}'::jsonb)
           || '{"tenant_id": "00000000-0000-0000-0000-000000000001"}'::jsonb
     where email = '<office-user-email>';
    ```
    Documented in README by Task 5. No new env vars — `SUPABASE_URL`/`SUPABASE_JWT_SECRET`/`SUPABASE_ANON_KEY` already exist; `NEXUS_TENANT_ID` remains for the machine seam, seeds, and tests.

## Execution notes

- **Blocking user input**: the Task 5 ops step (dashboard user + claim SQL) — flag it when reached; everything before it is buildable and testable with minted HS256 tokens.
- Reuse: `mint_tenant_jwt` (`tests/conftest.py:46`) is the canonical token shape — extend it with an optional `email` kwarg rather than writing a second minter; TestClient app-fixture pattern in `test_chat_api.py`; signed-webhook fixture helpers in `test_webhook_ingress.py`; gated-action queue/approve pattern in `test_approval_gate.py`.
- The JWKS client is lazy + cached (module singleton): offline test runs never touch the network because test tokens are HS256; don't fetch JWKS at import or startup.
- Scope discipline: do not restyle anything (6b) beyond a functional login form; do not touch `execute_tool`, approvals internals, or the SSE contract; do not add a "dev mode" auth bypass — fail closed is the house rule, and the test harness mints real-shaped tokens instead.
- Background tasks (`tenant_tx`) already receive `tenant_id` captured at request time — no changes there; verify ingestion still passes the *JWT-derived* tenant through (it comes from the same dependency).
