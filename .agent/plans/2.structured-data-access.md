# Plan 2: Structured Data Access

**Complexity: ⚠️ Medium.** Not on CLAUDE.md's default-🔴 list (no MCP server, approval gate, or harness — those are M3/M5/M8). The tool registry and read tools are linear and independently testable; the one risky piece is rewriting the chat turn into a multi-step agentic loop. If execution stalls, the natural split point is after Task 4 (tool layer complete and tested, chat still pure-RAG); Tasks 5–7 are the chat/frontend integration.

## Context

Module 1 delivered a running app: FastAPI on the RLS-subject `nexus_app` role (`db.tenant_tx`/`tenant_conn` set the tenant GUC per transaction), SSE chat with always-on RAG (`chat_service.stream_chat_turn`), Voyage embeddings, ingestion pipeline, and a Vite frontend. `chat_messages.content` stores Anthropic content-block JSON **verbatim**, deliberately forward-compatible with the `tool_use`/`tool_result` blocks this module introduces. Module 0 seeded a full demo data set (leads in all 5 statuses, 3 clients, 5 resources with overlapping regions/qualifications, 8 schedules) — the read tools have real rows to answer from on day one.

Module 2 builds the **structured data access layer** per the PRD: named, parameterized read tools over the entity schema, a read-only text-to-SQL reporting tool, and — per user decision — wires them into chat as a proper agentic tool loop, with document retrieval converted from always-injected context into a `search_documents` tool the model routes to.

**User-locked decisions**:
1. Tools are wired into chat this module (Anthropic `tools` param, `tool_use`/`tool_result` blocks, multi-step turn) — not deferred to M3's MCP server.
2. RAG becomes a tool: `search_documents` replaces per-turn context injection; the model routes each question to structured tools, vector search, or both (the PRD success criterion).
3. Reads only — no write tools until Module 5's approval gate exists to resolve them. Text-to-SQL enforcement: existing `nexus_app` connection inside `SET TRANSACTION READ ONLY` + statement validation; no new Postgres role or ops step.

**Non-goals**: MCP server exposure of these tools (M3 — it will wrap this registry), webhook connectors (M3), write tools and `pending_actions` execution (M5), auth (M6), n8n nodes (M7), the matching harness (M8 — `can_we_serve_this_lead` is explicitly NOT a Module 2 tool), hybrid search/reranking (M10). No new migrations and no new env vars.

## Key design decisions (locked in — do not re-litigate during build)

1. **Tool layer lives in `backend/app/services/tools/` with a single execution seam.** `core.py` defines `ToolDef` (dataclass: `name`, `description`, `input_schema` JSON Schema, async `handler(conn, args) -> ToolResult`, `safe: bool`) and `execute_tool(conn, tenant_id, name, args) -> ToolResult`. `ToolResult` = `{summary: str, data: Any}` — `summary` is the plain-language line for user-facing surfaces and the audit event; `data` is what gets JSON-serialized into the `tool_result` block. `execute_tool` is the **only** entrypoint (chat now, MCP in M3, n8n nodes in M7): it validates args are a dict, refuses `safe=False` tools with a "gated tools arrive in Module 5" error, catches handler exceptions into an `is_error` tool result, and **always writes one `events` row** (`source_system='chat'` passed by caller, `event_type='tool.called'`, payload `{tool_name, summary, input, error?}` — technical detail belongs in the payload per CLAUDE.md; summary is the plain-language part). All M2 tools are `safe=True` (read-only).
2. **Registry split mirrors the re-templating seam.** `registry.py` holds `register()`, `get_tool()`, `anthropic_tool_defs()` (list of `{name, description, input_schema}` dicts ready for the API, with `cache_control: {"type":"ephemeral"}` on the **last** tool def so the whole tools array caches). Core tools (`documents.py`, `reporting.py`) are business-agnostic and never reference vertical concepts. **`entities.py` is the vertical seam** — senior-care read tools plus `SQL_SCHEMA_DOC` (the table/column reference injected into the reporting tool's description); a new vertical replaces `entities.py` the same way it replaces the entity migration.
3. **Entity read tools (7), generic naming, handlers are plain parameterized SQL on the already-tenant-scoped connection — RLS does all tenant filtering, no `tenant_id` in any input schema:**
   - `list_leads(status?, region?, search?, limit=20)` — region by name (ILIKE join to `regions`), search ILIKE on name/email/phone; returns rows + region name.
   - `get_lead(lead_id)` — lead + region name + its `external_ids` rows.
   - `list_clients(status?, search?, limit=20)`.
   - `get_client(client_id)` — client + `external_ids` + next 5 upcoming schedules (joined resource name).
   - `list_resources(qualification?, region?, search?, limit=20)` — qualification/region filters resolve names → ids against the reference tables, then `= any(qualification_ids)`; returns resolved qualification/region **names**, not UUID arrays.
   - `get_resource_availability(resource_id)` — availability jsonb + upcoming `scheduled` schedules (the PRD's `get_caregiver_availability`, generic name per the re-templating rule).
   - `list_schedules(client_id?, resource_id?, status?, date_from?, date_to?, limit=50)` — joined client/resource names, ordered by `start_time`.
   All UUID params validated with `uuid.UUID()` before hitting SQL (clean tool-result error, not a psycopg exception). Timestamps serialized ISO-8601; `data` must be JSON-serializable (shared `_jsonable` helper).
4. **`search_documents(query, top_k=8)` (core, `documents.py`)** wraps the existing `retrieval.retrieve_chunks` (add an optional `top_k` cap ≤ 8 to its signature; default unchanged). `data` = numbered source list `{n, document_id, filename, chunk_id, chunk_index, snippet, chunk_text}`; numbering is **turn-global** (the chat loop passes a starting index so a second search in the same turn continues [9], [10]… instead of restarting at [1]).
5. **`run_report(sql, purpose)` (core, `reporting.py`) — read-only text-to-SQL with three independent layers:**
   - *Guard* (`sql_guard.py`, pure function `validate_report_sql(sql) -> str`): strip trailing semicolon; reject multiple statements; must start with `select` or `with`; reject forbidden keywords as word-boundary matches anywhere (`insert|update|delete|merge|truncate|drop|alter|create|grant|revoke|copy|execute|call|do|vacuum|analyze|set|reset|listen|notify|refresh|lock|comment|security|pg_read|pg_sleep`), reject `for update/share/no key update/key share` and `into`; extract table identifiers (regex on `from`/`join` targets) and require every one to be in the **allowlist**: the six entity tables + `events`, `tasks`, `pending_actions`, `external_ids`, `documents` (NOT `document_chunks` — embedding column, and content is `search_documents`' job; NOT `chat_*`, `tenants`).
   - *Postgres enforcement*: executed on a dedicated `tenant_tx` where the first statements are `set transaction read only` and `set local statement_timeout = '5s'` — even a guard bypass cannot write or hang the pool.
   - *Output cap*: `fetchmany(201)`, truncate to 200 rows, set `truncated: true` in `data` (`{columns, rows, row_count, truncated}`).
   Tool description embeds `SQL_SCHEMA_DOC` from `entities.py` and instructs: single SELECT, tenant filtering is automatic (never filter by tenant_id), for aggregate/reporting questions the parameterized tools can't answer.
6. **Chat turn becomes an agentic loop** (rewrite of `stream_chat_turn`): persist user message → load history → loop (max **5** iterations): call `client.messages.stream(model, system, tools=anthropic_tool_defs(), messages=…)`; stream text deltas as today; on `stop_reason == "tool_use"`, persist the assistant message (content blocks **verbatim**, incl. `tool_use`), then for each tool block: emit SSE `tool` event → `execute_tool` on a fresh `tenant_tx` → emit SSE `tool_result` event; persist ONE user-role message whose content is the `tool_result` blocks (verbatim — the existing `role in ('user','assistant')` check accommodates this, matching the Anthropic contract); append both to the in-memory message list and continue the loop. On `stop_reason == "end_turn"` (or iteration 5, where the call is made with `tool_choice={"type":"none"}` to force an answer): persist final assistant message with accumulated citations + usage metadata and finish. History replay for later turns is automatic — messages were stored verbatim.
7. **System prompt & caching**: single static system block (persona + tool-usage guidance: route structured questions to entity tools, document questions to `search_documents`, cite search results `[n]` inline, use `run_report` only for aggregates; answer directly without tools when history suffices) with `cache_control` — the per-turn retrieved-context block is **gone**. Cache breakpoints: last tool def + system block. Usage metadata sums input/output tokens across all loop iterations.
8. **SSE contract (additive — existing events unchanged)**: `start` → per tool call `tool {name, label, tool_use_id}` and `tool_result {tool_use_id, summary, is_error, sources?}` → `citations {sources}` (now emitted once, before the final text stream, aggregating all `search_documents` results this turn; empty list if none — frontend behavior on empty unchanged) → `text {delta}`* → `done {assistant_message_id, usage}`. `label` is a plain-language progress line derived from the tool + args (e.g. "Searching documents for 'morning visits'…", "Looking up leads…") — a `TOOL_LABELS` map in `chat_service.py`; **no raw JSON in any user-facing field** (args/data stay in LangSmith and the `events` payload).
9. **Citations persistence**: assistant message `citations` column keeps the same source-dict shape as M1 (turn-global `n`); `metadata` gains `tool_calls: [{name, summary, is_error}]` so the frontend can render tool activity on history reload without parsing content blocks.
10. **Frontend (minimal, additive)**: `MessageList` renders only `text` blocks of `user`/`assistant` messages and **hides** messages whose content is entirely `tool_result` blocks; new `ToolActivity.tsx` renders a muted chip row (label + ✓/✗) from live `tool`/`tool_result` SSE events and, on reload, from `metadata.tool_calls`. `lib/sse.ts` parser already generic (named events) — only `ChatPage` state handling grows. Sources panel unchanged.
11. **Observability**: `@traceable` on `execute_tool` (`run_type="tool"`, name = tool name) and keep the `chat_turn` chain span — the LangSmith trace must show `chat_turn` → interleaved model calls and tool spans (CLAUDE.md validation rule). No config changes; no new env vars; README untouched.

## Tasks

### Task 1 — Tool core: `ToolDef`, registry, `execute_tool` audit seam

**Validation**: `backend/tests/test_tools_entities.py::test_execute_tool_seam` (nexus_app-gated, existing `_require` skip pattern) — register a throwaway tool, `execute_tool` returns its result AND an `events` row with `event_type='tool.called'` + plain-language summary exists; a `safe=False` throwaway is refused; a raising handler yields `is_error=True` and still writes the event.

## Execution notes

- **No blocking user input for Tasks 1–5** beyond what Module 1 already required: nexus_app-gated tests need `NEXUS_APP_DB_URL` in `.env`; Task 7's live validation needs `ANTHROPIC_API_KEY`/`VOYAGE_API_KEY` (+ LangSmith key for the trace check). Same `_require()` skip pattern keeps the suite green regardless.
- Reuse: `tenant_tx` (`app/db.py`), `log_event` (`services/events.py`), `traceable` re-export (`app/llm.py`), conftest fixtures (`mint_tenant_jwt`, `db`, `set_tenant`, `_require`), basis-vector pattern (`tests/test_retrieval.py`), scripted-stream pattern (`tests/test_chat_api.py`).
- Anthropic specifics: tool defs use `input_schema` (not `parameters`); `cache_control` goes on the last tool AND the system block; `tool_result` blocks live in a **user**-role message and must reference `tool_use_id`; when streaming with tools, accumulate the final message via `stream.get_final_message()` to persist content blocks verbatim.
- Scope discipline: no write tools, no MCP transport, no `pending_actions` writes, no `can_we_serve_this_lead` composite (M8), no hybrid search. If a tool feels like it needs to *change* anything, it belongs in Module 5.
- Non-technical-user rule: every string that can reach the UI (`summary`, `label`) is plain language; args/rows/errors detail goes in `events.payload` and LangSmith only.
