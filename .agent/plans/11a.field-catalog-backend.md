# Plan 11a: Field catalog & calculation backend

**Complexity: ⚠️ Medium** — additive vocabulary/schema work plus one small, well-tested engine semantics change; no migrations, no new env vars, no new endpoints. Fallback split point: Tasks 1–2 (catalog + function, purely additive) vs Task 3 (engine change).

Parent: `.agent/plans/11.automation-field-tokens.md`. Requires Modules 7–8 built. 11b consumes this sub-plan's `field_catalog`.

## Context

The builder's field knowledge today is `Vocabulary.field_roots` (`["trigger","entity","context"]`) and `field_suggestions` — one flat, unlabeled, trigger-blind list built in `routers/automations.py::_build_vocabulary` (core trigger paths + `trigger.payload.*` keys pooled across all observed event types + every entity table's columns via `services/automations/entities.py::entity_field_suggestions`). 11b needs to answer, for the trigger the user actually picked: *which fields exist, what are they called in plain language, and which entity is this run about?* That structure is served here. Alongside it: the one function the user selected (`days_until`), the condition-value rendering inconsistency found in the audit, and a draft prompt that teaches the agent the same catalog so drafted recipes use real paths.

**User-locked** (parent plan): score results context-only; `days_until` is the only new function; recipe format unchanged.

**Non-goals**: any new step type or recipe field; score persistence/tooling; `count_events`/`calculate`/`tier` (backlog); removing `field_roots`/`field_suggestions` (kept — nearly free to derive, and `FieldCombobox` consumes them until 11b upgrades it).

## Key design decisions (locked in — do not re-litigate during build)

1. **`Vocabulary` gains `field_catalog`** (new Pydantic models in `schemas.py`; built in `_build_vocabulary`):
   ```jsonc
   "field_catalog": {
     "trigger_fields": [ {"path": "trigger.event_type", "label": "Event type"}, … ],   // 5 core fields, static
     "payload_by_event": { "lead.created": [ {"path": "trigger.payload.phone", "label": "Phone"}, … ] },
     "entities": { "lead": {"label": "Lead", "fields": [ {"path": "entity.status", "label": "Status"}, … ]} },
     "event_entity": { "lead.created": "lead", "applicant.stage_changed": "applicant" }
   }
   ```
   *Why this shape*: `payload_by_event` + `event_entity` let 11b filter to the selected trigger; `entities` keyed by type keeps a `lead.created` session from seeing applicant columns. One `FieldRef {path, label}` model is reused everywhere.
2. **`payload_by_event` from observed events, grouped**: `select event_type, jsonb_object_keys(payload) as k from events where jsonb_typeof(payload)='object' and source_system <> 'automation' group by 1, 2` — the existing pooled query with `event_type` added. Labels humanized (`_humanize("hours_per_week")` → "Hours per week"); the automation-sourced exclusion mirrors the event-type facet query.
3. **`event_entity`**: observed mapping (`select event_type, entity_type … where entity_type is not null`, most-frequent per event type) **unioned with** the prefix heuristic `event_type.split('.')[0] in ENTITY_TABLES` so core-known-but-unobserved types (`_KNOWN_EVENT_TYPES`) still map. Observed wins on conflict. *Why*: empirical where possible, sensible before any events exist.
4. **`entities` from the vertical seam**: `services/automations/entities.py` gains `ENTITY_LABELS: dict[str,str]` (`{"lead": "Lead", "client": "Client", "resource": "Caregiver", "schedule": "Visit", "applicant": "Applicant"}`) and `entity_catalog(conn) -> dict` returning `{type: {label, fields:[{path,label}]}}` from `information_schema` per table (skip `tenant_id`; keep `id` — it feeds tool inputs like `lead_id`). `entity_field_suggestions` re-derives its flat list from this (one query path, no drift). *Why seam*: entity labels are vertical content; core only humanizes.
5. **Core label helper `_humanize`** lives beside `_build_vocabulary` (snake → sentence case); the seam may override per-column labels later if a real need appears — not built now.
6. **`days_until`** in `services/automations/functions.py`: mirror of `days_since` (same `_parse_dt`), returns `(then - now).days` — whole days until the date, negative if past. Description written for the builder dropdown ("How many days from now until a date — e.g. a credential expiry…").
7. **Condition values render templates, unresolvable → condition false** (never a run failure): in `engine.py::_eval_condition`, `value` passes through `render(value, scope)`; a `TemplateError` makes the condition evaluate **false**. *Why false, not fail-loud*: `_get_path` already treats a missing *field* as found=False (condition false), and entry conditions run before any run exists to fail — "a comparison against a value that isn't there is simply not true" is one rule for both entry and step conditions. `wait_until` keeps its existing behavior (it already rendered conditions; align it to the same false-on-TemplateError rule). `field` is never rendered — it stays a path.
8. **Draft prompt** (`services/automations/draft.py`): the TEMPLATING/RULES sections gain the catalog — per-event payload paths (only for event types the vocabulary lists), per-entity fields, the event→entity note ("for trigger X the entity is a Lead with fields: …"), `days_until` arrives via the existing functions listing. Keep it compact (paths only, no labels — the agent doesn't need them). *Why*: drafts must reference paths that resolve, or the drafted recipe fails at run time.
9. **No migration, no new env vars, no frontend changes here** (11b owns the consumers; `field_suggestions` keeps today's frontend working unmodified in the interim).

D6.

### Task 3 — Condition-value template rendering
D7.

**Validation**: gated cases in `backend/tests/test_automation_engine.py`: a condition step `{field: "entity.region_id", op: "eq", value: "{{trigger.payload.region_id}}"}` passes when they match and stops the run (condition-false path, not run-failure) when the payload key is absent; an entry condition with a `{{context.…}}` value (context empty at entry) skips the automation rather than crashing the dispatcher; a plain literal value behaves exactly as before (regression guard on the existing condition tests).
