# Plan 14a: WelcomeHome CRM Sync

**Complexity: ⚠️ Medium** — the largest sub-plan of Module 14: it lands the shared connector-sync infrastructure (loop + ingest seam) and the full WelcomeHome integration. **Fallback split**: Tasks 1–4 (infrastructure, WH client, migration) stand alone and unblock 14b/14c; Tasks 5–7 (mapping, runner, backfill) are the WH content pass.

See the parent plan `.agent/plans/14.external-connectors.md` for the shared conventions (ingest seam, sync loop, secrets-in-env, fixtures discipline) and the planning-time API findings — they are assumed here, not restated.

## Context

WelcomeHome is the CRM system of record (account 18754, one community, id 65648). The API has **no webhooks**; sync is polling `GET /api/exports/community/{cid}/table/{table}` (live paginated CSV, `filters[updated_at_after]`, `Link`-header cursor, ≤3 reuses of a cursor per minute) with JSON reference endpoints for config (`/stages`, `/activity_types`, `/lead_sources`, `/prospect_fields`). Auth: `Authorization: Token token={WELCOMEHOME_API_KEY}` (verified live, HTTP 200). User-locked scope: prospects + stages + backfill; activities/messages/calls → lead timelines; call transcriptions → RAG; residents + influencers/relationships. One-way inbound.

**Live stage vocabulary → Nexus lead statuses** (locked mapping; the Nexus funnel does NOT change this module — re-templating the Leads view to mirror WH stage names is a possible fast-follow, noted in Future Plans):

| WelcomeHome stage (`system_type`, position) | Nexus `leads.status` |
|---|---|
| Inquiry (`new_lead`, 0) | `new` |
| Contact Attempted (1), Contact Made (2) | `contacted` |
| Home Visit Scheduled (3), Home Visit Completed (`visit`, 4) | `qualified` |
| Start of Care (`move_in`, 5) | `converted` |
| discarded / close_reason set | `lost` |

The map is **config, not code**: keyed by `system_type` where present, else position band, so a renamed WH stage doesn't break it. Unknown stage ⇒ leave status unchanged + `connector.sync_failed`-style warning event (never crash).

## Key design decisions (locked in — do not re-litigate during build)

1. **Sync loop core** (`services/connectors/sync.py`): `SyncRunner` protocol (`source`, `enabled(settings)`, `run(conn, state) -> new_state`), a module registry, and `connectors_cycle()` invoked from a lifespan task every `NEXUS_CONNECTORS_POLL_SECONDS` (default 120). One `tenant_tx` per runner run; cursor state read/written on the `(tenant, source_system)` row of `connector_state` (state key layout per source, e.g. `{"cursors": {"Prospects": "...", "Activities": "..."}}`). Failure ⇒ `connector.sync_failed` event with plain summary + technical detail, runner skipped this cycle, loop continues. `connector_sync` traceable chain span per cycle with per-runner child spans. Justification: mirrors the proven M7 loop pattern exactly.
2. **Ingest seam** (`services/connectors/ingest.py`): move the post-verification body of the webhook route (receipt event → adapter.normalize → route_normalized_event, incl. the existing trace span) into `ingest_payload(source, payload, headers)`. The route keeps verify + HTTP concerns; runners call it directly. Behavior-identical: **all existing `test_connectors` tests must pass unmodified.**
3. **Resolution gains an update path**: `NormalizedEvent` gets `updates_entity: bool = False` + reuses `attributes`; `entity_writers.py` gains `UPDATERS = {"lead": update_lead}` (patch only non-null attributes; status changes go through `services/views/leads.update_lead_status` so `lead.stage_changed` fires with `source_system='welcomehome'` via the existing contextvar seam — sequences and timelines react for free). On a `matched` outcome with `updates_entity`, resolution applies the updater before logging. No updater registered ⇒ today's log-only behavior. Justification: the M3 WellSky docstring already named "applying field updates" as the real adapter's job; this is the minimal core extension that keeps unknown types safe.
4. **Migration `20260724000000_entities_crm_sync.sql`** (vertical seam): `alter table public.leads add column if not exists zip text, add column if not exists address text, add column if not exists background text;` plus new table `lead_contacts` (`id uuid pk`, `tenant_id`, `lead_id fk → leads on delete cascade`, `name text not null`, `relationship text`, `phone text`, `email text`, `is_primary boolean default false`, `source text`, timestamps) with the standard 4-policy RLS + `set_updated_at`; no Realtime (timeline events cover UI freshness). Residents: the *primary* resident's name/details feed the lead row (a WH "prospect" is the deal; the resident is the care recipient); additional residents and all influencers become `lead_contacts` rows (`relationship` = WH relationship label or `resident`). Justification: keeps `leads` the single pipeline entity (M9 unchanged) while giving related people a queryable home.
5. **WH client** (`wh_client.py`): httpx-based; `ping()`, `reference(name)` for the JSON config endpoints, and `export_pages(table, updated_at_after)` — an async generator yielding parsed CSV row-dicts page by page, following the `Link` header and sleeping to respect the 3/min cursor-reuse limit. Timeouts + bounded retries; raises a typed error the runner converts to `connector.sync_failed`.
6. **Mapping** (`wh_map.py`): pure functions, fixture-tested — `map_prospect(row, refs) -> (attributes, status)` (name from primary resident else prospect fields; phone/email; `source` = lead-source name; zip/address/background from resident/custom fields discovered at build time via `/prospect_fields`; `notes`), `map_activity(row, refs) -> NormalizedEvent` (`lead.activity_logged`, summary "«Type name»: first line of notes", `detail` carries type/direction/notes/completed_at; activity `record_type/record_id` links it to the prospect's external id), `map_contact(row)` for Influencers/extra Residents. `refs` (stages/activity_types/lead_sources/prospect_fields) are fetched once per cycle and cached in the runner. Which activity types represent messages/calls/transcriptions is **discovered at build time from the live `activity_types` list** (Task 5 records the chosen type→treatment table in code comments + README); transcription-bearing activities route their text to ingestion (decision 8).
7. **Events used**: `lead.created` (auto-create, existing), `lead.updated` (update path), `lead.stage_changed` (via `update_lead_status`), `lead.activity_logged` (new — declared in `EVENT_ENTITY_TYPES`/`EVENT_PAYLOAD_FIELDS` and `_KNOWN_EVENT_TYPES` so automations/tokens see it). Activity idempotency: the Activities cursor only moves after a committed batch, and each activity event's `detail.wh_activity_id` lets the runner skip rows already ingested in a partially-replayed page (checked against the last-batch id set kept in cursor state — cheap, no new table).
8. **Transcriptions → RAG**: `services/ingestion` gains an `ingest_text(title, text, entity_type, entity_id, source)` entry (same parse→chunk→embed pipeline minus file parsing; document row `storage_path` null-able or a `text://` marker per existing schema constraints — build checks the `documents` DDL and takes the minimal path). Only activities whose type is in the transcription set and whose notes exceed a threshold (~500 chars) are ingested, tagged to the lead. Justification: chat retrieval over call transcripts was the user-locked ask; reusing the pipeline keeps chunks/embeddings/citations uniform.
9. **Backfill** (`scripts/backfill_welcomehome.py`): operator-run, idempotent, resumable (its own cursor key in `connector_state`), ordered Prospects → Residents → Influencers → Activities, prints per-table counts, never prints record contents. Events for backfilled history are written with their original `occurred_at`. Rate-limit aware (the 3/min cursor rule). Justification: one-time import is an ops action, not a loop concern.
10. **New settings**: `welcomehome_api_key`, `welcomehome_community_id` (default `"all"`), `nexus_connectors_enabled` (default true), `nexus_connectors_poll_seconds` (default 120), `welcomehome_backfill_since` (optional ISO date bound for Activities backfill). All optional; the WH runner is enabled only when the key is set.

## Tasks (strict order)

### Task 1 — Config + WH client + live smoke
Settings additions (decision 10); `wh_client.py` (decision 5) with sanitized CSV/JSON fixtures modeled on the live shapes (spec: `swagger.yaml`; re-download if needed).
**Validation**: offline `test_wh_client.py` — export pager follows a fixture `Link` chain across 3 pages and parses CSV rows; env-gated live test `test_wh_live_ping` asserts `ping()["account_id"] == 18754`.

### Task 2 — CRM-sync migration + seam threading
Push the migration (decision 4); seed a couple of `lead_contacts` rows; thread `lead_contacts` awareness where the seams expect it (`SQL_SCHEMA_DOC`; the M11 entity catalog picks the new lead columns up automatically from `information_schema` — verify, don't assume).
**Validation**: gated schema pytest — new lead columns exist, `lead_contacts` RLS blocks the probe tenant, cascade delete works; vocabulary field catalog now lists `entity.zip` for leads.

### Task 3 — Ingest-seam refactor
Extract `ingest_payload` (decision 2); rewire the webhook route.
**Validation**: the **existing** `test_connectors` suite passes unmodified; a new direct-call test ingests a signed-fixture payload via `ingest_payload` and sees the same receipt + resolution events as the HTTP path.

### Task 4 — Sync loop core
`sync.py` (decision 1) + lifespan wiring + `NEXUS_CONNECTORS_*` flags; a fake in-test runner proves the contract.
**Validation**: gated `test_connector_sync.py` — cursor advances on success; a raising runner writes `connector.sync_failed` and doesn't block a healthy runner in the same cycle; disabled flag ⇒ no cycles; uvicorn boot smoke with the loop on and off.

### Task 5 — WH mapping + resolution update path
`wh_map.py` (decision 6) with the stage map table above; `base.py`/`resolution.py`/`entity_writers.py` changes (decision 3); `lead.activity_logged` declarations (decision 7). Build-time discovery: pull the live `activity_types` list and record the message/call/transcription type mapping.
**Validation**: offline `test_wh_map.py` — prospect fixture → attributes (resident name wins, source label resolved, zip/background mapped); each stage-map band; activity fixture → `lead.activity_logged` with plain summary; gated resolution test — a matched `lead.updated` with `updates_entity` patches the row and a WH stage move lands `lead.stage_changed` with `source_system='welcomehome'`.

### Task 6 — WH runner + backfill + transcription ingestion
`wh_runner.py` (incremental: Prospects, Residents, Influencers, Activities with per-table cursors, refs cache, contact upserts); `ingest_text` seam (decision 8); `backfill_welcomehome.py` (decision 9).
**Validation**: offline runner test over fixture pages (two cycles: second cycle fetches only post-cursor rows, no duplicate events); gated `ingest_text` test (document + chunks rows, entity-tagged, retrievable via `search_documents`); **live**: run the backfill against the real account, then `list_leads` over chat/MCP returns real prospects, a lead profile timeline shows imported activities, and a transcription is retrievable in chat. (Real data lands in the DB — summaries in test output only, never dumped to the console.)

### Task 7 — Wrap-up
README: connectors section (env vars, sync-loop behavior, backfill runbook, stage-map table); `.env.example`; full `pytest backend/tests` green; `npm run build` untouched-but-verified.
**Validation**: full pytest green; a live incremental cycle observed end-to-end — change a prospect's stage in WelcomeHome, within one poll interval the lead's status updates and `lead.stage_changed` appears in the Event Log with `source_system='welcomehome'`, and any bound stage sequence fires (LangSmith `connector_sync` span present).
