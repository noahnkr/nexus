# Plan 4: Event Log

**Complexity: ✅ Simple.** Not on the default-🔴 list. Read-only surface over the existing immutable `events` table that every module already writes to — no new writers, no new core tables, one guarded migration line (Realtime publication). Linear, single-pass executable.

## Context

Modules 0–3 made `events` the system's audit trail: document lifecycle (`document.uploaded/processing/ready/failed`), chat turns (`chat.message.completed`), every tool call from chat and MCP (`tool.called`, with `source_system` distinguishing them), webhook receipts (`webhook.received`), and connector resolutions (`lead.created`, `call.completed`, `email.received`, …). RLS scopes it per tenant; a `forbid_mutation` trigger plus SELECT+INSERT-only policies keep it append-only. What's missing is the user-facing surface: PRD interface #5, the business-facing counterpart to LangSmith.

Audit of existing rows (verified in code, 2026-07-16): `tool.called` and all connector events set a plain-language `payload.summary`; `document.*` and `chat.message.completed` do **not** — so the Event Log derives summaries at read time rather than expecting them.

**User-locked decisions**:
1. **Filters + entity drill-down** — source system, event type, date range, *and* `(entity_type, entity_id)` filtering so "everything that happened to this lead" works; the `events_entity_idx` index already covers it. Drill-down is URL-addressable (query params) so later entity views can deep-link.
2. **Pure feed** — no aggregate/stats strip; aggregation and "needs attention" surfaces belong to Control Center Home (Module 6).

**Non-goals** (name the module): task/approval surfaces and resolving anything from this page (M5); the unified needs-attention queue, nav shell rework, and auth (M6); entity detail pages that would link *into* the drill-down (M6+/M9 plugin views); LangSmith trace links in the UI (developer detail stays in LangSmith itself); no new event writers and **no backfill** of summaries onto existing rows (events are immutable). No new env vars.

## Key design decisions (locked in — do not re-litigate during build)

1. **Migration `supabase/migrations/<ts>_eventlog_realtime.sql`**: add `public.events` to the `supabase_realtime` publication, guarded exactly like `20260715014929_ingestion_infra.sql` does for `documents` (idempotent re-run). Nothing else — RLS SELECT policy already exists, and Realtime respects it via the tenant JWT the frontend already fetches.
2. **Summary derivation is read-path only** (`services/event_summaries.py`, core, business-agnostic): `summarize_event(event_type, source_system, payload) -> str` with precedence: non-empty `payload.summary` → per-type template for the known core types (`document.uploaded` → "Document '{filename}' uploaded", `document.processing/ready/failed` likewise from `payload.filename`, `chat.message.completed` → "Assistant replied in chat", `webhook.received` → "Received a webhook from {source_system}") → generic fallback: humanize the event_type (`lead.created` → "Lead created"). Pure function, no vertical seam needed — connector/tool events self-describe via `payload.summary`, which stays the convention for all future writers (CLAUDE.md gains this line).
3. **Events API (`routers/events.py`), tenant-scoped via the standard `tenant_conn` dependency — RLS does all filtering:**
   - `GET /api/events?source_system=&event_type=&entity_type=&entity_id=&since=&until=&cursor=&limit=50` — `limit` capped at 100; `since`/`until` ISO-8601; keyset pagination ordered `(created_at desc, id desc)` with an opaque cursor (base64 of `created_at|id`, both applied as a row-comparison `(created_at, id) < (%s, %s)`), immune to inserts between pages. Response `{"events": [...], "next_cursor": str | null}`; each event: `id, created_at, source_system, event_type, entity_type, entity_id, summary` (derived per D2), `payload` (raw jsonb — the PRD-sanctioned technical detail lives here, inline because rows are small at this scale; no separate detail endpoint).
   - `GET /api/events/facets` → `{"source_systems": [...], "event_types": [...]}` via `select distinct` (fine at low-hundreds scale; keeps filter dropdowns business-agnostic instead of hardcoding a vertical's event names).
   - Pydantic response models (`EventOut`, `EventPage`, `EventFacets`) in `schemas.py`; router registered in `main.py`.
4. **Frontend — route `/events`, nav item "Event Log"** (ScrollText icon) in the existing `AppShell` nav (M6 reworks the shell; this is just a third entry):
   - `pages/EventLogPage.tsx`: filter state ↔ URL query params (so `?entity_type=lead&entity_id=…` deep-links a drill-down), initial fetch + "Load more" button driven by `next_cursor` (explicit button, not a scroll observer — simpler and testable), live tail via Supabase Realtime INSERT subscription on `events` reusing `IngestionPage`'s token/channel pattern; a live row is prepended only if it matches the active filters (client-side predicate mirroring the server filters); derive its summary client-side is NOT done — live rows arrive without the derived summary, so the page calls `summarize` … no: keep one source of truth — live Realtime payloads are rendered with `payload.summary ?? humanized event_type` via a tiny `fallbackSummary()` in `lib/events.ts` that mirrors only the generic fallback; on the next fetch the server-derived summary replaces it. (Realtime is a convenience layer; the API is the truth.)
   - `components/events/EventFilters.tsx` (selects fed by the facets endpoint + date inputs + active entity-filter chip with clear button), `components/events/EventRow.tsx` (timestamp, source badge, plain-language summary, entity chip that applies the drill-down filter; expandable `<pre>` with pretty-printed payload JSON — the sanctioned technical-detail view), skeleton loading state.
   - `lib/api.ts` gains `listEvents(params)`, `getEventFacets()` + types.
5. **No LangSmith work**: this is a pure read path with no agent involvement; `@traceable` adds nothing (the events being *displayed* already carry their trace context in LangSmith).

## Execution notes

- **Blocking user input**: none beyond the standing Module 1 env (`NEXUS_APP_DB_URL` for gated tests; `SUPABASE_DB_URL` + linked project for `supabase db push`; frontend `.env` + running stack for browser/live checks). No new env vars, no new ops steps.
- Reuse: `tenant_conn` (`app/db.py`), fixtures `_require`/`db`/`app_db`/`set_tenant`/tenant ids (`tests/conftest.py`), app-client test pattern (`tests/test_chat_api.py`), Realtime token + channel pattern (`frontend/src/pages/IngestionPage.tsx`), badge/table/skeleton UI primitives (`frontend/src/components/ui/`), guarded-publication SQL (`supabase/migrations/20260715014929_ingestion_infra.sql`).
- Scope discipline: read-only module — no new `log_event` call sites, no summary backfill (immutable), no stats/aggregates (M6), no task actions (M5). If a summary reads badly, fix the template in `event_summaries.py` or the *future* writer's `payload.summary` — never mutate stored events.
- Non-technical-user rule: the feed line is always the plain-language summary; raw JSON exists only inside the row expander (the PRD-sanctioned technical detail), pretty-printed, never in the summary line or chips.
