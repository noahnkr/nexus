- **`16b.clients-view-frontend.md`** (⚠️ Medium) — the `/clients` directory page with a census strip (authorized vs scheduled vs delivered hours — the revenue-leakage number), the `/clients/{id}` care-overview profile (info, care card, family contacts, hours, assigned caregivers, visits with EVV badges, client documents, timeline), and the schedule board's EVV surfaces (check-in/out in the drawer, late/missed badges).

Build order: **16a → 16b** (16b renders 16a's API exclusively — no backend work in 16b).

**Requires Modules 0–12 built** (all are): the M12 schedule seam + board carry the EVV columns and drawer actions; the M9/M10 view patterns (directory + profile + metrics + shared `SmartSummary`/`EntityTimeline`/`FunnelStrip`-style strip) are reused wholesale; the M13 `ui/Select` is used for every new dropdown. **Independent of Modules 14–15** (neither is built): nothing here touches connectors or the 15b Knowledge page — 16b adds its tagged-upload UI on the client profile only, so the Ingestion page rework in 15b lands without collision. Connector-fed EVV (GoTo telephony clock-ins, WellSky) later fills the same `check_in_at`/`check_out_at` columns through Module 14's ingest path.

## Context

Clients exist as a table (M0) with read tools and a gated `update_client_status`, but have **no view surface at all** — the operational heart of the business (who we serve, who serves them, are we delivering the hours we're paid for) is invisible outside chat. The PRD's Module 16 summary names three feature groups: **active census** (count of active clients by region/payer + authorized vs scheduled vs delivered hours — the authorized-minus-delivered gap is direct revenue leakage), **per-client care overview** (care plan, assigned caregivers, schedule, family contacts, status active/hospital-hold/discharged), and **EVV** (clock-in/out, missed/late visits — legally mandated for Medicaid-funded home care in most states).

What exists to build on: the `schedules` table with the full M12 status machine and the single-writer transition seam (`services/views/schedule.py`); `document_chunks` already has `entity_type`/`entity_id` columns (the parent `documents` table doesn't — this module fixes that asymmetry); the cached entity-summary service (`services/views/summary.py` + `entity_summaries`); the applicants router as the canonical directory/profile API shape; `clients` already gained `address`/`zip`/`languages`/`preferences` in M12a.

**User-locked decisions (2026-07-19):**
1. **In-app EVV-lite**: `check_in_at`/`check_out_at` on schedules, recorded from the visit drawer (and gated agent tools); late/missed computed **at read time** (no detector loop, no new engine machinery); connector-fed EVV plugs into the same columns in Module 14+.
2. **Care plans = documents + RAG**: upload gains optional canonical-entity tagging, the client profile lists that client's documents, chat searches them; plus one free-text `care_summary` field for at-a-glance context. No structured care-plan editor (Future Plans).
3. **Delivered hours = actuals, fallback scheduled**: for completed visits, check-in→check-out duration when both stamps exist, else the scheduled duration.
4. **Two sub-plans**, 16a backend → 16b frontend.

**Design decisions locked at planning time** (do not re-litigate during build):
1. **Clients become the fourth sanctioned vertical surface** — same pattern-core/content-seam split as Leads/Caregivers/Schedule: `services/views/clients.py` + `routers/clients.py` + the frontend pages are re-templating-seam members. CLAUDE.md's surface list is amended at planning time.
2. **Client statuses become `active` / `hospital_hold` / `discharged`** (replacing `paused`/`ended`; data-migrated `paused→hospital_hold`, `ended→discharged`). One writer for status changes: `views/clients.change_status()` emits `client.status_changed` — the REST PATCH and the `update_client_status` tool both delegate (the `move_stage`/schedule-seam precedent), so sequences and timelines see every move regardless of source.
3. **Census math is one seam function** (`census_metrics`), Monday-week window matching the board: authorized = Σ `authorized_hours_per_week` over active clients; scheduled = Σ duration of the week's planned visits (`scheduled`/`completed`/`no_show`/`called_out`; `open` reported separately as unfilled hours); delivered = Σ over completed visits of actual-else-scheduled duration; leakage = max(authorized − delivered, 0). Deterministic SQL — no LLM anywhere near the numbers.
4. **`documents.entity_type`/`entity_id` is a core change** (chunks already carry them; the parent must too so "this client's documents" is one query). Upload accepts the optional tag, validates the entity exists, and the pipeline stamps chunks with it — retrieval and chat citations work unchanged.
5. **Check-out completes the visit**: `check_out` sets `check_out_at` *and* `status='completed'` in one transition (`schedule.checked_out` event). `set_outcome` remains for manual bookkeeping when no clock data exists. New event types `schedule.checked_in`/`schedule.checked_out` + `client.status_changed` join the known-types/entity maps so automations can trigger on them (e.g. WHEN a visit is checked out THEN …).
6. **EVV flags are read-time and rule-based**: a `scheduled` visit with no check-in is `late` after a 15-minute grace, `missed` after its end time (in-seam constants, plain-language). No stored flag, no cron writer — `no_show` stays the explicit terminal record.

**Non-goals** (name the destination): connector-fed EVV / telephony clock-ins (Module 14b+); structured care-plan editor with goals/ADLs (Future Plans); billing/payroll export of delivered hours (Future Plans); referral-source dashboard (Module 17); credential expiry & workforce views (Module 18); a Home census StatCard (Future Plans — Home stays as-is this module); client deletion (never — statuses end a client, history stays).
