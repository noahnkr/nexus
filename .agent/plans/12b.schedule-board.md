# Plan 12b: Smart Staffing — schedule board & call-out flow

**Complexity: ⚠️ Medium** — the first calendar-grid UI in the app (everything so far is tables/cards), but it consumes a finished API and reuses the shell/drawer/dialog/Realtime patterns. Fallback split point: Tasks 1–2 (read-only board, independently shippable) vs Tasks 3–5 (actions + roster + wrap-up).

Parent: `.agent/plans/12.smart-staffing-scheduling.md`. Requires 12a complete (week feed, transitions, candidates, roster, notify).

## Context

12a leaves a fully-working scheduling/matching backend with no face. This sub-plan ships the `/schedule` week board — caregivers as rows, an Open-shifts row pinned on top, visits as status-colored blocks — and the owner's call-out loop: click the called-out visit's replacement → ranked candidates with plain-language reasons → assign → optionally queue the gated SMS notify → approve in Tasks. Plus the create dialog (with weekly repeat), a minimal caregiver drawer (the only roster-editing surface — address/zip/languages/traits/availability now feed matching), the Home open-shifts card, and the documented call-out automation recipe.

**User-locked** (parent plan): week board; owner-picks-then-gated-SMS; repeat-weekly create; no broadcast path in the UI.

**Non-goals**: drag-and-drop rescheduling (Future Plans — actions go through the drawer); day/month views (week only); a full roster page (drawer only); seeded automations (M8 lock — the call-out recipe is README-documented); client profile pages; mobile layout beyond not-broken.

## Key design decisions (locked in — do not re-litigate during build)

1. **`lib/schedule.ts`**: week helpers (Monday-start `weekStartOf`, `addWeeks`, day columns array), `VISIT_STATUS_META` (label/tone per status: `scheduled` accent, `open` warning, `called_out` muted-striped, `completed` success-muted, `cancelled`/`no_show` muted-ghost), types mirroring the 12a payloads, and `api.ts` additions (`getScheduleWeek`, `createVisits`, `patchVisit`, `callOutVisit`, `assignVisit`, `getCandidates`, `getRoster`, `patchRosterMember`, `notifyCaregiver`).
2. **`SchedulePage`** (`/schedule`, nav **Schedule** with lucide `CalendarDays` after Caregivers): `PageHeader` + "New visit" button; week navigation `◀ This week ▶` with `?week=YYYY-MM-DD` round-tripped to the URL (the M9 filter↔URL precedent); one `getScheduleWeek` fetch feeding the whole board; Supabase Realtime `postgres_changes` on `schedules` → debounced refetch (the M8 precedent); `EmptyState` when the roster is empty.
3. **`ScheduleBoard`** — CSS grid, columns Mon–Sun (+ a sticky name column), rows: pinned **Open shifts** row (warning-toned, shown only when the week has `open` visits) above one row per caregiver sorted by name, each name cell showing `hours_this_week` under the name and opening the caregiver drawer. Cells stack `VisitBlock` chips (start–end + client name, status tone per D1, `replaces_schedule_id` visits get a "covering" hint) sorted by start time — **day-column blocks, not hour-positioned geometry** (visits are 2–10h; a time-scaled axis is complexity the board doesn't need). Horizontal scroll inside the board on narrow viewports.
4. **`VisitDrawer`** (sheet, right side — the app's drawer precedent): header = client name + day/time + status badge; body = caregiver (or "Unassigned"), required qualifications as chips, notes, technical expander with the raw row. Actions by state:
   - `scheduled` (future): **Call out** (confirm dialog: "Record a call-out for Alicia and open a replacement shift?" → `callOutVisit` → drawer follows to the replacement shift), **Reassign** (opens the candidate list), **Cancel** (confirm).
   - `scheduled` (past): **Mark completed** / **No-show** (PATCH outcome).
   - `open`: **`CandidateList`** fetched on open — ranked cards: name, score badge, `reasons` as plain chips, `warnings` amber; **Assign** on each → on success, an inline "Text ⟨name⟩ about this shift?" prompt with a prefilled editable message (visit day/time/client) → **Queue text** calls `notifyCaregiver` → amber "Text queued for approval" chip linking to `/tasks` (the 5b queued-chip precedent). Declining just closes — assignment already happened.
   - `called_out`/`cancelled`/terminal: read-only + a link chip to the replacement/original via `replaces_schedule_id`.
5. **`VisitCreateDialog`**: client select (from facets/roster feed), optional caregiver select ("Leave unassigned → creates an open shift" hint), date + start/end time inputs, required-qualification multi-select, notes, and **"Repeat weekly until"** date input (≤12 weeks out, mirroring the backend cap with client-side validation); on submit `createVisits` → toast with created count. Errors (series conflict) surface the backend's plain message verbatim.
6. **`CaregiverDrawer`** (from the board name cell): contact, address/zip, `languages`/`traits` tag editors (simple chip input), availability editor as per-day time-range rows writing the existing `{"mon":["08:00-16:00"]}` shape, hours-this-week readout; save → `patchRosterMember` (one `resource.updated` in the timeline/Event Log). *This is deliberately the only roster surface* — a full roster view stays in Future Plans.
7. **Home card**: 12a's week feed already counts open shifts cheaply — add `open_shifts` to `GET /api/home/summary` (count of `open` visits with `start_time >= now()`) and a warning-toned `StatCard` ("Open shifts", `CalendarDays`, deep-link `/schedule`) on the Home grid.
8. **Documented call-out recipe** (README, not seeded — M8 lock): WHEN `schedule.called_out` → tool `find_available_caregivers` (schedule id from `{{trigger.payload.replacement_schedule_id}}`) → safe `create_task` titled with the top candidate — plus the gated `send_sms` variant as the "AI dispatch" extension. Buildable end-to-end in the M8 builder with M11 tokens; the wrap-up live walk builds it for real.

## Execution notes

- **Blocking user input**: the Module 6 office user + running stack for every browser validation (the standing situation); nothing else new.
- Reuse (file paths): drawer/sheet + confirm patterns from `components/tasks/` and `components/caregivers/`; Realtime debounce from `AutomationsPage.tsx`; queued-amber-chip from `components/chat/ToolActivity.tsx`; `StatCard` from `components/home/`; URL-param state from `LeadsPage.tsx`; tag-chip editing can crib the qualification multi-select in `ApplicantCreateDialog.tsx`.
- Keep board day-columns dumb: no drag-and-drop, no hour-scaled positioning, no visit resizing — actions live in the drawer.
- All copy plain-language (reasons/warnings straight from the API); scores show as a small badge, never a table of numbers; raw row JSON only in the technical expander.
- Both themes: status tones must come from the semantic tokens (`--warning`/`--success`/`--info`) — no hex literals (the 6b convention).
