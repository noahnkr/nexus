# Plan 7: Core Automations Framework (parent)

**Complexity: 🔴 Complex** — CLAUDE.md defaults Module 7 to complex. It is the largest pure-backend module yet: a new recipe vocabulary, two new core tables, a step-executing engine with durable state, and three asynchronous machineries (event triggers, cron, wait/wake) plus an approval pause/resume integration. Per the planning rule it is **broken into two sub-plans**:

- **`7a.recipe-model-and-engine.md`** (⚠️ Medium) — the synchronous foundation: migration (`automations`, `automation_runs`, `pending_actions.automation_run_id`), the WHEN/IF/THEN recipe vocabulary + validation, template rendering, the function registry, the step-executing engine core (runs a recipe until it completes, parks at a delay, or pauses at an approval gate), and the REST CRUD + manual-run API.
- **`7b.triggers-scheduler-durability.md`** (⚠️ Medium) — the asynchronous half: in-app engine loops (event-trigger dispatcher with a durable cursor, cron scheduler, due-wait waker, stale-run recovery) and the approval resume/cancel hook in `services/approvals.py`.

Build order: **7a then 7b** — 7b's loops advance runs that only exist because 7a's engine and API do. If 7a stalls, its API task is a clean fallback split (recipes validate and execute synchronously via curl before any background machinery exists). If 7b stalls, the dispatcher (Task 2) alone already delivers the headline use case shape (event-triggered runs); cron/waker can land separately.

## Context

Everything the engine needs already exists as a seam:

- `execute_tool()` (`services/tools/core.py`) is the single audited execution path — automation steps call it with `source_system='automation'`, so tool steps are audited, and **gated tools gate automatically** (the run pauses; nothing new needed at the gate).
- `services/approvals.py` is the only `approved_action_id` caller — the natural place for the run resume/cancel hook once `pending_actions` knows which run queued it.
- `log_event` (`services/events.py`) is the single events writer; the dispatcher watches the `events` table (poll + durable cursor), so *anything* that lands in the audit trail — webhooks, chat tool calls, approvals — can trigger an automation.
- `llm.py.get_anthropic()` (LangSmith-wrapped) backs the content-generation step; `settings.chat_model` is the default model, with a cheap-model option per the Haiku-for-high-volume stack rule.
- The lifespan already hosts a long-running service (`mcp session_manager.run()`) — the engine loops follow the same pattern.
- `deps.get_machine_tenant_id()` is the machine-path tenant seam — the engine loops run under it (single-tenant phase; multi-tenant iteration is a documented future seam).

**User-locked decisions (2026-07-16):**
1. **IF conditions are declarative only** — field comparisons (path, operator, value; AND-list) over `trigger.*` / `entity.*` / `context.*`. Scoring-style logic = a *function step* computes a value into `context`, then a declarative condition compares it. No function predicates, no LLM-evaluated conditions this phase.
2. **Step failure ⇒ fail the run + plain-language review task** (the `resolution.py` pattern) plus an `automation.run_failed` event. No retries this phase.
3. **One active run per (automation, entity)** — a new trigger while one is in flight is skipped with an `automation.run_skipped` event; enforced by a partial unique index. Entity-less automations are unconstrained.
4. **Module 7 is engine + REST only** — no builder UI, no agent-facing automation tools; the Automations Center (grid, recipe builder, agent drafting) is M8. A manual `POST /api/automations/{id}/run` endpoint makes the whole engine curl-testable.

**Non-goals** (name the module): any UI including run-history views (M8); agent tools to create/trigger automations and the describe-it drafting flow (M8); the pre-defined lead/caregiver stage sequences and their constrained builders (M9/M10); vertical scoring functions (M10 registers them into 7a's function registry); real SMS/email delivery (the gated placeholders queue exactly as they do from chat — real credentials arrive with M9/M10 connector work); step branching beyond linear stop-guards, per-automation concurrency overrides, retries/TTLs (future, when a real automation needs them); fuzzy matching (M11); multi-tenant engine iteration (post-validation — the loop-under-machine-tenant seam is where it lands).

## Shared conventions (both sub-plans)

- **Every step effect goes through an existing seam**: tool steps via `execute_tool` (never handlers directly), audit via `log_event`, LLM via `get_anthropic()`. The engine adds orchestration, never a second execution path.
- `source_system='automation'` on everything the engine writes (tool calls, run lifecycle events) — Event Log and LangSmith distinguish automation activity for free.
- Run lifecycle event vocabulary (all with `payload.summary` naming the automation in plain language): `automation.run_started`, `automation.run_completed`, `automation.run_failed`, `automation.run_skipped`, `automation.run_cancelled`.
- **One transaction per step**: the engine commits `context`/`step_index` after each step (via `tenant_tx`), so a crash mid-run resumes at the next step instead of replaying side effects.
- **Loop guard**: the event dispatcher never matches events whose `source_system='automation'` — automations cannot trigger automations this phase (prevents cycles; funnels trigger off connector/user/chat events).
- Recipes are validated, declarative JSON — no code and no LLM output in the control path; the only LLM surface is the `generate` step's *content*.
- Plain language everywhere users will look (M8 renders these): automation names, run summaries, review-task titles; raw recipe JSON is API/expander material.
