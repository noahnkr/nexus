# Plan 11b: Token inputs & builder UX

**Complexity: ⚠️ Medium** — frontend-only, but the contenteditable token input is genuinely fiddly; it is isolated behind a pure, vitest-covered tokenizer and one component so the risk is contained. Fallback split point: Tasks 1–2 (tokenizer + `TokenText`, independently testable) vs Tasks 3–5 (picker, integration, calculation editor, read mode).

Parent: `.agent/plans/11.automation-field-tokens.md`. Requires 11a built (`field_catalog` in the vocabulary). The stage-sequence builders (`StageSequencePage`) and draft review (`DraftBox`) reuse these components and inherit everything with no extra work.

## Context

Today the builder shows raw `{{trigger.payload.phone}}` text in inputs, `TemplateInsert` appends incomplete stubs the user must finish blind, `FieldCombobox` offers one flat unlabeled path list, and `weighted_score` degrades to a raw-JSON textarea. This sub-plan replaces that surface: template references render as labeled atomic chips inside the inputs, insertion is cursor-positioned from a searchable picker grouped by *the selected trigger's* actual fields, and `weighted_score` gets a field × weight editor. The stored recipe string is unchanged throughout — chips are a view, not a format.

**User-locked** (parent plan): keep the insert-button + dropdown pattern; user never types dotted paths (picker supplies them; a custom-path escape hatch remains for power users); score results context-only — no score UI beyond the step editor.

**Non-goals**: recipe-shape changes; expression syntax; drag-and-drop; score persistence/display; changes to `RunTimeline`'s technical context expander (raw JSON stays raw there by design — it's the technical detail surface).

## Key design decisions (locked in — do not re-litigate during build)

1. **`lib/template.ts` — pure tokenizer, no React**: `parseTemplate(s: string): Segment[]` (`{kind:"text"|"token", value}` — token value is the trimmed path), `serializeTemplate(segs): string`, and `labelForPath(path, catalog?, contextKeys?)`. Label resolution: catalog match (trigger fields, payload of any event, entity fields — prefixed "Lead — Status" when the entity is known) → `context.<k>` → "Step result: k" → fallback humanized tail (`trigger.payload.phone` → "Phone"). Round-trip guarantee: `serializeTemplate(parseTemplate(s)) === s` for every string, including malformed braces (kept as text). *Why pure*: this is the only intricate logic; vitest covers it exhaustively and `TokenText` stays thin.
2. **Vitest added** (`vitest` dev dep, `"test": "vitest run"` script). *Why now*: the tokenizer is the first frontend logic where a browser check can't prove correctness; `npm run build` + browser stays the validation for everything visual.
3. **`TokenText.tsx` — the chip input**: a controlled contenteditable (`value: string` in, `onChange(string)` out via parse/serialize), `multiline?: boolean` (replaces `Input` vs `Textarea`), chips as `contentEditable={false}` inline spans (primary-tinted pill, label from `labelForPath`, title tooltip = raw path), atomic backspace/delete, `insertToken(path)` exposed via ref (picker inserts at the caret; falls back to append when unfocused), paste as plain text re-parsed (pasted `{{…}}` becomes chips). Number-ish fields: `SchemaForm`'s existing behavior is preserved — plain numbers keep the native number input; a field switches to `TokenText` when templated (the `isTemplate` branch that exists today).
4. **`FieldPicker.tsx` replaces `TemplateInsert` (deleted)**: same trigger-button pattern (Braces icon), popover with a search box and grouped, labeled entries (label primary, mono path secondary):
   - **From the trigger event** — 5 core fields + `payload_by_event[trigger.event_type]` (event triggers only).
   - **The Lead / Applicant / …** — `entities[event_entity[event_type]]` fields; for cron/manual (or unmapped events) the group is replaced by a one-line hint: *"Scheduled and manual runs aren't about a specific record, so there are no record fields here."* — the audit's "what is entity for" answered in-place.
   - **Earlier step results** — `contextKeys` as "Step result: message".
   - **Custom path…** — footer row revealing a mono input (escape hatch; free text still valid everywhere).
5. **One `FieldContext` prop object** `{vocabulary, trigger, contextKeys}` threaded from `AutomationBuilderPage`/`StageSequencePage` through `StepList` → `StepCard` → `SchemaForm`/`ConditionChips` (replacing today's bare `contextKeys` prop). *Why*: the picker needs the trigger everywhere; one object stops the prop sprawl at three renames.
6. **Integration sweep** (behavior identical, inputs upgraded): `SchemaForm` string fields → `TokenText` + `FieldPicker`; `StepCard` generate prompt → multiline `TokenText` + `FieldPicker`; `ConditionChips` value `Input` → single-line `TokenText` (11a made condition values render); `FieldCombobox` (condition *field* side — a path, not a template) keeps free text but its popover shows grouped labeled entries from the same catalog data instead of the flat `field_suggestions`.
7. **"Run a calculation"** — plain-language function UX, recipe JSON unchanged (`function` step, `weighted_score`):
   - `StepList` add-menu relabel: "Compute a value" → **"Run a calculation"**; new-step default is `{type:"function", function:"weighted_score", args:{weights:{}, inputs:{}}, save_as:"score"}`.
   - `lib/recipe.ts` gains `FUNCTION_LABELS` (`weighted_score` "Calculate a score", `days_since` "Days since a date", `days_until` "Days until a date", `now` "Current date & time") used by the function select and `describeStep`.
   - **`CalculationEditor.tsx`** renders when `function === "weighted_score"` (others keep `SchemaForm`): rows of *[value — single-line `TokenText`, typically one chip or a literal number] × [weight — number input]*, add/remove row; factor keys auto-slugged from the chip's label or path tail (deduped `_2` suffix), invisible to the user; writes `{weights, inputs}`; a live formula line ("Score = 3 × Hours per week + 2 × Days since created"); `save_as` defaulted to `score` with helper copy "Use it later as Step result: score — e.g. in a condition or message."
   - Opening an existing recipe whose `weights`/`inputs` keys mismatch or hold exotic values falls back to the raw `SchemaForm` (never data loss).
8. **Read-mode labels**: `describeStep`/`readDetail` (StepCard), `describeCondition`, and `RunTimeline`'s step titles pass template strings through a `labelizeTemplate(s, catalog?)` helper (`lib/template.ts`) that replaces each `{{path}}` with the chip label text — "Send SMS to Phone" instead of "Send SMS to {{trigger.payload.phone}}". Raw JSON stays raw in technical expanders.

## Execution notes

- **Blocking user input**: none new — browser checks need the running stack + Module 6 office user (standing caveat since M6).
- Reuse: popover open/close pattern (`TemplateInsert`/`AddStepMenu` — click-outside + ref), `FieldCombobox` keyboard navigation (arrow/enter/escape) for the picker's search results, badge/pill styling from `Badge` variants, `isTemplate` branch in `SchemaForm` for the number-vs-token switch.
- contenteditable discipline: keep ALL string logic in `lib/template.ts`; `TokenText` only maps segments ↔ DOM nodes and reports caret ops. If a browser quirk bites (IME, Safari selection), degrade to append-at-end insertion rather than growing the component.
- Scope discipline: no backend edits (11a done); `RunTimeline` context expander stays raw JSON; don't labelize the Event Log (its summaries are already server-derived plain language); `FieldCombobox` keeps free-text entry — suggestions guide, never constrain.
