# Plan 13: Automation Builder Enhancements

**Complexity: ⚠️ Medium** — no migrations, no engine or recipe-format changes, no new deps; the work is one new shared UI component, a verify-and-fix pass on the builder's field scope, and a large-but-mechanical frontend sweep (~28 native `<select>` sites). It touches the automations *builder* frontend only — none of the framework surfaces (engine, tools, gate) that force the 🔴 default — so it stays a single plan. **Fallback split point** if it stalls: Tasks 1–4 (Select component + builder scope/UX) stand alone; Tasks 5–6 (app-wide sweep + wrap-up) are a clean second pass.

**Build order note**: Module 12 builds first (user decision 2026-07-18). At M13 build time, the sweep in Task 5 also covers any native selects Module 12b introduced under `frontend/src/components/schedule/` (its plan calls for client/caregiver selects in `VisitCreateDialog`).

## Context

Module 13's PRD summary was written before Module 11 existed, and M11 already delivered most of what it describes: the trigger-aware field catalog (labeled core trigger fields, per-event payload fields, per-entity fields, event→entity map), token chips in every template input, and condition values that render templates — including for the fixed-trigger lead/caregiver sequence builders. What remains is (a) the builder's IF section behaving sensibly per trigger type, (b) a user-reported field-scope bug, and (c) the app-wide custom dropdown rollout.

**User-locked decisions (2026-07-18):**
1. **Gate IF on event triggers**: the entry IF section only renders for event triggers — cron/manual runs have no trigger/record fields, so entry conditions on them can only silently fail today.
2. **Plain-language trigger labels**: the event dropdown shows human labels ("Lead stage changed") with the raw type (`lead.stage_changed`) as secondary mono text.
3. **The reported scope bug**: "no fields appearing in the dropdown for the IF condition — there should be fields relating to the trigger event." Static reading says the M11 catalog path is correct end-to-end, but M11b's browser validation never ran; Task 2 verifies live and fixes the root cause. Two statically-confirmed gaps get fixed regardless: `FieldCombobox` silently drops hint-only groups (the user sees *nothing* instead of an explanation), and the entry IF section renders for triggers that have no fields at all.
4. **Hand-rolled `ui/Select`** — no Radix, no new deps; consistent with the repo's existing popover patterns (FieldPicker, UserMenu, AddStepMenu).
5. **Full sweep** — every native `<select>` in the app is replaced, including ScheduleBuilder, SchemaForm enums, and DateTimePicker's month/year.

**Non-goals** (name the destination): field value tokens without curly-brace syntax in non-builder inputs (Future Plans); "Run manually" button + chat-triggered manual runs (Future Plans); additional calculation functions (Future Plans); Settings view (Future Plans); any engine, recipe-schema, or `validate_recipe` change (recipes with entry conditions on cron/manual triggers stay *valid* — this is builder UX, not a format change); nested submenus in `Select` (no current site needs one — groups cover every case; add nesting only when a real site demands it).

## Key design decisions (locked in — do not re-litigate during build)

1. **`ui/Select.tsx` API** — a button-plus-listbox popover, one component for the whole app:
   ```tsx
   interface SelectOption<T extends string = string> {
     value: T;
     label: string;
     description?: string;         // secondary line, muted
     mono?: string;                // secondary mono text (raw event types, cron parts)
     icon?: ComponentType<{ className?: string }>;
     dot?: string;                 // CSS color/class for a leading status dot
     disabled?: boolean;
   }
   interface SelectGroup<T extends string = string> { label?: string; icon?: ComponentType; options: SelectOption<T>[] }
   function Select<T extends string>(props: {
     value: T | ""; onChange: (v: T) => void;
     options?: SelectOption<T>[];  groups?: SelectGroup<T>[];   // one of the two
     placeholder?: string;         // shown when value is ""
     searchable?: boolean;         // filter input pinned at top (event-type lists)
     clearable?: boolean;          // adds a leading "placeholder" option that emits ""
     size?: "sm" | "md";           // h-8 / h-9 to match existing selectClass sites
     disabled?: boolean; className?: string; align?: "start" | "end";
   })
   ```
   Trigger button styled like the current `selectClass` (border-input, bg-background, ring focus) with a chevron; renders the selected option's icon/dot + label. Popover: absolute-positioned `z-30` card like FieldPicker (no portal), `max-h-64 overflow-y-auto`, outside-click close via the existing `mousedown` listener pattern. A11y/keyboard: `aria-haspopup="listbox"`, `aria-expanded`, `role="listbox"`/`role="option"` + `aria-selected`, Arrow/Home/End navigation, Enter selects, Escape closes and returns focus, printable-character type-ahead. Justification: matches every current site's needs (icons, colored dots, groups, descriptions, search) with zero deps and one consistent style.
2. **`fieldGroups` moves to `frontend/src/lib/fields.ts`** — it's a pure function currently living in `FieldPicker.tsx`; extracting it lets vitest cover the scope logic directly and keeps `FieldContext` importable without a component file. `FieldPicker`/`FieldCombobox` re-import from the lib (re-export the type from FieldPicker for back-compat if simpler). Justification: the scope bug needs a regression net that doesn't require a browser.
3. **Plain-language labels live in the frontend** (`lib/recipe.ts`, beside the existing `sourceLabel`): `EVENT_TYPE_LABELS` curated map for core-known event types + humanized fallback (`applicant.stage_changed` → "Applicant stage changed"), and `OPERATOR_LABELS` (`eq` → "is", `neq` → "is not", `contains` → "contains", `gt`/`gte`/`lt`/`lte` → "is greater than"/…, `exists` → "has a value", `not_exists` → "is empty"). Recipe JSON and the vocabulary endpoint are unchanged — labels are a view, exactly like M11's tokens. Justification: no backend surface change; the vocabulary already ships raw types and the M9/M10 seam rule keeps core from interpreting vertical names.
4. **IF gating is builder-UX only**: `AutomationBuilderPage` renders `ConditionChips` only when `trigger.type === "event"`; switching the trigger type away from event with conditions present asks `window.confirm` (the `applyDraft` precedent) and clears them; a one-line muted hint replaces the section ("Conditions apply when an event starts the run — scheduled and manual runs go straight to the steps."). **Edit-mode exception**: an existing non-event automation that already has entry conditions shows the IF section with a warning banner instead of hiding saved data. `StageSequencePage` is untouched (its trigger is always an event). Backend validation unchanged.
5. **Field-scope fix is verify-first**: Task 2 asserts the catalog end-to-end (gated pytest on `/api/automations/vocabulary` content → vitest on `fieldGroups` → live browser check on the IF dropdown), fixes whatever breaks, and lands the two known UX fixes: `FieldCombobox` renders hint-only groups and an explicit "No fields for this trigger" empty row (mirroring `FieldPicker`'s hint behavior) instead of showing no popover.
6. **Sweep is total**: every native `<select>` goes through `Select`; the per-file `selectClass` constants are deleted as their last select disappears. End-state check is mechanical: `rg '<select' frontend/src` returns nothing. `CreateTaskDialog`'s `datetime-local` input and checkboxes are out of scope (not dropdowns).
7. **Per-site customization inventory** (what each conversion adds beyond a straight swap):
