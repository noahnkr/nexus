# Plan 15c: Automation Touches — Formula Builder & Manual Runs

**Complexity: ⚠️ Medium** — two contained framework additions (a pure function + a safe tool with one small engine extension) and one builder component. The recipe format, `validate_recipe`, conditions, and the gate are untouched. Kept ⚠️ only because both changes ride existing, well-tested seams; anything that grows beyond them re-triggers the 🔴 rule. **Fallback split point**: Tasks 1+3 (formula) and Tasks 2+4 (manual runs) are independent pairs.

Parent: `.agent/plans/15.finishing-touches.md`. Build after 15a/15b (independent of both; last keeps framework risk isolated).

## Context

- **"Calculate a score" renders as raw JSON**: `weighted_score`'s `weights`/`inputs` args are free-form objects, so `SchemaForm` falls back to raw-JSON textareas. (PROGRESS credits M11b with a `CalculationEditor`, but it was never committed — no such component exists in the tree or git history; correct PROGRESS when ticking boxes.) The user wants a real formula: field tokens + math operators, output saved to context.
- **Manual automations pretend to be pausable**: `POST /api/automations/{id}/run` already ignores `status` entirely (`routers/automations.py::run_now`, `skip_conditions=True`), yet the grid shows manual-trigger automations with an Active/Paused toggle that does nothing meaningful for them. The PRD asks for a Run button instead, plus an MCP-reachable way for the agent to start a manual automation.
- **Engine facts that shape the design**: `start_run` inserts the run `'running'` and its docstring requires the caller to COMMIT before `advance_run`; a tool handler runs inside `execute_tool`'s savepoint on the caller's uncommitted transaction — so a tool cannot call `advance_run` directly. The M7b waker already advances any `waiting` run whose `wake_at` is due, every poll cycle.

## Key design decisions (locked in — do not re-litigate during build)

1. **`formula` function** (registered in `services/automations/functions.py`, parser in a sibling `services/automations/formula.py`): args `{"formula": "<expression>"}`. Grammar: decimal numbers, `+ - * /`, parentheses, unary minus, and `round(expr[, digits])` — evaluated by a hand-rolled tokenizer + recursive-descent parser (**no `eval`, no `ast`-exec**). Field values arrive via the existing engine template pass (`{{trigger.record.hourly_rate}} * 1.5` renders to `"22.5 * 1.5"` before the handler runs). Errors are plain `ValueError`s the engine surfaces verbatim ("Couldn't compute the formula: 'pending' is not a number", "Division by zero"), failing the run per M7 semantics. Returns a float rounded to 4 dp (the `weighted_score` precedent). `weighted_score` **stays registered** — existing recipes keep running and old steps still open in the builder via the SchemaForm fallback.
2. **Deferred start**: `start_run` gains `defer: bool = False` — when true the run inserts with `status='waiting', wake_at=now()` instead of `'running'` (same conditions/concurrency logic; `step_log` notes "queued by {source}"). The M7b waker picks it up within one poll (`NEXUS_AUTOMATIONS_POLL_SECONDS`). Justification: reuses the durable machinery instead of inventing post-commit hooks; a manual run via the tool starting a few seconds later is fine.
3. **`run_automation` tool** (`services/tools/automations.py`, registered in the tools bootstrap `__init__` order): **safe** — starting a run has no direct external effect (any gated step still parks `waiting_approval`). Input `{"automation": "<name or id>", "entity_type"?: string, "entity_id"?: string}`; name match is case-insensitive exact against this tenant's automations. Refusals (all `ToolInputError`, plain language): automation not found (list available manual ones in the message); `trigger.type != "manual"` ("…runs on its own trigger — it can't be started this way"); `current_invocation()["source_system"] == "automation"` ("Automations can't start other automations") — the loop-guard rule extended to the tool layer. On success: `start_run(..., skip_conditions=True, defer=True)`; concurrency-guard `None` → non-error result explaining a run is already active. Summary: "Started '<name>' — it will run within a few seconds; check its run history." `data` carries `run_id`. Add a `TOOL_LABELS` entry ("Run an automation").
4. **Vocabulary excludes `run_automation` from tool steps**: the builder's step palette must not offer a tool that always refuses under `source_system='automation'` — filter it in the vocabulary endpoint's tool list (one-line exclusion, same spot that annotates safety). Chat and MCP list it normally via the registry.
5. **`FormulaEditor.tsx`** (builder): shown for function steps whose `function == "formula"` — a `TokenText`-based single-line input for the formula (field tokens insert atomically via the existing `FieldPicker` flow), a row of operator/paren insert buttons (`+ − × ÷ ( ) round`, inserting `*`/`/` etc. at the caret), live syntax validation via a mirrored parser in `lib/formula.ts` (same grammar, treats `{{…}}` tokens as number placeholders) with a plain inline error, and the existing `SaveAsField` for the output name. The add-step menu's "Run a calculation" pre-selects `formula`; the function `Select` still lists `weighted_score` (legacy) which keeps its SchemaForm rendering. Read mode (`readDetail`/`describeStep` in `lib/recipe.ts`) labelizes the formula's tokens like other templates.
6. **Manual-run UX**: when `trigger.type === "manual"`: `AutomationCard` replaces the Active/Paused badge with a neutral "Manual" badge, hides the pause/resume toggle, and shows a **Run** button (calls the existing `runNow` API; success toast links to the run on the detail page; 409 → "already running" toast). `AutomationDetailPage` gets the same Run button in its header. Backend unchanged; the builder's save flow unchanged (a manual automation's stored `status` becomes irrelevant UI-wise).

## Tasks (strict order)

### Task 1 — `formula` backend
`formula.py` parser + `FunctionDef` registration per decision 1 (input_schema: single required `formula` string with a description that teaches the draft LLM the grammar and that `{{tokens}}` are allowed).
**Validation**: offline `test_functions.py` cases — precedence (`2+3*4` → 14), parens, unary minus, division, `round(10/3, 2)`, whitespace tolerance; plain errors for a stray word, unbalanced paren, and division by zero; a template-rendered numeric string evaluates. Vocabulary endpoint (existing gated test file) lists `formula`.

### Task 2 — Deferred start + `run_automation` tool
Engine `defer` per decision 2; the tool per decision 3; label; vocabulary exclusion per decision 4.
**Validation**: gated — `execute_tool("run_automation", {"automation": "<manual one>"})` as `source_system='chat'` creates a `waiting` run with `wake_at <= now()`, then one `waker_once()`/`run_cycle()` advances it to `completed` (reuse `test_automation_scheduler.py` fixtures); refusals: non-manual trigger, unknown name (message lists manual automations), `source_system='automation'`; second call while active → non-error "already running" result; vocabulary tool list omits `run_automation` while the MCP-visible registry includes it.

### Task 3 — `FormulaEditor` + `lib/formula.ts`
Per decision 5.
**Validation**: vitest `lib/formula.test.ts` — mirrors Task 1's grammar cases plus token-placeholder handling and error positions; `npm run test` + `npm run build` clean. - **Reuse (file paths)**: engine/waker test fixtures in `backend/tests/test_automation_scheduler.py` and `test_automation_engine.py`; `TokenText` + `FieldPicker` insertion flow from `components/automations/TokenText.tsx`/`FieldPicker.tsx` (see `SchemaForm`'s `TokenField` usage); `SaveAsField` in `StepCard.tsx`; vitest patterns from `lib/template.test.ts`; `runNow` API call from `lib/api.ts` (used by the detail page already).
