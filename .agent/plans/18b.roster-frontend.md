# Plan 18b: Roster Tab & Compliance Frontend

**Complexity: ⚠️ Medium** — a tabbed rework of one existing page, three new components reusing StatCard/table/drawer patterns, and an extension of the shared `CaregiverDrawer`. No backend work (18a's API is complete), no migrations, no new deps. **Fallback split point**: Tasks 1–2 (tabs + roster view) stand alone; Tasks 3–5 (drawer, regression, walk) are a clean second pass.

Parent: `.agent/plans/18.workforce-compliance.md`. Build after 18a.

## Context

18a shipped: `GET /api/workforce/roster` (metrics + per-caregiver rows with hours/available/utilization/credentials), credentials CRUD (`credential.added/updated/removed` on the resource entity), roster PATCH accepting `status` (emits `resource.status_changed`), the safe `list_expiring_credentials` tool, and inactive exclusion from matching/board. The frontend has: `CaregiversPage` (hiring pipeline: `FunnelStrip` + `HiringMetrics` + directory table), the shared `CaregiverDrawer` (contact/tags/availability editors, used by the schedule board), `?tab=` URL patterns, `StatCard`, Realtime debounced refetch, and `ui/Select`.

User-locked: Roster is a tab on `/caregivers` (no new nav entry); retention/at-risk deferred; per-credential granularity = read-time badges.

## Key design decisions (locked in — do not re-litigate during build)

1. **`lib/workforce.ts`**: `CREDENTIAL_STATUS_META` (valid=success, expiring=warning, expired=destructive, no_expiry=muted; labels), `RESOURCE_STATUS_META` (active/inactive), `fmtUtilization` (null → "—", caps display at 999%), `fmtDaysLeft` ("expires in 30 d" / "expired 10 d ago"). Pure module, vitest-covered.
2. **Tabs on `CaregiversPage`**: `?tab=pipeline|roster` (default pipeline; Knowledge-page `?tab=` precedent). Pipeline tab = the existing page content, moved unmodified into `PipelineTab`. Roster tab = new `RosterTab`. Tab strip under the `PageHeader`; deep links (`/caregivers?tab=roster`) work.
3. **`ComplianceStrip`** (top of Roster tab): four `StatCard`s from `metrics` — Active caregivers (inactive count as subline), Average utilization, Expiring credentials (warning tone when > 0), Expired credentials (destructive tone when > 0). All numbers server-computed.
4. **`RosterTable`**: one row per caregiver (active first, then inactive muted) — name, status pill, hours this week / available hours, utilization bar (hand-rolled, warning tone > 100%, "—" when no availability), `CredentialBadges` (compact chips per credential using status meta; expired/expiring first). Search box + status filter (`ui/Select`: All / Active / Inactive) client-side (roster scale is small). Row click opens the caregiver drawer.
5. **`CaregiverDrawer` extension** (shared component — schedule board keeps working unchanged):
   - New **Credentials** section: list rows (qualification name, issued/expires, status chip, notes) + `CredentialsEditor` add/edit/delete (qualification `Select` from facets, date inputs, notes; duplicate-pair 409 surfaced inline; delete with confirm).
   - New **status toggle** (Active/Inactive with confirm on deactivate: "Removed from the schedule board and matching; visits history is kept") → roster PATCH; drawer badge reflects it.
   - Both sections render wherever the drawer is opened (board or Roster tab) — one component, no fork.
6. **Realtime**: Roster tab subscribes to `resources` and `resource_credentials` (debounced refetch, view-page precedent). Board regression: deactivated caregiver drops off the board roster on its existing Realtime refetch.
7. **No pipeline changes, no nav changes, no client-side compliance math** — utilization/status/credential states all come from 18a responses.
