# Plan 9: Leads View & Marketing Funnel (parent)

**Complexity: 🔴 Complex** — Module 9 is not on CLAUDE.md's default-complex list, but it pairs the first vertical entity API + page family (directory, profiles, an LLM smart summary) with a core schema change (`automations.binding`) and a constrained builder composed from M8's components. Per the planning rule it is **broken into two sub-plans**:

- **`9a.leads-api-directory.md`** (⚠️ Medium) — the leads REST API (list/create/edit + `lead.stage_changed` events), the on-demand AI smart summary, and the `/leads` directory + `/leads/{id}` profile pages with entity event timeline.
- **`9b.funnel-and-stage-sequences.md`** (⚠️ Medium) — the core `automations.binding` column, the funnel strip + metrics widgets, and the per-stage outreach sequence builder on the M7 engine.

Build order: **9a then 9b** — 9a stands alone as a useful lead directory and establishes the page, router, and event vocabulary 9b's funnel and sequences hang off. If 9b stalls, its Task 1–2 (binding + metrics backend) are a clean fallback split — both curl-testable before any funnel UI exists.

**Requires Modules 7 + 8 built**: sequences are ordinary M7 automations; the constrained builder composes 8b's components (`StepList`, `StepCard`, `SchemaForm`, `ConditionChips`, `TemplateInsert`) and the vocabulary endpoint; the Automations Center renders bound sequences with a binding chip.

## Context

Modules 7–8 delivered the automation platform; this module instantiates PRD interface #7 — the first vertical dashboard view. The **pattern is core** (entity directory + profile + funnel strip + per-stage sequences + metrics widgets — M10 re-instantiates it for caregivers); the **content is seam** (lead stages, outreach steps, the leads router, stage configs). Leads today are reachable only through chat tools and webhook auto-creation; office staff have no place to see the pipeline, work a lead, or attach outreach.

What exists to build on: `leads` table with `status in (new, contacted, qualified, converted, lost)` (Module 0); `lead.created`/`lead.updated` events from the connector seam (3b) and `update_lead_status` as a gated tool (5a); `GET /api/events` with entity drill-down (M4); the M7 engine + REST and the M8 builder components/vocabulary; the M6 design system and Realtime-refetch pattern; `settings.fast_model` + traced `llm.get_anthropic()`.

**User-locked decisions (2026-07-17):**
1. **Funnel strip + directory on one page**: clickable stage segments (counts) filter the directory table below; stage moves happen via a stage selector in the lead profile. No kanban, no dnd lib.
2. **Sequence↔stage linkage = core `automations.binding` jsonb** (e.g. `{"view":"leads","stage":"contacted"}`) with one-sequence-per-stage enforced by a partial unique index. Business-agnostic core change M10 reuses verbatim.
3. **Smart summary = on-demand, no persistence**: generated when a profile opens (fast model over the lead row + recent entity events), lazily with a skeleton. No cache columns, no invalidation machinery.
4. **Lead writes = create + stage moves + basic-field edits** as human REST writes (Tasks-page precedent, `source_system='user'`). **No delete** — leads end as converted/lost, keeping funnel history honest.

**Non-goals** (name the module): the caregivers instantiation (M10); fuzzy matching / "can we serve this lead" decisions (M11); lead scoring functions (M10 introduces the scoring-function seam); free-form automation building (M8 owns it — this module's builder is deliberately narrower); referral-source ROI dashboard and census views (Future Plans backlog); webhook-driven stage *updates* (connectors record `lead.updated` but don't move stages this phase); bulk lead operations and CSV import.

## Shared conventions (both sub-plans)

- **Stages are `leads.status`** — no new column, no parallel stage table. The ordered funnel config (labels, order, terminal flags) lives in the vertical seam: `backend/app/services/views/leads.py` + `frontend/src/lib/leads.ts`.
- **`services/views/` is the vertical view-services seam** (stage config, metrics, smart summary); `routers/leads.py` and the leads pages/components are seam content too. M10 adds `views/caregivers.py` alongside — core never imports vertical view code.
- **Every stage change emits `lead.stage_changed`** (`payload: {from, to, summary}`, entity-linked) from whichever writer moved it — the REST PATCH and the `update_lead_status` tool handler. Plain-language `payload.summary` per CLAUDE.md. Automation-sourced stage changes are never re-dispatched (M7 loop guard) — sequences can't chain off each other; deliberate, documented.
- **Sequences are ordinary automations** — created/edited through the standard validated API (binding is just a column on the row), executed by the untouched M7 engine, visible in the Automations Center with a binding chip. No second engine path, no shadow tables.
- Plain language everywhere: stage labels ("Contacted", not `contacted`) via the seam config; raw JSON only behind technical expanders.
- New frontend deps: none. New env vars: none.
