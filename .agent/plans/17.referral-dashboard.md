# Plan 17: Referral-Source Dashboard

**Complexity: ⚠️ Medium** — one vertical migration (a small enrichment table), one seam + router pair following the M9/M16 precedents, and one new page with hand-rolled visuals. Deliberately **no** tool-layer, approval-gate, or automations-framework involvement (the 🔴 default doesn't apply): no new agent tools — `referral_partners` joins `SQL_SCHEMA_DOC` so chat answers referral questions through the existing read-only `run_report`. Single plan, strict task order. **Fallback split point**: Tasks 1–3 (backend) stand alone; Tasks 4–6 (frontend) are a clean second pass.

**Requires Module 16a's migration** (`clients.authorized_hours_per_week` feeds the hours-won revenue proxy) — build after Module 16, or at minimum after its Task 1. Independent of Modules 14–15; when 14a's WelcomeHome sync lands, its mapped source strings enrich automatically (join-by-name needs no linking logic).

## Context

The PRD's Module 17 summary: which partners (hospitals, senior-living communities, discharge planners) send leads that actually convert — referral ROI drives where the owner spends relationship time. What exists: `leads.source` is free text written by every lead path (manual create, webhooks, the future WH sync); `views/leads.py::funnel_metrics` already computes top-sources and days-to-convert (the SQL house style to extend); `clients.lead_id` links each won client back to its lead, and M16 adds `authorized_hours_per_week` — so "hours/week won per partner" is one join away. There is no partner record anywhere: a source is just a string, with no category, no contact person, no notes, and no per-source conversion view.

**User-locked decisions (2026-07-19):**
1. **Enrichment by name**: a new `referral_partners` table (name unique per tenant) joined to leads by **exact source-name match**. No lead schema change, no backfill, no FK — connector-written source strings keep working untouched, and an unmatched source can be promoted to a tracked partner in one click.
2. **Hours-won revenue proxy**: per partner, the summed `authorized_hours_per_week` of clients whose `lead_id` traces to that source's converted leads (all linked clients — a discharged client was still won business). Ties referral ROI to the census's revenue number.
3. **Own `/referrals` page** with a nav entry — metrics strip + sortable partner table + partner detail drawer.
4. **Hand-rolled bars** for trends (FunnelStrip/HoursCard precedent) — no chart library.

**Design decisions locked at planning time** (do not re-litigate during build):
1. **Migration `20260728000000_entities_referral_partners.sql`** (VERTICAL seam, additive + idempotent):
   ```sql
   create table if not exists public.referral_partners (
     id           uuid primary key default gen_random_uuid(),
     tenant_id    uuid not null references public.tenants(id),
     name         text not null,
     category     text check (category in
                    ('hospital','senior_living','discharge_planner','home_health','community','other')),
     contact_name text, phone text, email text, notes text,
     created_at   timestamptz not null default now(),
     updated_at   timestamptz not null default now(),
     unique (tenant_id, name)
   );
   ```
   Standard 4-policy RLS + `set_updated_at` trigger + `(tenant_id, name)` covered by the unique; added to the `supabase_realtime` publication (guarded). `category` nullable = untyped partner.
2. **Seed enriches, never restructures**: two partner rows ("St. Mary's Hospital" → hospital, "Sunrise Senior Living" → senior_living) and the sources of two *existing* seeded leads updated to match (Harold Byrne `referral` → `St. Mary's Hospital`; Estelle Ferraro `referral` → `Sunrise Senior Living`). Estelle is a converted lead linked to a seeded client with authorized hours, so hours-won demos immediately — **no new clients or visits**, so M16a's hand-computed census assertions stay green. Any existing test pinning the `referral` source string gets its assertion updated in the same task.
3. **Seam `services/views/referrals.py`** (re-templating-seam member): `PARTNER_CATEGORIES` (key/label config + `category_label()`), and `referral_metrics(conn, months=6) -> dict` — one deterministic SQL pass (funnel_metrics style, empty tenant → zeroes/nulls, never a 500) returning:
   - `sources`: one row per distinct non-empty `leads.source`, left-joined to `referral_partners` by exact name — `{source, partner: {id, category, contact_name, phone, email, notes} | null, leads_total, in_pipeline, converted, lost, conversion_rate, avg_days_to_convert, hours_won, last_lead_at, monthly: [{month, count}]}` (monthly = `date_trunc('month')` buckets over the last `months` months, zero-filled). Partners with zero leads still appear (tracked-but-quiet partners are exactly the relationship signal).
   - `totals`: `tracked_partners`, `leads_last_30_days`, `total_hours_won`, `best_converter` (`{source, conversion_rate}` among sources with ≥ 3 leads; null below threshold).
   `avg_days_to_convert` reuses the `lead.stage_changed→converted` events join from `funnel_metrics`; `hours_won` = `sum(c.authorized_hours_per_week)` over `clients c join leads l on c.lead_id = l.id` grouped by `l.source`.
4. **Router `routers/referrals.py`** (JWT tenant-scoped, `source_system='user'`, wired in `main.py`):
   - `GET /api/referrals/metrics?months=6` (months clamped 1–24) → decision-3 payload.
   - `GET /api/referrals/partners` (name-ordered), `POST /api/referrals/partners` (name required/trimmed; duplicate name → 409), `PATCH /api/referrals/partners/{id}` (all fields incl. rename — renames simply re-join, by design), `DELETE /api/referrals/partners/{id}` (allowed — deleting a partner only un-enriches its source string; funnel history lives on leads and is untouched).
   - Events on the partner entity: `referral_partner.created` / `updated` / `deleted`, each with a plain `payload.summary` ("Referral partner 'St. Mary's Hospital' (hospital) added"); `updated` names changed fields; no-op PATCH emits nothing.
5. **Threading, no tools**: `SQL_SCHEMA_DOC` (`services/tools/entities.py`) gains the table so `run_report` answers "which referral sources convert best?"; `services/automations/entities.py` `ENTITY_TABLES`/`ENTITY_LABELS` gain `referral_partner` and `routers/automations.py`'s known event types gain the three new ones (timeline/automation vocabulary resolve cleanly). **No new agent tools** — a dedicated safe metrics tool waits for a real need (revisit if chat usage shows `run_report` falling short). Update any vocabulary/catalog test assertions that pin entity or event-type lists.
6. **Frontend — `/referrals` page** (nav entry **Referrals**, `Handshake` icon, between Leads and Caregivers):
   - `ReferralMetricsStrip`: four `StatCard`s — Tracked partners, Leads (last 30 days), Best converter (rate + source subline), Hours/wk won (info tone).
   - `MonthlyTrendBars`: hand-rolled monthly bar row for all leads (last 6 months, month labels, theme tokens) — a shared small component also rendered per-partner in the drawer.
   - `PartnerTable`: one row per source — name + category chip (tracked) or a muted "untracked" state with a **Track** button (opens the create dialog prefilled with the source name), leads, converted, conversion rate, hours won, last lead (relative), 6-month sparkline bars. Client-side sortable headers (default: hours won desc, then leads); all data from the single metrics call.
   - `PartnerDrawer` (VisitDrawer pattern, row click): editable contact fields + category `Select` + notes, per-partner `MonthlyTrendBars`, that source's recent leads (name, stage pill, relative time, linking `/leads/{id}`), Delete with confirm ("stops tracking; leads keep their source"). Create/edit through `PartnerDialog` (shared by Track + New partner buttons).
   - Realtime on `referral_partners` and `leads` (debounced refetch, view-page precedent). `lib/referrals.ts`: category meta (labels + dot tones), `fmtHoursWon`, sort helpers — vitest-covered.

**Non-goals** (name the destination): partner-facing outreach/automation sequences (a WHEN `referral_partner.*` automation is already possible via the builder — nothing shipped here); applicant/hiring source analytics (Module 18's workforce views); fuzzy/alias source matching (Future Plans — revisit when WH sync surfaces messy strings); a `referral_partner_id` FK on leads (Future Plans, only if rename-stability becomes a real problem); dedicated safe metrics tool (future need); charts library (user decision — hand-rolled).
