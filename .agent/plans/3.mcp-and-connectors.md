# Plan 3: MCP Server & External Connectors (parent)

**Complexity: 🔴 Complex** — CLAUDE.md defaults Module 3 to complex, and it genuinely spans two independent concerns: exposing the tool layer outward (MCP) and normalizing external systems inward (connectors + entity resolution). Per the planning rule it is **broken into two sub-plans, each independently executable**:

- **`3a.mcp-server.md`** (⚠️ Medium) — MCP server mounted in FastAPI, exposing the Module 2 tool registry over Streamable HTTP.
- **`3b.connector-ingress.md`** (⚠️ Medium) — webhook ingress, connector adapter seam, entity resolution with auto-create + task fallback, five placeholder adapters documented against the real systems.

Build order: **3a then 3b** (3a is smaller and has zero schema changes; 3b carries the one migration). Neither depends on the other — if 3a stalls, 3b can proceed and vice versa.

## Context

Modules 0–2 delivered: canonical schema with RLS (`nexus_app` role, tenant GUC via `db.tenant_tx`/`tenant_conn`), `external_ids` for cross-system identity, immutable `events`, `tasks`/`pending_actions`, ingestion, SSE chat, and a tool registry where **`execute_tool()` is the single audited execution seam** (`backend/app/services/tools/core.py`) — designed from day one to be the thing MCP (this module) and n8n (M7) call into.

Module 3 per the PRD: an MCP server exposing Modules 1–2 tools to external clients (Claude clients now, n8n custom nodes in M7), plus connector adapters that normalize inbound external-system events into canonical entities via `external_ids` before anything else is written.

**User-locked decisions:**
1. **Connectors = mixed approach.** The user has credentials for five real systems (WelcomeHome CRM, GoTo Connect VoIP/SMS, WellSky EHR, Gmail, Google Calendar) but live configuration is deferred. This module builds the ingress + adapter + resolution architecture validated with **placeholder payloads**, while each adapter's file documents the researched real integration flow so the real adapter later replaces only that file's internals — never the seam.
2. **MCP mounted in FastAPI** — official `mcp` Python SDK, Streamable HTTP transport at `/mcp` inside the existing backend. One process, reuses pool + tenant seam + `execute_tool`. Static bearer token (`NEXUS_MCP_TOKEN`) until Module 6 auth.
3. **Unresolved inbound events: auto-create + task fallback.** Events explicitly representing a new entity (e.g. `lead.created`) auto-create the canonical row + `external_ids` mapping. Events *referencing* an entity we can't resolve (call from unknown number) create a review task linked to the originating event — no business-table write. Fuzzy matching/dedup stays in Module 8.

**Non-goals** (name the module): write/action tools and `pending_actions` execution (M5); outbound connector actions — send SMS/email, update CRM (M5 gated tools); Event Log UI (M4); auth/JWT tenancy (M6); n8n nodes and scheduled pollers (M7 — pollers will POST into this module's ingress); fuzzy entity matching/dedup harness (M8); live credentials/config for the five real systems (post-module ops, per system); frontend changes (none in this module).

## Integration research (2026-07-16) — how each real system actually delivers data

| Source | `external_ids` category | Real delivery | Real flow (documented in each adapter's docstring) |
|---|---|---|---|
| **WelcomeHome** (CRM) | `crm` | Webhook, full payload | Account-scoped API token in `Authorization` header; REST API (`/api/ping` smoke test) + **webhook-subscriptions endpoint** for lead lifecycle events. Test accounts available via their support. Docs: https://crm.welcomehomesoftware.com/api-docs/index.html |
| **GoTo Connect** (VoIP/SMS) | `phone` | Webhook **or WebSocket**, full payload | OAuth2. Create a **Notification Channel** (webhook URL or WebSocket), then subscribe it to the Call Events API (`/call-events/v1/subscriptions`). Channels expire → renewal cursor in `connector_state`. Outbound SMS (`messaging.v1.send` scope) is an M5 gated tool, not this module. Docs: https://developer.goto.com/guides/GoToConnect/14_HOW_useNotificationChannelApi/ |
| **WellSky Personal Care** (EHR, ex-ClearCare) | `ehr` | Webhook (FHIR payloads); partner-gated; poll/export fallback | Connect API access is granted via a WellSky account rep. Webhook subscriptions deliver **FHIR-shaped resources** (Patient/Practitioner/Encounter), sometimes minimal notifications needing a fetch-back. If webhook access isn't granted, fallback is an external poller (n8n, M7) that POSTs into this same ingress. Docs: https://apidocs.clearcareonline.com/ |
| **Gmail** (email) | `email` | **Ping + fetch-back** (Pub/Sub push) | OAuth2. `users.watch` → Cloud Pub/Sub topic → push subscription POSTs `{emailAddress, historyId}` — a watermark, **no message content**. Real adapter fetch-back: `users.history.list` since the stored `historyId` (cursor in `connector_state`); `watch` must be renewed ≤ every 7 days. Docs: https://developers.google.com/workspace/gmail/api/guides/push |
| **Google Calendar** (calendar) | `calendar` | **Ping + fetch-back** (watch channel) | OAuth2. `events.watch` channel POSTs a bodyless ping (`X-Goog-Resource-State: sync` on handshake, `exists` on change). Real adapter fetch-back: `events.list` with stored `syncToken` (cursor in `connector_state`); 410 → full resync; channels expire → renewal. Docs: https://developers.google.com/workspace/calendar/api/guides/push |

**Three delivery shapes → seam requirements** (these drive 3b's design):
1. *Full-payload webhook* (WelcomeHome, GoTo, WellSky) — normalize directly from the POST body.
2. *Ping + fetch-back* (Gmail, GCal) — the webhook carries a watermark only; `normalize()` must be **async** (room for the API call-back), adapters need **ack-only** results (handshake/sync pings produce no event), and durable **cursors** need a home → new core table `connector_state`.
3. *Poll/export* (WellSky fallback, GoTo WebSocket alternative) — lives **outside** the core (n8n in M7, or manual trigger); the poller re-posts results into the same `/api/webhooks/{source}` ingress, so the core stays webhook-shaped and the CLAUDE.md "manual or webhook/event-triggered only" rule holds.

Placeholder payloads carry the data inline **as if the fetch-back already happened** — the real Gmail/GCal adapters add the fetch-back call inside `normalize()` without changing the seam.

## Shared conventions (both sub-plans)

- Connector source names: `welcomehome`, `goto`, `wellsky`, `gmail`, `gcal`. `events.source_system` uses the **connector name** (precise audit); `external_ids.source_system` uses the **category** (`crm`/`phone`/`ehr`/`email`/`calendar` — one live system per category this phase; re-templating note: a second CRM would namespace external ids).
- New env vars: `NEXUS_MCP_TOKEN` (3a), `NEXUS_WEBHOOK_SECRET` (3b). Both optional at boot (feature 401s/refuses without them), matching the config.py philosophy.
- Every MCP tool call and every inbound webhook writes `events` rows (CLAUDE.md audit rule); plain-language summaries in `payload.summary`, technical detail in the rest of the payload.
- No frontend work; no changes to chat.
