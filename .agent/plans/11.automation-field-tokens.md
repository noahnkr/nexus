### Audit: how template fields work today, and where they fail the user

The run scope is `{trigger, entity, context}` (`services/automations/templates.py`, `engine.py::_build_scope`):

- **`trigger.*`** — the triggering event row: `event_type`, `source_system`, `entity_type`, `entity_id`, `created_at`, and `payload.*` (whatever the event carried).
- **`entity.*`** — the raw canonical row the run is *about* (the lead a webhook created, the applicant whose stage changed), resolved via the vertical seam `services/automations/entities.py::get_entity` from the trigger event's entity link. **On cron/manual triggers there is no triggering entity, so `entity` is empty** — yet the builder offers it anyway. This is the answer to "what is entity even for" — and the UI never explains it.
- **`context.*`** — earlier steps' `save_as` outputs.

Concrete UX failures found:

1. **`TemplateInsert` inserts incomplete stubs** (`{{trigger.payload.}}`, `{{entity.}}`) that the user must finish typing with no way to know what completions exist, and it appends to the end of the field rather than the cursor.
2. **Suggestions are one flat, global, unlabeled list** (`Vocabulary.field_suggestions`): `entity.*` mixes the columns of all five entity tables (a `lead.created` builder session is offered `entity.start_time` from schedules and `entity.stage` from applicants), and `trigger.payload.*` pools observed keys across *all* event types. Nothing is filtered by the trigger actually selected; everything renders as raw mono paths with no plain-language meaning.
3. **Raw `{{trigger.payload.phone}}` text sits in inputs** — unreadable to the target user, fragile to hand-edit (a typo'd path fails the run at execution time).
4. **`weighted_score` is unusable in the builder**: its `weights`/`inputs` object args fall through `SchemaForm` to a raw-JSON textarea — the marquee scoring feature requires hand-writing JSON with dotted template paths.
5. **Condition *values* are not template-rendered** in entry conditions or `condition` steps (only `wait_until` renders its conditions) — a quiet semantic inconsistency: `value: "{{context.score}}"` silently compares the literal string.
6. **Read mode leaks the same raw syntax**: step cards, run timelines, and recipe sentences print `{{paths}}` verbatim.

### What this module ships

1. **A trigger-aware, plain-language field catalog** served by the vocabulary endpoint: core trigger fields with labels, observed payload keys *per event type*, entity fields *per entity type* with a seam-supplied entity label ("Lead", "Applicant", …), and an event-type → entity-type map so the builder knows a `lead.created` run is about a Lead.
2. **Visual field tokens**: every template-accepting input renders `{{path}}` references as atomic, labeled, deletable chips; insertion happens at the cursor from a searchable, grouped picker. The stored recipe format is **unchanged** — chips are a view over the same `{{path}}` strings, so existing recipes, the draft agent, and the engine are untouched.
3. **"Run a calculation"**: the function step is presented in plain language, and `weighted_score` gets a dedicated editor — rows of *field × weight* — that writes the `{weights, inputs}` args without the user ever seeing JSON or dotted paths. Applicant scoring / lead value determination is this editor plus a condition or generate step on `{{context.score}}`.
4. **`days_until`** joins the function registry (credential-expiry / upcoming-date automations).

**User-locked decisions (2026-07-17):**
1. **Harness + scheduling deferred** — this module replaces them in the numbering; they move to the backlog.
2. **Score results are context-only** — no score column, no score display surface, no score-write tool. A calculation's result lives in run context; the recipe decides what becomes of it (condition, message, task). Persisting/displaying scores is future work if a real need appears.
3. **Calculation additions = `days_until` only.** Brainstormed candidates recorded to the backlog, not built: `count_events` (entity engagement counts), `calculate` (binary arithmetic), `tier` (threshold → label bucketing), `hours_between`.
4. Keep the field-insert button/dropdown pattern — but the user never types dotted paths; the picker shows plain labels and the catalog supplies the paths.

**Non-goals**: matching/decision harness and every scheduling surface (backlog); score persistence or display (backlog, per lock #2); new step types or recipe-shape changes (the recipe JSON is untouched); rich-text or expression syntax in templates (path lookups only, per CLAUDE.md); builder drag-and-drop; new env vars; migrations (none needed — everything is read-time).

## Shared conventions (both sub-plans)

- **The recipe format does not change.** Tokens serialize to the exact `{{path}}` strings the engine already renders; a recipe built before this module opens correctly in the new builder and vice versa.
- **Labels come from the catalog, never hardcoded in components**: the vertical seam (`services/automations/entities.py`) supplies entity labels and field labels; core humanizes (`snake_case` → "Snake case") as the fallback. Frontend components receive catalog data via props — no component imports vertical config.
- **Unknown paths never crash**: a chip for a path the catalog doesn't know renders the humanized path itself; parse/serialize round-trips arbitrary user text.
- **Plain language everywhere** (CLAUDE.md): picker groups read "From the trigger event", "The lead", "Earlier step results" — mono paths are secondary text.
